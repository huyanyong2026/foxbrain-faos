# VAFOX M6.5 Customer Brain Online Design

**状态：** Design Phase / 仅设计，不部署。  
**前置设计：** `DATA003.1_CUSTOMER_IDENTITY_DESIGN.md`、`DATA003.2_RECOMMENDATION_ENGINE_DESIGN.md`、`DATA003.4_GROWTH_LOOP_DESIGN.md`、`M5.1_DIGITAL_EMPLOYEE_FOUNDATION_DESIGN.md`、`M6.1_AI_RUNTIME_FOUNDATION_DESIGN.md`、`M6.2_EMPLOYEE_AI_WORKSPACE_MVP_IMPLEMENTATION_DESIGN.md`。  
**负责人：** Codex（设计开发）；WorkBuddy（环境）；Marvis（管理）；CTO（验收）。

> Customer Brain Online 是面向客户服务和已授权员工辅助的、可解释且只读的客户认知层。它组合既有 Customer Identity、受治理画像、购买与装备快照、户外旅程、社群关系、成长投影和推荐结果；不替代 SAP、Core 或活动系统的事实源。所有推荐均为 `advisory_only_requires_human_approval: true`，本设计不部署服务、导入数据或改变生产同步。

## 1. 目标、原则与硬边界

### 1.1 目标

1. 在同一 `organization_id` 内，以 DATA-003.1 的不可变 `customer_id` 建立客户的统一在线视图。
2. 将客户的户外兴趣、装备覆盖、活动成长和社群关系组织为带来源、时点、同意范围和质量状态的最小投影。
3. 为客户自助 Member Assistant、Employee Workspace 与 AI Runtime 提供只读、可引用、可审计的查询和建议。
4. 连接 DATA-004 Outdoor Community（活动/社群权威事实）和 DATA-003.4 Growth Loop（积分、等级及成长事实），但不复制或篡改其主数据。

### 1.2 设计原则

- **身份、组织和同意优先。** 先解析服务端身份，再同时判断组织隔离、RBAC、资源 ACL、客户关系、用途、Consent、字段策略和有效期；前端条件只能缩小结果。
- **事实不搬家。** Identity Vault、SAP/Core 购买事实、活动引擎活动事实、Growth Loop 积分账本和社群成员资格仍分别归原系统所有；本设计保存可失效的受控引用或读模型。
- **最小化与目的限制。** API 只返回完成当前服务、活动或经批准个性化所需字段；原始手机号、微信 ID、会员 ID 及令牌不进入通用日志、提示词或推荐载荷。
- **未知即补问。** 路线风险、天气、高海拔适应、健康条件、尺码、实时价格、库存、活动名额或装备状态未知时，返回 `needs_verification` 和人工确认项，绝不猜测。
- **建议与执行分离。** AI 可以解释、匹配、排序、生成待办草案；不能成交、触达、下单或写入任何权威业务系统。
- **可回放且可撤回。** 每一属性、推荐和访问都保留来源、版本、`as_of`、授权决定、规则版本和审计关联；撤回同意、失效关系或撤销知识后立即停止后续读取。

### 1.3 明确禁止

Customer Brain Online 不得直接或间接：

- 未经授权发送营销信息、群发、自动触达或规避退订/频控；
- 自动成交、自动报价、锁货、库存预留、自动下单、支付或创建购物车；
- 修改 SAP、SAP 主数据/交易或任何来源客户记录；
- 修改 Core 生产同步、同步规则或生产事实；
- 由 AI 自动确认活动资格、户外安全、高海拔适应性、积分、等级、身份合并或客户同意。

## 2. Customer Brain Architecture

```text
Customer Identity (DATA-003.1)
           ↓ identity + consent + relationship gate
Customer Profile
           ↓ authorized attribute projection
Purchase History (SAP/Core read-only facts)
           ↓ product and order references
Equipment Profile
           ↓ coverage / gap / upgrade hypotheses
Activity Journey (DATA-004 / Growth Loop)
           ↓ verified participation and route facts
Community Relationship (DATA-004 Outdoor Community)
           ↓ membership and matching eligibility
Growth System (DATA-003.4)
           ↓ points / Lv1–Lv5 / achievements
AI Recommendation (DATA-003.2 + Product Intelligence + AI Runtime)
```

| 层 | 主要内容 | 权威来源/接口 | 输出与控制 |
| --- | --- | --- | --- |
| Customer Identity | `customer_id`、身份链接、组织、顾问关系、逐目的 Consent。 | DATA-003.1 / Identity Vault。 | 只输出脱敏身份和允许字段；冲突身份不个性化扩展。 |
| Customer Profile | 基础、户外和消费画像及每属性的质量元数据。 | 客户声明、授权互动、受治理派生规则。 | 属性按用途/字段 ACL 投影；不能由模型静默回写。 |
| Purchase History | 已授权购买摘要、时间、品类、来源引用。 | SAP/Core 只读认证事实。 | 不把订单地址推断为常住城市；不承诺价格/库存。 |
| Equipment Profile | 已确认装备、使用场景、等级、覆盖缺口。 | 购买事实、客户明确声明、人工核验。 | `upgrade_opportunity` 只是可解释假设，需人工确认。 |
| Activity Journey | 路线、活动、日期、难度、成就、核验状态。 | DATA-004 / DATA-003.4。 | 未审核自报记录不作为等级、安全或个性化事实。 |
| Community Relationship | 兴趣、城市、经验、社群和成员状态。 | DATA-004 Outdoor Community。 | 仅返回有权且获同意的社群/成员资格；不泄露他人信息。 |
| Growth System | 积分账本摘要、等级、邀请/分享/打卡状态。 | DATA-003.4 Growth Loop。 | 账本只读、可冲正、反作弊/审核优先。 |
| AI Recommendation | 活动、社群、装备和下一动作候选。 | DATA-003.2、Product Intelligence、AI Runtime。 | 证据、限制、置信度和 `advisory_only` 必填。 |

每次请求经过：`认证 → 服务端派生身份/范围 → Consent 与关系门控 → 最小投影查询 → 规则/推荐 → 输出字段复检与 Citation → 追加式审计`。任一环节拒绝、过期或不可用时，后续组件不得读取未授权数据；应返回统一的 `403 POLICY_DENIED` 或不暴露对象存在性的受控结果。

## 3. Customer Identity Model

### 3.1 统一身份模型

`CustomerIdentity` 复用 DATA-003.1；`customer_id` 为不含业务意义、不可变、不可重用的 opaque ID。所有关系、旅程、积分、推荐和审计以 `organization_id + customer_id` 为强制边界。手机号和各类微信/会员标识仅在 Identity Vault 的加密值或版本化 token 中参与解析。

| 字段 | 类型 / 约束 | 说明 |
| --- | --- | --- |
| `customer_id` | string，必填，不可变 | 统一客户引用；不是手机号、订单号或会员号。 |
| `mobile` | 受限 PII，选填 | E.164 规范化的 Vault 引用；标准响应仅返回掩码。 |
| `wechat_id` | 受限 PII，选填 | 个人微信受保护引用；不得单独视作全局唯一。 |
| `enterprise_wechat_id` | 受限 PII，选填 | 需与企业微信租户及 `organization_id` 一起解释。 |
| `member_id` | 受限业务标识，选填 | 会员系统来源引用；允许版本化历史链接。 |
| `source_channel` | enum，必填 | `store`、`wechat`、`enterprise_wechat`、`member`、`ecommerce`、`order`、`service` 等。 |
| `advisor_id` | string，选填 | 当前主顾问投影；完整历史在受控关系实体中。 |
| `organization_id` | string，必填 | 租户/组织隔离与解析边界；不得自动跨组织合并。 |
| `consent_status` | enum，必填 | `granted`、`partial`、`denied`、`withdrawn`、`unknown` 的摘要，不能替代逐目的 Consent。 |

### 3.2 解析、隔离与隐私治理

1. **统一身份解析：** 仅在同一组织内以已验证手机号的精确 token 优先；其他标识必须有相互一致的批准证据。姓名、昵称、城市、装备、共同地址或模型相似度不能自动合并。冲突、多候选和共享联系方式进入人工审核，禁止拿合并前画像、积分或同意互相覆盖。
2. **权限隔离：** `organization_id` 必须由服务端认证上下文派生，并在每个读取、缓存键、检索、审计查询和下游调用中强制过滤。客户 ID 在 URL 中不构成授权；未授权、不可见和不存在使用同一拒绝语义。
3. **隐私治理：** Consent 以 `purpose`、`state`、证据、采集/到期/撤回时间管理。撤回后，后续个性化、活动匹配、社群匹配和员工明细读取立刻失效；仅保留依法需要的最小审计引用。身份原值禁止进入模型上下文、错误栈、分析事件和导出。

## 4. Customer Profile Model

`CustomerProfile` 是可解释的属性投影，不是身份事实源。每个属性共同携带：`value`、`source_type`、`source_ref`、`observed_at`、`updated_at`、`quality_state`、`confidence`、`allowed_purposes`、`expires_at` 和 `profile_version`。没有同意、证据或有效期的属性须省略而非填充推断值。

| 画像域 | 字段 | 设计与约束 |
| --- | --- | --- |
| 基础画像 | `age_band`（年龄段）、`city`、`interests[]` | 年龄仅返回分段；城市来自客户声明或受权事实，订单地址不能自动当常住地；兴趣需明确声明或授权行为证据。 |
| 户外画像 | `experience_level`、`activity_frequency`、`route_preferences[]`、`equipment_level` | 等级为 `newcomer/casual/regular/advanced/unknown`；频率以可说明窗口表达；路线偏好不等同安全能力；装备等级仅反映覆盖程度。 |
| 消费画像 | `brand_preferences[]`、`price_range`、`purchase_cycle` | 仅用获准目的内的聚合购买事实；价格区间为区间/未知，非支付能力；购买周期标明观测窗口和样本不足状态。 |

示例最小投影：

```json
{
  "customer_id": "cst_01J...",
  "profile": {
    "basic": {"age_band": "25_34", "city": "深圳", "interests": ["hiking", "camping"]},
    "outdoor": {"experience_level": "regular", "activity_frequency": {"band": "monthly", "window_days": 180}, "route_preferences": ["weekend_hike"], "equipment_level": "developing"},
    "consumption": {"brand_preferences": [{"brand_ref": "brand_kailas", "affinity": "observed"}], "price_range": "mid", "purchase_cycle": {"status": "insufficient_evidence"}}
  },
  "policy": {"purpose": "member_service", "decision": "allow_masked"},
  "freshness": {"as_of": "2026-07-22T00:00:00Z"}
}
```

## 5. Purchase History and Equipment Profile

### 5.1 Purchase History

购买历史只读取 SAP/Core 已认证事实，作为装备档案和消费画像的证据之一。返回默认是最小摘要：`purchase_ref`、`product_ref`、`brand_ref`、`purchased_at`、`channel`、`fact_version`、`as_of`；订单金额、支付、收货和其他敏感字段仅在必要且明确授权时另行处理。退货、取消、来源冲突或数据过期应保留状态，不得被推荐层掩盖。

### 5.2 Equipment Profile

| 字段 | 类型 / 来源 | 设计与控制 |
| --- | --- | --- |
| `product` | `product_ref`，必填 | 指向权威产品实体；不使用自由文本替代产品事实。 |
| `brand` | `brand_ref`，选填 | 由产品/购买来源确认。 |
| `purchase_time` | RFC 3339 / date，选填 | 仅为购买或确认时间，不推定实际磨损状态。 |
| `usage_scenario` | 受控标签数组 | 如 `hiking`、`mountaineering`、`camping`、`high_altitude`；来自客户声明或经审核规则。 |
| `equipment_level` | enum | `entry`、`developing`、`complete_for_declared_scenario`、`unknown`；只描述已知覆盖。 |
| `upgrade_opportunity` | 可选建议对象 | 含缺口、理由、证据、置信度、待确认条件和 `advisory_only`；不是购买指令。 |

装备记录还应包含 `verification_state`（`purchased`、`customer_confirmed`、`inferred_pending_review`、`retired`、`unknown`）、来源引用和有效期。客户自述“已拥有”可形成待核验记录；不得因为已购买就假设仍在使用、状态良好或适合高风险路线。产品兼容性由已发布 Product Intelligence 判断；实时库存和价格必须单独核实。

## 6. Outdoor Journey and Activity Engine Integration

### 6.1 户外成长轨迹

`OutdoorJourney` 连接 DATA-004 活动与 DATA-003.4 成长事实，包含：

| 字段 | 定义 |
| --- | --- |
| `route` | 经批准路线引用、名称投影或自建路线审核引用；包含版本与适用日期。 |
| `activity` | `activity_id`、活动类型、组织方、参与状态和来源引用。 |
| `date` | 已发生日期/时间窗及 `observed_at`；不以报名时间替代完成时间。 |
| `difficulty` | 活动引擎定义的受控难度及版本；不是医疗、天气或现场安全结论。 |
| `achievement` | 经验证的完成、徽章或成长成就；注明规则和审核状态。 |

状态为 `draft → submitted → verified|rejected|withdrawn`；只有 `verified` 记录可影响经验、活动频率、成就和等级。允许客户或授权人员提交反馈/更正候选，但只能进入审核，不能直接改写活动事实或积分。

### 6.2 DATA-004 Outdoor Community / Activity Engine 对接

```text
客户（经同意的最小画像 / 当前声明）
        ↓
兴趣（徒步、登山、露营、高海拔；明确或受权证据）
        ↓
路线（已发布、可见、适用的路线/活动候选）
        ↓
活动匹配（资格、城市、经验、时间、名额/安全待核实）
```

| 对接步骤 | 输入 | 结果 | 必须控制 |
| --- | --- | --- | --- |
| 兴趣归一 | 授权 `interests`、客户本次选择 | 受控活动标签：徒步、登山、露营、高海拔。 | 未知不推断；高海拔标签不表示具备参与条件。 |
| 路线检索 | 活动类型、城市/距离偏好、日期窗、授权范围 | DATA-004 已发布且可见的路线/活动候选。 | 先 ACL 后检索；路线资料不替代天气、许可和救援信息。 |
| 匹配与排序 | 明确经验、已核验装备覆盖、活动规则、客户偏好 | `fit`、理由、缺口、待确认问题。 | 不自动报名或判断健康/资格；名额、天气、领队确认均为 `verification_required`。 |
| 结果回流 | 报名/签到/完成等活动引擎事实 | Journey 引用及可审核 Growth 输入。 | 仅可信状态进入成长计算；不直接发积分。 |

## 7. Community Matching

```text
兴趣 → 城市 → 经验 → 社群
```

社群匹配首先取客户明确同意的兴趣，求交集于其允许使用的城市/活动范围，再根据经验证的经验等级匹配 DATA-004 可见社群。结果对象包含 `community_ref`、名称/简介的可见投影、匹配兴趣、城市、经验要求、活动类型、`fit_reason`、`confidence`、`membership_status`、`eligibility_status` 和核实项。

- `membership_status` 仅反映当前客户自身状态，绝不公开其他成员名单、联系方式或位置。
- `eligibility_status` 仅为建议：`potential_fit`、`needs_verification`、`not_matched`；不得自动加入社群、发送邀请或订阅通知。
- 城市、兴趣或经验缺失时优先询问客户或提供匿名通用社群目录；不能用购买能力、顾问关系或模型相似度暗中填补。
- 社群推荐/活动通知属于独立目的；没有有效 Consent 时只能提供客户主动查询的非个性化公开信息（若其本身可见）。

## 8. Growth & Points

Customer Brain 将 DATA-003.4 Growth Loop 作为唯一成长/积分投影来源，而不是自建可写账本。

| 积分来源 | 可接受事实 | 入账控制 |
| --- | --- | --- |
| 购买 | 已认证的购买/退款状态与活动规则引用。 | 需幂等键、规则版本、有效期；退款/撤销可冲正。 |
| 活动 | 报名、签到、领队确认、完成等可信活动事件。 | 未完成、争议或高风险活动待审核。 |
| 分享 | 已获内容/平台/可见性授权的内容事实。 | 禁止把营销发送或未经授权内容自动计分。 |
| 打卡 | 可验证、符合频控和反作弊规则的事件。 | 重复、伪造或位置/隐私风险进入审核。 |
| 邀请 | 已解析身份、归因窗口和反作弊通过的邀请案件。 | 禁止自邀、重复邀请或直接自动入账。 |

成长等级显示为 **Lv1–Lv5**。等级阈值、权益和升级规则由 Growth Loop 版本化规则决定；Customer Brain 只显示 `level`、积分摘要、进度、规则版本、`as_of` 与审核/限制状态。等级不是户外安全许可、信用判断或自动折扣/成交依据。

## 9. AI Member Assistant

Member Assistant 是客户自助的 AI Runtime 消费者。它只能使用当前客户本人认证会话中、按 Consent 和目的过滤后的最小上下文；不向客户泄露员工备注、其他客户、内部机会分数、成本、库存保留或安全策略细节。

| 客户查询 | 编排 | 安全输出 |
| --- | --- | --- |
| “我的装备适合去哪？” | 读取已核验装备、客户明确经验和可见路线知识，调用活动匹配。 | 给出适配场景候选、装备缺口、天气/路线/身体状况/活动资格核实项；不保证安全或自动报名。 |
| “我下一步需要买什么？” | 使用装备覆盖、明确场景/预算和已发布产品知识，调用 DATA-003.2。 | 给出可选装备角色/产品候选、理由、兼容限制、价格/库存待核实；不自动加购或下单。 |
| “适合参加什么活动？” | 读取兴趣、城市、经验证经验和 DATA-004 可见活动。 | 返回活动/社群候选、匹配理由、难度/资格/名额/天气待确认项；不自动报名或发送通知。 |

每次回答将事实、分析、假设、未知和建议分区；每个事实/建议附可见 `citation_id`、来源、版本、`as_of`、置信度和限制。没有足够授权或证据时，返回匿名通用指导、澄清问题或 `needs_verification`，而非访问个人历史。

## 10. Employee Integration

Employee Workspace 仅经 M6.1 AI Runtime 调用 Customer Brain，员工从不凭客户 ID 直接获得完整资料。服务端以员工身份、组织/门店范围、`advisor_id` 或已批准客户关系、用途、Consent 及字段 ACL 求交集。

| 员工视图 | 可显示内容 | 限制 |
| --- | --- | --- |
| 客户画像 | 脱敏展示名、户外/兴趣/消费的最小授权摘要、数据时点。 | 无关系、Consent 或字段权时不显示；不显示原始身份标识。 |
| 装备档案 | 已核验装备、覆盖摘要、使用场景、待确认项。 | 不以购买记录推断实际状态；不显示不必要订单细节。 |
| 推荐机会 | 经过授权的装备/活动/社群机会及理由、置信度、Citation。 | 仅候选，不是营销名单或成交指令；库存/价格/资格需人工核实。 |
| 下一动作 | 可编辑的服务问题、核实步骤或人工待办草案。 | 不能自动发送消息、创建订单、报名、写画像或改变积分。 |

员工看见的任何个人级推荐均带 `advisory_only_requires_human_approval`、`policy.decision`、`freshness` 与限制。客户新声明只能经独立、获同意的 Customer Brain 提交流程形成审核候选；本设计的读取与反馈 API 不会自动写回事实。

## 11. API Contract

### 11.1 通用约定

所有 API 由服务端从认证令牌派生 `actor_id`、`organization_id`、角色、门店/客户关系、用途及策略快照；请求提供的身份/范围不能提升权限。每个响应包含 `correlation_id`、`audit_id`、`policy`、`freshness.as_of`、`lineage` 和 `disposition: "advisory_only_requires_human_approval"`。响应只包含已授权字段；无权、对象不存在或不应披露存在性时统一 `403 POLICY_DENIED`。常用错误：`400 INVALID_REQUEST`、`401 UNAUTHENTICATED`、`403 POLICY_DENIED`、`409 CONTEXT_STALE`、`422 PURPOSE_OR_CONSENT_REQUIRED`、`503 DEPENDENCY_UNAVAILABLE`。

| API | 权限与用途 | 返回 | 重要约束 |
| --- | --- | --- | --- |
| `GET /api/customer/profile/{id}` | `customer:profile:read`；`member_self_service`、`service` 或获批 `personalization`。 | 最小 Customer Profile、Consent 限制、质量/来源/时点。 | `{id}` 必须是 `customer_id`；不得按手机/微信自由检索。 |
| `GET /api/customer/equipment/{id}` | `customer:equipment:read`；服务或获批个性化用途。 | 已授权 Equipment Profile、购买证据摘要、升级机会。 | 升级机会是建议；不返回原始订单/支付数据。 |
| `GET /api/customer/journey/{id}` | `customer:journey:read`；活动/服务或获批个性化用途。 | 已验证 Journey、成就、Growth 摘要和核验状态。 | 不返回其他成员数据；自报/草稿明确标记。 |
| `GET /api/customer/activity-recommend/{id}` | `customer:activity:recommend`；客户主动查询或有效个性化同意。 | 徒步、登山、露营、高海拔活动/社群候选、理由、缺口、核实项。 | 禁止自动报名、邀请、触达或安全结论。 |
| `POST /api/customer/feedback` | `customer:feedback:submit`；本人或有权员工的服务反馈用途。 | `202 Accepted`、`feedback_id`、`review_state`、`audit_id`。 | 只创建脱敏反馈/更正候选；不得直接写入画像、积分、活动或权限。 |

`GET` 支持受控的 `as_of`、`locale`、`cursor`、`limit`（1–50）；范围参数只能与服务端范围求交集。`POST /api/customer/feedback` 最小负载为 `customer_id`、`feedback_type`、`target_ref`、`statement`（可选最小文本）、`evidence_refs`（可选）、`occurred_at` 和 `correlation_id`；状态机为 `submitted → triaged → in_review → approved|rejected|withdrawn`。

## 12. Privacy & Permission

最终许可公式为：

```text
Authenticated identity ∩ RBAC capability ∩ resource ACL ∩ organization/data scope
∩ customer relationship (employee only) ∩ purpose ∩ consent ∩ field policy ∩ effective time
```

| 控制 | 设计 |
| --- | --- |
| Consent | 按 `service`、`member_self_service`、`personalization`、`activity`、`community`、`growth`、`feedback` 等目的保存证据与有效期；撤回后停止未来个性化处理和触达。 |
| RBAC + ACL | RBAC 设定能力上限；ACL 在具体客户、活动、社群、反馈和审计对象上逐项判定。客户本人只能访问本人；员工需有效关系与职责；管理层默认使用去标识化聚合。 |
| Data Scope | 强制组织、区域、门店、顾问关系、客户本人、时间窗、字段分类和明细/聚合粒度。导出、跨范围比较和高敏字段需要额外显式许可。 |
| Field minimization | PII 默认掩码/省略；模型只接收授权后的最小投影。任何缓存采用组织与策略快照隔离，并在 Consent/关系/策略变更时失效。 |
| Audit | 追加式记录主体受控引用、目的、授权决定、掩码、资源引用/哈希、来源/版本、推荐规则、反馈状态、时间与关联 ID；审计访问本身受 ACL 和保留期控制。 |
| Retention & rights | 执行适用的保留、删除、更正、导出和法务保全规则；更正/删除请求不可静默改写历史审计或权威来源事实，而应建立可回放案件。 |

## 13. 验收清单

- [x] **Identity：** 定义 `customer_id`、mobile/微信/会员/企业微信、来源、顾问、组织与 Consent；具备统一解析、组织隔离和隐私治理边界。
- [x] **Profile：** 定义基础、户外和消费画像以及属性来源、质量、有效期和目的限制。
- [x] **Equipment：** 定义产品、品牌、购买时间、使用场景、装备等级及可解释的升级机会。
- [x] **Journey：** 定义路线、活动、日期、难度、成就和可信状态。
- [x] **Activity Engine：** 定义连接 DATA-004 的客户→兴趣→路线→活动匹配，支持徒步、登山、露营、高海拔。
- [x] **Community：** 定义兴趣→城市→经验→社群匹配及成员隐私边界。
- [x] **Growth：** 对接 DATA-003.4，覆盖购买、活动、分享、打卡、邀请积分及 Lv1–Lv5。
- [x] **AI Member Assistant：** 覆盖装备去向、下一步购买和活动建议，并提供证据、限制和人工核实。
- [x] **Employee Integration：** 定义客户画像、装备档案、推荐机会与下一动作的受控只读视图。
- [x] **Privacy：** 定义 Consent、RBAC + ACL、Data Scope、字段最小化与 Audit。
