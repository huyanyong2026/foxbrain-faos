# VAFOX Control Server Phase 1 实施计划

> **文档性质：** 购买并交付 `control.vafox.com` 后的实施依据。<br>
> **阶段：** Phase 1。<br>
> **执行边界：** 本文仅定义实施计划、架构、配置目标、验证与回滚要求；不执行 SSH 修改、服务器操作、Docker 修改、数据库修改或任何生产变更。

## 0. 目标、范围与前提

### 0.1 目标

建设独立的 `control.vafox.com`，作为 VAFOX 2.0 的 **Control Plane Server**。Phase 1 建立统一、可审计的控制能力，为以下服务提供后端基础：

- Agent Hub；
- Agent Registry；
- Approval Center；
- Report Center；
- CEO Dashboard Backend。

### 0.2 Phase 1 服务器基线

| 项目 | 规划配置 |
| --- | --- |
| 域名 | `control.vafox.com` |
| 定位 | VAFOX Control Plane Server |
| CPU | 4 核 |
| 内存 | 16 GB |
| 存储 | 200 GB SSD |
| 带宽 | 5 Mbps |
| 操作系统 | Ubuntu LTS |

### 0.3 架构边界

```text
gateway.vafox.com
        |
        v
control.vafox.com
        |
--------------------------------
|              |               |
v              v               v
huyan          ai              core
CEO 入口       AI Workforce    Data Brain
```

- `gateway.vafox.com` 保持统一入口、流量路由与边界接入职责。
- `control.vafox.com` 负责控制面编排、注册、审批、报告和仪表盘后端，不承载业务工作负载主数据。
- `huyan` 承载 CEO 入口相关能力，`ai` 承载 AI Workforce，`core` 承载 Data Brain；三者均是控制面的受控目标或数据来源，而非控制面本体。
- Phase 1 不改变上述既有生产角色，不将 Control Plane 功能直接部署到 Huyan、AI 或 Core。

### 0.4 非目标

Phase 1 不包含跨节点业务迁移、数据库重构、高可用集群、自动化生产发布、对外开放管理接口或替换现有 Gateway。本阶段先形成可控、可观测、可审计的单节点控制面基础。

## 一、Control Server 定位

### 1.1 核心定位

`control.vafox.com` 是 VAFOX 2.0 的控制面中枢：接收经 Gateway 转发或内部系统提交的控制请求，维护 Agent 身份与能力目录，执行审批流状态管理，归集报告，并向 CEO Dashboard 提供经过授权的聚合数据与查询接口。

控制面应以“**协调、授权、记录、呈现**”为主，而不直接承担 Huyan 的 CEO 交互、AI 的推理/执行或 Core 的数据脑主职责。

### 1.2 为什么独立于 Huyan

独立部署的原因如下：

1. **职责隔离：** Huyan 是 CEO 入口，优先保障对话、指令接收与入口体验；Control Server 管理跨域 Agent、审批和报告，两类职责的变更节奏、资源模型和权限模型不同。
2. **风险隔离：** 审批、注册或报告服务异常时，不应影响 CEO 入口的可用性；同样，Huyan 的入口流量或故障不应中断 Control Plane 的审计与恢复能力。
3. **权限最小化：** 控制面需要持有面向 `huyan`、`ai`、`core` 的受控访问凭据与策略。将这些高权限能力从入口服务器分离，可缩小凭据暴露面并便于独立审计。
4. **资源隔离：** Agent 编排、报告汇总、Dashboard 查询及日志保留会产生独立的 CPU、内存、I/O 和存储消耗，不能与 Huyan 入口竞争资源。
5. **演进隔离：** 后续扩展多 Agent、多节点、审批策略、报表和 CEO Dashboard 时，可独立扩容或升级控制面，不要求同步改动 Huyan。
6. **可审计性：** 以 `control.vafox.com` 作为唯一控制记录汇聚点，有利于统一追溯请求、审批、执行引用、结果和责任边界。

## 二、Phase 1 资源规划

### 2.1 资源承载结论

在严格限定 Phase 1 并发、日志、报表保留与后台任务频率的前提下，**4C16G、200 GB SSD、5 Mbps** 可以承载以下初版能力：

- Agent Hub；
- Registry；
- Approval；
- Reports；
- Dashboard 初版。

这是一套单节点、控制面优先的容量基线，不等同于高可用或大规模实时分析容量承诺。

### 2.2 建议资源预算

| 资源 | 建议预算 | 规划说明 |
| --- | ---: | --- |
| CPU | 4 vCPU | API、审批状态机、轻量任务调度与报表聚合共用；避免在本机运行大模型推理、批量数据训练或高并发 ETL。 |
| 内存 | 16 GB | 为服务进程、缓存、操作系统页缓存与短时报告任务预留余量；应为每个组件设置上限。 |
| SSD | 200 GB | 供应用、审计日志、报告产物、备份包和短期缓存使用；不作为大规模原始数据湖或镜像仓库。 |
| 带宽 | 5 Mbps | 适合控制指令、状态、审批和结构化报告；大文件、镜像和批量原始数据需排队、限速或转移到专用存储。 |

### 2.3 存储分配目标

| 数据类别 | 初始配额/策略 | 说明 |
| --- | --- | --- |
| 应用与运行时 | 30 GB 以内 | 包含服务版本、依赖与受控运行时缓存。 |
| 审计与服务日志 | 40 GB 上限 | 分级保留、轮转、压缩并配置告警。 |
| 报告与 Artifact | 50 GB 上限 | 报告按日期/任务归档，超期转移到备份存储。 |
| 本地备份暂存 | 50 GB 上限 | 仅用于加密备份的短期暂存；不得成为唯一备份副本。 |
| 操作系统与预留空间 | 至少 30 GB | 保持升级、故障排查和恢复空间；磁盘使用率不得长期逼近阈值。 |

### 2.4 运行控制

- 为每项服务设置 CPU、内存、文件描述符和重启策略上限，避免单服务耗尽主机资源。
- 将报告生成、归档和备份安排在低峰窗口，并限制并发任务数。
- 对请求体、上传附件、报表导出和日志单文件大小设置边界，适配 5 Mbps 带宽。
- Phase 1 只保存控制面元数据、必要审计记录与报告产物；业务原始数据仍归属 `huyan`、`ai` 或 `core`。

## 三、软件组件规划

| 组件 | 主要职责 | Phase 1 输入/输出 | 核心控制要求 |
| --- | --- | --- | --- |
| Agent Hub | 协调 Agent 任务、健康状态、任务引用和受控分发。 | 输入：经授权的任务请求；输出：任务状态、执行引用、健康摘要。 | 不直接绕过审批执行高风险动作；所有任务具备可追溯 ID。 |
| Registry Service | 管理 Agent、节点、能力、版本、状态与信任信息。 | 输入：受控注册/心跳；输出：可用 Agent 与能力目录。 | 注册需认证；变更保留审计；禁止匿名或未授权注册。 |
| Approval Service | 管理审批单、策略匹配、授权状态、有效期和拒绝原因。 | 输入：待审批动作；输出：批准/拒绝/过期状态与审批记录。 | 默认拒绝；授权绑定主体、动作、目标、时间窗口与范围。 |
| Report Service | 汇聚健康、任务、审批和异常报告，生成可检索的结构化记录。 | 输入：已验证报告；输出：报告索引、摘要、导出产物。 | 校验来源和完整性；保留生成时间、来源和关联任务 ID。 |
| Dashboard Backend | 为 CEO Dashboard 提供最小必要的聚合查询、状态卡片和趋势数据。 | 输入：已授权查询；输出：脱敏后的仪表盘数据。 | 只读优先、RBAC、查询审计、数据最小化。 |
| API Gateway | 为控制面组件提供统一鉴权、路由、限流、版本与审计入口。 | 输入：来自 Gateway/内部服务的 API 调用；输出：受控转发或拒绝。 | TLS、认证、限流、请求 ID、审计；不替代 `gateway.vafox.com` 的公网入口职责。 |

### 3.1 组件交互原则

```text
请求/事件
   |
API Gateway --> Approval Service --> Agent Hub --> 受控目标（huyan / ai / core）
   |                   |                  |
   |                   v                  v
   +--> Dashboard Backend <--- Report Service <--- 执行/健康报告
                 ^
                 |
          Registry Service（身份、能力、状态）
```

- 涉及控制动作的请求，先经 API Gateway 认证和审计，再由 Approval Service 判定是否具备有效授权。
- Agent Hub 仅可选择 Registry 中已登记、状态健康且权限匹配的 Agent/目标。
- Report Service 仅接收具有来源身份、任务引用和完整性信息的报告。
- Dashboard Backend 不直接访问受控节点，不直接发起控制动作。

## 四、目录规划

控制面统一根目录规划为：`/opt/vafox-control/`。

```text
/opt/vafox-control/
├── agents/
├── registry/
├── approvals/
├── reports/
├── artifacts/
├── logs/
└── backups/
```

| 目录 | 规划用途 | 管理要求 |
| --- | --- | --- |
| `agents/` | Agent Hub 配置、任务定义、运行元数据与受控工作区。 | 按 Agent/任务 ID 隔离；不存放明文长期密钥。 |
| `registry/` | Agent、节点、能力、版本和信任状态的服务数据与配置。 | 数据变更须可审计；仅服务账户可写。 |
| `approvals/` | 审批策略、审批记录、授权凭据引用及不可变审计副本。 | 追加式保留；禁止普通服务绕过审批直接写入“已批准”状态。 |
| `reports/` | 健康、任务、审批和异常报告及其索引。 | 按日期与任务 ID 归档；配置保留和导出策略。 |
| `artifacts/` | 经过审查的计划产物、部署包、校验清单和版本清单。 | Artifact 必须含版本、校验和、来源与审批引用。 |
| `logs/` | API、服务、安全、审计和作业日志。 | 统一时间、轮转、压缩、访问控制和异常告警。 |
| `backups/` | 加密备份的短期暂存、备份清单和恢复演练记录。 | 不作为唯一副本；备份应转移至独立受控存储。 |

目录与权限实施时应使用专用服务账户和服务组，避免以 `root` 日常运行应用；`/opt/vafox-control/` 不应与 Huyan、AI、Core 的应用目录、运行时或 Secrets 混用。

## 五、安全加固计划

### 5.1 SSH 与主机访问

- 使用 SSH Key 登录，密钥采用个人或角色化管理，禁止共享账户与共享私钥。
- 禁止 `root` 使用密码登录；日常管理使用具名低权限账户，通过受审计的受控提权完成必要维护。
- 禁止密码暴力尝试与不受限管理源；管理入口应按组织网络策略限制来源并记录登录审计。
- 建立离职、角色变更和密钥泄露时的密钥吊销与轮换流程。

### 5.2 最小权限

- 为 API Gateway、Agent Hub、Registry、Approval、Report 与 Dashboard Backend 分配独立服务身份或等效权限边界。
- 每个组件仅获取完成职责所需的目录、网络、API 和凭据访问权。
- 控制目标访问应按 `control → huyan`、`control → ai`、`control → core` 分别授权；不得使用跨节点通用高权限凭据。
- 高风险操作采用默认拒绝、显式批准、短时有效、最小目标范围与可撤销的授权模型。

### 5.3 审计日志

- 审计认证、授权、审批、策略变更、Agent 注册、任务提交、目标调用、报告接收、导出和管理操作。
- 每条审计记录至少包含时间、请求 ID、主体、动作、目标、结果、关联审批 ID 与来源地址（在适用时）。
- 统一使用 UTC 时间，限制审计日志的修改和删除权限，并配置保留、归档和完整性校验策略。
- 安全告警至少覆盖重复认证失败、越权尝试、审批绕过、未知 Agent 注册、异常出站连接和磁盘空间不足。

### 5.4 Secret 管理

- Secret 不写入代码仓库、Artifact、普通日志、报告正文或未加密备份。
- 使用受控的 Secret 管理机制保存 API 密钥、节点凭据、签名密钥和数据库连接信息；应用只通过短期引用或受限注入读取。
- 分离开发、测试和生产 Secret；按组件、环境和目标节点最小化分发。
- 建立 Secret 轮换、撤销、访问审计和泄露处置流程；变更后验证旧凭据失效。

### 5.5 基线加固与更新

- 采用受支持的 Ubuntu LTS 版本，并建立安全更新评估、维护窗口、变更审批与回滚记录流程。
- 仅开放必需端口与协议，默认拒绝不需要的入站访问；服务间流量使用加密传输和身份验证。
- 关闭不必要服务，配置时间同步、资源告警、磁盘告警和备份失败告警。

## 六、网络规划

### 6.1 逻辑连接

```text
control.vafox.com
   ├── control → Huyan：CEO 入口状态、已批准控制请求与结果回传
   ├── control → AI：AI Workforce Agent 状态、受控任务与结果回传
   └── control → Core：Data Brain 状态、受控查询/任务与结果回传
```

### 6.2 网络原则

- `control → Huyan`、`control → AI`、`control → Core` 均应使用独立的身份、授权范围与审计记录。
- 默认只允许 Control Server 发起完成 Phase 1 所必需的出站连接；目标节点不得因方便而获得对 Control Server 的广泛反向管理权限。
- 控制接口应走 TLS，加密传输并校验证书/服务身份；禁止以明文协议传递凭据、审批令牌或报告。
- API Gateway 对控制面接口实施认证、授权、限流、请求大小限制、超时和请求 ID 传播。
- 5 Mbps 带宽优先保障审批、状态、告警和小型结构化报告；大文件走异步队列、限速、压缩或专用存储，不阻塞控制指令。
- 未经批准，不开放公共管理面、数据库端口、内部服务调试端口或跨节点任意访问。

### 6.3 最小连通性矩阵

| 来源 | 目标 | Phase 1 允许用途 | 要求 |
| --- | --- | --- | --- |
| `gateway.vafox.com` | `control.vafox.com` | 受认证的控制面 API 转发。 | TLS、路由白名单、限流、审计。 |
| `control.vafox.com` | `huyan` | 已批准的控制请求、健康状态与报告交换。 | 最小端口、目标专属凭据、操作审计。 |
| `control.vafox.com` | `ai` | 已批准的控制请求、健康状态与报告交换。 | 最小端口、目标专属凭据、操作审计。 |
| `control.vafox.com` | `core` | 已批准的控制请求、健康状态与报告交换。 | 最小端口、目标专属凭据、操作审计。 |
| CEO Dashboard 客户端 | Dashboard Backend | 经 Gateway 暴露的只读聚合查询。 | 身份认证、RBAC、查询审计、脱敏。 |

## 七、部署流程

以下为购买并交付服务器后的标准实施顺序。每一步均须有责任人、变更记录、审批状态和验证结果；任一步失败即停止后续步骤并进入回滚评估。

### 7.1 准备

1. 确认 `control.vafox.com` 的域名、Ubuntu LTS 版本、4C16G/200 GB SSD/5 Mbps 资源和交付边界。
2. 确认 Gateway、Huyan、AI、Core 的责任人、联络窗口、目标接口、数据分类及最小网络需求。
3. 完成服务清单、端口清单、权限矩阵、Secret 清单、日志保留策略、备份目标和变更审批单。
4. 制定维护窗口、失败停止条件、验收负责人及回滚负责人。
5. 准备经过审查的版本清单、配置模板、校验和、发布说明和回滚包；不得使用未经审查的临时 Artifact。

### 7.2 安装

1. 按获批的 Ubuntu LTS 基线完成操作系统初始化与安全基线配置。
2. 创建专用服务身份、受控目录结构 `/opt/vafox-control/`、日志与备份路径，并应用最小权限。
3. 安装经过批准的运行时、服务管理、监控与日志组件；每个软件版本应记录来源和校验结果。
4. 部署 API Gateway、Registry Service、Approval Service、Report Service、Agent Hub 与 Dashboard Backend 的最小可用版本。
5. 为各服务设置独立配置、资源上限、健康检查、启动顺序和失败恢复策略。

### 7.3 配置

1. 配置 TLS、服务身份、API 认证、RBAC、审计字段、限流和超时。
2. 配置 Secret 管理引用，验证应用不在配置文件、日志或环境导出中暴露明文 Secret。
3. 配置 Agent/节点注册策略、审批策略、报告格式、Artifact 校验和与保留周期。
4. 配置 `control → Huyan`、`control → AI`、`control → Core` 的最小连通性、专属凭据和网络规则。
5. 配置备份、日志轮转、告警、监控与恢复演练记录；备份应加密并转移至独立位置。

### 7.4 验证

1. 验证主机资源、时间同步、磁盘预留、服务进程、健康检查和日志写入均符合基线。
2. 验证未认证请求、无权限请求、过期审批、越权目标与未知 Agent 注册均被拒绝并留下审计记录。
3. 验证合法 Agent 可完成注册、心跳、能力查询和健康状态上报。
4. 验证一条模拟的“提交—审批—受控任务引用—报告回收—Dashboard 查询”闭环，不触发生产业务变更。
5. 验证备份可用性、配置恢复步骤和服务恢复步骤；记录演练结果。

### 7.5 接入

1. 首先以只读健康状态和报告上报方式接入 Huyan、AI、Core，观察稳定性和审计完整性。
2. 再接入 CEO Dashboard 初版的只读聚合接口，确保 Dashboard 不直接控制目标节点。
3. 每次新增 Agent、节点、接口或动作类型均独立完成权限评审、审批策略、测试和验收。
4. Phase 1 稳定运行后，基于容量、错误率、审批时延与运维反馈决定后续功能和扩容计划。

## 八、回滚方案

### 8.1 回滚触发条件

出现以下任一情况时，停止接入或发布并启动回滚评估：认证/授权绕过、Secret 泄露风险、审计不可用、目标节点异常、数据完整性异常、服务持续不健康、磁盘逼近容量阈值或性能显著影响控制链路。

### 8.2 配置备份

- 每次变更前备份已审批的服务配置、策略、目录清单、版本清单、关键权限设置和 Secret 引用元数据（不备份明文 Secret）。
- 为备份记录版本、时间、责任人、校验和、适用服务和恢复顺序。
- 配置备份与报告/审计备份分层保存，并定期验证可读性和完整性。

### 8.3 版本回退

- Artifact 必须可定位到明确版本和校验和，保留当前稳定版与至少一个已验证的前一稳定版。
- 回退前冻结新任务、保留现场日志和审计证据，并确认数据库/状态兼容性（如未来引入持久化组件）。
- 按“API Gateway → 核心服务 → 后台作业 → Dashboard Backend”的受控顺序切换至已验证版本，并执行健康检查。
- 不得以删除日志、清空数据或覆盖审批记录作为回退手段。

### 8.4 服务恢复

1. 隔离故障组件或停止其接收新请求，保持审计证据。
2. 使用经验证的配置备份与稳定 Artifact 恢复受影响服务。
3. 恢复 Registry、Approval、Report 的一致性检查，再恢复 Agent Hub 和 Dashboard 查询。
4. 验证认证、审批拒绝路径、审计、健康检查与报告查询后，才逐步恢复对 Huyan、AI、Core 的接入。
5. 记录故障时间线、影响范围、恢复结果和预防措施；未完成复盘不得扩大接入范围。

## 九、扩容条件：4C16G 升级至 8C32G

当以下任一条件持续出现，且经日志、监控和容量评审确认不是单点配置错误或异常任务造成时，应规划从 **4C16G 升级到 8C32G**：

| 维度 | 4C16G 扩容触发条件 | 8C32G 的目标 |
| --- | --- | --- |
| CPU | 正常业务时段 CPU 持续超过 70%，或报告/编排高峰导致控制 API 响应显著变慢。 | 为 API、编排、报告和后台任务提供隔离余量。 |
| 内存 | 可用内存长期低于 25%，发生频繁交换、OOM 或服务因内存压力重启。 | 为服务缓存、并发查询和短时任务保留安全余量。 |
| 延迟与并发 | 审批、注册、报告或 Dashboard 在既定 SLO 下无法稳定响应，且优化并发/缓存后仍无改善。 | 支持更高并发 Agent、审批和查询量。 |
| 报告处理 | 报告汇总、归档或导出长期挤占在线 API 资源，无法在低峰窗口完成。 | 将在线服务与后台处理资源更好隔离。 |
| Agent/节点规模 | 接入 Agent、目标节点、心跳或任务量超出单节点基线，并造成队列积压或审计延迟。 | 为增加的控制面实体与队列处理提供余量。 |
| 预留增长 | 计划在同一阶段新增多个 Dashboard 视图、策略评估或经批准的自动化能力。 | 在功能上线前预留容量，避免带压迁移。 |

扩容决策必须基于至少一个完整业务周期的 CPU、内存、磁盘 I/O、网络、队列长度、错误率、P95/P99 延迟和日志增长趋势；升级前后均需记录基线并执行回归验收。

## 十、上线验收标准

上线应满足以下全部标准：

1. **资源与系统：** `control.vafox.com` 已确认满足 4C16G、200 GB SSD、5 Mbps、Ubuntu LTS 基线，且 CPU、内存、磁盘和时间同步健康。
2. **角色隔离：** Huyan、AI、Core 的现有职责未被替换或混入 Control Server；Control Server 不承载其业务主数据或推理工作负载。
3. **组件可用：** Agent Hub、Registry、Approval、Report、Dashboard Backend、API Gateway 均已部署最小可用版本并通过健康检查。
4. **身份与权限：** SSH Key、禁用 root 密码登录、专用服务身份、RBAC、最小权限与 Secret 管理已按计划验证。
5. **网络最小化：** Gateway 到 Control、Control 到 Huyan/AI/Core 的链路仅开放获批用途，并具有 TLS、身份验证、限流和审计。
6. **审批闭环：** 无审批、越权、过期或目标不匹配的动作均被拒绝；有效审批可完整追溯到主体、目标、范围、时间窗口和结果。
7. **审计与报告：** 关键控制行为均产生可查询审计记录；报告包含来源、时间、任务/审批关联和完整性信息；Dashboard 仅展示获授权的聚合数据。
8. **备份与回滚：** 配置备份、版本回退和服务恢复流程已完成演练或桌面推演，并有可核验记录。
9. **可观测性：** 服务健康、认证失败、审批异常、未知注册、磁盘容量、备份失败和关键链路错误均具备日志或告警。
10. **变更治理：** 发布版本、配置版本、审批记录、验收结果、责任人和已知风险均已归档；未满足项不得以“临时例外”直接上线。

## 十一、禁止事项

本计划为实施依据，不是执行授权。未经独立、明确的变更审批与维护窗口，严禁执行以下事项：

- SSH 修改；
- 任何服务器操作；
- Docker 修改；
- 数据库修改；
- 生产变更。

上述禁止范围包括但不限于：登录并修改主机配置、创建/删除目录、安装或卸载软件、修改 SSH 配置、变更用户或权限、重启服务、修改容器/镜像/编排、变更数据库结构或数据、开放端口、写入生产 Secret、部署版本、切流、执行自动化任务或修改 Huyan/AI/Core/Gateway 的生产配置。

后续任何实际操作均必须以单独的、经审批的实施单为准；实施单应明确目标主机、操作内容、责任人、时间窗口、验证步骤、停止条件与回滚步骤。
