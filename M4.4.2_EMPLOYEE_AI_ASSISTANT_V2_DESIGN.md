# VAFOX M4.4.2 Employee AI Assistant V2 Design

**状态：** Design Phase / 仅设计，不部署。<br>
**前置设计：** `M4_KNOWLEDGE_FOUNDATION_REPORT.md`、`M4.3.8_PRODUCT_INTELLIGENCE_PLATFORM_DESIGN.md`、`M4.4.1_SALES_KNOWLEDGE_ENGINE_DESIGN.md`、`DATA003.3_EMPLOYEE_AI_ASSISTANT_DESIGN.md`。<br>
**负责人：** Codex（设计开发）；WorkBuddy（环境）；Marvis（管理）；CTO（验收）。

## 1. 目标、原则与非目标

Employee AI Assistant V2（EAA V2）是供已认证员工使用的**只读决策辅助与人工可控工作台**。它在员工的授权范围内，将 Customer Brain、Sales Knowledge、Product Intelligence 和 Retail Brain 的已治理信息整理为可引用的客户服务、产品讲解、场景装备、连带需求及跟进建议。

1. **先授权、后读取。** 每次请求由服务端从可信身份计算组织、门店、角色、RBAC、ACL、用途和客户关系；前端不能提交或扩大这些范围。
2. **证据优先。** 对产品事实、销售主张、场景规则和经营信号的每一项业务主张都必须返回 `citation`、版本、数据时点、置信度和限制。
3. **最小化与人工确认。** 仅处理当前服务所需的最小授权客户投影；未知信息保持 `unknown`。员工在对客沟通前确认需求、尺码、价格、库存、天气/路线及适用性。
4. **建议不执行。** 每个响应固定为 `advisory_only_requires_human_approval: true`；AI 不能替员工联系客户、成交或写入业务事实。
5. **可回放且可纠正。** 请求、策略决定、输入快照引用、规则/模型版本、引用和员工反馈均可审计；反馈只能经审核形成新知识。

本设计不建设 CRM、订单、营销自动化、库存/SAP/Core 主数据或生产同步能力；不把模型推断写成客户、产品、库存或销售事实。

### 明确禁止

- 自动发送营销、活动邀请或任何外部消息；
- 自动成交、自动下单、预留/调拨库存、自动报价或改价；
- 修改 SAP；
- 修改 Core 生产同步、同步规则或生产数据；
- 将没有引用、未批准、过期或越权的信息作为建议依据。

## 2. Employee AI Assistant Architecture

```text
Employee Context（可信身份、门店、角色、权限、用途）
                         ↓ policy gate + minimization
Customer Brain（同意、客户关系、购买/装备最小投影）
                         ↓ authorized customer context
Sales Knowledge（已审核的需求发现、话术、异议与组合规则）
                         ↓ cited sales guidance
Product Intelligence（已发布产品事实、适配、限制、兼容性）
                         ↓ product candidates and evidence
Retail Brain（门店库存/销售只读快照、经营机会）
                         ↓ fresh, scoped operating signals
AI Advice（规则约束优先；生成仅整理解释）
                         ↓ editable advisory response
Employee Action（人工查看、核实、编辑、沟通或记录）
                         ↓ explicit result and feedback
Feedback Loop（脱敏反馈 → 审核 → Sales Memory / 知识优化）
```

| 层 | 职责与输入 | 输出 | 强制边界 |
| --- | --- | --- | --- |
| Employee Context | 从认证、员工目录和授权策略计算当前访问范围。 | 不可由客户端覆盖的 `AuthContext`。 | 失效、调店、撤权后下一请求重新计算。 |
| Customer Brain | 在同意、用途、客户关系都通过后提供最小客户投影。 | 客户画像摘要、购买摘要、明确装备和偏好状态。 | 不返回原始 PII；无权时不泄露对象存在。 |
| Sales Knowledge | 检索已批准且 ACL 可见的发现问题、方案、话术和异议知识。 | 带版本/引用的销售指导。 | Draft、过期或无证据内容不能进入回答。 |
| Product Intelligence | 用发布的产品知识筛选适配、限制、兼容和替代。 | 产品候选、事实与限制。 | 不替代 SKU、价格、库存或订单事实。 |
| Retail Brain | 读取授权门店的库存、销售和机会聚合快照。 | 时点化的可售性、重点商品及机会信号。 | 只读；库存不是承诺或预留。 |
| AI Advice | 先执行硬约束/权限过滤，再编排解释和补问。 | 可编辑建议、引用、限制、置信度。 | 不用生成内容填补未知事实或安全结论。 |
| Employee Action | 员工核实、采纳、编辑、拒绝或在批准系统中另行完成业务动作。 | 受控结果码和可选最小化摘要。 | 助手无发送、交易、下单或写 SAP/Core 权限。 |
| Feedback Loop | 将显式反馈去标识化、聚类并送人工审核。 | Sales Memory 草稿、质量指标、知识改进候选。 | 未审核内容不得自动改变知识、排序或员工评价。 |

### 2.1 Advice 生成门控

1. 验证 `Authorization`、`X-Correlation-Id`、`X-Purpose` 和请求体，再从令牌建立 `EmployeeContext`。
2. 先评估组织隔离、RBAC、资源 ACL、门店范围、客户关系、同意和用途；再检索各 Brain。
3. 排除未批准/失效/越权来源、`not_suitable` 产品、安全约束冲突、过期零售快照及未满足的必需输入。
4. 先返回必要补问、风险提示或“请人工核实”；仅在证据充分时组装候选方案。
5. 将客户明确陈述、知识事实、零售快照和推理建议分区呈现，附 Citation、鲜度和人工核实清单。

## 3. Employee Context Model

Employee Context 是每次请求的短期策略快照。`assigned_customers` 仅为授权关系引用，不能被用于枚举客户；`sales_history` 为范围内聚合或权威事实引用，而非客户明细副本。

| 字段 | 类型 | 定义 | 控制 |
| --- | --- | --- | --- |
| `employee_id` | string | 员工目录的稳定 ID。 | 从认证主体绑定，客户端不可覆盖。 |
| `store_id` | string[] | 当前工作门店及授权门店范围。 | 请求目标门店须在同组织的授权集合内。 |
| `role` | string[] | 如 `employee`、`store_manager`、`regional_manager`、`executive`。 | 仅受治理角色服务签发。 |
| `permission` | string[] | 如 `customer:read`、`advice:generate`、`feedback:write`。 | 能力须同时通过角色与 ACL/范围检查。 |
| `assigned_customers` | reference[] | 有效顾问/服务/转交关系。 | 顾客读取需关系、同意和用途共同满足。 |
| `sales_history` | aggregate/reference[] | 员工授权范围内的聚合表现或销售事实引用。 | 不用于自动归因、佣金判断或员工自动考核。 |
| `training_level` | controlled enum | 已完成的受控培训等级及有效期。 | 仅用于提示需要复核/升级人工处理，不能放宽权限或安全边界。 |

```json
{
  "employee_id": "emp_01J...",
  "organization_id": "org_vafox_cn",
  "store_id": ["store_chengdu_01"],
  "role": ["employee"],
  "permission": ["opportunities:read", "customer:read", "advice:generate", "feedback:write"],
  "assigned_customers": [{"customer_id": "cst_01J...", "relationship_role": "advisor", "valid_to": null}],
  "sales_history": [{"period": "2026-07", "metric_ref": "salesagg_01J..."}],
  "training_level": {"level": "outdoor_sales_l2", "valid_to": "2027-01-31"},
  "policy_snapshot": "pol_01J...",
  "evaluated_at": "2026-07-21T00:00:00Z"
}
```

## 4. Assistant Capabilities

### 4.1 Customer Assistant

**输入：** 经授权的客户信息最小投影、购买历史摘要、客户本次明确的活动需求、已有装备和员工补充的当前会话事实。缺失同意、关系或用途时降级为不含客户信息的通用场景建议。

**输出：**

| 输出 | 内容与呈现规则 |
| --- | --- |
| 客户画像 | 已明确的经验、偏好、预算/复用倾向与数据状态；区分“客户明确”“历史事实”“未知”，不推断敏感属性。 |
| 需求判断 | 活动、环境、时长、负重、预算、已有装备的确认状态、优先级和补问。 |
| 推荐方案 | 需求角色、产品候选/替代、适配理由、限制、组合覆盖和 Citation。 |
| 跟进建议 | 由员工决定的下一步、应确认问题、可选活动邀请草稿和复购机会；绝不自动发送。 |

客户画像的保留状态须为 `request_scoped`；员工如要记录客户明确声明，必须走独立、受同意治理的 Customer Brain 流程，不能由 EAA V2 自动回写。

### 4.2 Product Sales Assistant

员工询问“产品怎么介绍？”时，助手仅基于对应型号/变体的已发布产品知识和销售知识返回以下结构；型号、尺码、价格、库存和政策必须由员工现场核实。

| 区块 | 返回内容 |
| --- | --- |
| 产品定位 | 品类、目标需求、适用场景和不适用边界。 |
| 场景价值 | “场景/问题 → 有证据功能 → 条件与限制”的解释。 |
| 技术解释 | 规格/技术机制及其来源，不把营销语推导为绝对性能。 |
| 销售话术 | 可编辑、条件化、非贬损的开场/解释/确认草稿。 |
| 异议处理 | 价格、比较、线上价格、犹豫等的澄清问题、可验证比较维度、禁止表述与引用。 |

### 4.3 Scenario Assistant

```text
目的地
  ↓ 路线、海拔、季节、许可/天气待确认
活动
  ↓ 徒步 / 登山 / 露营及强度、时长、人数
环境
  ↓ 温度、风雨、地形、湿度、昼夜、负重
装备需求
  ↓ 功能角色、必选/可选、兼容性、风险与补问
产品组合
  ↓ 已发布候选、替代、覆盖缺口、Citation、人工核实
```

| 覆盖场景 | 必须确认 | 方案护栏 |
| --- | --- | --- |
| 川西 | 路线、海拔、季节、天数、温度、是否露营、负重与经验。 | 多日山地和多变天气仅输出角色级覆盖及已证实适配；不作天气、安全或通行保证。 |
| 冈仁波齐 | 转山计划、海拔适应、健康/专业要求、天气、天数和已有装备。 | 标为高海拔场景；缺少专业安全信息时抑制结论并提示人工/专业确认。 |
| 徒步 | 路线、时长、地形、湿滑、负重、经验。 | 核实鞋服尺码、天气和库存；不将城市轻量方案说成技术登山方案。 |
| 登山 | 技术难度、地形、气象、负重、经验和必要专业装备。 | 不替代领队、救援、医疗或路线安全判断；不满足硬约束即不推荐。 |
| 露营 | 人数、过夜、温度、营地、车载/徒步及燃料条件。 | 不承诺营地许可、气候安全或燃料使用；兼容性需引用支持。 |

### 4.4 Cross Sell Assistant

```text
经客户确认的已有装备
            ↓ 角色/适配/兼容/场景覆盖检查
缺口分析（required / optional / unknown）
            ↓ 预算、客户意愿和库存快照过滤
组合推荐（互补角色、替代、理由、Citation、核实项）
```

连带建议的触发条件是已确认的功能或兼容缺口，而非销售目标、员工历史或模型推测。输出先说明“已有覆盖”“待确认”和“缺口”；客户无需求、预算不适合、存在安全/兼容不确定性时不推送产品，只给开放式确认问题。不得自动加购、创建购物车、锁货或将推荐解释为促销。

### 4.5 Follow-up Assistant

| 项目 | 设计 |
| --- | --- |
| 互动记录 | 员工显式填写的最小化结构化记录：时间、渠道类别、已确认需求、已展示方案、客户许可状态、`outcome_code`。自由文本可选且先经敏感信息处理。 |
| 下一动作 | 由员工创建/修改的人工待办、截止时间和所需核实项；没有外部发送执行器。 |
| 活动邀请 | 仅在客户同意与用途允许时生成可编辑草稿和适用理由；员工在批准渠道中自行决定是否发送。 |
| 复购机会 | 基于授权购买摘要、生命周期或已确认消耗/场景缺口形成的候选，明确为机会而非客户意愿或成交预测。 |

## 5. Store Manager Dashboard

店长仪表盘是门店范围内的只读、聚合视图；客户条目仅在店长同时具备客户关系、用途与字段 ACL 时可下钻。所有指标均标注统计窗口、`as_of`、数据新鲜度和来源。

| 模块 | 展示 | 操作边界 |
| --- | --- | --- |
| 销售机会 | 授权门店内待人工跟进的客户/场景机会及理由、置信度、失效时间。 | 不自动分配、触达或标记成交。 |
| 库存提醒 | 只读库存快照、低可售/过期数据提醒、替代类别。 | 不补货、调拨、预留或承诺库存。 |
| 重点商品 | 已发布主推候选、适用场景、证据和门店表现聚合。 | 不绕过客户需求、适配或安全约束。 |
| 客户机会 | 最小化且授权的机会计数/队列，支持按客户关系下钻。 | 不用来建立未经同意的营销名单。 |
| 员工建议 | 培训覆盖、引用/反馈质量、待确认事项等聚合建议。 | 不用于自动绩效、排班、奖惩或佣金决定。 |

## 6. Sales Feedback Loop

```text
销售结果（权威系统引用或员工受控结果码）
                         ↓
反馈（采纳/编辑/拒绝、原因、最小化摘要、引用问题）
                         ↓
Sales Memory（去标识化、聚类、标记证据缺口的 Draft）
                         ↓
知识优化（销售负责人 + 知识审核 + 必要时产品/合规复核）
                         ↓
approved 新版本，供后续检索
```

| 字段 | 规则 |
| --- | --- |
| `feedback_id`、`advice_id`、`knowledge_ref`、`version` | 关联当时建议和证据快照，支持回放，不覆盖历史。 |
| `action` | 受控值：`accepted`、`edited`、`rejected`、`insufficient_context`、`citation_issue`、`product_unavailable`。 |
| `outcome_ref?` | 仅引用已授权的权威销售/互动事实；AI 或员工摘要不能自行宣称成交。 |
| `employee_summary?` | 可选、最小化、先进行 PII/敏感信息检测；不进入通用审计日志。 |
| `review_state` | `submitted → triaged → draft → in_review → approved|rejected`。 |

负反馈不自动下架知识、改变排序或评价员工。质量指标使用去标识化聚合，例如 Citation 完整率、信息缺口率、编辑/拒绝原因和审核时效。

## 7. API Contract

**共同约束：** JSON、RFC 3339 UTC、`Authorization`、`X-Correlation-Id`、`X-Purpose` 为必需头；组织、员工、角色和门店范围全部由服务器派生。请求不得携带客户端声明的角色、权限、客户原始 PII 或可写入 SAP/Core 的字段。所有成功 Advice 响应带 `advisory_only_requires_human_approval: true`。

### `GET /api/employee/opportunities`

用途：读取当前员工（或授权店长）的机会队列。查询参数：`store_id?`、`window?`、`status?`、`limit`（1–50）、`cursor?`；`store_id` 必须在授权范围内。

```json
{
  "items": [{
    "opportunity_id": "opp_01J...",
    "scope": "assigned_customer",
    "customer_ref": "cst_01J...",
    "reason": "已确认的雨天徒步需求仍缺少防雨角色",
    "next_questions": ["确认实际路线与预算"],
    "freshness": {"as_of": "2026-07-21T00:00:00Z", "status": "fresh"},
    "citation": ["cit_01"],
    "confidence": {"level": "medium", "rationale": "天气待确认"}
  }],
  "next_cursor": null,
  "advisory_only_requires_human_approval": true
}
```

### `GET /api/employee/customer/{id}`

用途：读取一个已授权客户的最小投影。`id` 仅接受受控 `customer_id`，不接受手机号、微信号或名称检索。响应按字段 ACL 脱敏；无权或不存在均返回不泄露存在性的 `404 RESOURCE_NOT_AVAILABLE`。

```json
{
  "customer_id": "cst_01J...",
  "profile": {"experience": "regular", "preferences": ["lightweight"], "unknown": ["budget"]},
  "purchase_history": [{"product_ref": "prd_...", "purchased_at": "2026-04-01"}],
  "existing_gear": [{"product_ref": "prd_...", "confirmation": "customer_confirmed"}],
  "consent": {"purpose": "in_store_service", "status": "allowed"},
  "as_of": "2026-07-21T00:00:00Z"
}
```

### `POST /api/employee/advice`

用途：生成 Customer、Product、Scenario、Cross Sell 或 Follow-up 的只读建议。`customer_id` 可选；只有经授权且符合用途时才可解析。`intent` 为受控值：`customer`、`product_sales`、`scenario`、`cross_sell`、`follow_up`。

```json
{
  "intent": "scenario",
  "customer_id": "cst_01J...",
  "store_id": "store_chengdu_01",
  "context": {"destination": "川西", "activity": "hiking", "environment": ["variable_weather"], "existing_gear_refs": ["prd_..."]},
  "question": "如何准备 7 天徒步装备？"
}
```

```json
{
  "advice_id": "adv_01J...",
  "sections": [{"kind": "needs", "content": "先确认路线、海拔、温度、负重与露营安排。", "citation": ["cit_01"]}],
  "recommendations": [{"role": "rain_shell", "product_refs": ["prd_..."], "limitations": ["请核实尺码、库存与天气"], "citation": ["cit_02"]}],
  "clarifying_questions": ["是否需要露营？"],
  "freshness": {"retail_as_of": "2026-07-21T00:00:00Z"},
  "confidence": {"level": "medium", "rationale": "路线与温度未知"},
  "citation": ["cit_01", "cit_02"],
  "advisory_only_requires_human_approval": true
}
```

### `POST /api/employee/feedback`

用途：提交对某个 Advice 的显式反馈。要求 `Idempotency-Key`；只创建受审核的反馈记录，不触发外部消息、订单、库存或知识发布。

```json
{
  "advice_id": "adv_01J...",
  "action": "edited",
  "reason_code": "insufficient_context",
  "employee_summary": "客户确认单日徒步，需补问降雨情况。",
  "outcome_ref": "salefact_01J..."
}
```

返回 `202 Accepted` 和 `{ "feedback_id": "fb_01J...", "review_state": "submitted" }`。`400` 表示 schema/幂等键错误；`403 POLICY_DENIED` 表示权限、用途或关系不通过；`409` 表示请求与幂等键冲突；`429` 表示限流；`503` 表示上游事实源不可用并且响应不使用陈旧事实伪装成功。

## 8. Permission: RBAC + ACL

RBAC 决定可用功能，ACL 决定特定客户、知识、产品、门店和字段是否可见；二者必须同时允许。策略还评估 `organization_id`、当前门店、客户关系、用途/同意、资产品牌/部门、生命周期和时间有效性。筛选只能缩小集合，不能授予权限。

| 角色 | 可读范围 | 可执行的助手操作 | 明确不可做 |
| --- | --- | --- | --- |
| 员工 | 自己门店、已分配/有效服务关系客户的最小投影；可见产品/知识。 | 查看机会、生成建议、提交自身反馈和人工跟进草稿。 | 跨店客户、受限资料、团队聚合、自动外发或交易。 |
| 店长 | 授权门店聚合和符合关系/用途的客户范围。 | 查看 Dashboard、审核门店机会、提出培训/知识缺口反馈。 | 跨门店数据、绕过客户字段 ACL、自动分配/考核。 |
| 区域 | 授权区域的跨店去标识化聚合和已授权经营信号。 | 查看区域趋势、提出知识优化建议。 | 任意客户明细、跨区域访问、自动经营执行。 |
| 管理层 | 授权组织级去标识化聚合、风险和知识质量指标。 | 查看经营/质量概览与审核状态。 | 不受限客户 PII、修改来源事实、以 AI 输出执行交易。 |

审计事件至少记录主体/委托主体、请求用途、策略版本与决定、资源类型/脱敏资源 ID、命中知识版本/Citation、鲜度、`advice_id` 和关联 ID。日志不得记录完整客户资料、原始 PII、受限来源正文或不必要自由文本。

## 9. Citation、版本与数据质量

每个 `citation` 至少包含 `citation_id`、`source_id`、`knowledge_or_product_id`、来源版本、SHA-256、不可变 locator（页/段/行/单元格）、捕获时间、ACL 和派生血缘。Citation 与正文采用同一 ACL：无权时既不返回内容，也不泄露来源标题或 locator。仅 `approved`/`Published` 且在有效期内的内容进入默认回答；新版本不可原地覆盖旧版本，历史 Advice 保留当时引用。

响应必须分别标注：`customer_confirmed`、`authoritative_fact`、`knowledge_claim`、`retail_snapshot`、`inference` 与 `unknown`。置信度表示证据和上下文完整性，绝不是成交概率；库存/价格/政策必须附 `as_of` 与鲜度，超过策略阈值则降级为“请人工核实”。

## 10. 验收清单

- [x] Employee Context：定义七个必填字段、来源、重新求值和最小化控制。
- [x] Customer Assistant：定义授权输入与画像、需求、方案、跟进输出。
- [x] Product Assistant：覆盖定位、场景价值、技术解释、话术与异议处理。
- [x] Scenario Assistant：覆盖川西、冈仁波齐、徒步、登山、露营及安全护栏。

- [x] Follow-up：定义互动记录、下一动作、活动邀请和复购机会，且不自动发送。
- [x] Dashboard：定义店长的销售机会、库存提醒、重点商品、客户机会和员工建议。
- [x] Feedback Loop：定义销售结果、反馈、Sales Memory、审核与知识优化。
- [x] Permission：定义 RBAC + ACL、门店/关系/用途/同意及审计控制。
- [x] Citation：定义可回链引用、版本、ACL、鲜度和可回放要求。
