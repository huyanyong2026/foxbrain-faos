# VAFOX M4.3.6 OSPREY Product Intelligence Instance Design

**状态：** Design Phase / 仅设计，不部署。
**定位：** OSPREY = **背负系统专家**；本实例将路线、天数、负重、体型和活动场景转化为可解释的背包方案，而非简单的商品检索。
**前置设计：** `M4.3_PRODUCT_KNOWLEDGE_DESIGN.md`、`M4.3.2_KNOWLEDGE_ASSET_CENTER_DESIGN.md`、`M4.3.3_PRODUCT_INTELLIGENCE_DESIGN.md`、`DATA003.2_RECOMMENDATION_ENGINE_DESIGN.md`、`DATA003.3_EMPLOYEE_AI_ASSISTANT_DESIGN.md`。
**负责人：** Codex（设计开发）；WorkBuddy（环境）；Marvis（管理）；CTO（验收）。

## 1. 目标、原则与边界

本设计定义 OSPREY 的只读商品智能实例：在经过审批且可引用的 OSPREY 产品知识上，先判断背负需求，再匹配场景和授权客户画像，最后结合门店库存生成供员工复核的推荐。推荐是装备适配建议，不是医疗、高反、路线安全或成交保证。

| 原则 | 设计要求 |
| --- | --- |
| 背负优先 | 容量、实际负重和背负系统是硬约束；品牌偏好、销量和库存只能在合格候选间排序。 |
| 证据优先 | 每个产品事实、背负结论和适用边界须有版本化 Citation；缺证据或冲突即降级、补问或抑制。 |
| 场景先于商品 | 先收集路线、天数、住宿/补给、装备体积、海拔/地形和天气信息，再讨论具体型号。 |
| 客户最小化 | 仅使用已授权、用途匹配的聚合画像；不以原始 PII、性别刻板印象或未经确认的体型推断推荐。 |
| 只读且人工确认 | 结果恒为 `advisory_only_requires_human_approval`；员工须现场确认躯干长度、腰带贴合、实际试背、天气、路线和实时库存。 |

### 明确禁止

- 修改 SAP，或修改 Core 生产同步、同步规则、来源事实；
- 自动下单、报价、锁定/扣减库存、调拨、补货、改价或创建采购订单；
- 自动销售、自动触达客户、代表客户或员工完成交易；
- 将背包建议描述为防止受伤、高反或其他安全风险的保证；
- 用未发布资产、营销文案臆测规格，或无 Citation 地发布产品事实。

## 2. OSPREY Product Intelligence Architecture

```text
Brand Knowledge（已审批 OSPREY 资料、技术手册、尺码/背负说明）
                         ↓ citation extraction + review
Product Intelligence（稳定产品/SKU 只读映射、版本化事实）
                         ↓
Backpack Intelligence（容量、负重、背负结构、躯干/腰带、通风）
                         ↓
Scenario Matching（路线、天数、住宿、地形、海拔、装备体积）
                         ↓
Customer Matching（授权体型、经验、活动等级、装备水平）
                         ↓
Recommendation（硬过滤 + 可解释排序 + 备选与补问）
                         ↓
Retail Integration（授权库存、销售聚合、门店陈列建议）
```

| 层 | 输入 | 输出 | 强制控制 |
| --- | --- | --- | --- |
| Brand Knowledge | KAC 中 `Published`、有效且 ACL 可见的 OSPREY 一手资料与已审核培训资料。 | 可定位的产品主张、版本、来源等级和 Citation。 | Draft、失效、越权资料不可作为推荐依据。 |
| Product Intelligence | `product_id`、系列、产品变体、只读 SKU 映射。 | 已审批的产品事实与适用边界。 | 不覆盖 SAP 商品主数据；`unresolved` SKU 不可作可售首选。 |
| Backpack Intelligence | 容量、重量、背负系统、框架、负重范围、躯干和腰带信息。 | 可计算的背负能力、适配范围与试背核对项。 | 规格必须有字段级证据；不能从系列名猜测。 |
| Scenario Matching | 行程、住宿、补给、环境、地形、装备清单。 | 容量带、负重带、结构要求、风险复核清单。 | 未提供关键条件时只给区间/补问，不给确定型号。 |
| Customer Matching | 明示或授权的身高、躯干测量、体型、经验和装备水平。 | 适配/排除信号、尺码试背步骤。 | 身高不是躯干长度替代；无授权时改为非个性化。 |
| Recommendation / Retail | 合格候选及授权库存、销售、门店快照。 | 排名建议、组合、库存核验与经营建议。 | 库存仅只读快照；不承诺有货、不执行交易或补货。 |

## 3. Backpack Intelligence Entity Schema

`BackpackIntelligenceEntity` 是派生的发布视图，不是商品主数据、实时库存表或客户档案。所有可发布主张都有 `citation[]`；`confidence` 衡量证据与输入完整度，**不是**舒适度或成交概率。

| 字段 | 类型/规则 |
| --- | --- |
| `product_id` / `brand_id` | 必填稳定 ID；品牌为 `brand:osprey`。 |
| `series` | `{series_id, name, positioning}`；必须有品牌知识证据。 |
| `capacity` | `{liters, capacity_band, expandable_liters?, evidence_refs}`；保留标称容量和容量带。 |
| `weight` | `{value, unit, basis, evidence_refs}`；`basis` 如 `listed_weight`，不得混用不同尺码/版本。 |
| `carry_system` | `{name, adjustment, suspension_features, limitations, evidence_refs}`；描述已证实的背负系统。 |
| `frame_structure` | `{type, material?, load_transfer_role, evidence_refs}`；例如内部框架/周边框架，但不推断未证实材料。 |
| `load_range` | `{min_kg?, max_kg?, source_kind, fit_status, evidence_refs}`；缺少官方范围时为 `unresolved`，不可伪造。 |
| `torso_fit` | `{size_system, torso_range_cm?, adjustment_range?, fitting_required, evidence_refs}`。 |
| `hip_belt` | `{type, adjustability, fit_notes, evidence_refs}`；腰带需员工试贴合确认。 |
| `ventilation` | `{backpanel_design, fit, limitations, evidence_refs}`；通风是条件化体验，不承诺体感。 |
| `material` | `[{component, material, claim, evidence_refs}]`；仅记录经批准技术主张。 |
| `scenario` | `ScenarioFit[]`：活动、环境、难度、`suitable/conditional/not_suitable`、限制。 |
| `activity_type` | 受控数组，如 `day_hiking`、`trekking`、`backpacking`、`high_altitude_trekking`。 |
| `customer_profile` | 匿名适配条件、经验/装备水平、排除项；不是 PII。 |
| `citation` / `confidence` | 字段级血缘及 `score/level/basis/limitations`，所有响应必须回传。 |

### 3.1 SKU Mapping 与库存链接

一个 `product_id` 可关联多个只读 `sku_id`（尺码、颜色、版本）；推荐先选择产品模型，再由授权门店的 `sellable` SKU 快照确认可售变体。映射须含 `sku_id`、`variant_attributes`、`mapping_status`、`catalog_as_of`、`source_ref`；`ambiguous/unresolved/discontinued` 映射不能显示为可成交首选。库存链接包含 `store_id`、`sellable_state`、`inventory_as_of`、`freshness_status` 和来源引用，绝不写入或预留库存。

## 4. 背负系统知识与容量模型

```text
容量（装备体积、天数、补给）
       ↓
负重（总重量及重量分布）
       ↓
背负结构（框架、悬挂、腰带、调节）
       ↓
用户体型（躯干、腰围、试背、经验）
       ↓
路线难度（距离、爬升、海拔、地形、天气）
       ↓
推荐型号（经 Citation、SKU 和库存核验的候选）
```

### 4.1 决策规则

1. **容量先估算，非按天数硬套。** `required_volume` 来自睡眠系统、炊具/食物、衣物、补水、雨具和共享装备体积；天数只用于补给与容错估算。返回容量区间而不是虚假的精确升数。
2. **负重决定结构门槛。** 估算总重须由客户和员工确认；仅当产品的 Citation 支持负重范围/结构能力时才判断合格。无可靠 `load_range` 时标记 `load_capacity_unverified`。
3. **结构决定负重传递。** 评估框架、悬挂、腰带和躯干调节是否满足已证实需求；不能以“更大包”替代合适的腰带与躯干贴合。
4. **体型决定尺码与试背，不决定能力。** 身高仅作初筛；必须量取躯干长度、检查腰带髂嵴受力、肩带/负重调节带和满载试背。没有现场试背时，建议为条件性。
5. **路线提高保守程度。** 长距离、高海拔、连续爬升、技术地形和多变天气提高装备、补水和人工安全复核要求；不对路线安全作判断。

| 容量带 | 典型输入（须员工复核） | 建议背负要求 | 输出纪律 |
| --- | --- | --- | --- |
| `15_25l` | 短距离、单日、轻量补水/雨具/保暖层。 | 日用舒适性与稳定性；不将其用作多日重装默认选择。 | 缺少装备清单时仅为初步范围。 |
| `25_40l` | 单日长线或需更多保暖、补水、摄影/季节装备。 | 贴背稳定、肩/腰受力和通风需试背。 | 依据路线与装备体积决定上/下沿。 |
| `40_55l` | 多日徒步、住宿/补给明确且睡眠/衣物装备受控。 | 需有已证实的框架、腰带和躯干调节信息。 | 不因“7 天”自动固定到同一容量。 |
| `55_70l` | 多日自带睡眠/炊具、较大体积或较重补给。 | 重装型负重传递、可调躯干及腰带贴合必须核验。 | 新手或体型不适配时先减少装备/分担，不盲目上大包。 |
| `70l_plus` | 远征或大量共享装备等经确认的特殊需求。 | 仅在官方证据、实际负重、试背与路线计划充分时条件性推荐。 | 需专项人工复核；不是高海拔的默认答案。 |

## 5. Scenario Matching Model

场景输入先规范化为 `route_profile`，再计算容量带、背负系统条件和产品候选。天气、封路、海拔风险和路线开放状态是时变事实；本设计不接入实时权威源时，必须要求员工另行核实。

| 场景 | 最小输入 | 规则输出 | 强制补问/限制 |
| --- | --- | --- | --- |
| 深圳周末徒步 | 距离、时间、装备需求；可补充爬升、天气、补水。 | 通常从 `15_25l` 或 `25_40l` 区间开始；侧重稳定、透气/舒适、雨具与补水空间。推荐具有已证实日用背负特征的 OSPREY 候选。 | 确认单/双日、爬升、降雨、补水量和是否携带摄影装备；不以城市地点替代路线难度。 |
| 川西 7 天徒步 | 天数、住宿方式、负重需求；另需自带睡眠/炊具、补给方式和共享装备。 | 按装备体积和总重在 `40_55l` 与 `55_70l` 间选择；要求经证实的框架、腰带、可调躯干和负重范围。 | “7 天”不足以定容量；确认住宿、补给、季节、天气、总重和分担方案。 |
| 冈仁波齐转山 | 高海拔、长距离、装备需求；另需行程安排、支持方式、保暖/补水和经验。 | 以实际装备与支持条件确定容量带；优先已证实的稳定负重传递、贴合与分层收纳需求，输出高海拔人工复核提示。 | 强制确认路线、实时天气、体能、适应/医疗与当地安全要求；背包不构成高反或安全保障。 |

**匹配顺序：** 不适配/失效知识/缺关键证据/不可售 SKU → 排除；容量和负重范围 → 背负结构与试背 → 路线约束 → 客户偏好 → 库存健康与销售稳定性排序。

## 6. Customer Matching

```text
Customer（授权或显式提供）
        ↓
身高 + 躯干长度 + 体型 + 经验 + 活动等级 + 装备水平
        ↓
Backpack Recommendation（容量带、尺码试背、产品候选、限制）
```

| 信号 | 使用方式 | 不可替代的人工动作 |
| --- | --- | --- |
| 身高/体型 | 仅用于尺寸初筛和背负接触点问题提示。 | 测量躯干、检查腰带位置和肩带调节。 |
| 经验/活动等级 | 用于决定是否需要更简单的方案、更多说明和保守补问。 | 确认客户实际路线、训练和负重经验。 |
| 装备水平 | 用于估算已有睡眠/炊具/衣物的体积与兼容缺口。 | 让客户展示或逐项确认装备，不猜测拥有物。 |
| 偏好/预算 | 仅在结构、适配和安全边界满足后排序。 | 明确可接受的取舍和价格范围。 |

**示例：女性 160cm、新手徒步。** 系统不得按性别或身高直接指定型号，也不得因“新手”自动推送大容量重装包。若为深圳单日、装备轻且无露营需求，先给出较小容量带和试背要求；若多日行程确有体积/负重需求，则根据躯干、腰带贴合、实际清单和可证实负重能力评估较大容量候选，并可建议减重或分担装备。

## 7. Gear Package Integration

`GearPackage` 是审核过的角色级清单，不是 SKU 捆绑、报价、库存预留或自动加购。每个角色只在产品知识、适配和库存核验通过后显示候选。

| 装备包 | 背包角色 | 配套角色 | 输出 |
| --- | --- | --- | --- |
| 深圳周末徒步包 | 对应轻量/日用容量带的 OSPREY 背包。 | KAILAS 服装（透气/雨具等经审核角色）+ 适配地形的鞋类组合 + 补水。 | 背包 + 服装 + 鞋类候选、缺口、尺码/天气/库存核验。 |
| 川西 7 天装备包 | 按住宿与负重确定的 OSPREY 多日背包。 | KAILAS 分层保暖/防风雨角色 + 支撑型徒步鞋 + 睡眠、补水、照明角色。 | 容量/负重理由、共享装备分担、未满足角色及人工复核。 |
| 冈仁波齐转山包 | 按实际装备量和支持方式选择的 OSPREY 背包候选。 | KAILAS 保暖/防风雨层 + 合适鞋类 + 防晒、补水、应急角色。 | 高海拔装备辅助方案和安全复核清单，不作为安全许可。 |

任一必选角色无经批准候选、尺寸未确认、库存陈旧或关键限制不明时，返回 `package_incomplete_requires_human_review`，不得以无关商品填充。

## 8. Retail Brain 与 DATA-003.2 Integration

```text
Customer → Activity → Backpack Intelligence → Inventory → Recommendation → Employee AI Assistant
```

| 节点 | 只读输入 | 输出与约束 |
| --- | --- | --- |
| Customer | 同意、用途、门店范围内的最小授权画像，或匿名需求。 | 个性化或非个性化背包需求；无授权时不使用客户历史。 |
| Activity | 路线、天数、负重、住宿、环境和装备清单。 | 场景需求、补问和风险复核项。 |
| Backpack Intelligence | 已发布 OSPREY 实体、Citation、容量/结构规则。 | 合格候选、排除原因、可解释评分。 |
| Inventory | 授权门店的 `sellable` 状态、鲜度、库存健康、销售聚合。 | 可核验候选/替代项；陈旧或未知时不宣称有货。 |
| Recommendation | 上述时点化上下文和规则版本。 | 单品/装备包建议、理由、限制、置信度与人工确认。 |
| Employee AI Assistant | 员工身份、门店、用途和短时授权。 | 面向员工的问答/卡片；不执行订单、库存或外部消息。 |

### 8.1 经营建议

Retail Brain 可按授权门店、时间窗和容量带生成以下**建议**：

- 哪些容量带销售更好：显示销量、窗口、样本量、异常标记和 `sales_as_of`，不把销量视作客户适配证据；
- 哪些型号需要人工补货评估：结合可售库存、库存健康、销售速度和 OSPREY 场景覆盖，输出“请计划人员复核”，不创建补货；
- 哪些门店适合陈列：依据授权门店的场景需求、销售覆盖与可售 SKU，建议陈列/培训复核，不创建任务或调拨。

每条建议包含 `knowledge_as_of`、`inventory_as_of`、`sales_window`、`rule_version`、Citation、`confidence`、`human_action_required=true`。

## 9. 员工销售助手场景

**问题：**“客户去四姑娘山 5 天，需要多大包？”

**AI 输出模板（设计示例，非真实产品/库存）：**

1. **路线分析：** 已知为 5 天，尚不足以确定容量。请确认住宿方式、是否自带睡眠/炊具、补给和共享装备、预计总重、季节/天气、每日距离/爬升，以及客户躯干和试背条件。
2. **容量建议：** 若住宿/补给降低自带体积且装备受控，可从 `40_55l` 评估；如需自带较多睡眠、炊具或补给，则评估 `55_70l`。这是待确认区间，不是按天数的固定结论。
3. **型号推荐：** 仅返回在有效 OSPREY 知识中，容量、已证实背负结构、负重范围、躯干/腰带适配均通过硬过滤，且当前授权门店库存快照可核验的候选；以 `product_id`/`sku_id` 而非杜撰型号展示。
4. **推荐理由：** 对每个候选展示“实际容量/负重需求 → 已证实框架、腰带、调节特征 → 试背/库存事实”，并附 Citation、版本、库存时间、置信度、排除原因和人工确认清单。

固定结尾：**“请在装入接近实际负重后试背，并由员工核实路线、天气、尺码与门店实时库存；本建议不构成安全或库存承诺。”**

## 10. Citation、知识治理与权限

### 10.1 Citation 最小结构与发布门槛

每项智能结论必须可追溯到 `citation_id`、`asset_id`、`asset_version`、`content_sha256`、`locator`、`chunk_id`、`claim_path`、`captured_at`、来源等级和提取/审核血缘。响应还必须带知识/规则版本、`confidence`、输入完整度、限制和关联 ID。上游资料撤销、归档、过期或发生未解决冲突时，关联实体停止默认检索并进入复审。

```json
{
  "citation_id": "cit:osprey:manual:v3:p12",
  "asset_id": "asset:osprey:manual",
  "asset_version": "v3",
  "content_sha256": "...",
  "locator": "page 12 / carry-system section",
  "claim_path": "carry_system.adjustment",
  "captured_at": "2026-07-21T00:00:00Z",
  "confidence": {"score": 0.84, "level": "high", "basis": ["approved primary asset"], "limitations": ["customer fit requires trial"]}
}
```

### 10.2 权限与数据治理

所有请求须认证，并由服务端按组织、门店、角色、用途、客户同意、资产 ACL、字段策略和数据鲜度决定可见性；客户端不能扩大范围。审计记录决策、调用者、版本、引用、时间和结果码，不记录不必要 PII、完整自由文本或受限资产正文。AI 可起草提取和摘要，但不能自动发布知识、提升来源等级、修改 ACL 或裁决冲突。

## 11. API Contract

**共同规则：** JSON、UTC RFC 3339、`Authorization`、`X-Correlation-Id`、`X-Purpose`、`X-Organization-Id`；服务端派生 ACL 与门店范围。所有端点均为只读决策支持，不接受订单、价格、库存写入字段。

### 11.1 `GET /api/backpacks/{id}`

读取单个已发布背包智能实体。需要 `backpack:read`。

**响应关键字段：** `entity`（第 3 节 Schema）、`sku_mapping`（按调用者范围过滤）、`knowledge_as_of`、`citation`、`confidence`、`disposition: advisory_only`。失效/越权返回安全的 `404` 或 `403`，不泄露资产存在性。

### 11.2 `GET /api/backpacks/search`

按受控条件检索已发布实体。需要 `backpack:read`。

**允许查询参数：** `activity_type`、`capacity_min_l`、`capacity_max_l`、`load_kg`、`carry_system`、`torso_fit`、`store_id`、`cursor`、`limit`（1–50）。服务端忽略任何客户端伪造的 ACL/客户字段。响应包含候选、匹配/排除标签、Citation 摘要、库存鲜度状态，不以搜索结果承诺库存。

### 11.3 `POST /api/backpacks/recommend`

生成一次时点化推荐快照。需要 `recommendation:generate` 和批准用途 `customer_service|clienteling|analytics`；`customer_id` 可选且仅在授权/同意后使用。

```json
{
  "customer_id": "optional-authorized-reference",
  "store_id": "authorized-store-reference",
  "route_profile": {
    "activity_type": "trekking",
    "location": "western_sichuan",
    "days": 5,
    "distance_km": null,
    "lodging": "unknown",
    "estimated_load_kg": null,
    "gear_requirements": ["sleep_system_unknown", "rain_layer"],
    "altitude_context": "unknown"
  },
  "customer_fit": {"torso_length_cm": null, "experience": "newcomer", "gear_level": "unknown"},
  "options": {"include_gear_package": true, "max_candidates": 3}
}
```

响应包含 `recommendation_id`、`input_completeness`、`questions_to_confirm`、`capacity_recommendation`、`carry_system_requirements`、`candidates`、`excluded_candidates`、`gear_package`、`inventory_context`、`citations`、`confidence`、`limitations`、`rule_version`、`as_of` 和 `disposition: advisory_only_requires_human_approval`。库存不可用/陈旧、关键输入缺失或无有效证据时返回降级结果或 `422 INSUFFICIENT_CONTEXT`，绝不创建订单、预留或补货。

## 12. Recommendation Test 与验收标准

设计验收采用脱敏、合成产品/库存/客户夹具；不得以虚构型号、真实客户 PII 或生产写操作测试。

| 验收项 | 验收证据 |
| --- | --- |
| Backpack Intelligence Schema | Schema 含全部规定字段，字段级 Citation、版本和置信度校验通过。 |
| Capacity Model | 单日、7 天住宿差异和高海拔案例均先输出容量带/补问，非仅按天数固定。 |
| Carry System Model | 负重、框架、腰带、躯干和试背为硬过滤/限制；缺负重证据不推荐为合格重装。 |
| Scenario Matching | 深圳、川西 7 天、冈仁波齐三类输入输出容量、背负要求、候选与风险复核。 |
| Customer Matching | 160cm 女性新手案例不自动推荐过大重装包；未授权画像降级为非个性化。 |
| Gear Package | 三类装备包输出 OSPREY + KAILAS 服装 + 鞋类角色；缺必选角色显式不完整。 |
| Citation | 每个候选与理由有版本、locator、血缘和置信度；失效 Citation 被抑制。 |
| SKU Mapping | 产品到变体的只读映射可追溯；模糊 SKU 不作可成交首选。 |
| Inventory Link | 仅使用授权、鲜度合格的 `sellable` 快照；陈旧/未知不宣称有货。 |
| Recommendation Test | 返回规则分项、排除原因、人工确认和 `advisory_only`，且无写操作。 |
| Employee Assistant | “四姑娘山 5 天”回答含路线分析、容量区间、证据化候选和推荐理由。 |

## 13. 交付结论

M4.3.6 交付的是 OSPREY 背负系统专家的**设计契约**：以可追溯产品知识、保守的容量/背负推理、场景与客户适配、只读零售事实和员工人工确认构成闭环。本文件不部署服务、模型、连接器、同步或自动化业务动作。
