# VAFOX 2.0 Agent 注册标准

**文档状态：** 规范（Normative）
**适用范围：** VAFOX 2.0 内所有 Agent、其控制平面、接入网关及被授权的业务系统。
**规范性用语：** “必须”表示强制要求；“应”表示推荐要求；“可”表示可选实现。

---

## 1. Agent Registry 定位

### 1.1 定义

Agent Registry（Agent 注册中心）是 VAFOX 2.0 的**唯一受控 Agent 身份与治理目录**。所有机器人必须完成注册并处于允许状态后，才能通过 `gateway.vafox.com` 接入 VAFOX 架构、接收任务、访问数据或请求执行动作。

VAFOX 的逻辑控制链路如下：

```text
gateway.vafox.com
        |
huyan.vafox.com（CEO Control Plane）
        |
-------------------
|                 |
ai.vafox.com   core.vafox.com
```

注册中心应由 CEO Control Plane 统一治理，并为网关、AI 服务和核心业务服务提供可验证的 Agent 身份、状态、能力、权限与审计依据。

### 1.2 注册中心职责

Agent Registry 必须：

1. 为每个 Agent 分配全局唯一、不可复用的 Agent ID。
2. 保存 Agent 的元数据、能力声明、权限授权、审批状态和生命周期状态。
3. 向接入方提供可验证的注册与状态结论。
4. 拒绝未注册、已暂停、已弃用或权限不匹配的 Agent 请求。
5. 保留 Agent 版本、授权和状态变更的完整审计轨迹。
6. 支持以下规划中的 Agent 类型及后续扩展：CEO、Ops、Health、Connectivity、Report、Inventory、SAP、Finance、CRM。

### 1.3 设计边界

Agent Registry 负责“**谁可以以何种身份、在何种条件下做什么**”，不替代业务系统、模型运行时或具体工作流引擎。任何运行时实际访问与执行仍必须在请求时进行身份、状态、权限和审批校验。

## 2. Agent Metadata 标准

每个注册记录必须包含以下最小元数据；缺少任一必填字段的记录不得进入 Registered 状态。

| 字段 | 必填 | 标准与约束 |
|---|---:|---|
| Agent ID | 是 | 全局唯一、稳定、不可复用的标识符。建议格式：`agt_<domain>_<name>_<environment>_<id>`；不得承载密钥或个人敏感信息。 |
| 名称 | 是 | 面向人类可读的名称；应清晰说明业务职责，例如 `Inventory Agent`。 |
| 版本 | 是 | 使用语义化版本（`MAJOR.MINOR.PATCH`）；版本变更必须可追溯，并与能力及权限变更关联。 |
| 类型 | 是 | 受控枚举：`CEO`、`Ops`、`Health`、`Connectivity`、`Report`、`Inventory`、`SAP`、`Finance`、`CRM` 或经治理批准的扩展类型。 |
| 负责人 | 是 | 对 Agent 的业务、技术和安全治理负责的个人或团队标识；必须可联系且可追责。 |
| 创建时间 | 是 | 使用 UTC、ISO 8601 格式记录首次创建时间；创建后不得覆盖。 |
| 状态 | 是 | 仅可使用本规范定义的生命周期状态：Draft、Registered、Approved、Active、Suspended、Deprecated。 |

注册记录还应包含：描述、所属业务域、部署环境、运行端点、认证方式、依赖系统、数据分类、风险等级、最近审核时间、变更单号及弃用替代方案（如适用）。

## 3. Agent Capability 标准

### 3.1 能力分级

Agent 能力必须以最小权限方式显式声明，不得由名称、类型或提示词隐式推断。标准能力按风险由低到高分为：

| 能力 | 定义 | 示例 |
|---|---|---|
| 观察（Observe） | 读取经授权的状态、事件、指标或文档，不改变任何数据或系统状态。 | Health Agent 查询服务健康指标。 |
| 分析（Analyze） | 对已授权数据进行计算、关联、诊断、预测或风险判断，输出结构化分析结果。 | Inventory Agent 分析缺货风险。 |
| 建议（Recommend） | 基于观察或分析生成可供人或系统评估的建议，不直接触发外部变更。 | Finance Agent 提出预算调整建议。 |
| 执行（Execute） | 发起会改变数据、配置、流程或外部系统状态的受控动作。 | Ops Agent 创建已批准的运维工单。 |

### 3.2 声明要求

每项能力声明必须至少包含：能力等级、业务范围、输入来源、输出目标、允许动作、禁止动作、数据分类、所需权限、审批策略、幂等要求、超时与回滚策略（执行能力必填）。

1. Observe、Analyze 与 Recommend 不得被解释为 Execute 权限。
2. Execute 必须细化到具体动作、目标系统、资源范围与操作条件；不得以“完全访问”或“任意操作”等宽泛描述注册。
3. 能力变更必须触发重新审核；扩大数据范围、系统范围或执行范围时，必须在生效前完成审批。
4. Agent 输出的建议必须标明其依据、置信度或不确定性（如适用），并与实际执行动作保持可追溯关联。

## 4. Agent Permission 模型

### 4.1 总体原则

权限必须以 Agent ID、版本、环境、资源和动作共同绑定，并遵循默认拒绝、最小权限、职责分离、短时有效和可审计原则。权限授予不等同于执行批准。

### 4.2 数据权限

数据权限必须定义：

- 数据域与数据分类（例如公开、内部、机密、受限）；
- 允许的操作（读取、写入、导出、删除）；
- 数据对象或字段范围；
- 环境范围与保留期限；
- 脱敏、聚合、下载和跨域传输限制。

涉及 Finance、CRM、SAP、健康或其他敏感数据时，应默认使用最小字段集、脱敏结果和只读访问；访问原始受限数据必须具备明确业务理由和批准记录。

### 4.3 系统权限

系统权限必须限制到指定服务、API、租户、命名空间、队列、数据库或业务对象。每项授权必须注明目标系统、接口、认证主体、允许方法、资源范围、网络边界和有效期。

`gateway.vafox.com` 必须在转发前校验 Agent 注册状态与凭据；`huyan.vafox.com` 应作为跨域协调、治理和审批策略的控制入口；`ai.vafox.com` 与 `core.vafox.com` 必须分别独立执行其资源侧权限校验。

### 4.4 执行权限

执行权限仅可授予处于 Active 状态、已批准相应 Execute 能力且具有目标系统动作授权的 Agent。执行授权必须至少限定：

- 动作名称与允许参数范围；
- 目标资源、环境和业务域；
- 最大影响范围、频率与金额/数量阈值（如适用）；
- 幂等键、超时、重试、回滚或补偿策略；
- 审批策略和紧急停止机制。

高风险、不可逆、跨系统、涉及资金、主数据、生产配置或外部发送的动作，必须默认要求人工审批，除非经正式风险评估批准了受限的预授权策略。

### 4.5 审批要求

审批必须与具体 Agent、版本、能力、动作范围和有效期绑定。审批人不得审批其本人提交的高风险执行申请。审批结论必须为批准、拒绝或退回补充，并记录理由、时间和审批身份。

运行时必须在执行前重新校验审批是否仍有效；已过期、被撤销、范围不匹配或条件不满足的审批不得用于执行。

## 5. Agent 生命周期

Agent 状态及允许转换如下：

| 状态 | 含义 | 允许的运行行为 |
|---|---|---|
| Draft | 设计或配置中的未完成记录。 | 不得接入、访问数据或执行任务。 |
| Registered | 元数据与基础身份已登记，等待治理审核。 | 不得在生产环境运行；仅可进行受控注册验证。 |
| Approved | 能力、权限及风险已通过审核，但尚未正式启用。 | 可进行批准范围内的测试；不得自动获得生产执行权。 |
| Active | 已完成上线批准并可在授权范围内运行。 | 可执行已授权能力；每次请求仍须运行时校验。 |
| Suspended | 因风险、异常、变更或治理要求暂停。 | 必须阻断新任务、数据访问和执行；保留审计查询。 |
| Deprecated | 已停止使用并进入保留期。 | 不得接收新任务或新授权；仅保留审计与迁移所需访问。 |

标准转换为：`Draft → Registered → Approved → Active → Suspended/Deprecated`。Suspended Agent 在完成整改、复测和重新批准后可恢复为 Approved 或 Active；Deprecated 不得直接恢复，若确有需要必须以新的版本或新的注册记录重新走完整流程。

状态、版本、权限或负责人发生变更时，必须写入审计记录。任何状态为 Suspended 或 Deprecated 的 Agent，其凭据和会话必须被撤销或阻断。

## 6. Agent Audit

### 6.1 审计记录范围

所有 Agent 请求均必须产生可关联的审计记录，至少包括：

1. **调用记录：** 请求时间、调用方、Agent ID、版本、身份验证结果、关联 ID、目标服务、请求类型和授权判定。
2. **执行记录：** 被请求动作、输入摘要、审批引用、目标资源、执行时间、幂等键、执行主体、前后状态及回滚/补偿信息。
3. **结果记录：** 成功、失败、拒绝、部分完成或超时状态，以及安全脱敏后的结果摘要和输出位置。
4. **异常记录：** 权限拒绝、审批失效、策略冲突、模型或工具错误、超限、可疑行为、系统故障和人工干预。

### 6.2 审计质量要求

审计记录必须使用 UTC 时间、不可变的事件标识和端到端关联 ID。不得在审计日志中记录密钥、令牌、完整敏感个人信息或不必要的原始业务数据。日志应支持按 Agent ID、版本、负责人、时间、能力、审批单和关联 ID 检索。

高风险执行、权限变更、状态变更和异常处置必须具备可复核的证据链。审计保留期限、访问控制与导出规则应遵循 VAFOX 数据治理及适用法规中要求更严格的一方。

## 7. Agent 接入流程

### 7.1 标准流程

| 阶段 | 必需活动 | 退出条件 |
|---|---|---|
| 创建 | 定义业务目标、负责人、类型、版本、风险等级及能力草案。 | Agent 处于 Draft，且信息完整可评审。 |
| 注册 | 提交最小元数据、身份凭据方案、能力声明、权限申请及审计方案至 Registry。 | 生成唯一 Agent ID，状态变为 Registered。 |
| 审核 | 由业务、技术、安全及权限责任方核验范围、风险、数据处理与审批策略。 | 审核结论为批准，状态变为 Approved。 |
| 测试 | 在隔离或非生产环境验证认证、授权、审计、异常处理、审批及回滚策略。 | 测试证据通过，遗留风险得到接受或关闭。 |
| 上线 | 完成生产变更批准，配置最小生产权限、监控、告警、限额及紧急停止措施。 | 状态变为 Active，并可在批准范围内运行。 |
| 停用 | 停止新任务，撤销凭据与权限，迁移依赖，保留审计，并标记替代方案。 | 状态变为 Deprecated；必要时先进入 Suspended。 |

### 7.2 上线门禁

进入 Active 前必须确认：注册信息完整、负责人有效、所需审核已完成、测试证据可用、权限最小化、审批策略生效、审计链路可用、告警与停用机制已验证。任一门禁不满足时不得上线。

## 8. 安全原则

1. **禁止未注册 Agent 运行：** 未获得有效 Agent ID 和 Active 状态的 Agent，不得通过 VAFOX 网关接入、访问 VAFOX 资源或执行任何业务动作。
2. **禁止越权访问：** Agent 只能访问其已授权的数据、系统、环境、资源和动作；不得使用共享身份、借用其他 Agent 权限或绕过资源侧校验。
3. **禁止绕过审批执行：** 需要审批的动作必须在审批有效、范围匹配且条件满足时执行；不得以提示词、人工口头指令、缓存结果或批处理方式绕过审批。
4. **默认拒绝：** 注册状态、身份、版本、能力、权限、审批或审计任一项无法验证时，系统必须拒绝请求并记录异常。
5. **最小影响：** Agent 应优先观察、分析和建议；执行动作必须最小化影响范围，并具备限额、超时、停止与回滚/补偿能力。
6. **持续验证：** 每次敏感访问与执行均应进行运行时身份、状态、权限和审批校验；不得因历史成功而永久信任。
7. **可追责与可停用：** 每个 Active Agent 必须有明确负责人、可审计活动和可立即生效的停用路径。

---

## 9. 合规判定

满足本规范的最低条件为：Agent 已注册、状态有效、能力明确、权限最小化、执行审批可验证、审计完整且可停用。任何不满足上述条件的 Agent 或请求均应视为不合规，并被阻断、暂停或进入整改流程。
