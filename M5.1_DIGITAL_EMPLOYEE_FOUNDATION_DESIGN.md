# VAFOX M5.1 Digital Employee Foundation Design

**状态：** Design Phase / 仅设计，不部署。  
**前置能力：** M3 Enterprise Memory、M4 Knowledge Brain、M4.3 Product Intelligence Platform、M4.4 Sales Knowledge Engine、M4.4 Employee AI Assistant Design。  
**负责人：** Codex（设计开发）；WorkBuddy（环境）；Marvis（管理）；CTO（验收）。

## 1. 目标、原则与明确边界

M5.1 定义 VAFOX 数字员工的基础平台：一个以可信员工身份、最小权限、已治理知识和人工决策为中心的只读 AI Operating System。它将跨 Brain 的观察结果组织成可引用的任务、提醒和建议，交由人类用户确认、执行并反馈；它不是自主业务执行器。

1. **身份与权限先行。** `employee_ai_id` 由服务端映射到受治理的角色、组织、门店/品牌/部门和 ACL；客户端不能声明或扩大访问范围。
2. **证据优先。** 业务主张必须包含来源、版本、定位、数据时点、置信度和限制；未知、过期、草稿或无 Citation 内容不得被包装为事实。
3. **建议与执行分离。** 数字员工只发现、解释、排序和提出待办。任何对外沟通或业务写操作均由人类在获批系统中独立完成。
4. **最小化与隔离。** 跨组织默认拒绝；客户数据、品牌资料和门店经营数据按字段、资源和用途共同控制，只返回完成当前任务所需投影。
5. **可回放与可撤回。** 每次建议、任务、人工动作和反馈都关联策略快照、输入引用、模型/规则/知识版本和 Citation；知识撤销后不再用于新建议。
6. **学习须经审核。** 反馈形成候选记忆或知识 Draft，绝不自动改变事实、权限、规则、知识发布状态或员工绩效。

### 非目标与禁止事项

本设计只定义平台、数据契约与治理要求，**不部署**任何服务、模型或连接器。M5.1 严禁 AI：

- 自主执行交易、自动采购、自动下单、自动补货或自动调拨；
- 自动发送营销、客户消息或任何外部通信；
- 修改 SAP、Core 生产同步、同步规则或生产数据；
- 将生成文本、未审核反馈或无证据推断写成客户、商品、库存、销售或品牌事实。

## 2. Digital Employee Architecture

```text
Memory Brain（企业记忆：已治理事件、决策、反馈与历史）
                              ↓
Knowledge Brain（已审核知识、政策、流程、Citation 与版本）
                              ↓
Product Intelligence（商品事实、生命周期、适配与品牌资料）
                              ↓
Customer Brain（同意、关系、最小客户投影与客户洞察）
                              ↓
Retail Brain（门店销售、库存、运营信号的只读快照）
                              ↓
Digital Employee Layer（策略门控、运行时、任务、建议、审计）
                              ↓
Human User（确认、执行、纠正、反馈）
```

| 层 | 负责内容 | 向上提供 | 强制边界 |
| --- | --- | --- | --- |
| Memory Brain | 已治理企业记忆、决策结果、反馈模式与时间线。 | 带保留期和可见范围的记忆引用。 | 不以记忆覆盖权威事实；Draft 不能直接指导业务。 |
| Knowledge Brain | 审核通过的 SOP、政策、销售知识、规则与 Citation。 | 版本化、可检索的知识片段。 | 仅 `Published`、有效且 ACL 可见内容可用。 |
| Product Intelligence | 产品、品牌资产、适配、限制、生命周期和只读 SKU 映射。 | 产品候选、证据、限制与版本。 | 不承载实时库存/价格，不写 SAP/Core。 |
| Customer Brain | 同意、用途、客户关系及最小授权投影。 | 最小画像、购买摘要、授权客户洞察。 | 无同意、关系或用途时不泄露对象存在。 |
| Retail Brain | 库存、销售、运营和机会信号的时点化聚合。 | 门店/品类提醒和经营机会。 | 只读，库存不是可售承诺或预留。 |
| Digital Employee Layer | 运行时策略、上下文编排、任务、建议、工作台与审计。 | 人工可审阅的建议、任务、解释和 Citation。 | 不持有源系统写权限或自治执行器。 |
| Human User | 审阅、确认、在批准系统执行、提供结果与纠错。 | 明确动作、结果码、最小反馈。 | 人工动作不等于 AI 已执行；必须独立记录。 |

### 2.1 请求处理与门控

```text
Authenticated User + requested AI employee
  ↓ identity binding / organization isolation
Runtime Context（server-derived）
  ↓ RBAC + ACL + purpose + field filtering
Authorized Brain queries（parallel, read-only）
  ↓ freshness / publication / citation / policy checks
Observation + reasoning constraints
  ↓
Task or advisory response（human approval required）
  ↓
Immutable audit event + optional human feedback
```

拒绝优先：缺少身份、用途、组织匹配、角色授权、资源 ACL、有效同意、有效知识版本或必要 Citation 时，服务返回受控错误或降级为泛化建议；不得通过错误信息暴露受限资源是否存在。

## 3. Digital Employee Runtime

数字员工是一个受治理的运行时身份，不是可任意调用工具的通用 Agent。每个请求构造不可由前端覆盖的 `EmployeeAIRuntimeContext`，有效期仅覆盖当前请求/会话；持久数据仅保存引用和审计所需的最小快照。

| 字段 | 类型 | 定义与控制 |
| --- | --- | --- |
| `employee_ai_id` | string | 数字员工稳定 ID，如 `aiemp_sales_store_01`；绑定受治理模板，不由用户选择任意提示词。 |
| `role` | enum[] | `sales_ai`、`store_manager_ai`、`procurement_ai`、`product_manager_ai`、`ceo_cockpit_ai`；由角色目录签发。 |
| `organization` | object | `organization_id`、部门、门店/区域、品牌和法人范围；必须与认证主体同组织。 |
| `permissions` | string[] | 有效能力集合，例如 `retail:read`、`advice:create`；需同时满足 RBAC、ACL、用途和字段策略。 |
| `knowledge_scope` | object | 可检索知识集合、状态、语言、品牌/门店和分类范围；只含已发布可见版本。 |
| `business_scope` | object | 客户、门店、品类、区域、时间窗口和用途限制；服务端求交集。 |
| `available_tools` | ToolGrant[] | 仅列出只读检索、分析、任务/反馈记录工具及参数范围；不存在交易型工具。 |
| `memory_scope` | object | 可读记忆命名空间、保留期、脱敏级别和写入候选路径；不可跨租户。 |

```json
{
  "employee_ai_id": "aiemp_store_mgr_cn_01",
  "role": ["store_manager_ai"],
  "organization": {"organization_id": "org_vafox_cn", "store_ids": ["store_cd_01"], "brand_ids": ["brand_a"]},
  "permissions": ["retail:read", "product:read", "task:read", "advice:create", "feedback:create"],
  "knowledge_scope": {"statuses": ["Published"], "collections": ["retail_sop", "product_knowledge"], "languages": ["zh-CN"]},
  "business_scope": {"purpose": "store_operations", "customer_access": "aggregate_only", "as_of_max_age_minutes": 1440},
  "available_tools": [{"name": "retail_snapshot.read", "mode": "read", "scope": "store_cd_01"}, {"name": "task.feedback.create", "mode": "write_candidate"}],
  "memory_scope": {"namespaces": ["store_operations"], "read": "approved_or_authorized", "feedback_write": "review_queue"},
  "policy_snapshot": "pol_01J...",
  "evaluated_at": "2026-07-21T00:00:00Z"
}
```

`available_tools` 必须为 allow-list。工具调用记录 `tool_name`、参数摘要/hash、策略决定、输入版本和输出引用；运行时不得授予 `order:create`、`purchase:create`、`message:send`、`sap:write` 或 `core:write`。

## 4. 角色体系

| 数字员工 | 主要能力 | 默认可读范围 | 输出与人工边界 |
| --- | --- | --- | --- |
| Sales AI Employee | 客户分析、产品推荐、销售辅助。 | 已同意且有关联的客户最小投影、销售知识、产品知识、所属门店只读快照。 | 需求补问、方案草稿、连带机会和跟进待办；员工确认后自行沟通，绝不自动触达/成交。 |
| Store Manager AI | 经营分析、库存提醒、团队管理辅助。 | 授权门店的聚合销售/库存、运营知识、经最小化处理的团队信号。 | 经营摘要、风险、培训/核实待办；不自动排班、评价、奖惩、调拨或补货。 |
| Procurement AI | 商品分析、采购建议、品牌资料管理。 | 授权品牌/品类资料、商品生命周期、聚合销售和库存趋势。 | 采购研究、资料完整性和人工审批建议；不创建采购单、供应商订单或 SAP 变更。 |
| Product Manager AI | 商品生命周期、商品策略、品牌管理。 | 授权品牌资产、产品知识、生命周期、聚合经营指标。 | 生命周期风险、策略候选、审核待办；不发布知识、不改主数据、不改价格。 |
| CEO AI Cockpit | 经营摘要、风险提醒、机会发现。 | 经治理的组织/区域聚合指标和批准的高层知识。 | 引用化管理摘要与需核实事项；不下达自动指令，不下钻越权客户或员工个人数据。 |

角色是能力模板，不是数据豁免。人类用户必须同时拥有该数字员工的使用权和请求资源的实际访问权；例如 CEO Cockpit 只能读取被明确授予的聚合视图，不能因职称绕过客户同意或字段 ACL。

## 5. AI Employee Workspace

工作台首页统一呈现当前人类用户**有权查看**的数字员工卡片与以下五个区块。所有卡片显示 `as_of`、新鲜度、置信度、Citation 数、范围标签及“建议，需人工确认”。

| 区块 | 内容 | 交互与控制 |
| --- | --- | --- |
| Today Tasks | 今日待确认、待补信息、待复核、已逾期任务。 | 用户可查看、确认、拒绝、调整优先级或标记外部完成；不能以“完成”隐含 AI 执行。 |
| Business Opportunities | 客户/商品/门店/品牌的机会候选及证据。 | 先展示理由、限制、数据时点和下一步问题；客户项受同意与关系门控。 |
| Alerts | 库存新鲜度、知识过期、生命周期、政策或风险提醒。 | 告警可确认/静默（需理由和期限），不触发自动业务动作。 |
| Recommendations | 角色相关的可编辑建议、替代方案和人工核实清单。 | 支持采纳/编辑/拒绝与 Citation 展开；无 Citation 不显示为可采纳建议。 |
| Learning | 用户反馈状态、待审核的知识改进候选和已发布改进摘要。 | 仅展示本人/授权范围的最小信息；提交反馈进入审核队列。 |

建议卡的固定结构为：`observation`（来源事实）→ `reasoning`（规则和假设）→ `recommendation`（人工可选动作）→ `limitations`（未知/风险）→ `citations`（可定位证据）→ `human_confirmation_required=true`。

## 6. Task Management

### 6.1 AI 生成任务生命周期

```text
任务发现（已授权观察 + 规则/阈值/知识）
  ↓ 去重、范围与证据门控
任务优先级（影响 × 紧迫性 × 置信度 × 风险）
  ↓
人工确认（接受 / 编辑 / 拒绝 / 延后）
  ↓
人工完成反馈（结果码、证据引用、原因）
  ↓
审核式学习（候选记忆/知识 Draft）
```

| 阶段 | 要求 |
| --- | --- |
| 任务发现 | 来源必须是授权的观察、已发布规则或人工创建请求；保存 `source_refs`、范围、检测时间和重复键。不得以敏感推断或未批准知识生成任务。 |
| 任务优先级 | `priority = P0–P3` 由可解释的影响、截止时间、数据鲜度、置信度与风险策略计算；AI 不可用个人绩效、佣金或敏感属性排序。 |
| 人工确认 | `proposed → accepted|edited|rejected|snoozed`；接受只创建人工待办，不授权数字员工执行。拒绝/延后要求受控原因或可选说明。 |
| 完成反馈 | 用户记录 `completed_by_human`、结果码、完成时间、外部系统事实引用（如适用）和最小备注；不能将模型输出当作完成证据。 |
| 学习 | 反馈先脱敏、分流和人工审核，状态为 `submitted → triaged → draft → approved|rejected`。 |

```json
{
  "task_id": "aitask_01J...",
  "employee_ai_id": "aiemp_sales_store_01",
  "type": "customer_follow_up_candidate",
  "title": "人工确认雨天徒步需求是否仍有效",
  "status": "proposed",
  "priority": "P2",
  "scope": {"organization_id": "org_vafox_cn", "store_id": "store_cd_01", "customer_ref": "cst_01J..."},
  "source_refs": ["retail:signal:01J", "knowledge:sales:published:v4"],
  "citations": ["cit_01", "cit_02"],
  "freshness": {"as_of": "2026-07-21T00:00:00Z", "status": "fresh"},
  "human_confirmation_required": true,
  "execution": {"mode": "human_only", "prohibited_actions": ["message_send", "order_create"]}
}
```

## 7. AI Operating Loop

```text
Observation
    ↓
Reasoning
    ↓
Recommendation
    ↓
Human Action
    ↓
Feedback
    ↓
Learning
```

| 环节 | 输入/输出 | 控制 |
| --- | --- | --- |
| Observation | 从授权 Brain 读取事实、快照、信号和已发布知识。 | 校验组织、用途、ACL、同意、版本、Citation 与新鲜度；明确标记未知。 |
| Reasoning | 使用受控规则、检索证据和模型整理优先级、假设与限制。 | 规则约束优先；模型不得臆造事实、越权检索或隐去不确定性。 |
| Recommendation | 输出建议、备选、风险、待确认问题和 Citation。 | 固定 `advisory_only_requires_human_approval=true`；没有充分证据则只建议核实。 |
| Human Action | 人类采纳、编辑、拒绝，或在批准系统中进行独立动作。 | 数字员工没有执行令牌；外部动作以权威系统审计为准。 |
| Feedback | 记录采纳状态、修改、结果引用、Citation 问题和最小说明。 | 对原始 PII/自由文本做最小化与检测；不得自动影响员工评价。 |
| Learning | 形成度量、记忆候选或知识更新 Draft。 | 人工审核、版本化发布、回滚和审计后才可影响未来检索。 |

## 8. Memory & Knowledge Access

| 资源 | 可访问内容 | 额外门控 | 返回形式 |
| --- | --- | --- | --- |
| 企业记忆 | 已批准或被明确授权的决策、经验、反馈聚合。 | 命名空间、保留期、创建者可见性、脱敏级别。 | 记忆引用、摘要、状态与版本。 |
| 知识库 | 已发布 SOP、政策、销售知识、操作指引。 | `Published`、有效期、知识 ACL、语言/品牌/部门范围。 | Citation、版本、定位、限制。 |
| 产品智能 | 已发布产品、品牌资料、适配、生命周期。 | 品牌/品类 ACL、发布状态、产品有效期。 | 产品事实候选及字段级 Citation。 |
| 客户数据 | 同意且有关系的最小画像、购买/互动摘要。 | 组织、用途、同意、关系、字段 ACL、保留期。 | 最小投影；无权不暴露存在性。 |
| 销售知识 | 已审核需求发现、话术、组合和异议处理知识。 | 门店/区域/品牌 ACL、培训或角色要求。 | 可编辑建议草稿及 Citation。 |

访问决策按每个资源和字段实时计算：`allow = org_match ∧ RBAC ∧ ACL ∧ scope ∧ purpose ∧ consent_if_required ∧ publication/freshness`。只要任一条件不成立，默认拒绝或返回去标识化聚合。Digital Employee Layer 不复制全量源数据，也不将一次请求的数据自动写入长期记忆。

## 9. Permission Model（RBAC + ACL）

RBAC 定义“谁可以尝试何种能力”，ACL 定义“该能力可以访问哪一个资源及字段”；二者必须同时成功。策略还要叠加组织、用途、同意、时间、品牌、门店和数据分类约束。

| 人类角色 | 默认数字员工访问 | 典型 RBAC 能力 | ACL/范围示例 |
| --- | --- | --- | --- |
| 员工 | Sales AI Employee | `advice:create`、`task:read`、`feedback:create`、`product:read` | 本人、授权门店、已关联且同意的客户最小字段。 |
| 店长 | Store Manager AI、授权 Sales AI | `retail:read`、`team_aggregate:read`、`task:manage` | 本门店聚合数据；客户下钻仍需关系/用途/字段 ACL。 |
| 采购 | Procurement AI | `brand_asset:read/write_draft`、`product_lifecycle:read`、`procurement_advice:create` | 获授权品牌/品类资料和聚合指标；Draft 写入不等于发布或采购。 |
| 商品经理 | Product Manager AI | `product_strategy:read`、`knowledge_review:request`、`lifecycle:read` | 授权品牌、产品线及审核队列；无 SAP/Core 写权限。 |
| 管理层 | 授权 Cockpit、Store Manager AI | `business_aggregate:read`、`risk:read` | 指定区域/部门的去标识化聚合，默认无客户 PII。 |
| CEO | CEO AI Cockpit | `executive_summary:read`、`opportunity:read` | 明确授权的全局聚合；并不绕过知识、客户或字段级 ACL。 |

策略服务必须记录 allow/deny 决定、匹配的角色/ACL、脱敏规则和 `policy_snapshot`。权限变更、调店、离职、品牌撤权和同意撤回在下一请求立即生效；活动会话使用短 TTL 并在高风险读取前重新评估。

## 10. Audit & Governance

审计日志为仅追加、可检索、访问受控的事件流；日志中的客户或商业敏感内容采用引用、哈希或最小化摘要，不写入不受控自由文本。保留策略按数据分类和法务要求配置。

| 审计对象 | 最低记录字段 |
| --- | --- |
| AI 建议记录 | `advice_id`、`employee_ai_id`、用户/组织、请求目的、输入引用、观察、推理规则/模型版本、输出 hash、置信度、限制、Citation、`policy_snapshot`、时间。 |
| 用户操作记录 | 用户 ID、任务/建议 ID、`accepted|edited|rejected|snoozed|completed_by_human`、前后差异、理由、时间、关联权威结果引用。 |
| 知识来源 | `source_id`、来源类型、原件 hash、定位、所有者、ACL、来源可信级别和提取版本。 |
| 版本 | 知识/产品/策略/提示模板/规则/模型的版本、有效期、审批人、发布时间、撤销和回滚关系。 |
| Citation | `citation_id`、资源、版本、不可变定位（页/段/单元格/字段）、引用时状态与访问决定。 |
| 工具与权限 | 工具名、参数摘要/hash、允许/拒绝、脱敏、错误码、相关请求与关联 ID。 |

治理控制包括：知识发布四眼复核（适用时）、职责分离、周期性 ACL 复核、过期/撤销内容阻断、异常访问告警、审计导出审批和 Citation 可回放。任何建议均可按 `advice_id` 回放到其当时允许使用的输入与版本；回放展示仍应重新执行当前查看者的 ACL。

## 11. API Contract

**共同约束：** JSON；RFC 3339 UTC；必需请求头 `Authorization`、`X-Correlation-Id`、`X-Purpose`。服务端从认证信息计算组织、用户、角色、权限和范围，忽略客户端提交的特权字段。所有建议响应包含 `advisory_only_requires_human_approval: true`，所有写入仅创建建议/反馈/审核候选，绝不触发业务交易。

### `GET /api/ai-employees/{id}`

读取当前用户可使用的数字员工公开运行时投影。`id` 为 `employee_ai_id`；无权或不存在均返回 `404 AI_EMPLOYEE_NOT_AVAILABLE`，避免枚举。

```json
{
  "employee_ai_id": "aiemp_store_mgr_cn_01",
  "role": "store_manager_ai",
  "display_name": "门店经营数字员工",
  "organization_scope": {"store_ids": ["store_cd_01"]},
  "capabilities": ["operations_analysis", "inventory_alerts", "team_support"],
  "workspace_sections": ["today_tasks", "business_opportunities", "alerts", "recommendations", "learning"],
  "advisory_only_requires_human_approval": true,
  "as_of": "2026-07-21T00:00:00Z"
}
```

### `GET /api/ai-employees/tasks`

读取当前用户在授权数字员工与业务范围内可见的任务。查询：`employee_ai_id?`、`status?`、`priority?`、`scope_type?`、`limit`（1–50）、`cursor?`。服务端执行字段脱敏、客户关系和用途验证。

```json
{
  "items": [{
    "task_id": "aitask_01J...",
    "employee_ai_id": "aiemp_store_mgr_cn_01",
    "status": "proposed",
    "priority": "P1",
    "summary": "库存快照超过新鲜度阈值，请人工核实",
    "source_refs": ["retail:snapshot:01J..."],
    "citations": ["cit_01"],
    "human_confirmation_required": true,
    "as_of": "2026-07-21T00:00:00Z"
  }],
  "next_cursor": null
}
```

### `POST /api/ai-employees/advice`

创建一次受控建议请求。请求仅接受目标、受控资源引用和用户明确提供的非特权上下文；不能传入 `organization`、`permissions`、原始客户 PII、执行指令或写 SAP/Core 字段。

```json
{
  "employee_ai_id": "aiemp_sales_store_01",
  "intent": "product_recommendation",
  "resource_refs": ["cst_01J...", "prd_01J..."],
  "context": {"activity": "hiking", "environment": ["wet"], "unknown": ["budget"]}
}
```

```json
{
  "advice_id": "advice_01J...",
  "status": "completed",
  "observation": [{"statement": "客户已确认雨天徒步需求", "citation": "cit_01"}],
  "reasoning": {"rules_version": "rules:recommendation:v4", "assumptions": ["预算未知"]},
  "recommendations": [{"text": "先确认路线、预算与尺码，再比较已发布候选", "citations": ["cit_01", "cit_02"]}],
  "limitations": ["库存与价格需由人工实时核实"],
  "human_confirmation_required": true,
  "advisory_only_requires_human_approval": true
}
```

### `POST /api/ai-employees/feedback`

记录用户对建议或任务的反馈，作为审核候选。写入不改变知识、模型、权限、订单、采购或外部消息状态。

```json
{
  "advice_id": "advice_01J...",
  "task_id": "aitask_01J...",
  "action": "edited",
  "reason_code": "insufficient_context",
  "human_action": "completed_by_human",
  "outcome_ref": "crm_interaction_01J...",
  "summary": "已人工确认预算后调整了候选。"
}
```

响应为 `202 Accepted`，返回 `feedback_id`、`review_state: submitted` 和 `audit_event_id`。服务端先进行权限检查、关联对象检查、长度限制、敏感内容检测和去标识化；不满足时返回 `400 VALIDATION_ERROR`、`403 POLICY_DENIED` 或不泄露资源存在性的 `404 RESOURCE_NOT_AVAILABLE`。
