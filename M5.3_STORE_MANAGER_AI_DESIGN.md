# VAFOX M5.3 Store Manager AI Design

**状态：** Design Phase / 仅设计，不部署。

**前置设计：** `M5.1_DIGITAL_EMPLOYEE_FOUNDATION_DESIGN.md`、`M5.2_EMPLOYEE_AI_WORKSPACE_DESIGN.md`、Retail Brain、Product Intelligence Platform、Customer Brain。

**负责人：** Codex（设计开发）；WorkBuddy（环境）；Marvis（管理）；CTO（验收）。

## 1. 目标、原则与边界

M5.3 定义面向店长的 AI 驾驶舱。它在 M5.1 受治理 Digital Employee Runtime 和 M5.2 Workspace 的基础上，将授权门店的销售、库存、客户、员工及陈列观察组织为可解释的今日经营摘要、人工待办和建议。Store Manager AI 是**只读经营辅助**，由店长核实、决策并在已批准的权威系统中独立执行。

1. **上下文与权限先行。** 每个请求由服务端从认证身份计算门店、品牌、区域、用途、RBAC、ACL 和字段策略；客户端传入的 `store_id` 只能是授权范围的子集。
2. **证据、时点和限制可见。** 每一项洞察都携带 `as_of`、新鲜度、数据窗口、数据/知识 Citation、置信度、假设和未知项；过期或无证据内容只能提示人工核实。
3. **建议与执行分离。** AI 可以发现、排序、解释、生成可编辑的人工待办及记录反馈，不能写入业务系统或执行业务动作。
4. **最小化与公平。** 客户和员工明细按用途、关系、同意、字段 ACL 及数据最小化过滤；不得以敏感属性、佣金或不可解释推断评价、排序或奖惩员工。
5. **可回放、可撤回。** 输出保留策略快照、输入版本/引用、规则或模型版本及反馈审计；权限、同意、Citation 或知识版本失效时，卡片必须撤回、抑制或降级。

### 明确禁止

本设计不部署服务、模型、连接器或生产同步，严禁 Store Manager AI：

- 自动销售执行、成交、报价、下单、锁货、预留、营销触达或外部消息；
- 自动库存修改、补货、调拨、采购、创建采购单或供应商订单；
- 修改 SAP、Core 生产同步、同步规则或任何生产数据；
- 自动排班、员工奖惩、绩效定级或把 AI 推断写成员工/客户/商品业务事实。

## 2. Store Manager AI Architecture

```text
Retail Brain（授权门店销售、库存、运营只读快照）
Product Intelligence（商品、品牌、生命周期与陈列知识）
Customer Brain（同意、关系和最小客户机会投影）
Employee Foundation（受治理团队信号、培训与知识覆盖）
                              ↓
Store Manager AI
（Store Context + 策略门控 + 洞察编排 + Citation + 审计）
                              ↓
Sales Insight | Inventory Insight | Customer Opportunity
Employee Insight | Task Management | Visual Merchandising
                              ↓
Store Manager Action
（人工查看、核实、接受/编辑/拒绝建议；在批准系统独立执行）
                              ↓
Feedback Loop
（最小反馈 → 脱敏/聚类 → 人工审核 → 版本化知识或规则候选）
```

| 层 | 职责 | 输出 | 强制控制 |
| --- | --- | --- | --- |
| Retail Brain | 提供销售、库存、运营信号的时点化只读快照。 | 门店、品牌、品类、SKU 聚合观察。 | 不承诺可售库存，不提供写入工具。 |
| Product Intelligence | 提供产品事实、生命周期、品牌资料及已发布陈列标准。 | 商品、替代品、陈列规则及 Citation。 | 不改价格、商品主数据或发布状态。 |
| Customer Brain | 在同意与用途约束下生成最小客户机会投影。 | 授权客户画像、历史摘要、兴趣和机会理由。 | 无同意/关系/用途时不泄露客户存在性。 |
| Store Manager AI | 求交集 Store Context，执行策略、鲜度、证据和去重门控。 | 洞察、告警、建议、任务和可回放审计。 | allow-list 只读查询；所有建议需人工确认。 |
| Store Manager Action | 店长审阅并在获批系统中实际处理。 | `accepted`、`edited`、`rejected`、`completed_by_human` 等状态。 | “完成”只表示人工完成，不代表 AI 执行。 |
| Feedback Loop | 将反馈变为候选改进。 | 审核队列、质量指标、版本化 Draft。 | 未审核反馈不改变生产知识、模型、规则或业务事实。 |

### 2.1 请求处理与输出门控

```text
Authenticated user + requested store scope + X-Purpose
  ↓ server-derived Store Context / organization isolation
RBAC + ACL + consent + field filtering + read-tool allow-list
  ↓
Authorized source snapshots and published knowledge
  ↓ freshness / reconciliation / citation / deduplication checks
Insight, alert, recommendation, or human task
  ↓
Immutable audit record + optional action feedback
```

缺少身份、用途、组织匹配、资源 ACL、客户同意、有效数据时点或必要 Citation 时，服务必须拒绝、返回聚合结果，或标记为 `needs_verification`；错误信息不得暴露受限对象是否存在。所有响应固定包含 `policy_snapshot`、`as_of` 与 `advisory_only_requires_human_approval: true`。

## 3. Store Context Model

`StoreContext` 是单次请求有效的服务端派生上下文，不可由浏览器覆盖。它将人、门店、品牌、时间窗、数据可见范围及目的绑定，并取认证主体范围与请求范围的交集。

| 字段 | 类型 | 定义与控制 |
| --- | --- | --- |
| `store_id` | string | 稳定门店 ID；仅能访问主体已授权门店。 |
| `store_type` | enum | 例如 `flagship`、`mall`、`outlet`、`franchise`；用于指标基线，不作人员评价依据。 |
| `location` | object | 国家/区域/城市/商圈/时区等最小运营信息；精度受角色与数据策略约束。 |
| `staff` | object | 授权团队成员引用、角色、班次覆盖和培训覆盖摘要；默认聚合，不展示敏感人事字段。 |
| `brands` | string[] | 已授权品牌/产品线集合；品牌资料必须同时通过品牌 ACL。 |
| `sales_target` | object | 已批准目标、周期、币种和来源版本；不得将草案或 AI 预测写成目标。 |
| `inventory_scope` | object | 可读仓/门店/SKU/品类范围、可用快照时点与数据质量。 |
| `customer_scope` | object | 允许的客户聚合级别、关系/同意/用途和字段策略；默认最小投影。 |

```json
{
  "store_id": "store_cd_01",
  "store_type": "mall",
  "location": {"country": "CN", "city": "Chengdu", "timezone": "Asia/Shanghai"},
  "staff": {"team_refs": ["emp_ref_01"], "roles": ["sales_associate"], "view": "aggregate_with_authorized_detail"},
  "brands": ["brand_a"],
  "sales_target": {"period": "2026-07", "amount": 500000, "currency": "CNY", "source_ref": "target:v7"},
  "inventory_scope": {"locations": ["store_cd_01"], "categories": ["outerwear", "footwear"], "as_of_max_age_minutes": 60},
  "customer_scope": {"purpose": "store_operations", "access": "consented_relationship_filtered"},
  "policy_snapshot": "pol_01J..."
}
```

## 4. Daily Store Dashboard

驾驶舱首页将今天的经营情况浓缩为可追溯卡片，先呈现高影响、高紧迫且证据完整的事项。它不替代 POS、ERP、WMS 或人事系统；任何数字都应明确指标口径、比较窗口和刷新时间。

| 区块 | 内容 | 必填说明 | 人工操作 |
| --- | --- | --- | --- |
| `sales` 今日销售 | 销售额、目标进度、交易数、客单和同比/环比（可用时）。 | 币种、目标版本、时间窗、`as_of`、缺失数据。 | 查看明细、核实口径、创建复盘任务。 |
| `trend` 趋势 | 时段、日/周趋势与异常变化。 | 基线、比较窗口、统计显著性/置信度和限制。 | 展开原因候选，不将相关性作为因果。 |
| `opportunity` 机会 | 商品、客户、品类或服务机会候选。 | 观察事实、规则/知识、授权范围、下一步问题。 | 创建人工跟进或确认任务。 |
| `risk` 风险 | 缺货、库存异常、数据过期、任务逾期、陈列不符。 | 严重度、影响范围、数据时点、Citation。 | 确认、静默（理由+期限）、升级或创建待办。 |
| `recommendation` 建议 | 有序的店长人工行动方案和替代方案。 | 依据、假设、未知、置信度、禁止自动执行标识。 | 接受、编辑、拒绝、要求核实。 |

建议摘要固定回答：**发生了什么、证据是什么、潜在影响是什么、店长下一步应人工核实什么**。若库存或销售数据超过允许新鲜度，摘要优先显示数据质量风险，避免用旧数据生成运营建议。

## 5. Sales Insight

Sales Insight 基于 Retail Brain 的授权只读快照，服务于经营复盘和人工决策。它不承诺实时交易事实、不自动归因员工表现，也不生成强制销售指标。

| 分析 | 设计 | 输出与限制 |
| --- | --- | --- |
| 销售趋势 | 按小时/日/周、目标周期和可比门店/历史基线分析销售额、交易数与转化（数据具备时）。 | 标注比较窗口、促销/营业时长等已知混杂因素；“异常”是核实候选而非因果结论。 |
| 品牌贡献 | 聚合各授权品牌的销售额、交易数、增长及目标贡献。 | 品牌 ACL、目标版本和不可比条件必须可见。 |
| 品类贡献 | 以批准品类树汇总销售和增长，识别品类结构变化。 | 品类映射版本、未分类 SKU 和数据缺口应显示。 |
| 客单变化 | 比较平均订单金额及其分布窗口，关联交易数而非个人客户推断。 | 小样本或退款延迟时标注低置信度。 |
| TOP 商品 | 列出按明确指标排序的 SKU/款式及趋势、库存快照、生命周期限制。 | 排名口径、SKU 可见性和库存时点必须明确；不等于补货或陈列指令。 |

每张销售洞察卡至少含 `metric`、`value`、`comparison`、`time_window`、`as_of`、`data_quality`、`reasoning`、`limitations`、`citations` 和 `confidence`。当门店类型、营业日或促销规则发生变化，系统应提示基线不可直接比较。

## 6. Product & Inventory Insight

本模块连接 **Product Intelligence**（商品、生命周期、品牌规则、替代关系）和 **Inventory Data**（门店/仓位只读库存快照）。输出始终是供人工核实的建议，**不执行库存修改**。

| 输出 | 触发与内容 | 建议的人工下一步 | 硬边界 |
| --- | --- | --- | --- |
| 缺货提醒 | 可售库存低于已发布阈值、货架缺货信号或库存快照不一致。 | 核对实际库存、检查数据时点和替代商品。 | 不自动锁货、补货、调拨或修改数量。 |
| 库存风险 | 低周转/积压、异常波动、库存过期、生命周期或尺码结构风险。 | 核实盘点、商品状态、陈列与品牌策略。 | 不改库存状态、价格、主数据或处置指令。 |
| 补货建议 | 需求趋势、销售速度、阈值、交期和可用范围形成的候选。 | 复核销售/库存、审批规则与供给约束后，在权威流程人工处理。 | 不创建采购单、补货单或供应商订单。 |
| 调拨建议 | 授权地点间的供需不平衡候选、替代影响和时点。 | 核实两端库存、政策、运输和审批后人工提交。 | 不创建或执行调拨、预留或跨店写入。 |

库存建议须显示 `sku_ref`、适用门店/仓位、库存快照 `as_of`、阈值/规则版本、需求窗口、替代商品、已知供给限制、数据质量、Citation 和 `advisory_only_requires_human_approval=true`。库存快照过期、SKU 映射冲突或无法确认可售状态时，降级为“请人工核实”，不得计算确定性数量。

## 7. Customer Opportunity

Customer Opportunity 连接 **Customer Brain**，且仅在客户同意、店长/门店关系、合法用途、组织范围和字段 ACL 均满足时返回客户级条目；否则仅返回去标识的聚合机会，或不返回任何对象。

| 客户字段 | 设计与限制 |
| --- | --- |
| 画像 | 仅显示已确认、为当前用途所需的偏好、场景、经验或预算声明，按来源状态区分；未知保持 `unknown`。 |
| 历史购买 | 显示授权购买摘要、类别/商品引用和时间窗；历史购买不是当前拥有、兴趣或购买意愿的证明。 |
| 活动兴趣 | 仅展示客户明确表达或经授权记录的兴趣，含确认时间及来源。 |
| 升级机会 | 使用“观察事实 → 已发布产品/销售知识 → 待核实机会假设”的解释链；不以敏感推断或自动评分决定。 |
| 下一动作 | 只提出人工可执行的开放式确认问题、服务准备或跟进待办；不自动发送消息、创建订单或营销活动。 |

客户卡必须有 `customer_ref`（日志优先哈希/引用）、`opportunity_reason`、`next_action`、`consent_basis`、`as_of`、`citations`、`confidence` 和 `limitations`。客户撤回同意、关系终止或权限变化后，卡片及相关待办应立即从后续视图移除并停止生成新建议。

## 8. Employee Insight

Employee Insight 用于店长辅导和运营支持，而非自动化人事管理。默认显示团队聚合；员工级明细仅在角色、组织、人事 ACL、用途及当地政策允许时提供最小必要信息。不得基于 AI 输出进行自动排班、奖惩、晋升、解聘或绩效定级。

| 维度 | 可见内容 | 受控建议 |
| --- | --- | --- |
| 销售表现 | 授权周期内的团队/个人已批准指标、趋势和数据完整性。 | 安排人工复盘、核实目标口径或分享已发布实践；不自动归因。 |
| 培训需求 | 已完成培训、有效期、角色必修覆盖及任务相关的学习缺口。 | 推荐已发布微学习或人工辅导；完成学习不自动改变资格。 |
| 知识掌握 | 测验/确认学习等授权的覆盖度与知识问题趋势。 | 建议复习具体已发布知识并保留 Citation；不把推断当能力事实。 |
| 跟进情况 | 已接受任务的状态、逾期、人工完成或阻塞原因摘要。 | 建议澄清优先级、移除阻塞或升级；不自动分配、关闭或惩罚。 |

个人洞察应清楚区分源系统事实、员工自报、AI 假设和未知项，并可由授权人员提出数据更正。对样本过小或指标缺失的团队，只展示数据质量提示或聚合层级更高的趋势。

## 9. Task Center

Task Center 将跨洞察的 AI 候选组织为店长可审阅的人工任务。任务仅来自授权观察、已发布规则/知识或人类显式创建；按照 `store_scope + source + subject_ref + action_type + time_window` 去重。

| 字段 | 设计 |
| --- | --- |
| `task_source` 任务来源 | `sales_signal`、`inventory_alert`、`customer_opportunity`、`employee_insight`、`vm_check`、`human_created`；包含来源引用。 |
| `priority` 优先级 | `P0`–`P3`，由影响、紧迫性、风险、数据鲜度和置信度形成；不可用敏感属性或佣金排序。 |
| `reason` 原因 | 分列观察事实、规则/知识依据、假设和限制，附 Citation。 |
| `execution_status` 执行状态 | `proposed → accepted|edited|rejected|snoozed → completed_by_human|expired`；接受不触发业务执行。 |
| `feedback` 反馈 | 受控码 `accepted`、`edited`、`rejected`、`insufficient_context`、`citation_issue`、`outcome_recorded` 加可选最小化备注。 |

任务卡包含建议人工动作、截止/升级条件、操作者、状态时间线和 `human_confirmation_required=true`。静默或延后必须记录理由与到期时间；证据撤销、数据过期、同意撤回或权限失效时，任务被抑制、失效或移除。外部系统中的实际操作只可作为人工结果引用记录，不能由本模块发起。

## 10. Visual Merchandising Assistant

Visual Merchandising Assistant 连接已发布且 ACL 可见的**品牌陈列知识**，帮助店长对门店照片进行人工检查。照片、人员和客户可能包含个人信息，采集、保留、访问、跨境及训练使用必须遵守当地政策；不合规、无授权或包含不应处理主体的图片不得进入分析。

| 能力 | 设计 | 输出限制 |
| --- | --- | --- |
| 门店照片检查 | 店长选择授权门店、区域与照片，系统读取元数据、拍摄时间和适用标准版本。 | 不作人脸识别、身份识别、员工行为评估或客户画像。 |
| 陈列标准匹配 | 将可见货架/区域与品牌陈列规则、产品可见性及必需元素进行证据化比对。 | 图片质量不足、遮挡、标准缺失时返回待核实项，不宣称合规/不合规事实。 |
| 整改建议 | 输出按影响排序的人工检查清单，如补充价签核实、调整分区、检查主推款可见性。 | 不自动修改陈列、创建库存操作或向员工下达强制命令。 |

检查结果包含 `inspection_id`、`store_id`、`zone_ref`、`photo_ref`、`captured_at`、`standard_version`、`observations`、`confidence`、`limitations`、`recommended_human_checks`、`citations` 和保留期标签。店长可提交“已人工核实/不适用/照片问题”等反馈；反馈先进入审核队列，不自动改品牌标准。

## 11. Store AI Q&A

店长可用自然语言对授权的当前 Store Context 提问。Q&A 先解析意图和时间窗，再执行服务端范围门控、只读查询、Citation/鲜度检查，最后输出“事实、判断、限制、建议人工下一步”。无充分证据时必须明确回答“无法确认”并给出核实路径，而非编造结论。

| 示例问题 | 预期回答形态 |
| --- | --- |
| 今天最重要三件事？ | 返回最多三项按影响/紧迫性/证据完整度排序的人工任务，说明优先级理由、时点和 Citation。 |
| 哪些商品需要动作？ | 返回缺货、库存风险、补货/调拨候选及替代品，区分“需核实”与“建议”，绝不修改库存。 |
| 哪些客户值得跟进？ | 仅返回已授权客户机会或聚合机会，说明已确认事实和开放式下一动作，绝不自动触达。 |

Q&A 不得绕过 Dashboard 或 API 的数据范围；敏感、越权或执行型问题应拒绝或改写为允许的只读解释和人工流程提示。回答、引用、策略快照和用户反馈均应审计。

## 12. API Contract

**共同约束：** JSON、RFC 3339 UTC；`Authorization`、`X-Correlation-Id`、`X-Purpose` 必填。服务端派生组织、角色、门店、品牌、客户和字段范围；客户端不能提交特权角色、权限、原始 PII 或执行参数。所有读取端点只读，均支持 `store_id?`（授权范围子集）、`as_of?`、`limit`、`cursor?`，并返回 `policy_snapshot`、`as_of`、`citations` 和 `advisory_only_requires_human_approval: true`。

### `GET /api/store/dashboard`

读取单门店或授权聚合的每日经营摘要。可选参数：`store_id?`、`date?`、`include=tasks,risks,recommendations`。

```json
{
```

```json
  "advisory_only_requires_human_approval": true
}
```

### `POST /api/store/action-feedback`

记录人类对洞察、任务、告警、建议或陈列检查的最小化反馈。它只写入受审核的反馈候选，不执行任何业务动作，也不自动改变知识或规则。

```json
{
  "subject_type": "inventory_alert",
  "subject_id": "ialert_01J...",
  "action": "edited",
  "status": "completed_by_human",
  "reason_code": "inventory_verified",
  "note": "已在批准系统人工核实",
  "external_result_ref": "authority:record:opaque_ref"
}
```

响应返回 `feedback_id`、`review_status: "submitted"`、`recorded_at` 和 `policy_snapshot`。自由文本先按 PII/敏感数据、用途和保留期处理；`external_result_ref` 仅保存最小引用，不复制权威系统内容。

## 13. Permission Model: RBAC + ACL

RBAC 决定角色可使用的模块，ACL 决定其是否能读取具体门店、品牌、商品、客户、照片、员工字段及知识版本。二者必须同时允许；另外客户数据还须通过同意、关系和用途门控。管理层不因职位自动绕过字段 ACL 或客户同意。

| 角色 | 典型 RBAC 能力 | ACL 与数据边界 | 禁止 |
| --- | --- | --- | --- |
| 店员 | 查看本人任务、授权销售辅助和最小客户机会。 | 仅本人关联、已同意客户；无团队人事/跨店库存明细。 | 访问全店经营、员工洞察、客户全量或任何执行接口。 |
| 店长 | 查看所属门店 Dashboard、销售/库存、授权客户机会、团队训练/任务和陈列检查。 | 仅所属店、授权品牌和最小员工信息；客户逐项门控。 | 自动排班、评价、补货、调拨、采购或写 SAP/Core。 |
| 区域经理 | 查看被授予区域的门店汇总、趋势、风险与跨店候选。 | 默认聚合；下钻须有每店/品牌 ACL，客户与员工细节继续受控。 | 因区域角色绕过客户同意或执行库存/采购操作。 |
| 管理层 | 查看批准的组织/区域经营汇总、重大风险与治理指标。 | 默认去标识、聚合与最小字段；细节需显式 ACL。 | 越权下钻个人客户/员工、修改生产系统或自动决策。 |

服务端记录授权决定、拒绝原因类别（不泄露资源存在性）、策略快照、字段脱敏结果和访问审计。照片、客户、员工和品牌知识应分别应用资源 ACL 和保留策略。

## 14. Feedback Loop and Governance

```text
店长人工反馈（采纳/编辑/拒绝/结果/引用问题）
                    ↓
最小化、脱敏、去重与质量标注（仅生成 Draft）
                    ↓
业务负责人 + 知识/合规审核（职责分离）
                    ↓
批准/拒绝、版本化发布、Citation、回滚
                    ↓
仅未来请求使用已发布且仍有效的版本
```

反馈状态为 `submitted → triaged → draft → in_review → approved|rejected`。AI 可以总结重复问题、标注证据缺口和提出候选规则，但不得自动修改阈值、排序、品牌陈列标准、员工记录、库存事实或源系统数据。治理指标至少覆盖数据鲜度、Citation 覆盖率、建议采纳/编辑/拒绝率、过期撤回率、权限拒绝率和人工核实结果；这些指标不可直接作为员工绩效结论。
