# VAFOX M6.4 KAILAS Live Pilot Implementation Design

**状态：** Design Phase / 仅设计，不部署。  
**前置设计：** `M6.1_AI_RUNTIME_FOUNDATION_DESIGN.md`、`M6.2_EMPLOYEE_AI_WORKSPACE_MVP_IMPLEMENTATION_DESIGN.md`、`M6.3_KAILAS_PRODUCT_INTELLIGENCE_ONLINE_DESIGN.md`。  
**负责人：** Codex（设计开发）；WorkBuddy（环境）；Marvis（管理）；CTO（验收）。

## 1. 目标、原则与边界

本 Pilot 验证 KAILAS 产品智能能否在南山、航苑、振兴三家授权门店中，以可引用、可审计、人工复核的方式支持一线导购与店长经营判断。它是一个受范围约束的**只读建议 Pilot**：验证知识、上下文、建议、反馈和指标闭环，不上线交易或自动化执行能力。

1. **证据优先。** 产品事实、SKU 关系、推荐理由和经营数字均须携带可见的 Citation、版本与数据时点；证据不足时明确说明未知或要求人工核实。
2. **最小范围。** 身份、门店、品牌、客户授权、数据字段和时间窗均由服务端派生；请求参数只能收窄，不能扩大范围。
3. **建议不执行。** 所有推荐固定标记 `advisory_only_requires_human_approval=true`。员工在向客户沟通前必须复核需求、尺码、路线、天气、价格及实时可售库存。
4. **只读事实源。** 产品、销售、库存、会员及购买记录均通过已治理连接器读取快照；Pilot 不成为 SAP、Core、POS、库存或 CRM 的事实主系统。
5. **客户最小化。** Customer Brain 仅使用已同意、用途匹配且 ACL 允许的最小字段；无个性化授权时仅提供匿名场景建议。

### 明确禁止

- 自动成交、自动营销、自动采购、自动下单；
- 创建、修改、提交订单、报价、支付、库存预留、扣减、调拨、补货或价格；
- 修改 SAP，或修改 Core 生产同步、同步规则或生产数据；
- 将未经审核的员工反馈、模型文本或 Citation 不完整内容写入产品事实。

## 2. Pilot Architecture

```text
KAILAS Product Intelligence
        ↓  已发布产品事实、SKU、场景组合、Citation
AI Runtime
        ↓  身份/范围/权限、只读路由、Citation 校验、审计
Employee Workspace
        ↓  导购问答、产品卡、组合建议、反馈入口
Store Manager AI
        ↓  门店销售机会、库存提醒、商品建议
Retail Brain
        ↓  已授权销售/库存/门店快照与聚合分析
Feedback Loop
        ↓  审核后的 Sales Memory 与 Product Intelligence 优化候选
```

| 层 | Pilot 职责 | 输入 | 输出 | 强制控制 |
| --- | --- | --- | --- | --- |
| KAILAS Product Intelligence | 提供 KAILAS 与 MONT 的已发布产品智能。 | 品牌资料、产品知识、SKU 映射、场景/搭配规则。 | 产品定位、卖点、限制、装备角色、Citation。 | 仅 `published`、有效、ACL 可见资产；不得把生成内容当产品事实。 |
| AI Runtime | 为每个请求编排授权上下文与只读查询。 | 认证用户、问题、服务端身份、收窄筛选。 | 策略化回答、来源、置信度、`audit_id`。 | RBAC ∩ ACL ∩ scope ∩ purpose；无写工具；Citation 检查失败即降级。 |
| Employee Workspace | 呈现给员工的可编辑建议和核实项。 | Runtime 响应。 | 方案、销售话术、补问项、反馈。 | 不持有特权、不直连 Brain；显示数据时点与人工复核提示。 |
| Store Manager AI | 将门店聚合信号转化为经营复核建议。 | 授权门店销售、库存、商品覆盖。 | 销售机会、库存提醒、商品建议。 | 仅聚合/授权明细；不自动生成采购、调拨或任务。 |
| Retail Brain | 提供时点化的零售只读事实。 | 销售、库存、门店范围。 | 销售窗口、库存健康度、商品覆盖/缺口。 | 明示 `as_of`、质量和范围；不把历史快照表述为实时可售承诺。 |
| Feedback Loop | 收集、整理、审核并形成受控优化候选。 | 员工反馈与关联审计。 | 审核队列、Sales Memory 候选、知识改进建议。 | 反馈不自动生效、不改事实源/权限/模型/业务系统。 |

**请求路径：** Employee Workspace 将问题及可选 `store_id`、`brand_id`、时间窗提交至 Runtime；Runtime 从可信身份派生 `pilot_context`，先授权再检索 Product Intelligence、Customer Brain 和 Retail Brain；Citation Engine 对每项主张覆盖校验后才返回。Audit Engine 固化输入摘要、授权决定、来源和版本；反馈仅能关联既有 `audit_id`。

| --- | --- | --- | --- |
| 南山 | `store:nanshan` | 一线装备组合、MONT 介绍、门店经营查询验证。 | 本店 KAILAS/MONT 商品、库存及销售；已授权客户最小投影。 |

| 振兴 | `store:zhenxing` | 店长经营建议与反馈闭环验证。 | 本店 KAILAS/MONT 商品、库存及销售；已授权客户最小投影。 |

### 3.2 服务端上下文契约

| 上下文 | 定义 | 服务端派生与限制 |
| --- | --- | --- |
| `store_context` | `{store_id, store_name, channel, region, sellable_scope, inventory_as_of, sales_window}`。 | 来自用户门店任职/资源 ACL 与只读 Retail 快照。客户端 `store_id` 仅可求交集；库存状态陈旧、未知或非 `sellable` 时不得宣称有货。 |
| `employee_scope` | `{employee_id, role, assigned_store_ids, capabilities, customer_relation_scope, policy_snapshot}`。 | 来自认证、目录、RBAC、ACL 与用途。员工只能访问本人授权门店、关联且已同意的客户最小字段。 |

| `data_scope` | `{domains, field_policy, aggregation, date_range, consent_required, freshness_policy}`。 | 允许产品知识/SKU/Citation、销售/库存/门店快照、以及授权会员摘要/购买记录/活动兴趣；按角色限制字段、粒度、时间窗及保留期。 |

```json
  "employee_scope": {"role": "employee", "assigned_store_ids": ["store:nanshan"]},
  "brand_scope": {"brand_ids": ["KAILAS", "MONT"], "product_status": ["published"]},
  "data_scope": {"domains": ["product", "retail"], "aggregation": "store", "consent_required": true}
}
```

## 4. Data Integration

所有连接均是逻辑设计的**只读接口**。每次读取返回来源系统、数据版本/批次、`as_of`、质量状态、范围和 Citation/血缘引用；连接不可用、超过鲜度阈值、越权或字段不完整时，Runtime 只返回受控降级和人工核实项。

### 4.1 Product Intelligence

```text
KAILAS 品牌资料 + 产品知识 + MONT 产品智能
              ↓ approved extraction / review
产品实体与 SKU 只读映射
              ↓ field-level Citation
KAILAS Product Intelligence → AI Runtime
```

| 数据 | 最小内容 | 连接/使用方式 | Citation 要求 |
| --- | --- | --- | --- |
| 产品知识 | 品牌定位、品类、技术、场景、限制、卖点、搭配。 | 检索已发布 KAILAS/MONT 产品实体和审核过的组合模板。 | 每个产品事实与关键推荐理由关联资产、版本、hash、locator/claim path。 |
| SKU | `sku_id`、产品实体关系、门店/渠道可售状态引用。 | 只读映射候选产品到 SKU；不创建/修改商品主数据。 | 映射版本与来源批次必须返回；映射不确定不可列为可售首选。 |
| Citation | `citation_id`、source、version、locator、as_of、visibility、claim_type。 | Runtime 返回可见来源摘要，供员工展开复核。 | ACL 不足时不泄露受限正文；无 Citation 的事实不得输出为确定主张。 |

### 4.2 Retail Brain

| 数据域 | 最小字段 | 分析产出 | 控制 |
| --- | --- | --- | --- |
| 销售 | 门店、SKU/品类、数量/金额聚合、日期窗口、活动标记、数据质量。 | 场景/品类机会、组合渗透、趋势与异常待复核。 | 默认门店聚合；明确 `sales_window`，不作为个人绩效结论或需求预测。 |
| 库存 | 门店、SKU、`sellable`、可用量区间/状态、健康度、`inventory_as_of`。 | 缺口、陈旧、低覆盖和候选替代品的核实提醒。 | 仅只读；不锁货、不扣减、不调拨；未知/陈旧即降级。 |
| 门店 | 门店 ID、渠道、营业状态、授权品牌/品类、陈列/服务能力（若已批准）。 | 本店范围过滤、建议本地化和跨店聚合边界。 | 不从模型推断门店属性；门店 ACL 先于数据读取。 |

### 4.3 Customer Brain

| 数据域 | 最小字段 | 允许用途 | 控制 |
| --- | --- | --- | --- |
| 会员 | `customer_ref`、同意状态/用途、会员等级（如政策允许）、偏好摘要。 | 在客户明确授权的导购辅助中调整追问与推荐。 | 不返回姓名、电话、地址等原始 PII；无同意时走匿名模式。 |
| 购买记录 | 已购产品角色/SKU 引用、购买时间、门店（最小必要）。 | Cross Sell 识别已有装备和角色缺口。 | 需客户关系、同意、用途、字段 ACL 同时通过；不能对无关客户检索。 |
| 活动兴趣 | 明确声明的活动、目的地/行程偏好、经验等级（如已授权）。 | 场景澄清、装备组合和人工跟进建议。 | 不臆测兴趣或能力；高风险场景必须提示路线、天气、体能与专业安全人工确认。 |

## 5. Employee Pilot Scenarios

所有场景输出包含：`answer_type`（事实/建议/未知）、`citations[]`、`data_as_of`、`confidence`、`limitations[]`、`human_action_required=true`、`audit_id`。系统必须展示“库存、尺码、路线、天气及客户最终需求请由员工现场核实”。

### Scenario 1：客户第一次川西 7 天

| 项目 | 设计 |
| --- | --- |
| 输入 | `destination`（川西）、`duration_days`（7）、`budget`、`experience`；建议补问季节/天气、路线/海拔、徒步强度、尺码、已有装备和健康/安全准备。 |
| 处理 | 识别多日/温差/风雨等场景需求角色；过滤未发布、不适配、预算不符或库存不可核验候选；将授权本店 `sellable` 快照作为排序信号而非承诺。 |
| 输出：装备组合 | 按角色给出分层保暖、风雨防护、徒步鞋、背负、补水/防晒等“必选/可选/待核实”组合及替代候选；不是自动套装或订单。 |
| 输出：推荐理由 | 说明每个角色与已批准的多日徒步环境、客户经验和预算的匹配依据、限制和 Citation。 |
| 输出：销售话术 | 生成条件化、可编辑的话术，例如先确认路线/天气/尺码，再说明产品取舍；不得承诺安全、库存、价格或性能绝对效果。 |
| 通过条件 | 员工可看见每项理由的证据和人工核实项；关键输入缺失、证据不足或库存陈旧时，答案降级为补问/核实清单。 |

### Scenario 2：MONT 产品介绍

| 输出 | 设计 |
| --- | --- |
| 定位 | 基于已发布 MONT 品牌/系列资料说明目标用户、品类与定位边界，不作无证据的高低端比较。 |
| 场景 | 显示经批准的适用、条件适用和不适用场景，以及需要确认的环境条件。 |
| 卖点 | 以“技术/设计 → 用户价值 → 条件或限制”呈现；每项绑定产品 Citation。 |
| 搭配 | 返回审核过的角色级搭配、兼容/替代条件和库存核实提示；不自动加购。 |
| 通过条件 | 产品事实、话术与搭配均可追溯到 MONT 已发布资料；无证据内容明确标注未知。 |

### Scenario 3：Cross Sell

```text
已有装备（客户声明或授权购买记录）
              ↓
角色级覆盖与兼容性检查
              ↓
缺口（必选 / 可选 / 待确认）
              ↓
组合推荐（本店可核验候选 + 理由 + Citation）
```

| 项目 | 设计 |
| --- | --- |
| 输入 | 客户明确的已有装备，或经同意且权限允许的购买记录；活动/目的地、预算、尺码及偏好。 |
| 缺口判断 | 按已批准装备模板比较“已有角色—场景需要角色”，并显示兼容限制；记录缺口原因，不能仅由销量或模型猜测。 |
| 组合推荐 | 对每个缺口提供 KAILAS/MONT 候选、适配理由、替代项、实时库存核实事项和 Citation；可拒绝不必要/不适配的加购建议。 |
| 通过条件 | 无客户授权时不读取购买记录；缺口不是促销指令，员工可采纳、编辑或拒绝并提交原因。 |

### Scenario 4：店长经营查询

| 输出 | 设计 |
| --- | --- |
| 销售机会 | 基于本店/授权跨店的商品、场景和组合聚合，识别可复核的覆盖或搭配机会；附销售窗口、基线、数据质量和不确定性。 |
| 库存提醒 | 展示候选 SKU/角色的 `sellable`、健康度、库存时点和陈旧/低覆盖提醒；不视为补货、调拨或采购命令。 |
| 商品建议 | 建议人工复核陈列、培训资料、场景组合覆盖或商品经理评估，附产品知识与零售数据引用。 |
| 通过条件 | 店长仅看授权门店；结果可追溯、可解释，且不生成采购单、库存动作或人员绩效结论。 |

## 6. Pilot Feedback Loop

```text
员工反馈（采纳/编辑/拒绝、评分、原因、纠错证据）
        ↓
AI 整理（脱敏、去重、分类、关联 audit/citation/version）
        ↓
人工审核（门店负责人/商品经理按职责审批）
        ↓
Sales Memory（已审核、版本化、可撤销的经验候选）
        ↓
Product Intelligence 优化（知识缺口、规则/资料改进候选）
```

1. 员工提交 `feedback_type`（`accepted|edited|rejected|incorrect|missing_information`）、评分、可选原因和纠正依据；不得直接提交或复制不必要客户 PII。
2. AI 仅将反馈整理为候选：去重、脱敏、归类到产品/SKU/场景/门店和关联的 `audit_id`、Citation、规则/知识版本；它不判断事实真伪、更不自动发布。
3. 审核人核验来源、适用范围、时效、冲突和权限后，做出 `approved|rejected|needs_evidence` 决定并记录理由、审核人和版本。
4. 已批准的 Sales Memory 作为带来源、门店/品牌范围和有效期的经验线索；产品事实仍须由权威产品资产审核发布。撤销、过期和冲突记忆不可继续作为默认依据。
5. Product Intelligence 优化以受控变更候选进入既有知识审核流程；上线前需重新进行 Citation、ACL、回归场景和版本验收。

## 7. Pilot KPI

KPI 仅用于 Pilot 效果评估和改进，不单独作为员工奖惩、自动决策或采购依据。指标按门店、角色、场景、产品范围、时间窗与数据质量分组；样本量不足、数据延迟或缺失需同时展示。

| 类别 | KPI | 定义/口径 |
| --- | --- | --- |
| 使用 | 登录次数 | 授权用户在统计窗口内的成功 Workspace 会话数；去重规则明确。 |
| 使用 | 查询次数 | 通过 Runtime 完成授权处理的有效查询数，排除健康检查与重复失败重试。 |
| 使用 | 建议采纳率 | `accepted / (accepted + edited + rejected)`；编辑采纳另报，未知反馈不计入分母。 |
| 业务 | 客单价 | 授权门店/品牌范围内成交金额 ÷ 成交单数，与对照窗口同口径比较。 |
| 业务 | 连单率 | 同一交易中符合定义的多件/多角色购买单占比；公式及排除退款规则版本化。 |
| 业务 | 装备组合率 | 含经定义的两个或以上装备角色的交易占比，不把 AI 展示或推荐视为成交。 |
| 业务 | 客户跟进完成率 | 经授权且进入人工跟进流程的记录中，按既有业务规则标记完成的比例；AI 不自动完成。 |
| AI | Citation 完整率 | 返回的可验证事实/建议主张中，具有有效、可见、版本化 Citation 的比例。 |
| AI | 推荐准确率 | 由抽样人工审核或有定义的反馈结果确认“场景、产品、限制均正确”的推荐比例；须区分数据可用性与模型原因。 |
| AI | 人工评分 | 员工对有用性、可解释性、准确性和安全提示的 1–5 分评分，报告样本量与分布。 |

建议看板同时呈现：分母、去重方法、统计窗口、对照基线、`sales_as_of`/`inventory_as_of`、规则版本、样本量和异常注记。KPI 不满足预设阈值时，只触发人工复盘/暂停扩大范围建议，绝不触发自动业务动作。

## 8. Audit 与可回放

每次查询、指标读取、反馈与报告生成均创建不可变 `audit_id`。审计数据最小化、访问受 ACL 与保留策略约束，并以哈希/引用保存必要敏感输入，避免把客户 PII 或未授权来源原文写入通用日志。

| 记录项 | 内容 |
| --- | --- |
| 用户 | 认证主体引用、活动角色、组织/门店范围快照；不记录不必要身份明文。 |
| 问题 | 原始问题的受控存储或脱敏摘要、意图、请求用途和请求时间。 |
| 回答 | 结构化回答、限制、置信度、人工复核提示和输出策略决定。 |
| 来源 | Citation ID、来源系统、资源引用、版本、locator、数据时点、可见性决定。 |
| 建议 | 候选、排序/规则版本、排除原因、库存/销售数据质量和 `human_action_required`。 |
| 反馈 | 类型、评分、采纳/编辑/拒绝原因、脱敏内容、审核状态和关联 `audit_id`。 |
| 版本 | Runtime、提示/规则、产品知识、SKU 映射、策略、模型与指标口径版本。 |

可回放必须能回答：谁在何时、以何种授权范围、对什么问题，读取了哪一版本的哪些可见证据，产生何种建议，用户如何反馈，以及后续审核是否改变了可用记忆或知识候选。审计回放本身不授予查看原始客户记录或受限知识的权限。

## 9. API Contract

所有端点须认证，服务端忽略任何试图声明角色、扩大门店/品牌/客户范围或绕过用途的字段。响应不包含写入/执行 URL；所有建议是只读结果。错误使用统一受控代码：`UNAUTHENTICATED`、`FORBIDDEN_SCOPE`、`CONSENT_REQUIRED`、`DATA_STALE`、`EVIDENCE_INSUFFICIENT`、`VALIDATION_ERROR`。

### `POST /api/pilot/kailas/query`

**用途：** 员工/店长提交场景、产品、Cross Sell 或经营查询。

```json
{
  "question": "第一次川西 7 天，预算 5000，经验新手，请给装备组合",
  "scenario_type": "western_sichuan_7_day",
  "context": {
    "store_id": "store:nanshan",
    "brand_id": "KAILAS",
    "customer_ref": "cust_ref_optional",
    "destination": "western_sichuan",
    "duration_days": 7,
    "budget": {"amount": 5000, "currency": "CNY"},
    "experience": "newcomer"
  }
}
```

`context` 为候选收窄条件，`customer_ref` 需单独通过同意/关系检查。成功响应至少包含 `audit_id`、`scope_applied`、`answer`、`recommendations[]`、`citations[]`、`data_as_of`、`confidence`、`limitations[]`、`human_action_required:true`、`advisory_only_requires_human_approval:true`。

### `GET /api/pilot/kailas/metrics`

**用途：** 读取当前用户授权范围内的 Pilot KPI。

**查询参数：** `store_id?`、`brand_id?`、`from`、`to`、`group_by?`。服务器将参数与 `data_scope` 求交集，默认返回最小必要聚合。响应包含 `metrics[]`（名称、值、分子、分母、口径版本）、`scope_applied`、`data_as_of`、`sample_size`、`quality_notes[]`、`audit_id`。

### `POST /api/pilot/kailas/feedback`

**用途：** 提交对某次建议的反馈，进入审核队列。

```json
{
  "audit_id": "aud_01J...",
  "feedback_type": "edited",
  "rating": 4,
  "reason_code": "inventory_needs_verification",
  "comment": "员工现场复核后替换了尺码候选。"
}
```

响应包含 `feedback_id`、`status:"pending_review"`、`redaction_status`、`linked_audit_id` 与 `received_at`。接口不接受“发布知识”“修改 SKU”“执行交易”等动作字段。

### `GET /api/pilot/kailas/report`

**用途：** 生成面向店长、商品经理或管理层的只读 Pilot 报告。

**查询参数：** `from`、`to`、`store_id?`、`brand_id?`、`format?=json|markdown`。响应包括范围、KPI、场景覆盖、反馈审核状态、Citation 完整性、数据质量、风险/限制、人工行动建议与报告版本；不含客户 PII、自动采购建议或可执行业务动作。

## 10. Permission：RBAC + ACL

最终权限为：`RBAC capability ∩ resource ACL ∩ store_context ∩ brand_scope ∩ data_scope ∩ purpose ∩ field policy`。角色只是能力上限，不覆盖客户同意、门店/品牌 ACL、数据时效或字段掩码。

| 角色 | 可用范围与能力 | 明确禁止 |
| --- | --- | --- |
| 员工 | 本人授权门店的 KAILAS/MONT 产品问答、匿名场景建议；已关联且同意客户的最小画像/Cross Sell；提交反馈。 | 越店数据、未关联客户、经营跨店分析、采购/交易/营销动作。 |
| 店长 | 授权门店的员工能力，加本店销售/库存聚合、经营查询、门店报告和反馈初审。 | 未授权门店/客户明细、自动补货、调拨、采购或员工自动考核。 |
| 商品经理 | 授权品牌/品类及门店商品表现、产品知识缺口、审核后的优化候选和报告。 | 修改 SAP/Core、直接发布未审核事实、订单/价格/库存写入。 |
| 采购 | 授权品牌/品类/区域的库存与销售聚合、人工复核的供给信号。 | 创建/修改采购单、供应商下单、自动采购或越权客户读取。 |
| 管理层 | 授权组织/区域的去标识化 KPI、风险、汇总报告和 Pilot 决策材料。 | 绕过 ACL/客户同意、默认查看 PII，或用 AI 执行经营交易。 |

ACL 还需覆盖具体门店、品牌、产品资产、SKU 映射、客户关系、反馈候选、Sales Memory、审计事件和报告。拒绝时返回统一最小化错误，不泄露受限资源是否存在。

## 11. Pilot 报告与验收

### 11.1 报告结构

`GET /api/pilot/kailas/report` 的报告建议按以下顺序呈现：

1. 报告范围、门店、品牌、时间窗、数据时点和策略/规则版本；
2. 使用、业务、AI KPI 及分母、基线、样本量、质量注记；
3. 四类场景的查询量、采纳/编辑/拒绝、Citation 覆盖和人工评分；
4. 门店经营信号、库存鲜度/覆盖风险与**人工复核**建议；
5. 反馈队列、审核结果、Sales Memory/产品智能优化候选；
6. 权限拒绝、审计完整性、已知限制、边界合规与下一阶段人工决策。
