# VAFOX M5.4 Procurement AI Design

**状态：** Design Phase / 仅设计，不部署。  
**前置设计：** `M5.1_DIGITAL_EMPLOYEE_FOUNDATION_DESIGN.md`、`M5.2_EMPLOYEE_AI_WORKSPACE_DESIGN.md`、`M5.3_STORE_MANAGER_AI_DESIGN.md`、M4.3 Product Intelligence Platform、M4.3.2 Knowledge Asset Center。  
**负责人：** Codex（设计开发）；WorkBuddy（环境）；Marvis（管理）；CTO（验收）。

## 1. 目标、原则与边界

M5.4 定义 Procurement AI：在已授权的品牌、品类、供应商、门店和知识范围内，将品牌资料、产品情报、销售和库存的**只读、时点化**数据组织成新品研究、预测与库存洞察、SKU 生命周期判断及采购建议，供采购人员人工决策。它不替代采购制度、商品管理、供应商谈判、ERP/WMS/SAP 或 Core；任何订单、交易和主数据动作均由人类在获批权威系统中独立完成。

1. **上下文、权限与组织隔离先行。** 服务端从认证身份派生 Procurement Context，并对请求范围求交集；浏览器提交的品牌、品类、供应商、门店或 SKU 仅可缩小范围，不能授予权限。
2. **证据、时点与口径可见。** 每项洞察必须显示 `as_of`、数据窗口、指标口径、数据质量、来源 Citation、知识/规则版本、置信度、假设和未知项。无证据、过期或冲突数据只能输出“需人工核实”。
3. **建议与交易严格分离。** AI 仅产生可编辑的研究结论、风险和采购建议；人工确认仅记录审阅意图或结果，绝不触发订单、供应商交易或系统写入。
4. **安全最小化。** 供应商商业资料、价格、订货条件和品牌受 ACL、用途、保密级别、字段策略及保留期控制；日志优先使用引用或哈希，避免保留不必要的合同/联系人内容。
5. **可回放且反馈需审核。** 输出、输入快照、策略快照、模型/规则版本和人工反馈均需审计。反馈先脱敏、去重并进入审核，未审核内容不得改变生产知识、模型排序、采购规则或业务事实。

### 明确禁止

本设计不部署服务、模型、连接器或生产同步，且严禁 Procurement AI：

- 自动采购、自动补货、自动生成、批准或发送采购订单；
- 自动下单、自动供应商交易、议价、签约、付款、承诺交期、预留或锁定供给；
- 修改 SAP；
- 修改 Core 生产同步、同步规则或任何生产数据；
- 修改商品主数据、价格、供应商主数据、库存、生命周期状态或品牌资料发布状态。

## 2. Procurement AI Architecture

```text
Brand Asset Center（经 Knowledge Asset Center 发布、版本化、ACL 控制的品牌资产）
                                  ↓
Product Intelligence（产品主档投影、类目、属性、SKU 生命周期、竞争映射）
                                  ↓
Sales Data（授权销售快照、趋势、售罄与退货聚合）
                                  ↓
Inventory Data（授权库存、在途/覆盖投影、库存时点与质量）
                                  ↓
Procurement AI（Context + 策略门控 + 研究/预测/建议编排 + Citation + 审计）
                                  ↓
Buyer Decision（人工查看、核实、接受/编辑/拒绝；在权威系统独立执行）
                                  ↓
Feedback Loop（最小反馈 → 脱敏/聚类 → 人工审核 → 版本化知识/规则候选）
```

| 层 | 职责 | 输出 | 强制控制 |
| --- | --- | --- | --- |
| Brand Asset Center | 从 Knowledge Asset Center 读取已发布、有效的品牌、订货、培训和新品资料。 | 带版本、有效期、ACL 与 Citation 的资产投影。 | 不改变原资产、发布状态或保密级别。 |
| Product Intelligence | 提供商品、SKU、类目、属性、生命周期、替代与竞争映射。 | 已授权商品事实和规则。 | 不改主数据、价格、生命周期事实。 |
| Sales Data | 提供销售、销量、折扣、退货、售罄等聚合快照。 | 趋势、需求候选及数据质量。 | 只读；不把异常直接归因于品牌、门店或人员。 |
| Inventory Data | 提供库存、可售状态、在途（如授权）和覆盖天数的时点投影。 | 库存覆盖、积压/缺货风险。 | 不承诺实时可售量，不写库存。 |
| Procurement AI | 在范围内融合信号，执行鲜度、冲突、Citation 和风险门控。 | 产品洞察、订单建议、告警、问答和审计。 | 只读工具 allow-list；输出一律咨询性质。 |
| Buyer Decision | 采购员或有权人员审阅与核实建议。 | 人工决定、理由、外部结果引用。 | 决定不执行交易；实际操作在权威流程中独立完成。 |
| Feedback Loop | 将显式反馈形成可审核改进候选。 | 审核队列、质量指标、版本化 Draft。 | 未审核反馈不得影响生产结论或自动化。 |

### 2.1 请求与输出门控

```text
Authenticated user + X-Purpose + requested scope
  ↓ server-derived Procurement Context / organization isolation
RBAC + resource ACL + supplier confidentiality + field filtering
  ↓
Authorized source snapshots + Published/effective brand assets
  ↓ freshness / reconciliation / lifecycle / citation / conflict checks
Procurement insight, new-product assessment, or advisory recommendation
  ↓
Immutable audit record + optional human decision feedback
```

缺少身份、组织匹配、用途、资源 ACL、有效资产版本、必要数据时点或 Citation 时，服务必须拒绝、返回允许的聚合结果，或标为 `needs_verification`；错误不得泄露受限品牌、供应商或 SKU 是否存在。每个响应固定包含 `policy_snapshot`、`as_of` 和 `advisory_only_requires_human_approval: true`。

## 3. Procurement Context Model

`ProcurementContext` 是一次请求有效的服务端派生上下文。它绑定采购主体、角色、范围、用途及适用策略，且不能由客户端覆盖。

| 字段 | 类型 | 定义与控制 |
| --- | --- | --- |
| `procurement_ai_id` | string | 本次 AI 会话/编排实例稳定 ID；关联模型、规则、策略和审计版本，不作为人员身份。 |
| `buyer_id` | string | 经认证的采购主体 ID；日志可使用受控引用/哈希。 |
| `role` | enum[] | `buyer`、`merchandise_manager`、`brand_owner`、`executive` 等有效角色；由身份系统派生。 |
| `brand_scope` | object | 可见品牌/产品线及地域、时间、字段限制；请求范围必须是其子集。 |
| `category_scope` | object | 已批准品类树版本、品类/SKU 范围和聚合级别。 |
| `supplier_scope` | object | 允许读取的供应商引用、资料类型、保密等级和字段 ACL；无权时不暴露对象存在。 |
| `permission` | object | RBAC 权限、资源 ACL、用途、导出限制、字段脱敏及策略快照。 |
| `knowledge_scope` | object | 可检索的已发布、有效、未撤销知识版本、来源及 Citation 范围。 |

```json
{
  "procurement_ai_id": "procai_01J...",
  "buyer_id": "buyer_ref_01",
  "role": ["buyer"],
  "brand_scope": {"brands": ["brand_a"], "regions": ["CN"], "field_policy": "commercial_restricted"},
  "category_scope": {"taxonomy_version": "cat_v12", "categories": ["footwear", "outerwear"]},
  "supplier_scope": {"supplier_refs": ["supplier_ref_01"], "asset_types": ["order", "new_product"], "confidentiality_max": "internal"},
  "permission": {"purpose": "procurement_planning", "export": false, "acl_snapshot": "acl_01J..."},
  "knowledge_scope": {"states": ["published"], "effective_at": "2026-07-21T00:00:00Z", "citation_required": true},
  "policy_snapshot": "pol_01J..."
}
```

## 4. 新品分析助手（New Product Analysis Assistant）

新品分析助手用于采购研究和立项前评估，不构成引入、定价、配额或订单决定。它只使用授权范围内、已发布且有效的资料；市场信息必须有来源、地域、观察日期和许可状态。

| 输入 | 设计与门控 |
| --- | --- |
| 品牌资料 | 品牌定位、产品线、订货资料、上市节奏、培训资料和新品资料；必须来自 Brand Asset Center 的已发布版本。 |
| 产品资料 | Product Intelligence 中的 SKU/款式属性、功能、尺寸/颜色、建议价格、类目、替代关系及版本。 |
| 市场信息 | 授权市场研究、竞品资料、地区/渠道趋势和公开资料；记录来源、采集日期、适用市场及不确定性。 |

| 输出 | 内容 | 人工核实重点 |
| --- | --- | --- |
| 产品定位 | 目标场景、功能价值、品牌/品类位置和证据链。 | 品牌最新策略、实际产品状态及本地合规。 |
| 客户匹配 | 与已确认客群需求、价格敏感度和场景趋势的聚合匹配假设。 | 不使用敏感推断；样本、渠道和时效是否适用。 |
| 门店适配 | 按门店类型、区域、现有品类结构和历史需求的适配候选。 | 门店计划、空间、陈列和实际需求。 |
| 价格带 | 建议零售价区间与相邻产品/竞品区间的比较。 | 税费、汇率、促销、品牌定价政策和最终价格审批。 |
| 竞争分析 | 可比竞品、差异点、空白点与证据缺口。 | 比较维度和竞品资料时点；相关性不是胜率保证。 |
| 上市建议 | 分阶段测试、首批规模区间、门店/渠道候选、观察指标和停止条件。 | 人工审批预算、供给、合规和实际下单；不生成订单。 |

新品结论固定分列 `facts`、`analysis`、`assumptions`、`unknowns`、`risks`、`citations` 与 `confidence`。低置信度、市场资料过期或品牌资料冲突时，只提出补充研究问题并标记 `needs_verification`。

## 5. 采购辅助模型（Order Advice）

```text
Sales（销量、销售额、售罄、退货与数据质量）
  ↓
Season（已发布季节日历、节假日、上市窗口与可解释季节因子）
  ↓
Inventory（库存快照、覆盖、在途投影、状态与时点）
  ↓
Store Demand（授权门店/区域需求、门店类型和需求信号）
  ↓
Procurement Recommendation（候选数量区间、优先级、风险和核实清单）
```

模型先执行 SKU/品类映射、数据完整性、库存时点、季节适用性和供应约束检查，再计算候选。它不把预测伪装成事实，也不得以单一销售峰值、过期库存或未核实在途数据生成确定性数量。

| 输出 | 必填内容 |
| --- | --- |
| 采购建议 | `recommendation_id`、品牌/品类/SKU 或聚合范围、建议数量**区间**、目标覆盖窗口、优先级、销售/季节/库存/门店需求依据、替代方案、Citation 与置信度。 |
| 风险提示 | 缺货、积压、季节错配、退货异常、资料过期、供应约束未知、数据冲突、预测不稳定和品牌策略变化；说明严重度、影响与人工核实步骤。 |
| 人工确认 | `pending|accepted|edited|rejected|needs_verification`、操作者、时间、理由和可选外部流程引用；`accepted` 不创建采购单或供应商订单。 |

建议的可解释计算结构为：`需求候选 = 已授权销售趋势 × 季节调整 × 门店需求权重 − 可用库存/已确认在途投影`，再应用品牌策略、最小订货量等**已发布且授权可见**的约束。所有输入须显示时间窗和版本；未确认在途、最小订货量或交期只能作为显式假设。

## 6. SKU Lifecycle Management

SKU 生命周期是 Product Intelligence 的受治理分析视图，采购 AI 可以建议人工复核阶段，但**不能直接修改 SKU 生命周期状态**。

```text
New → Growth → Hero → Stable → Risk → Retire
```

| 阶段 | 观察信号 | 采购 AI 的咨询输出 | 人工控制 |
| --- | --- | --- | --- |
| `New` | 上市期、资料完整度、首批销售、早期退货与门店反馈。 | 建议观察窗口、数据缺口和小范围验证问题。 | 不自动扩大铺货或补单。 |
| `Growth` | 持续增长、扩展门店需求、库存覆盖和复购/退货趋势（可用时）。 | 识别补货候选和能力/供给核实项。 | 人工确认需求是否可持续。 |
| `Hero` | 多窗口稳定高贡献、售罄与门店需求证据充分。 | 提醒高影响缺货风险、替代与保护性研究。 | 不自动预留、锁货或下单。 |
| `Stable` | 销售和库存处于已批准基线附近。 | 提供维持覆盖的候选区间及偏离提醒。 | 人工复核季节和品牌策略。 |
| `Risk` | 低周转、积压、退货异常、资料/供应不确定或趋势反转。 | 说明风险证据、替代方案和待核实原因。 | 不自动降价、清货或停采。 |
| `Retire` | 已发布退市计划或经审批的持续退出信号。 | 提醒剩余库存、替代覆盖和资料失效日期。 | 生命周期变更、清理和采购停止必须走权威审批。 |

阶段判断必须记录 `lifecycle_source`（权威事实或 AI 复核候选）、观察窗口、阈值/规则版本、异常因素和 Citation。若权威状态与分析候选冲突，权威状态优先，AI 只显示差异待办。

## 7. Brand Asset Integration

Brand Asset Center 通过 **Knowledge Asset Center** 接入资产，采用引用式读取，保留发布、版本、有效期、撤销、密级和 ACL；不会复制为无治理的提示词附件。

| 支持资产 | 用途 | 版本与访问控制 |
| --- | --- | --- |
| 品牌资料 | 品牌定位、产品线、渠道/区域策略和品牌规范。 | 仅 `Published` + 有效 + ACL 允许版本可用于结论。 |
| 订货资料 | 订货季、产品册、MOQ、交期、订货窗口和限制（如已授权）。 | 商业保密字段独立 ACL；过期版本不得作当前事实。 |
| 培训资料 | 产品卖点、适配边界、陈列/上市教育与常见问题。 | 用作解释辅助，不能替代商品事实或合规审批。 |
| 新品资料 | 新品定位、上市节奏、图文、推荐场景和对比资料。 | 标记草案/已发布/撤销及适用地区，草案不自动进入建议。 |
| 版本管理 | 资产 ID、版本、语言、有效期、审批人、变更说明、Citation 和回滚指针。 | 输出携带精确版本；撤销后未来检索立即停止并使相关卡片失效。 |

资产发生发布、撤销、到期或 ACL 变化时，系统应重新评估依赖它的洞察；无法确认有效版本时，隐藏结论或降级为“请人工核实”，而非沿用缓存内容。

## 8. Forecast & Inventory Insight

Forecast & Inventory Insight 仅提供可解释的采购洞察和风险观察，不承诺实际供给或销量。

| 输入维度 | 设计 |
| --- | --- |
| 销售趋势 | 按 SKU/款式/品类/品牌及门店聚合，展示销量、销售额、售罄、退货与比较窗口；标识促销、缺货、门店开闭等可用混杂因素。 |
| 库存覆盖 | 基于库存时点和明确需求窗口估算覆盖天/周数，分开显示可售、不可售、在途和未知状态。 |
| 季节因素 | 使用已发布季节日历、节假日和上市窗口；因子来源、适用区域和置信度必须可见。 |
| 门店需求 | 以授权门店类型、区域、历史需求和已批准计划形成聚合候选，不以未经授权的员工/客户明细推断。 |

**采购洞察输出**至少包含：趋势方向及比较基线、库存覆盖及 `as_of`、季节/门店需求影响、潜在缺货或积压范围、候选采购区间、已知供应限制、数据质量、Citation、置信度、假设、未知项和建议人工核实步骤。数据不完整时，优先输出数据质量风险而非数量建议。

## 9. Procurement AI Workspace

采购工作台以“今日最重要、证据完整、需人工处理”的顺序显示授权范围内内容；它不替代商品企划、采购审批、订单管理或供应商协作系统。

| 区块 | 内容 | 卡片必填字段 | 允许交互 |
| --- | --- | --- | --- |
| 今日采购洞察 | 高影响趋势、覆盖变化、生命周期和待核实建议。 | 范围、原因、影响、`as_of`、Citation、置信度、限制。 | 查看、筛选、创建人工复核、反馈。 |
| 新品提醒 | 新品资料发布、上市窗口、资料缺口和评估候选。 | 资产版本、适用范围、截止/有效期、Citation。 | 打开新品分析、标记需核实、反馈。 |
| 库存风险 | 缺货、积压、覆盖异常、快照过期和 SKU 映射冲突。 | 严重度、影响范围、库存时点、规则/阈值版本。 | 确认、静默（理由+到期）、升级人工复核。 |
| 品牌更新 | 品牌资产新版本、撤销、策略变化、订货窗口和培训更新。 | 品牌、资产类型、版本、变更摘要、有效期、ACL。 | 查看受权资料、确认阅读、创建人工待办。 |
| 待审核资料 | 资料缺口、反馈形成的 Draft、版本冲突和审批待办。 | 状态、提交者引用、证据、影响范围、审批路径。 | 提交审核、批准/拒绝（具备独立权限时）；不由 AI 发布。 |

卡片按影响、紧迫性、数据鲜度、证据完整度及风险策略排序；不得使用供应商返利、个人绩效或不可解释偏好作为隐藏排序因子。所有卡片都有 `advisory_only_requires_human_approval=true`。

## 10. 采购 AI 问答

采购人员可以在当前 Procurement Context 内提问。问答先解析意图、时间窗和范围，再运行只读检索、ACL/字段过滤、数据鲜度和 Citation 检查，最后以“事实、分析、限制、建议人工下一步”回答。无充分证据时必须回答“无法确认”，并指出需补充的数据。

| 示例问题 | 预期回答形态 |
| --- | --- |
| 新品是否值得引入？ | 返回新品定位、市场/客户/门店适配假设、价格带、竞品比较、风险、Citation 和分阶段人工评估建议；不输出自动引入或订单。 |
| 某品牌是否适合深圳？ | 在授权品牌和深圳范围内比较区域市场资料、门店需求、现有品类与品牌资产，说明资料时点、样本局限和人工验证计划。 |
| 某品类是否需要增加？ | 显示销售趋势、库存覆盖、季节和门店需求、替代/生命周期影响及建议数量区间；不自动补货或创建采购单。 |

执行型、越权、保密级别不符或试图绕过审批的问题必须拒绝，或改写为允许的只读解释与人工流程提示。Q&A 回答、检索 Citation、策略快照和反馈均应审计。

## 11. API Contract

**共同约束：** JSON、RFC 3339 UTC；`Authorization`、`X-Correlation-Id`、`X-Purpose` 必填。服务端派生 `ProcurementContext`；客户端不得提交特权角色/权限、原始供应商联系人信息或交易执行参数。所有 GET 端点均只读，`brand_id`、`category_id`、`supplier_id`、`store_id`、`sku_id` 仅可作为已授权范围子集；所有响应均返回 `policy_snapshot`、`as_of`、`citations` 与 `advisory_only_requires_human_approval: true`。

## 12. 权限：RBAC + ACL

RBAC 决定角色的能力上限，ACL 决定具体品牌、品类、供应商、资产、门店和字段能否访问；两者必须同时允许。管理层角色不天然绕过供应商保密、品牌 ACL、数据最小化或审批职责分离。

| 角色 | 允许的默认能力 | ACL / 明确限制 |
| --- | --- | --- |
| 采购员 | 查看授权范围 Dashboard、产品洞察、订单建议、品牌资产；提交反馈和人工复核。 | 仅其品牌/品类/供应商/门店范围；不能发布资料、改变策略、下单或修改 SAP/Core。 |
| 商品经理 | 查看跨授权品类洞察、生命周期候选、资料质量和审核队列；按授权参与资料审核。 | 商品/品牌 ACL 与职责分离仍适用；不能自动改主数据、价格或采购订单。 |
| 品牌负责人 | 查看本品牌资产、品牌更新、新品研究与授权经营聚合；审核本品牌资料候选。 | 不能读取未授权供应商商业资料或跨品牌数据；不能自动发布或交易。 |
| 管理层 | 查看授权组织/区域的聚合趋势、风险、资料治理与待人工决策事项。 | 默认聚合和去标识化；不能因职位绕过资源 ACL、保密字段或执行边界。 |

字段 ACL 至少区分：公开品牌资料、内部经营聚合、受限订货条件、保密供应商商业字段和禁止导出字段。导出、分享、跨组织检索和自由文本附件需要独立策略判定并写入审计。

## 13. Feedback Loop、审计与验收边界

```text
人工反馈（接受/编辑/拒绝/结果/引用问题）
  ↓ 最小化、脱敏、去重和证据缺口标注
审核队列（采购/商品/品牌负责人 + 知识/合规职责分离）
  ↓ approved / rejected
版本化资产、规则或模型改进候选（可回滚）
  ↓ 仅后续请求使用已发布有效版本
```

审计记录包含请求主体引用、范围、用途、策略快照、输入数据/资产版本、Citation、生成版本、输出摘要、人工确认与反馈状态。审计用于安全、质量和回放，遵守保留期、访问控制和删除/撤销策略。
