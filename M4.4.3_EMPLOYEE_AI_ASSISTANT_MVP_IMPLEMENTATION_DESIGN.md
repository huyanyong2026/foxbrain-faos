# VAFOX M4.4.3 Employee AI Assistant MVP Implementation Design

**状态：** Implementation Design / 仅设计，不部署。
**前置设计：** `M4.3.9_PRODUCT_INTELLIGENCE_PLATFORM_MVP_IMPLEMENTATION_DESIGN.md`、`M4.4.1_SALES_KNOWLEDGE_ENGINE_DESIGN.md`、`M4.4.2_EMPLOYEE_AI_ASSISTANT_V2_DESIGN.md`。
**目标：** 将 Employee AI Assistant V2 收敛为可分阶段实施的员工服务 MVP：在服务端权限约束下，为员工提供客户机会、客户查询、产品/场景/连带销售建议、人工跟进、门店聚合看板和受审核反馈闭环。
**负责人：** Codex（设计开发）；WorkBuddy（环境）；Marvis（管理）；CTO（验收）。

> 本文只定义实现边界、数据契约、流程和验收方案；不部署服务、不迁移生产数据、不接入写操作。所有建议均为 `advisory_only_requires_human_approval: true`，须由员工核实并自行决定后续沟通或在获批系统中完成业务动作。

## 1. MVP Architecture

```text
Employee Portal
      ↓
Employee Context
      ↓
Customer Assistant
      ↓
Product Assistant
      ↓
Scenario Assistant
      ↓
Sales Advice
      ↓
Feedback Loop
```

| 组件 | MVP 实现职责 | 输入 | 输出 | 强制边界 |
| --- | --- | --- | --- | --- |
| Employee Portal | 已认证员工的机会列表、客户页、建议工作区、跟进表单、反馈入口和店长看板。 | 会话令牌、员工显式选择的客户/场景/产品、受控表单字段。 | 仅显示服务端已过滤的数据和可编辑建议。 | 前端不持有授权真相；不得提交角色、权限、跨店范围或原始 PII。 |
| Employee Context | 每个请求从可信身份、员工目录、门店授权和政策服务构建短期策略快照。 | `Authorization`、可信会话、用途。 | `AuthContext`、最小数据访问计划、审计关联 ID。 | 失效、调店或撤权后下一请求重新计算；客户端字段不能覆盖。 |
| Customer Assistant | 调用 Customer Brain 的授权最小投影与只读互动/购买摘要。 | 已通过关系、同意、用途校验的 `customer_id`。 | 画像、购买记录、装备档案、活动记录及机会。 | 无权或不存在不泄露对象存在性；无授权时降级为匿名通用建议。 |
| Product Assistant | 编排 Product Intelligence 与 Sales Knowledge 的已发布内容。 | 产品 ID/类别、客户明确需求、场景约束。 | 产品定位、场景价值、销售话术、异议处理、Citation。 | 不把知识当成价格、库存、SKU 或订单事实。 |
| Scenario Assistant | 用目的地、活动、环境和装备缺口执行规则优先的场景匹配。 | 受控场景上下文、已确认装备。 | 补问、角色级装备要求、候选/替代、限制。 | 不提供医疗、领队、路线、许可或天气安全结论。 |
| Sales Advice | 合并授权上下文、知识、产品事实和只读零售快照。 | 各助手已过滤的候选与证据。 | 可编辑建议、理由、引用、鲜度、置信度、核实项。 | 不发送消息、不创建购物车、不下单、不预留库存。 |
| Feedback Loop | 接收显式反馈，脱敏整理为待审 Sales Memory 草稿。 | Advice 快照、受控结果码、最小化摘要。 | 审核队列、质量指标、可追溯知识改进候选。 | AI 不审批；未审核反馈不改变知识、排序或员工评价。 |

### 1.1 运行时门控顺序

1. 网关验证令牌、`X-Correlation-Id`、`X-Purpose`、schema 和请求大小；服务端绑定 `employee_id`。
2. Context Service 计算组织隔离、门店范围、RBAC 权限、ACL、客户关系、同意和用途，生成不可由客户端修改的策略快照。
3. 仅向 Customer Brain、Product Intelligence、Sales Knowledge 和 Retail Brain 发出最小化、只读且带范围的查询。
4. 排除越权、草稿/未发布/过期知识、`not_suitable` 产品、冲突的安全/兼容约束、无效客户关系和陈旧零售事实。
5. 输入不足时优先返回补问或“请人工核实”；证据充分时才编排 `SalesAdvice`。客户明确陈述、权威事实、知识事实和 AI 推理建议必须分区展示。
6. 记录脱敏审计事件；员工可采纳、编辑、拒绝、记录跟进或提交反馈，但 Portal 无外发、交易或主数据写执行器。

## 2. MVP Scope and Dependencies

### 2.1 必须实现

| 能力 | 最小实现 | 完成定义 |
| --- | --- | --- |
| Employee Context | 服务端上下文解析、RBAC + ACL 策略门、门店/客户/用途校验。 | 客户端伪造 `employee_id`、`role`、`permissions` 或越权 `store_id` 均不能扩大读取范围。 |
| Opportunity Module | 当前员工或授权店长的分页机会队列与优先级排序理由。 | 每项包含 customer、history、need、recommendation、priority、next_action，且不包含不必要 PII。 |
| Customer Query Assistant | 客户最小投影页面/API。 | 可查询授权客户画像、购买记录、装备档案、活动记录和推荐机会；无权与不存在均不泄露。 |
| Product Sales Assistant | 已发布 Product Intelligence + Sales Knowledge 编排。 | 每项产品定位、场景价值、话术和异议处理都有 Citation、版本和限制。 |
| Scenario & Cross Sell | 五类场景流程、已装备覆盖/缺口/组合输出。 | 输出先有补问与硬约束过滤；不自动加购或推送。 |
| Follow-up | 人工创建互动记录、待办和客户提醒。 | 仅存储最小化结构化记录；不发送提醒或营销消息。 |
| Manager Dashboard | 门店范围的只读、聚合机会/商品/库存/跟进视图。 | 显示统计窗口、`as_of`、鲜度；下钻仍经客户 ACL。 |
| Feedback Loop | 反馈接收、敏感数据筛查、AI 草稿、审核状态。 | 未审核内容不能进入 Sales Memory 的可检索集合。 |

### 2.2 明确不在 MVP

- 自动营销、自动发送活动邀请、自动外呼或任何客户触达；
- 自动成交、自动下单、自动创建购物车、自动报价/改价；
- 库存预留、调拨、补货或把库存快照承诺为可售；
- 修改 SAP、修改 Core 生产同步、同步规则或生产数据；
- 自动发布知识、自动提升来源可信度、自动扩张 ACL 或把反馈自动用于绩效考核。

## 3. Employee Context Implementation

Employee Context 是请求范围的服务端策略快照，不是由 Portal 保存或信任的员工档案副本。所有字段取自可信身份、员工目录、授权策略及授权关系服务；下游仅收到完成最小化后的字段。

| 字段 | 类型 | 实现来源与含义 | 服务端控制 |
| --- | --- | --- | --- |
| `employee_id` | string | 认证主体绑定的稳定员工 ID。 | 忽略请求体同名字段；令牌主体与目录映射失败即拒绝。 |
| `store_id` | string[] | 当前工作门店和已授权门店集合。 | 请求目标门店必须属于集合且同组织；只可收窄不可扩张。 |
| `role` | controlled string[] | `employee`、`store_manager`、`regional_manager`、`executive` 等治理角色。 | 仅授权目录/策略服务签发；角色变更下一请求生效。 |
| `permissions` | controlled string[] | 例如 `opportunities:read`、`customer:read`、`advice:generate`、`followup:write`、`feedback:write`、`dashboard:read`。 | RBAC 是功能门槛，不能单独授予资源读取权限。 |
| `assigned_customers` | reference[] | 有效顾问、服务、转交关系引用。 | 不允许枚举；读取客户时还必须满足 ACL、用途和同意。 |
| `training_level` | controlled object | 培训等级、有效期、适用场景。 | 只用于要求复核、降级或升级人工处理；不得放宽权限。 |
| `sales_history` | aggregate/reference[] | 范围内聚合指标或权威销售事实引用。 | 不复制客户交易明细；不得自动归因、佣金计算或员工考核。 |

```json
{
  "employee_id": "emp_01J...",
  "organization_id": "org_vafox_cn",
  "store_id": ["store_chengdu_01"],
  "role": ["employee"],
  "permissions": ["opportunities:read", "customer:read", "advice:generate", "followup:write", "feedback:write"],
  "assigned_customers": [{"customer_id": "cst_01J...", "relationship_role": "advisor", "valid_to": null}],
  "training_level": {"level": "outdoor_sales_l2", "valid_to": "2027-01-31"},
  "sales_history": [{"period": "2026-07", "metric_ref": "salesagg_01J..."}],
  "policy_snapshot": "pol_01J...",
  "evaluated_at": "2026-07-21T00:00:00Z"
}
```

### 3.1 RBAC + ACL enforcement

RBAC 判定“能否调用能力”；ACL 判定“能否看见该客户、字段、知识、产品、门店或来源”；两者以及组织、门店、关系、同意、用途、生命周期和有效期必须同时允许。服务端策略决策遵循：

```text
trusted identity → Employee Context → RBAC capability check
                                      ↓
                      resource ACL + store + relationship + consent + purpose
                                      ↓
                              field minimization / allow or deny
```

- 资源筛选、`store_id`、客户 ID 和任何 UI 筛选器只能缩小集合，不得授予访问。
- `GET /api/employee/customer/{id}` 对不存在和无权均返回 `404 RESOURCE_NOT_AVAILABLE`，避免客户枚举；其他策略拒绝返回 `403 POLICY_DENIED` 而不返回受限内容。
- 审计记录主体/委托主体、策略版本/决定、用途、脱敏资源引用、结果/引用版本、鲜度和关联 ID；不记录完整客户画像、原始 PII、受限原文或非必要自由文本。

## 4. Customer Opportunity Module

机会模块用于安排**人工**服务优先级，不是自动营销名单或成交预测。机会仅从员工已授权的客户关系、客户明确需求、授权购买/装备摘要、已批准知识和只读业务事实生成；未知值保持 `unknown`。

| 输出字段 | 含义与约束 |
| --- | --- |
| `customer` | 授权范围内的最小显示引用/别名和关系状态；无客户授权时可为匿名机会。 |
| `history` | 已授权的购买、装备或互动摘要及 `as_of`；不复制原始 CRM 时间线。 |
| `need` | 客户明确需求、待确认需求和缺失问题，必须标出来源状态。 |
| `recommendation` | 角色级或产品候选建议、理由、Citation、限制和置信度。 |
| `priority` | `high|medium|low` 加受控理由、有效期和数据鲜度；不表示购买意愿或员工 KPI。 |
| `next_action` | 员工可执行的补问、试穿/现场核实或人工待办；不得是自动联系动作。 |

**优先级规则（可解释、可复核）：** 仅按已确认服务请求、授权客户关系、已确认装备缺口、建议有效期、预约/人工设置的截止时间和数据鲜度排序；不以敏感属性、未同意画像、员工销售目标或模型猜测排序。证据不足的机会须为 `priority=low` 或 `verification_required`。

```json
{
  "opportunity_id": "opp_01J...",
  "customer": {"customer_ref": "cst_01J...", "relationship_status": "assigned"},
  "history": {"equipment_summary": ["已有徒步鞋"], "last_interaction_at": "2026-07-18T09:00:00Z", "as_of": "2026-07-21T00:00:00Z"},
  "need": {"confirmed": ["川西徒步"], "to_confirm": ["路线、天数、天气、预算"]},
  "recommendation": {"roles": ["rain_shell", "hydration"], "citation_ids": ["cit_01"], "limitations": ["请核实尺码、路线与库存"]},
  "priority": {"level": "medium", "reasons": ["已确认活动，关键条件待确认"], "expires_at": "2026-07-28T00:00:00Z"},
  "next_action": {"type": "confirm_context", "questions": ["是否露营？", "预计温度和负重是多少？"]}
}
```

## 5. Customer Query Assistant

Customer Assistant 只接受受控 `customer_id`，不得按手机号、微信号、姓名或自由文本搜索客户。它将 Customer Brain 的授权最小投影与本次会话输入分开显示：

| 查询能力 | 返回内容 | 必要控制 |
| --- | --- | --- |
| 客户画像 | 已明确经验、偏好、预算状态、数据来源和未知项。 | 不推断敏感属性；字段 ACL 与同意/用途控制。 |
| 购买记录 | 授权购买摘要、产品引用、时间和权威事实引用。 | 不返回不必要价格/支付/订单详情；不将历史推断为当前意愿。 |
| 装备档案 | 客户确认、授权历史或当前会话声明的装备及确认状态。 | 需区分 `customer_confirmed`、`historical`、`unknown`；不得自动回写。 |
| 活动记录 | 授权互动的时间、渠道类别、结果码和最小摘要。 | 仅对有效服务用途开放；自由文本先脱敏。 |
| 推荐机会 | 当前已授权、有效、可解释的机会与补问。 | 不是营销任务；无证据时返回“暂无可验证机会”。 |

客户页面固定展示数据时点、来源状态、用途和“当前建议需员工确认”。员工要把客户新声明写入客户档案，必须使用独立、获同意且受治理的 Customer Brain 工作流；MVP 的 Advice/Follow-up 接口不自动写画像。

## 6. Product Sales Assistant

Product Sales Assistant 在 ACL 过滤后连接 **Product Intelligence**（产品定位、技术、适配、限制、兼容性）和 **Sales Knowledge**（发现问题、话术、异议、组合规则）。检索仅使用 `Published/approved`、有效期内、Citation 完整且调用者可见的版本。

| 输出 | 组成与呈现规则 |
| --- | --- |
| 产品定位 | 品类、目标需求、适用场景及不适用边界；每项为可引用产品事实。 |
| 场景价值 | “客户明确场景/问题 → 已证实功能 → 条件和限制”；不夸大绝对性能。 |
| 销售话术 | 可编辑的开场、说明、确认草稿；话术与事实分区，避免压力销售。 |
| 异议处理 | 价格、品牌比较、线上价格、犹豫、预算不足的澄清问题、可验证维度、禁止表述和证据。 |

输出必须带 `knowledge_version`、`citation_ids`、`confidence`、`limitations`、`clarifying_questions` 和 `advisory_only_requires_human_approval`。型号/变体、尺码、实时价格、促销、库存、售后政策都由员工在授权权威系统中现场核实；若知识、产品或零售快照无权/过期，助手只能降级或抑制相关主张。

## 7. Scenario Assistant

支持川西、冈仁波齐、徒步、登山和露营。目的地是用于发起补问与匹配的受控标签，不能被模型扩展为未经证实的路线、天气、许可或安全事实。

```text
Destination
      ↓
Activity
      ↓
Environment
      ↓
Equipment
      ↓
Recommendation
```

| 阶段 | MVP 行为 | 硬性护栏 |
| --- | --- | --- |
| Destination | 识别川西、冈仁波齐或通用目的地，要求确认路线、海拔、季节、天数和计划。 | 不将地名推断为确定天气、路线难度或许可状态。 |
| Activity | 选择徒步、登山或露营，收集时长、人数、负重、技术要求和经验。 | 活动/经验未知即标记 `unknown`，不擅自假设。 |
| Environment | 收集温度、风雨、湿度、地形、昼夜、海拔和季节。 | 不提供气象、医疗、救援或领队结论。 |
| Equipment | 先匹配必选/可选功能角色、兼容性和已有装备覆盖。 | 排除 `not_suitable`、安全冲突和无证据候选。 |
| Recommendation | 输出角色组合、产品候选/替代、缺口、理由、引用、限制和补问。 | 尺码、库存、天气、路线和专业安全均要求人工核实。 |

| 场景 | 最低必问 | 方案护栏 |
| --- | --- | --- |
| 川西 | 路线、海拔、季节、天数、温度、是否露营、负重、经验。 | 多日山地仅给有证据的角色覆盖；不保证通行、天气或安全。 |
| 冈仁波齐 | 转山计划、海拔适应、健康/专业要求、天气、天数、已有装备。 | 高海拔场景；专业安全信息不足时抑制结论并升级人工/专业确认。 |
| 徒步 | 路线、时长、地形、湿滑、负重、经验。 | 核实鞋服尺码、天气和库存；不把轻量方案说成技术登山配置。 |
| 登山 | 技术难度、地形、气象、负重、经验、专业装备。 | 不替代领队、救援、医疗或路线安全判断；硬约束不满足即不推荐。 |
| 露营 | 人数、过夜、温度、营地、车载/徒步、燃料条件。 | 不承诺营地许可、气候安全或燃料使用；兼容性需有证据。 |

## 8. Cross Sell Assistant

Cross Sell 是“补齐已确认装备缺口”的人工辅助，不能按销售目标、员工历史或推断偏好发起促销。

```text
已有装备（经客户确认/授权历史/当前会话）
                    ↓
装备缺口（required / optional / unknown）
                    ↓
组合推荐（互补角色、替代、理由、Citation、人工核实）
```

1. 汇总 `existing_gear`，保留每项的来源和确认状态；不把购买历史自动认定为仍持有或适用。
2. 以场景角色、兼容性、客户明确需求和安全约束做覆盖检查，输出“已有覆盖”“待确认”“必需缺口”“可选缺口”。
3. 在客户意愿、预算、授权库存快照都满足时，选择互补类别/产品候选与替代；库存仅展示 `as_of`，未知或陈旧时显示“请核实”。
4. 每项用开放式确认问题收束，例如“这次是否还需要考虑雨天外层？”；客户无需求或约束不明时不推荐产品。

不得自动添加商品、创建购物车、锁货、触发促销或将组合解释为套装价格承诺。

## 9. Follow-up Assistant

Follow-up Assistant 保存员工主动创建的、最小化的服务记录和待办；它不拥有消息、日历、营销或 CRM 自动执行权限。

| 能力 | 最小字段 | 规则 |
| --- | --- | --- |
| 互动记录 | `interaction_id`、`customer_id?`、时间、渠道类别、已确认需求、已展示建议引用、许可状态、`outcome_code`。 | 可选摘要先做 PII/敏感信息检测；不能自动写客户画像或宣称成交。 |
| 下一动作 | `followup_id`、`action_type`、`due_at?`、核实项、负责人（默认当前员工）、状态。 | 员工显式创建/修改/完成；不自动分配或执行外部发送。 |
| 客户提醒 | 门户内待办提醒、到期状态和可编辑提醒文本。 | 仅 Portal 内可见，不生成短信、微信、邮件或活动邀请。 |

`outcome_code` 使用受控值，如 `needs_confirmation`、`information_provided`、`customer_declined`、`follow_up_requested`、`product_unavailable`。如需关联销售结果，仅接受已授权权威系统的 `outcome_ref`；员工摘要和 AI 不能自行写成成交事实。

## 10. Store Manager Dashboard

店长 Dashboard 是**授权门店范围内只读、聚合**的运营辅助。客户级下钻必须再次通过店长的客户关系、用途和字段 ACL；无客户级权限时只显示去标识化计数。

| 模块 | 展示内容 | 约束 |
| --- | --- | --- |
| 销售机会 | 按优先级、有效期、待确认项和去标识化/授权客户队列汇总的机会。 | 不自动分配、触达、关闭或标记成交。 |
| 重点商品 | 已发布产品的场景适配、证据、门店表现聚合和数据窗口。 | 不绕过客户需求、适配或安全边界；不是自动促销清单。 |
| 库存提醒 | 只读库存快照、低可售/陈旧/不可用提示、替代类别。 | 不补货、调拨、预留或承诺库存。 |
| 员工跟进 | 待办数量、到期状态、缺失核实项和去标识化反馈质量聚合。 | 不用于自动绩效、排班、奖惩或佣金决定。 |

每个卡片显示 `store_scope`、统计窗口、`as_of`、`freshness.status`、来源和数据限制。区域/组织级角色默认仅见其授权范围内的去标识化聚合。

## 11. Feedback Loop

```text
员工反馈
      ↓
AI 整理
      ↓
审核
      ↓
Sales Memory
```

| 阶段 | MVP 处理 | 发布门 |
| --- | --- | --- |
| 员工反馈 | 员工针对 Advice 提交采纳/编辑/拒绝、原因码和可选最小摘要。 | Advice、员工和权限必须匹配；摘要通过敏感信息处理。 |
| AI 整理 | 对脱敏反馈按知识/场景聚类，形成“证据缺口、候选修订、需复核”草稿。 | 标记为 `draft`，保留输入版本与血缘；AI 不得补造 Citation。 |
| 审核 | 销售负责人和知识审核人审查来源、事实、限制、ACL 和冲突；安全/比较/政策内容可路由产品或合规复核。 | `submitted → triaged → draft → in_review → approved|rejected`。 |
| Sales Memory | 将批准后的新版本发布给 Sales Knowledge；历史 Advice 仍关联旧版本。 | 只允许 Citation 完整、ACL 正确、有效的 `approved` 版本检索。 |

反馈最小契约：`feedback_id`、`advice_id`、`knowledge_refs[]`、`version_refs[]`、`action`、`reason_code`、`employee_summary?`、`outcome_ref?`、`review_state`、`created_at`。负反馈不会自动下架知识、改变排序或评价员工。指标仅使用去标识化聚合，如 Citation 完整率、补问率、编辑/拒绝原因、审核时效和知识过期率。

## 12. API Contract

### 12.1 Shared contract

所有 API 使用 JSON、RFC 3339 UTC、`Authorization`、`X-Correlation-Id`、`X-Purpose`；写接口还要求 `Idempotency-Key`。组织、员工、角色、门店和最大可见范围全部由服务器派生。请求不得携带客户端声明的角色/权限、原始客户 PII、SAP/Core 写字段或执行指令。

通用错误：`400 INVALID_REQUEST`、`401 UNAUTHENTICATED`、`403 POLICY_DENIED`、`404 RESOURCE_NOT_AVAILABLE`、`409 IDEMPOTENCY_CONFLICT`、`422 VALIDATION_FAILED`、`429 RATE_LIMITED`、`503 DEPENDENCY_UNAVAILABLE`。依赖不可用或数据陈旧时不得伪装为新鲜事实；Advice 应降级为补问/人工核实或明确失败。

### 12.2 `GET /api/employee/opportunities`

**用途：** 返回当前员工的机会队列，或对具有 `dashboard:read` 的店长返回授权门店队列。
**查询参数：** `store_id?`、`status?`、`priority?`、`window?`、`limit=1..50`、`cursor?`；`store_id` 必须属于服务端计算的门店范围。

```json
{
  "items": [{
    "opportunity_id": "opp_01J...",
    "customer": {"customer_ref": "cst_01J...", "relationship_status": "assigned"},
    "history": {"as_of": "2026-07-21T00:00:00Z"},
    "need": {"confirmed": ["徒步"], "to_confirm": ["路线", "预算"]},
    "recommendation": {"roles": ["rain_shell"], "citation_ids": ["cit_01"]},
    "priority": {"level": "medium", "reasons": ["需求已确认，装备缺口待核实"]},
    "next_action": {"type": "confirm_context", "questions": ["预计是否下雨？"]}
  }],
  "next_cursor": null,
  "freshness": {"as_of": "2026-07-21T00:00:00Z", "status": "fresh"},
  "advisory_only_requires_human_approval": true
}
```

### 12.3 `GET /api/employee/customer/{id}`

**用途：** 读取单一已授权客户的最小投影。`id` 仅接受稳定 `customer_id`。
**返回：** `profile`、`purchase_history`、`existing_gear`、`activity_history`、`opportunities`、`consent`、`as_of`、字段可见性元数据。无权和不存在均为 `404 RESOURCE_NOT_AVAILABLE`。

```json
{
  "customer_id": "cst_01J...",
  "profile": {"experience": "regular", "preferences": ["lightweight"], "unknown": ["budget"]},
  "purchase_history": [{"product_ref": "prd_...", "purchased_at": "2026-04-01", "source": "authorized_history"}],
  "existing_gear": [{"product_ref": "prd_...", "confirmation": "customer_confirmed"}],
  "activity_history": [{"occurred_at": "2026-07-18T09:00:00Z", "channel": "in_store", "outcome_code": "needs_confirmation"}],
  "opportunities": [{"opportunity_id": "opp_01J...", "priority": "medium"}],
  "consent": {"purpose": "in_store_service", "status": "allowed"},
  "as_of": "2026-07-21T00:00:00Z"
}
```

### 12.4 `POST /api/employee/advice`

**用途：** 生成 Customer、Product Sales、Scenario 或 Cross Sell 的只读建议。
**受控 `intent`：** `customer`、`product_sales`、`scenario`、`cross_sell`。`customer_id` 可选，只有通过关系/同意/用途后才解析。

```json
{
  "intent": "scenario",
  "customer_id": "cst_01J...",
  "store_id": "store_chengdu_01",
  "context": {
    "destination": "川西",
    "activity": "hiking",
    "environment": ["variable_weather"],
    "existing_gear_refs": ["prd_..."]
  },
  "question": "如何准备 7 天徒步装备？"
}
```

```json
{
  "advice_id": "adv_01J...",
  "intent": "scenario",
  "sections": [{"kind": "needs", "content": "先确认路线、海拔、温度、负重与露营安排。", "citation_ids": ["cit_01"]}],
  "recommendations": [{"role": "rain_shell", "product_refs": ["prd_..."], "limitations": ["请核实尺码、库存与天气"], "citation_ids": ["cit_02"]}],
  "clarifying_questions": ["是否需要露营？"],
  "freshness": {"knowledge_as_of": "2026-07-21T00:00:00Z", "retail_status": "verification_required"},
  "confidence": {"level": "medium", "rationale": "路线与温度未知"},
  "advisory_only_requires_human_approval": true
}
```

### 12.5 `POST /api/employee/followup`

**用途：** 创建或更新当前员工的人工跟进记录/Portal 内提醒；不发送外部消息。
**权限：** `followup:write`，客户关联时还必须通过客户关系、用途和字段 ACL。

```json
{
  "customer_id": "cst_01J...",
  "advice_id": "adv_01J...",
  "interaction": {"occurred_at": "2026-07-21T10:00:00Z", "channel": "in_store", "outcome_code": "needs_confirmation"},
  "next_action": {"type": "confirm_context", "due_at": "2026-07-23T10:00:00Z", "questions": ["确认路线与预算"]},
  "employee_summary": "客户希望下次到店前先确认川西行程。"
}
```

**`202 Accepted`：** 返回 `followup_id`、`status=scheduled|completed`、`portal_reminder=true`、`created_at`。摘要经敏感信息筛查失败时返回 `422 VALIDATION_FAILED`；该接口绝不创建短信、微信、邮件、订单、库存或 CRM 自动任务。

### 12.6 `POST /api/employee/feedback`

**用途：** 提交对 Advice 的显式反馈，只创建受审核记录。
**权限：** `feedback:write`；必须有 `Idempotency-Key`。

```json
{
  "advice_id": "adv_01J...",
  "action": "edited",
  "reason_code": "insufficient_context",
  "employee_summary": "客户确认单日徒步，需补问降雨情况。",
  "outcome_ref": "salefact_01J..."
}
```

**`202 Accepted`：** `{ "feedback_id": "fb_01J...", "review_state": "submitted" }`。受控 `action` 为 `accepted|edited|rejected|insufficient_context|citation_issue|product_unavailable`；`outcome_ref` 仅是授权权威事实引用，不会由该接口自动宣称成交。

## 13. Data Quality, Citation and Audit

- 产品事实、销售主张、场景规则、组合关系和异议回应必须返回 Citation、版本、检索时间、置信度和限制；Citation 本身继续受 Asset/Knowledge ACL 过滤。
- 生命周期仅允许 `draft → in_review → approved/published → deprecated/archived`；默认检索与建议仅消费有效的 `approved/published` 版本。
- `confidence` 表示上下文、证据和数据鲜度的完整性，**不是**成交概率、客户价值或员工绩效。
- Retail Brain 只读快照须带 `as_of`、`status=fresh|stale|unavailable|verification_required`；陈旧、隔离、不可用或越权时不返回“有货”。
- 每个 Advice 保存输入最小化快照引用、策略/规则/模型版本、知识版本、Citation、过滤原因、输出、员工结果码和关联 ID，支持回放；审计不持久化完整 PII 或受限正文。

## 14. Security Boundaries

**RBAC + ACL 是强制的服务端控制**，并结合组织隔离、门店范围、客户关系、用途/同意、字段最小化、知识生命周期、来源 ACL 和审计。Portal 仅是显示层，不构成授权边界。

严格禁止：

- 自动营销；
- 自动成交；
- 自动下单；
- 修改 SAP；
- 修改 Core 生产同步。

同样禁止库存预留/调拨、改价、自动报价、自动外部消息、把模型推断回写为客户/库存/产品/订单事实，以及绕过人工审核发布 Sales Memory。
