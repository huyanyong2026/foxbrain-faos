# VAFOX Legacy Cleanup Plan

> 本文档仅用于规划 VAFOX 1.0 到 VAFOX 2.0 的架构迁移与遗留资产治理。  
> **禁止基于本文档直接执行任何服务器修改操作。**

## 1. 目标

规划 VAFOX 从早期建设阶段的 VAFOX 1.0 架构，平滑迁移至 VAFOX 2.0 标准架构。

当前涉及服务器：

| 角色 | IP | 域名 |
| --- | --- | --- |
| Huyan | `140.143.207.194` | `huyan.vafox.com` |
| AI | `1.13.254.217` | `ai.vafox.com` |
| Core | `139.199.174.36` | `core.vafox.com` |

## 2. 背景

VAFOX 早期建设阶段可能存在多轮试验性部署与迭代痕迹，包括但不限于：

- 不同版本脚本
- 不同部署方式
- 测试文件
- 临时配置
- 旧目录
- 旧架构遗留
- 未统一命名、未统一目录结构、未统一运维规范的资产

本计划的目标不是立即清理服务器，而是建立一套可审计、可回滚、可验证的 VAFOX 2.0 迁移规划。

## 3. 当前可能存在的 Legacy 类型

以下类型均属于迁移前需要盘点、标记、分类、确认归属的 Legacy 资产。

### 3.1 旧代码

可能包括：

- 早期版本业务代码
- 手工复制的项目目录
- 未纳入 Git 或版本管理的代码
- 临时实验分支代码
- 已废弃但仍在服务器上的服务代码
- 与当前生产版本不一致的 API、前端、任务脚本或集成代码

处理原则：

- 不直接删除
- 先确认是否仍被进程、定时任务、Docker、Systemd 或 Nginx 引用
- 可读性保留，必要时归档为只读包
- 与 VAFOX 2.0 标准目录结构对齐后再迁移

### 3.2 旧 Docker 配置

可能包括：

- 旧版 `docker-compose.yml`
- 旧版镜像与未使用容器
- 临时测试容器
- 未命名或命名不规范的 volume
- 重复网络、孤儿网络
- 与生产环境混用的测试服务

处理原则：

- 先盘点容器、镜像、volume、network 与端口映射关系
- 对承载数据的 volume 做高优先级保护
- 未确认用途前禁止 `docker rm`、`docker volume rm`、`docker system prune`
- 对 VAFOX 2.0 保留服务建立标准 Compose 或编排规范

### 3.3 旧 Nginx 配置

可能包括：

- 旧站点配置
- 反向代理临时配置
- 过期 upstream
- 测试域名或测试端口转发
- 未启用但仍留存的配置文件
- SSL 证书路径不统一的问题

处理原则：

- 先建立域名、端口、证书、后端服务映射表
- 保留当前线上可访问路径
- 禁止未验证直接 reload 或覆盖配置
- 迁移至 VAFOX 2.0 标准网关与域名路由规范

### 3.4 旧 Systemd 服务

可能包括：

- 手工创建的 `.service` 文件
- 已停用但仍 enabled 的服务
- 指向旧路径的服务
- 与 Docker 部署重复的服务
- 临时守护进程、脚本守护进程

处理原则：

- 先确认服务状态、启动命令、工作目录、用户权限和依赖关系
- 区分生产服务、辅助服务、遗留服务、废弃服务
- 禁止直接 disable、stop 或 remove
- VAFOX 2.0 中仅保留必要的标准化服务单元

### 3.5 旧脚本

可能包括：

- 部署脚本
- 初始化脚本
- 备份脚本
- 同步脚本
- 测试脚本
- 一次性迁移脚本
- 手工运维脚本

处理原则：

- 按用途、风险、是否仍被调用分类
- 对 SAP 同步、备份、数据处理类脚本优先保护
- 对危险命令脚本进行隔离标记
- VAFOX 2.0 中仅保留经过审计的标准运维脚本

### 3.6 旧 Release

可能包括：

- 旧版本压缩包
- 多个日期版本目录
- 手工备份包
- 未清理构建产物
- 旧前端构建目录
- 旧后端发布目录

处理原则：

- 建立 release 资产清单
- 标记当前线上运行版本
- 至少保留可回滚版本
- 历史版本归档后再决定是否废弃

### 3.7 临时测试文件

可能包括：

- 测试上传文件
- 临时日志
- 临时 SQL
- 临时 JSON、CSV、Excel
- 测试知识库材料
- 临时配置备份
- 调试输出文件

处理原则：

- 先判断是否包含业务数据、客户数据、SAP 数据或知识资产
- 含数据文件不得直接删除
- 无业务价值文件经确认后进入废弃候选清单
- 所有清理必须在备份与验收后执行

## 4. 三服务器资产分类

本节为迁移规划分类，不代表已经完成实际盘点。实际执行前必须通过只读方式采集资产清单。

### 4.1 Huyan：`140.143.207.194` / `huyan.vafox.com`

VAFOX 2.0 目标角色：**CEO Control Plane**。

#### 保留

- CEO 控制台相关核心服务
- 面向管理层的控制面 API 与前端入口
- 当前仍承担生产访问的 Nginx 路由
- 当前有效 SSL 证书与域名配置
- 与 VAFOX 控制面相关的身份认证、权限、审计入口
- 与 Core、AI 通信所需的安全配置

#### 迁移

- 分散在旧目录中的控制台代码
- 临时部署脚本中仍有效的发布逻辑
- 旧 Nginx 配置中仍承载生产流量的路由
- 指向旧服务路径的 Systemd 单元
- 未标准化的日志、备份与监控配置

#### 废弃候选

- 已不再使用的 Huyan 1.0 页面或接口
- 临时 Demo、PoC、Mock 服务
- 已被 VAFOX 2.0 控制台替代的旧入口
- 未被进程、端口、Nginx、Systemd 引用的旧 release
- 无业务价值的测试文件与调试文件

### 4.2 AI：`1.13.254.217` / `ai.vafox.com`

VAFOX 2.0 目标角色：**AI Workforce Platform**。

#### 保留

- AI Agent、AI Worker、模型调用相关核心服务
- Dify 相关服务与知识资产
- n8n 工作流与自动化编排资产
- Milvus 向量数据及其运行配置
- MinIO 对象数据及其运行配置
- 生产可用的模型网关、任务队列、调度组件

#### 迁移

- 分散的 AI 任务脚本与 Agent 脚本
- 旧版 Dify、n8n、Milvus、MinIO 部署配置
- 未统一命名的 Docker Compose 项目
- 测试模型配置中已转为生产使用的部分
- AI 服务间通信配置、环境变量、密钥引用方式

#### 废弃候选

- 过期模型测试脚本
- 已失效的模型 API 调试文件
- 未被工作流引用的临时知识库材料
- 不再运行的 PoC 容器与测试容器
- 重复的临时上传文件、日志、构建产物

### 4.3 Core：`139.199.174.36` / `core.vafox.com`

VAFOX 2.0 目标角色：**Enterprise Data Brain**。

#### 保留

- SAP 同步数据
- PostgreSQL 数据库及其备份链路
- 企业主数据、业务数据、指标数据
- 数据同步、数据清洗、数据服务 API
- 数据权限、数据审计、数据质量相关配置
- 与 Huyan、AI 通信的数据服务入口

#### 迁移

- 旧 SAP 同步脚本中仍有效的逻辑
- 旧数据处理脚本、定时任务、ETL 任务
- 旧 PostgreSQL 配置中仍需保留的参数
- 旧数据服务 API 中仍被调用的接口
- 旧备份、导出、报表生成逻辑

#### 废弃候选

- 已停用的数据测试表导出文件
- 临时 SQL、临时 CSV、临时 Excel
- 未被任务引用的旧 ETL 脚本
- 重复的历史 release 与调试日志
- 已被标准数据服务替代的旧接口服务

## 5. 数据保护原则

迁移前、中、后均必须执行数据保护优先原则。任何清理、移动、重命名、替换、升级、切换行为，都不得破坏以下资产。

### 5.1 必须保护的数据与资产

- **SAP 同步数据**
- **PostgreSQL 数据**
- **Milvus 数据**
- **MinIO 数据**
- **n8n 工作流**
- **Dify 知识资产**

### 5.2 数据保护要求

- 所有关键数据必须先完成备份，再进行迁移或切换。
- 备份必须可验证、可恢复、可追溯。
- 对数据库、对象存储、向量库、工作流平台、知识库资产分别建立备份策略。
- 不允许只备份配置而忽略底层 volume、数据库文件、对象文件或索引数据。
- 迁移前必须记录数据规模、版本、存储路径、容器挂载、账号权限和恢复方式。
- 迁移后必须进行数据一致性校验与业务可用性验证。

## 6. VAFOX 2.0 目标状态

### 6.1 Huyan：CEO Control Plane

目标定位：VAFOX 的 CEO 控制面与管理入口。

目标状态：

- 统一管理控制台入口
- 统一权限与身份认证策略
- 统一面向管理层的指标、任务、Agent 调度视图
- 标准化 Nginx / 网关路由
- 标准化部署目录、配置目录、日志目录、备份目录
- 与 AI、Core 的调用边界清晰

### 6.2 AI：AI Workforce Platform

目标定位：AI 员工、智能体、工作流与知识能力平台。

目标状态：

- AI Agent 与 AI Worker 统一部署
- Dify、n8n、Milvus、MinIO 等 AI 能力组件边界清晰
- 模型配置、提示词、知识库、工作流资产可备份、可迁移、可审计
- 与 Core 的数据访问边界清晰
- 与 Huyan 的控制指令和状态回传链路清晰
- 统一任务队列、日志、监控与告警规范

### 6.3 Core：Enterprise Data Brain

目标定位：企业数据大脑与数据服务中心。

目标状态：

- SAP 数据同步链路稳定、可监控、可恢复
- PostgreSQL 作为核心结构化数据底座
- 数据清洗、指标计算、数据 API 标准化
- 数据权限、审计、质量检查机制明确
- 为 AI 提供受控数据能力
- 为 Huyan 提供管理层指标与决策数据能力

## 7. 迁移策略

### 7.1 冻结旧版本

目标：防止迁移期间 Legacy 状态继续变化。

规划动作：

- 冻结旧 release 与旧部署目录
- 冻结临时脚本新增
- 冻结非必要配置修改
- 建立当前运行资产快照
- 标记线上当前版本与回滚版本

### 7.2 建立新标准

目标：先定义 VAFOX 2.0 标准，再迁移资产。

规划动作：

- 定义三服务器角色边界
- 定义标准目录结构
- 定义标准 Docker / Systemd / Nginx 管理方式
- 定义标准环境变量和密钥管理方式
- 定义标准日志、备份、监控、告警方式
- 定义标准发布、回滚、验收流程

### 7.3 逐步迁移

目标：降低一次性切换风险。

规划动作：

- 先迁移低风险、非核心入口
- 再迁移服务配置与部署方式
- 最后迁移核心数据链路与生产入口
- 每迁移一项，完成备份、变更记录、验证、回滚预案
- 保持 VAFOX 1.0 与 VAFOX 2.0 在过渡期内可对照验证

### 7.4 验证

目标：确认迁移后的服务、数据、链路与业务表现正确。

规划动作：

- 验证域名访问
- 验证 Nginx 路由
- 验证 Systemd / Docker 服务状态
- 验证数据库连接与数据一致性
- 验证 Milvus、MinIO、n8n、Dify 资产可用性
- 验证 Huyan、AI、Core 三服务器之间的调用链路
- 验证日志、备份、监控、告警可用性

### 7.5 切换

目标：在验证充分后进行受控切换。

规划动作：

- 制定切换窗口
- 明确切换负责人、验证负责人、回滚负责人
- 切换前完成最终备份
- 切换后完成冒烟测试与核心业务验证
- 保留旧版本只读回滚窗口
- 验收通过后再进入清理阶段

## 8. 风险控制

### 8.1 禁止事项

迁移期间明确禁止：

- **禁止直接删除** Legacy 文件、目录、容器、volume、数据库、对象存储、知识库或工作流。
- **禁止直接重装** 任一服务器。
- **禁止未备份迁移** 任何服务、数据、配置或工作流资产。
- **禁止未验证切换** 域名、端口、网关、数据库、任务调度或生产入口。

### 8.2 风险清单

| 风险 | 影响 | 控制方式 |
| --- | --- | --- |
| 误删数据 | 业务数据丢失 | 先备份、再验证、后清理 |
| 容器 volume 被误删 | Milvus、MinIO、PostgreSQL 等数据丢失 | 禁止 prune，建立 volume 清单 |
| Nginx 配置覆盖 | 生产入口不可用 | 配置版本化，切换前语法检查 |
| Systemd 服务误停 | 后台任务中断 | 建立服务依赖图，分批操作 |
| SAP 同步中断 | 企业数据不一致 | 同步任务冻结窗口与补偿机制 |
| Dify / n8n 资产丢失 | AI 工作流能力受损 | 导出工作流、备份数据库和文件资产 |
| 未识别旧脚本仍在运行 | 隐性任务失败 | 盘点 crontab、Systemd timer、进程与日志 |
| 一次性大切换 | 回滚困难 | 分阶段迁移、灰度验证 |

## 9. 最终验收标准

VAFOX 2.0 迁移完成前，必须满足以下验收标准：

### 9.1 架构验收

- Huyan 明确作为 **CEO Control Plane**。
- AI 明确作为 **AI Workforce Platform**。
- Core 明确作为 **Enterprise Data Brain**。
- 三服务器职责边界清晰，无明显职责混乱。
- VAFOX 2.0 标准目录、部署、配置、日志、备份规范已落地。

### 9.2 服务验收

- Huyan 控制台可正常访问。
- AI 平台核心服务可正常运行。
- Core 数据服务可正常运行。
- Nginx、Docker、Systemd 配置均经过验证。
- 跨服务器调用链路正常。

### 9.3 数据验收

- SAP 同步数据完整。
- PostgreSQL 数据完整且可恢复。
- Milvus 数据完整且可查询。
- MinIO 数据完整且可访问。
- n8n 工作流完整且可执行。
- Dify 知识资产完整且可检索。

### 9.4 运维验收

- 标准发布流程可执行。
- 标准回滚流程可执行。
- 备份与恢复流程经过验证。
- 监控与告警可用。
- 关键配置有版本记录。
- Legacy 资产已完成保留、迁移、废弃分类。

### 9.5 清理验收

- 所有废弃候选项均已确认不再被使用。
- 所有废弃候选项均已完成备份或归档。
- 清理操作有清单、有审批、有回滚方案。
- 清理后生产业务无异常。
- 清理后文档、资产清单、部署说明同步更新。

## 10. 执行边界声明

本文档只输出迁移规划，不包含任何服务器执行命令。  
在进入实际执行阶段前，必须另行制定详细执行 Runbook，并经过备份验证、风险评审和负责人确认。
