# VAFOX Control Plane Server 第一阶段安全加固方案

## 文档控制

| 项目 | 内容 |
| --- | --- |
| 目标系统 | `control.vafox.com` |
| 公网 IP | `114.132.55.178` |
| 内网 IP | `172.16.16.6` |
| 系统定位 | VAFOX Control Plane Server |
| 基础规格 | Ubuntu 24.04 LTS；4 vCPU / 16 GB RAM / 200 GB SSD |
| 当前状态 | 空白服务器；未安装 Docker、Nginx 或业务服务 |

## 一、目标

### 1.1 为什么必须先完成安全基线

Control Plane 是管理入口，未来可能保存审批上下文、任务定义、Agent 身份、短期访问凭据，并向其他 VAFOX 环境发出受控指令。若其操作系统、远程访问、网络暴露、权限边界和审计链未先完成最小安全基线，单一主机失陷即可扩大为跨环境的控制面风险。

因此，第一阶段的上线原则为：**先建立可验证、可审计、可回滚的最小可信控制面，再部署任何 Docker、Nginx、Agent 或业务服务。** 本阶段不将服务器视为生产可用，直至第十一章全部验收项通过并留下证据。

### 1.2 阶段性安全目标

1. 仅允许经过授权的管理员通过 SSH 密钥进入服务器，禁止 root 和密码远程登录。
2. 默认拒绝公网入站访问；仅开放经审批的 SSH、HTTP、HTTPS 服务端口。
3. 将人员管理、受控执行和 Agent 身份分离，遵循最小权限、职责分离和短期授权。
4. 确保系统补丁、时间、日志和审计链可持续运行，且日志可被安全留存。
5. 确保密钥、Token 和 API Key 不以明文出现在代码、配置仓库、脚本、命令历史、日志或备份中。
6. 在每次变更前具备可恢复的配置备份，在异常时可安全恢复 SSH、配置和防火墙访问。

### 1.3 范围与边界

- **纳入范围：** Ubuntu 基线、SSH、UFW、防更新策略、时间同步、日志与审计、账号权限、Secret、配置备份、变更审批及安全验收。
- **不纳入范围：** Docker/Nginx/业务服务安装、Control Plane 应用部署、Agent 实际接入、DNS 变更、生产流量切换和对其他服务器的远程操作。
- **网络边界：** 公网地址 `114.132.55.178` 用于受控入口；内网地址 `172.16.16.6` 仅供经批准的私网管理或后续受控服务通信。不得因存在内网地址而绕过身份认证、审计或审批。

## 二、SSH 安全设计

### 2.1 访问策略

| 控制项 | 第一阶段设计要求 |
| --- | --- |
| 登录方式 | 仅允许现代 SSH 公钥认证（建议 Ed25519）；每位管理员使用独立密钥对。 |
| root 远程登录 | 禁止；`PermitRootLogin no`。 |
| 密码登录 | 禁止；`PasswordAuthentication no`，同时禁用键盘交互式密码认证。 |
| 管理账号 | 建立命名到人的普通管理账号，例如 `vafox-admin-<id>`；禁止共享账号。 |
| 来源限制 | SSH 仅接受已登记管理出口 IP/CIDR；临时来源须有时限审批并在窗口结束后撤销。 |
| 暴力尝试防护 | 使用系统级限速与认证失败监控；可在后续经审批后增加 fail2ban，不以其替代密钥认证和来源限制。 |
| 空闲会话 | 设置合理的空闲断开及会话存活策略；保留满足故障处置的受控并发会话数。 |
| 密钥生命周期 | 密钥须登记所有者、用途、创建/到期时间和撤销状态；人员离岗、设备丢失或疑似泄露时立即撤销。 |

### 2.2 SSH 配置基线

SSH 守护进程配置应至少满足下列逻辑（具体文件落地、语法校验和重载属于后续审批变更）：

```text
PermitRootLogin no
PasswordAuthentication no
KbdInteractiveAuthentication no
PubkeyAuthentication yes
AllowGroups vafox-ssh-admin
MaxAuthTries 3
LoginGraceTime 30
X11Forwarding no
AllowTcpForwarding no        # 默认禁用；仅对明确批准的受控用途例外
```

- `AllowGroups` 中只应出现经审批的管理组；不得使用 `*` 或不受控的宽泛账号规则。
- 至少保留一个已验证的、非 root、具备 sudo 权限的应急管理账号及其离线保管的恢复密钥；不得在确认第二条可用登录路径前关闭当前会话或应用 SSH 配置。
- SSH 主机指纹应在受控资产清单中登记；管理员首次连接时必须通过独立可信渠道核验指纹，防止中间人攻击。

### 2.3 sudo 最小权限

- 管理员通过普通账号登录后才可使用 `sudo`；禁止日常直接 root 会话。
- sudo 权限按角色和命令白名单授予，优先使用 `/etc/sudoers.d/` 的独立、可审查片段；禁止无审批的 `NOPASSWD: ALL`。
- 可使用完整 sudo 的人员必须极少、实名、经双人审批，并定期复核；执行 Agent、应用服务账号和自动化账号不得拥有通用 sudo。
- sudo 日志须进入系统日志并纳入审计留存；对 sudoers 文件的所有变更须记录变更单和复核人。

## 三、防火墙设计

### 3.1 默认策略

UFW 采用“默认拒绝入站、默认允许出站、显式最小放行”的策略。任何新增端口均须说明服务归属、源地址范围、协议、有效期、风险评估、审批编号和回滚规则。

| 流量方向 | 基线策略 |
| --- | --- |
| 入站 | 默认拒绝。 |
| 出站 | 默认允许，但后续 Agent/应用上线前应按实际依赖收敛为允许列表。 |
| 转发/路由 | 默认拒绝；本阶段服务器不得充当网络转发节点。 |
| IPv6 | 若启用则实施与 IPv4 等价的规则；若不使用，须在变更记录中明确禁用状态与理由。 |

### 3.2 允许规则

| 服务 | 端口/协议 | 来源 | 第一阶段约束 |
| --- | --- | --- | --- |
| SSH | TCP/22（或获批替代端口） | 已登记的管理出口 IP/CIDR | 不允许向全网开放；仍必须通过 SSH 密钥认证。 |
| HTTP | TCP/80 | 公网 | 仅为未来 HTTP→HTTPS 跳转、证书签发或明确批准的临时用途预留；未部署服务时不应实际监听。 |
| HTTPS | TCP/443 | 公网 | 仅为未来经安全评审的 Web/API 入口预留；未部署服务时不应实际监听。 |

> 对 SSH 的“允许”是来源受限的允许，而非 `0.0.0.0/0`。HTTP/HTTPS 放行不等于安装 Nginx 或发布业务服务；在当前空白服务器阶段，端口可保持未监听。

### 3.3 禁止规则与控制

- 禁止所有未获明确批准的入站端口，包括数据库、缓存、Docker API、容器编排、监控管理面板、调试端口和内部 Agent 控制端口。
- 禁止公网访问私网管理接口；后续管理面服务应优先绑定 loopback 或内网，并另行定义源地址允许列表。
- 禁止通过临时 `allow any`、云安全组宽泛放行或未登记的旁路规则绕开 UFW 设计。
- UFW 规则、云安全组/ACL 与实际监听端口必须三方一致；任何一方的放宽都须走变更审批。

## 四、系统安全

### 4.1 自动更新策略

1. 启用 Ubuntu 24.04 LTS 的安全更新自动安装机制，覆盖操作系统安全补丁。
2. 内核、OpenSSH、关键加密库及其他可能影响远程访问或可用性的更新，须在维护窗口内完成复核，并依据变更策略安排重启。
3. 每月审阅更新状态、待重启状态和异常更新；紧急高危漏洞按安全事件流程提高处置优先级。
4. 第三方软件源应最小化、签名校验、明确所有者；未经审批不得添加未知 PPA、脚本安装源或不受控二进制包。
5. 更新前后均应生成配置备份和验证记录，失败时按第十章回滚。

### 4.2 时间同步

- 使用 `systemd-timesyncd` 或经批准的 NTP 客户端与可信时间源同步；至少配置多个可靠时间源并监控同步状态。
- 系统统一使用 UTC 记录安全日志；如展示本地时区，必须保留 UTC 原始时间。
- 时间偏差、时间服务停止和时间源不可达应产生告警，因为错误时间会破坏审计关联、证书验证和 Token 有效期判断。

### 4.3 系统日志

- 启用并保留 `systemd-journald`、认证日志、内核日志、UFW 日志及服务日志；配置持久化存储与合理轮转，防止重启后丢失关键证据或日志耗尽磁盘。
- 日志目录须限制写入和读取权限，普通应用/Agent 不得能够修改系统审计日志。
- 重要日志应在后续平台接入后安全转发到独立的集中日志系统；在此之前，须执行加密、访问受限的离机备份。
- 对认证失败、sudo 使用、UFW 拒绝、审计服务异常、磁盘容量阈值和日志篡改迹象建立告警规则。

### 4.4 审计

- 安装并启用 Linux 审计能力（例如 `auditd`），审计规则应覆盖认证、身份与组变更、sudoers、SSH 配置、UFW 规则、关键配置目录、服务单元和 Secret 元数据的变更。
- 审计服务应具备故障告警；审计规则的变更仅允许由最小范围的安全管理员通过审批实施。
- 审计记录应以只追加、限制访问和异机留存为目标；保留期须满足组织合规要求，第一阶段建议在线不少于 90 天、离线归档不少于 1 年，最终以合规负责人批准的策略为准。

## 五、权限模型

### 5.1 角色定义

| 身份 | 允许职责 | 禁止或限制 |
| --- | --- | --- |
| `root` | 仅限紧急恢复、底层系统修复和受审批的高权限维护。 | 禁止 SSH 远程登录、禁止日常工作、禁止运行常驻业务或 Agent。 |
| 管理员 | 以实名普通账号 SSH 登录；在批准范围内使用 sudo 执行系统维护、审查日志和应用已批准变更。 | 禁止共享账号/密钥、禁止直接持有生产 Secret 明文、禁止超出变更单范围操作。 |
| `vafox-exec` | 未来受控任务执行器的专用服务账号；仅能读取已审批 Artifact、写入自己的运行日志/报告，并调用明确允许的受控命令。 | 不可交互登录、不可通用 sudo、不可读取其他用户家目录或全量 Secret。 |
| Agent | 每个 Agent 使用独立、可撤销的机器身份；仅能访问其任务所需 API、队列和目录。 | 不可登录为 root/管理员、不可使用共享 Token、不可执行任意 shell 或跨环境横向访问。 |

### 5.2 授权原则

- 人员账号和机器账号绝不复用；每个身份均须可归属、可禁用、可审计。
- `vafox-exec` 和 Agent 的目录、日志、运行时配置使用独立所有权和最小文件权限；禁止通过同组可写目录提升权限。
- 受控执行必须关联审批编号、调用者身份、目标、命令/任务版本、开始结束时间和结果；未关联审批的执行必须拒绝。
- 权限授予采用到期复核机制：临时权限必须设置到期时间；至少每季度复核管理员、sudo、服务账号和 Agent 授权。

## 六、Secret 管理

### 6.1 管理对象与原则

Secret 包括但不限于 SSH 私钥、TLS 私钥、Token、API Key、数据库密码、Webhook 密钥、服务账户凭据、恢复密钥和加密密钥。所有 Secret 必须满足“最少持有、按需读取、可轮换、可撤销、可审计”。

- 使用经批准的 Secret Manager 或 KMS 保存生产 Secret；Control Plane 仅在运行时通过受控身份获取必要的短期凭据。
- 若第一阶段尚未具备 Secret Manager，不得因此将 Secret 写入服务器或项目文件；应将部署/Agent 接入保持在未上线状态，直至 Secret 管理能力通过验收。
- 配置中仅可保存 Secret 的引用、版本标识或非敏感参数，不得保存实际值。

### 6.2 明文禁止项

严禁在以下位置保存或输出明文 Secret：Git 仓库与提交历史、Markdown 文档、`.env` 文件、Shell 脚本、systemd 单元、容器镜像、命令行参数、终端历史、CI/CD 日志、应用日志、截图、工单附件、聊天工具及未加密备份。

### 6.3 生命周期和事件响应

1. Secret 创建、发放、读取授权、轮换、禁用和销毁均须记录责任人、用途、系统范围与时间。
2. 人员离岗、权限收回、设备丢失、日志误输出或疑似泄露时，立即吊销/轮换相关 Secret，并审计受影响调用。
3. 生产与非生产环境必须使用不同 Secret、不同权限边界和不同轮换记录。
4. 备份中的加密密钥须与被备份数据分离保存；恢复操作需经审批并记录。

## 七、备份设计

### 7.1 备份范围与频率

| 类别 | 备份内容 | 建议频率 | 安全要求 |
| --- | --- | --- | --- |
| 配置备份 | SSH、UFW、sudoers、系统更新、时间同步、日志、审计、用户/组及 systemd 配置。 | 每次变更前后；至少每日汇总。 | 加密、完整性校验、访问最小化。 |
| 安全配置备份 | 审计规则、日志轮转、告警规则、密钥/证书**引用和元数据**、权限清单。 | 每次安全变更前后。 | 不包含明文 Secret 或私钥。 |
| 运行证据备份 | 变更单、审批记录、验收结果、审计导出、恢复演练记录。 | 每次变更/演练后。 | 不可随意覆盖，保留版本与责任链。 |

### 7.2 存储与保留

- 备份采用“3-2-1”原则：至少三份副本、两种存储介质、一份异地或隔离副本；至少一份须具备不可变或防篡改能力。
- 备份传输和静态存储均须加密；备份访问与生产管理员权限分离，避免单一账号同时删除源数据和全部备份。
- 定期验证备份可读性、校验和和恢复可行性。仅“备份成功”不等于“可恢复”。

### 7.3 回滚准备

每个变更包必须包含：变更前配置快照、版本/校验和、变更步骤、验证步骤、回滚步骤、负责人、恢复密钥获取路径（不含密钥值）和预计恢复时间目标。任何无法提供回滚材料的变更不得进入实施窗口。

## 八、审计设计

### 8.1 必须记录的事件

| 事件类别 | 最低记录字段 | 处置要求 |
| --- | --- | --- |
| 登录 | 时间（UTC）、账号、来源 IP、认证方式、成功/失败、会话标识。 | 失败异常告警；定期审阅成功登录。 |
| 权限变化 | 操作人、目标账号/组、前后权限、sudoers 变更、审批编号。 | 高风险权限变更须双人复核。 |
| 配置变化 | 文件/对象、变更前后版本或校验和、执行人、工具、审批编号、结果。 | 与配置备份和变更报告关联。 |
| Agent 执行 | Agent 身份、任务/Artifact 版本、目标、授权范围、开始/结束时间、命令类别、结果与输出摘要。 | 未授权、越权或失败执行应告警并保留证据。 |

### 8.2 审计保护与复核

- 业务/Agent 日志和系统安全日志分别存储；Agent 只能写入自己的运行日志，不能删除或改写系统审计记录。
- 关键操作须将审批编号贯穿变更、执行、日志和报告，保证可追溯性。
- 安全负责人至少每月复核登录、权限、规则变更和异常 Agent 执行；高风险告警应按事件响应时限处理。
- 日志查询权限按需授予；任何导出包含敏感元数据的日志都须脱敏、审批并记录。

## 九、实施流程

> 以下是未来获批实施时必须遵循的流程，不是本文件授权的执行清单。

1. **检查：** 核对资产信息、Ubuntu 版本、当前监听端口、现有账号/组、SSH 可用性、云安全组、磁盘容量、时间同步、更新状态及变更冲突；确认目标确为 `control.vafox.com`（公网 `114.132.55.178`、内网 `172.16.16.6`）。
2. **备份：** 导出并加密保存变更前 SSH/UFW/sudoers/审计/日志/系统配置及账号权限清单，记录校验和与恢复位置；确认备份可读取。
3. **加固：** 在审批窗口内按最小可逆变更顺序实施：创建并验证实名管理员密钥访问 → 配置 sudo 最小权限 → 应用 SSH 基线 → 应用 UFW 默认拒绝与最小放行 → 配置更新、时间、日志和审计。每一步均须留存输出摘要和责任人。
4. **验证：** 使用独立会话验证管理员 SSH 密钥登录、root/密码登录拒绝、UFW 预期规则、必要端口、时间同步、更新服务、日志持久化、审计事件和备份恢复读取；严禁仅凭配置文件存在即判定成功。
5. **报告：** 形成实施报告，关联审批号、变更版本、验证证据、例外项、风险、遗留事项、回滚是否触发及验收签字。未通过验收时不得宣布上线。

### 9.1 变更闸门

- **实施前：** 方案评审、安全审批、维护窗口、回滚负责人和应急联络人齐备。
- **实施中：** 任一 SSH 可用性、规则一致性或日志审计验证失败，立即停止后续变更并进入回滚评估。
- **实施后：** 安全负责人和系统负责人共同确认验收；未关闭的高风险例外须有书面风险接受和到期整改计划。

## 十、回滚方案

### 10.1 通用原则

回滚优先恢复**受控管理访问**和**已知安全配置**，不得为图省事而永久开放公网 SSH、启用密码认证或重新允许 root 远程登录。回滚仅由经授权人员在事故/变更流程内执行，并完整记录原因、时间、操作人和恢复结果。

### 10.2 SSH 恢复

1. 保持当前已验证会话不退出，使用第二个独立会话验证新管理员密钥登录。
2. 若新 SSH 配置造成合法管理员无法访问，依据变更前备份通过云控制台/带外控制台恢复最后已知可用的 SSH 配置；该访问本身须受 MFA、审批和审计保护。
3. 恢复后只开放登记的管理来源，验证非 root、密钥认证可用；再次确认 root 和密码认证仍被禁止。任何临时例外必须设置明确到期时间并立即生成整改工单。

### 10.3 配置恢复

- 从已校验、加密的变更前配置快照恢复受影响的单一配置对象，优先最小范围恢复，避免覆盖变更窗口内的其他已批准修改。
- 恢复前核对备份版本、时间、校验和和适用系统版本；恢复后执行语法校验、服务健康检查和审计验证。
- 所有恢复动作须产生新的审计记录和变更报告，不得删除失败变更的证据。

### 10.4 防火墙恢复

- 若 UFW 规则导致管理访问中断，经带外控制台恢复到变更前已备份的最小规则集，并立即验证 SSH 仅对登记来源开放。
- 不得以“临时全放通”作为常规回滚手段；如紧急处置确需短时例外，须记录原因、责任人、范围和自动/人工关闭时间，并在恢复后复核监听端口与云安全组一致性。
- 防火墙恢复完成后，验证 HTTP/HTTPS 仅符合已批准设计，所有未授权端口继续被拒绝。

## 十一、安全验收标准

以下项目必须全部通过，或对例外获得书面风险接受、责任人和到期整改计划后，Control Plane 才可进入下一阶段部署评审：

| 编号 | 验收标准 | 证据 |
| --- | --- | --- |
| AC-01 | 系统资产、IP、域名、责任人和用途已登记，且与本方案一致。 | 资产清单与审批记录。 |
| AC-02 | root SSH 远程登录被拒绝；密码和键盘交互式认证被拒绝。 | 独立测试记录与 SSH 配置审查。 |
| AC-03 | 至少两条实名管理员 SSH 密钥登录路径已验证；密钥已登记且来源地址受限。 | 登录测试、密钥登记和 UFW 规则。 |
| AC-04 | 管理员以普通账号登录；sudo 最小权限已审查；无共享/通用管理账号。 | 账号、组、sudoers 审查记录。 |
| AC-05 | UFW 默认拒绝入站；仅 SSH（受限来源）、HTTP、HTTPS 按批准规则放行；无未授权监听端口。 | UFW、云 ACL 与端口扫描结果。 |
| AC-06 | 自动安全更新、时间同步、持久日志、日志轮转和审计服务均正常，异常可告警。 | 服务状态、时间偏差、日志与审计样例。 |
| AC-07 | 登录、权限变化、配置变化和 Agent 执行的审计字段完整，且日志受限访问并可留存。 | 审计查询样例、权限审查和留存策略。 |
| AC-08 | Secret Manager/KMS 路径已批准；仓库、配置、日志和备份中未发现明文 Secret。 | Secret 设计审查、扫描报告和抽样检查。 |
| AC-09 | 配置与安全配置备份完整、加密、可读且已完成恢复演练。 | 备份清单、校验和和演练报告。 |
| AC-10 | SSH、配置和防火墙回滚步骤已演练或在等价环境验证，恢复负责人已确认。 | 回滚演练记录。 |
| AC-11 | 本阶段未安装/启用未审批的 Docker、Nginx、业务服务或 Agent；下一阶段部署须另行审批。 | 软件清单、变更记录和验收签字。 |

## 十二、禁止事项

以下行为在第一阶段及后续 Control Plane 运维中明确禁止：

1. **禁止直接生产修改。** 未经方案、审批、维护窗口、备份和回滚准备，不得直接修改生产服务器。
2. **禁止绕过审批。** 不得以紧急、测试、临时排障或个人授权为由绕过既定变更和安全审批；紧急变更须走紧急流程并事后复盘。
3. **禁止密码共享。** 不得共享 SSH 私钥、账号密码、Token、API Key、MFA 恢复码或任何 Secret。
4. **禁止长期使用 root。** 不得以 root 作为日常登录、日常开发、应用运行、Agent 运行或自动化执行身份。
5. 禁止启用 root SSH、密码 SSH、未受限的公网 SSH 或 `NOPASSWD: ALL` 作为便利性措施。
6. 禁止将 Secret 明文写入代码、配置、日志、备份、工单、截图或聊天记录。
7. 禁止在未完成本方案验收前安装或暴露 Docker API、Nginx 管理接口、数据库、缓存、监控面板、Agent 控制端口或任何业务服务。
8. 禁止删除、关闭、篡改或规避审计日志、备份、告警和审批证据。

## 结论：Control Plane 安全上线标准

`control.vafox.com` 的第一阶段上线标准不是“服务器可连接”，而是“管理访问最小化、网络默认拒绝、权限与 Secret 分离、日志审计可追溯、配置可恢复，并且全部证据通过验收”。只有在本方案的安全验收标准完成、例外受控且获得正式批准后，才可为 Docker、Nginx、Control Plane 服务和 Agent 制定下一阶段的独立部署方案。
