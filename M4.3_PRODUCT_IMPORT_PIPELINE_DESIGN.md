# VAFOX M4.3 产品知识导入流水线设计

**状态：** Design only / 不部署。  
**范围：** 将受控产品资料转化为可审核、可引用、可检索的产品知识，并向只读零售事实和推荐服务提供输入。  
**非目标：** 本文不实现服务、迁移数据或改动生产环境；不修改 SAP、Core 生产同步或商品主数据，也不自动下单或自动销售。

## 1. 设计原则与边界

1. **来源优先。** 每个已发布产品属性、标签、卖点或比较结论都必须能追溯到版本化来源和精确 Citation；未知信息保持未知。
2. **人工发布。** 自动提取只产生候选实体和候选标签；只有审核通过的版本才能进入知识库、零售联动或推荐候选。
3. **主数据分离。** SAP SKU、库存及销售速度是权威零售系统的只读事实；产品知识不复制为主数据、更不回写。
4. **授权优先。** 导入身份、组织、部门、角色和可见性由可信服务端身份上下文派生，文件元数据和客户端参数只能收窄，不能放大权限。
5. **建议不执行。** 推荐仅供员工确认，不构成价格、库存、履约、订单或销售承诺。

### 明确禁止

- 修改 SAP；
- 修改 Core 生产同步、同步规则或来源事实；
- 自动下单、锁定库存、调拨、补货或改价；
- 自动销售、自动触达客户或替员工完成成交动作。

## 2. Product Data Flow

```text
PDF / DOCX / Excel (.xlsx) / Markdown
                 ↓
      Extract（提取、分段、定位）
                 ↓
 Product Entity（候选实体与属性）
                 ↓
       SKU Mapping（只读映射校验）
                 ↓
    Scenario Tag（受控场景标注）
                 ↓
 Citation（来源、位置、哈希、血缘）
                 ↓
 Knowledge Base（审核后索引与检索）
```

### 2.1 分阶段处理契约

| 阶段 | 输入与处理 | 输出 | 失败与控制 |
| --- | --- | --- | --- |
| Receive | 接收文件及可信 `AuthContext`；交叉验证扩展名、MIME、大小、恶意内容/加密状态。 | `import_id`、原文件对象、导入审计。 | 损坏、加密不可审查、超限、宏文件或不支持格式进入 `quarantined`。 |
| Extract | PDF 按页、DOCX 按标题/段落/表格、Excel 按工作表/单元格范围、Markdown 按标题/行提取。 | 标准化段落与不可变 locator。 | 无文本、OCR/解析失败或无法定位时不生成可发布知识。 |
| Entity | 规则/模型从片段提出产品、品牌、系列、类目、规格与增强字段候选。 | `ProductEntityDraft` 和字段级证据引用。 | 不允许把模型推断当作事实；冲突或置信度不足标记待审。 |
| SKU Mapping | 以已批准的 SAP 商品目录快照查询 SKU；解析来源中的 SKU 仅为候选键。 | `verified`、`unresolved` 或 `ambiguous` 映射。 | 未映射或多对一冲突不得发布为可售 SKU。 |
| Scenario Tag | 使用第 5 节枚举标注活动、用户与环境，并记录适配/限制。 | 可审核 `ScenarioTag[]`。 | 自由文本、无依据的安全适配或不合法枚举被拒绝。 |
| Citation | 为每一主张绑定来源版本、hash、位置和提取血缘。 | Citation 集合及完整性检查。 | 无 primary Citation、hash 或 locator 的项目不能批准。 |
| Review & Publish | 审核实体、SKU 状态、场景、引用和 ACL；批准版本写入知识读模型和索引。 | `approved` 知识版本。 | `draft/in_review/rejected/deprecated` 均不可作为默认检索或推荐输入。 |

导入生命周期为 `received → extracted → mapped → in_review → approved`，异常分支为 `quarantined` 或 `rejected`。每次重导入创建新版本，保留原件 hash、解析器版本、审核记录和版本间的替代关系；不得静默覆盖已批准实体。

## 3. Product Entity 与 SKU Mapping

### 3.1 实体层级

```text
SAP SKU（权威、只读）
        ↓ maps_to
Product Entity（知识中的稳定产品实体）
        ↓ belongs_to
Brand（品牌）
        ↓ has_series
Series（系列）
        ↓ classified_as
Category（类目）
```

`Product Entity` 是知识层稳定 ID，例如 `prd:kailas:mont-x-shell`；SKU 是售卖变体/颜色/尺码等零售标识，例如 `sap:sku:KG2131101-BLK-M`。一个产品实体可关联多个 SKU，SKU 在同一销售组织、渠道及有效期内最多映射一个当前产品实体。`brand`、`series`、`category` 是实体属性，不允许用 SKU 显示名称反向覆盖。

| 对象 | 必填字段 | 规则 |
| --- | --- | --- |
| `ProductEntity` | `product_id`、`product_name`、`brand_id`（无品牌须显式 unresolved）、`category`、`status` | 产品知识的聚合根；含生命周期、版本、ACL 和 Citation。 |
| `SkuMapping` | `sap_sku`、`product_id`、`source_snapshot_id`、`mapping_status`、`effective_from` | 只读引用 SAP 目录快照；状态为 `verified/unresolved/ambiguous/deprecated`。 |
| `Brand` | `brand_id`、名称 | 复用稳定品牌 ID，而非自由文本名称。 |
| `Series` | `series_id/name`、`brand_id` | 系列隶属品牌，可为空或 `unresolved`。 |
| `Category` | 受控 `category_code`、分类路径 | 示例：`footwear/hiking_boot`、`apparel/shell_jacket`、`equipment/tent`。 |

SKU 映射校验顺序：先以 `sap_sku + organization/channel + as_of` 精确匹配目录快照，再核对产品型号、变体属性和有效期，最后由审核者确认。映射响应必须携带目录快照时点；`unresolved`/`ambiguous` SKU 可以保留在草稿审计中，但不得作为库存查询或推荐的可售候选。

## 4. Knowledge Enhancement 模型

以下字段作为 `ProductEntity` 的受控知识增强；每项均含 `evidence_refs[]`，并区分“事实”“解释”“建议”。

| 字段 | 最小结构与约束 |
| --- | --- |
| 技术特点 `technical_features` | `{name, mechanism, benefit, limitation, evidence_refs}`；机制或规格不可用营销语臆测。 |
| 使用场景 `scenarios` | 第 5 节 `ScenarioTag`，声明 `fit=suitable/conditional/not_suitable` 与限制。 |
| 用户画像 `user_profiles` | `{level, needs, exclusions?, evidence_refs}`；仅匿名能力画像，不接收 PII。 |
| 销售卖点 `selling_points` | `{claim, audience, condition, evidence_refs}`；必须附适用条件，不得保证结果。 |
| 竞品区别 `competitive_differences` | `{competitor_ref?, dimension, statement, evidence_refs}`；仅比较可证实维度，禁止无证据贬损。 |
| 搭配关系 `pairings` | `{relation: complements/compatible_with/alternative_to, target_product_or_category, condition, evidence_refs}`；不是捆绑销售或自动加购。 |

推荐和展示时，系统必须同时返回主张的 Citation、限制条件及知识版本；若产品规格、竞争结论或场景存在冲突，降低置信度并要求人工核实，而不是合并为单一“事实”。

## 5. Scenario Taxonomy

`ScenarioTag` 是检索和匹配的共同语言。活动、用户和环境都使用下表受控值；允许在 `notes` 保留审核后的补充说明，但不能用自由文本替代枚举。

| 维度 | 受控值 |
| --- | --- |
| 活动 `activity` | `hiking`（徒步）、`mountaineering`（登山）、`camping`（露营）、`skiing`（滑雪）、`urban_outdoor`（城市户外）、`high_altitude`（高海拔）。 |
| 用户 `user_level` | `newcomer`（新人）、`intermediate`（进阶）、`professional`（专业）。 |
| 环境 `environment` | `shenzhen`（深圳）、`western_sichuan`（川西）、`tibet`（西藏）、`snow_mountain`（雪山）。 |

```json
{
  "activity": "hiking",
  "environment": ["shenzhen"],
  "user_level": ["newcomer", "intermediate"],
  "difficulty": "easy",
  "fit": "conditional",
  "requirements": ["breathability", "rain_protection", "hydration"],
  "limitations": ["出发前由员工确认天气、尺码和路线"],
  "evidence_refs": ["cit:source-01:page-3"]
}
```

`difficulty` 使用 `easy|moderate|challenging|expedition`，表示行程条件而非客户能力。高海拔场景不自动推断用户为专业级：海拔、天气、路线、时长或经验缺失时应降低匹配置信度，并输出人工安全确认要求。

## 6. Gear Package Model

`GearPackage` 是经审核的装备建议模板，既不是 SKU 套装，也不保留库存或触发促销。字段如下：

| 字段 | 定义 |
| --- | --- |
| `package_id`, `name` | 稳定 ID 和名称。 |
| `activity` | 必填，采用场景活动枚举。 |
| `environment` | 必填数组，采用环境枚举。 |
| `difficulty` | 必填行程难度。 |
| `products` | 必填数组 `{role, product_id? / category, quantity, required, alternatives[]}`；替代项必须有兼容依据。 |
| `budget` | `{currency, min, max, basis}`；为预算区间，不是实时价格。 |
| `reason` | 必填数组，说明需求覆盖、限制与每一条 Citation。 |

| 模板 | activity / environment / difficulty | products（角色级） | 预算与理由 |
| --- | --- | --- | --- |
| 深圳周末徒步 | `hiking` / `shenzhen` / `easy` | 透气层、轻量雨具、日用背包、补水容器、防滑徒步鞋 | 以人民币区间和当前目录价格核实；覆盖湿热、短途与降雨风险。 |
| 川西7天 | `hiking` / `western_sichuan` / `moderate` | 分层保暖、硬壳、徒步鞋、40L 级背包、睡眠/补水角色 | 预算随季节和路线调整；覆盖多日、温差和多变天气。 |
| 冈仁波齐转山 | `high_altitude` / `tibet` / `challenging` | 保暖层、防风雨层、支撑鞋、遮阳/补水、应急角色 | 仅为装备建议；必须人工确认高反风险、天气、路线和专业安全要求。 |

## 7. Citation 与知识库发布

每条 Citation 至少包含：`citation_id`、`document_id`、`source_version`、`content_sha256`、`locator`、`chunk_id`、`claim_path`、`captured_at`、`extractor_version` 和上游血缘。定位格式示例：PDF `page=12&chars=120-380`；DOCX `heading=Care&paragraph=4`；Excel `sheet=Shells&range=B12:F12`；Markdown `heading=规格&lines=42-55`。

批准门槛为：有效的主来源、可解析 locator 与 hash、合法受控字段、可审计 ACL、实体重复检查、SKU 映射状态和审核人决定。发布到 Knowledge Base 的索引载荷仅包含批准且有效版本的最小检索字段、ScenarioTag、实体 ID、Citation 指针和 ACL 过滤字段；原始文件与受限正文仍由来源存储按 ACL 保护。

## 8. Retail Integration

```text
已批准 Product Knowledge + Citation
                ↓ product_id
         SKU Mapping（只读）
                ↓ sap_sku
   库存 / 渠道可售 / 库存时点（只读）
                ↓
     销售速度（只读聚合，含时间窗口）
                ↓
推荐（理由、证据、鲜度、人工确认）
```

Retail Integration 只读取与调用者组织和门店范围相符的目录快照、库存、渠道可售与销售速度。销售速度应返回明确窗口（如 `last_7d`）和计算时点，不能被解释为需求预测或售罄承诺。推荐排序可使用库存可售、数据鲜度、销售速度和预算作为信号，但必须先满足产品知识场景约束和 Citation 完整性。

返回的推荐理由分开标示：产品知识事实、用户显式需求、零售库存事实和销售速度信号。库存缺失、陈旧、越权或依赖不可用时，`inventory_status` 为 `verification_required|stale|unavailable`，并抑制“有货”结论；系统绝不预留、扣减或修改库存。

## 9. Import、Query 与 Recommendation API

所有接口使用认证身份、服务器派生 ACL、`X-Correlation-Id` 和 RFC 3339 UTC 时间。导入接口为受控后台权限；查询/推荐为只读。客户端不得提交或覆盖组织、所有者、可见性、角色范围或零售事实。

| 接口 | 目的 | 请求重点 | 响应与控制 |
| --- | --- | --- | --- |
| `POST /api/product-imports` | 创建产品资料导入任务。 | multipart `file`；可选受控 `source_version`、`brand_hint`；身份来自网关。 | `202` + `import_id/status=received`；仅 `product_knowledge_importer` 或审核角色可用；不直接发布。 |
| `GET /api/product-imports/{import_id}` | 查看本组织授权范围内的导入状态和错误。 | 路径 ID。 | 阶段、候选数量、隔离原因、版本与审核状态；不泄露未授权文件。 |
| `POST /api/product-imports/{import_id}/review` | 提交审核决定。 | `approve/reject`、字段修正、理由。 | 新知识版本或拒绝记录；须 reviewer 权限与审计。 |
| `GET /api/products` | 产品查询。 | `query? brand_id? category? activity? environment? user_level? sku? limit? cursor?`。 | ACL 过滤后的实体摘要、SKU 映射状态、场景、Citation、知识鲜度。 |
| `GET /api/products/{product_id}` | 产品详情查询。 | `include=features,scenarios,pairings,citations`。 | 已授权批准版本、字段级 Citation 与限制；无权时不泄露存在性。 |
| `GET /api/products/recommendations` | 基于显式场景的只读推荐。 | `activity` 必填；`environment? difficulty? user_level? budget? store_id? channel?`。 | 候选、装备组合、理由、Citation、库存/销售速度时点、限制和 `requires_human_approval=true`。 |

错误采用：`400 INVALID_REQUEST`、`401 UNAUTHENTICATED`、`403 POLICY_DENIED`、`404 NOT_FOUND`（仅安全时）、`409 KNOWLEDGE_OR_MAPPING_STALE`、`422 REVIEW_REQUIRED`、`503 RETAIL_DEPENDENCY_UNAVAILABLE`。导入接口不接受 SAP 写入指令；推荐接口不接受客户原始 PII，也不产生订单、报价或销售动作。
