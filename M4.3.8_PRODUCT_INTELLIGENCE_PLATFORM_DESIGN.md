# VAFOX M4.3.8 Product Intelligence Platform Framework

**状态：** Design Phase / 仅设计，不部署。
**前置设计：** `M4.3_PRODUCT_KNOWLEDGE_DESIGN.md`、`M4.3.2_KNOWLEDGE_ASSET_CENTER_DESIGN.md`、`M4.3.3_PRODUCT_INTELLIGENCE_DESIGN.md`、`DATA003.2_RECOMMENDATION_ENGINE_DESIGN.md`。
**目标：** 将已治理的多品牌产品资产、产品知识、场景匹配、推荐能力与零售经营信号组织为跨品牌的 Product Intelligence Platform（PIP），向 Retail Brain 与 Employee AI Assistant 提供只读、可解释、可引用、受权限控制的决策支持。

## 1. 设计原则、范围与边界

1. **跨品牌统一，品牌边界不丢失。** 平台以统一产品模型支持检索、比较和场景匹配；每一项品牌资料、主张和 ACL 都保持所属 `brand_id`、来源和可见范围。
2. **先证据、后智能。** 仅 `Published`、有效、ACL 可见且带 Citation 的产品主张可进入产品智能、产品卡和推荐理由；AI 只能生成候选提取、摘要和草稿。
3. **事实不搬家。** SAP/ERP、POS、WMS、CRM 和 Core 仍分别是商品、库存、销售和客户事实源。PIP 保存受治理的知识实体、只读 ID 映射和时点化引用，不写回来源系统。
4. **建议不执行。** 所有结果均带 `advisory_only_requires_human_approval=true`。员工必须复核商品变体、尺码、价格、实时库存、天气/路线和客户实际需求后再沟通。
5. **最小权限与最小数据。** 推荐只接收 DATA-003.2 已通过同意与用途策略的最小化上下文；不保存或返回原始 PII，不以缺失信息猜测客户特征。
6. **可回放、可撤回。** 每个发布版本、推荐响应和经营建议均记录规则/模型版本、输入快照引用、Citation、数据时点、置信度和限制；归档或撤销后不得继续作为默认依据。

### 明确边界

本设计**只设计，不部署**，并明确禁止：

- 修改 SAP，或修改 Core 生产同步、同步规则、来源事实与生产数据；
- 自动采购、自动下单、自动补货、自动调拨、锁定或扣减库存；
- 自动销售、自动报价、自动改价、自动触达客户或替员工完成交易；
- 将 AI 生成内容、Draft/Review 内容、无 Citation 内容或越权内容发布为产品事实。

## 2. Platform Architecture

```text
Brand Asset
    ↓  extraction / metadata / citation / review
Product Knowledge
    ↓  approved claims / entity resolution / versioning
Product Intelligence Engine
    ↓  product cards / comparison / package candidates
Scenario Engine
    ↓  activity / environment / customer-level matching
Recommendation Engine (DATA-003.2)
    ↓  customer policy + inventory and sales snapshots
Retail Brain
    ↓  store and operating suggestions (read-only)
Employee AI Assistant
```

| 层级 | 核心职责 | 输入 | 输出 | 强制控制 |
| --- | --- | --- | --- | --- |
| Brand Asset | 接收与保存品牌、采购和运营资料原件。 | PDF、DOCX、PPT/PPTX、Excel、Markdown、Image。 | `asset_id`、内容 hash、原件版本、ACL、解析任务。 | 不因上传即成为知识或事实。 |
| Product Knowledge | 将审核后的证据组织为品牌、系列、产品、技术、场景和组合关系。 | 已解析资产与人工标注。 | 版本化 Product Entity、字段级 Citation。 | 仅 Published、有效、ACL 可见内容可检索。 |
| Product Intelligence Engine | 聚合产品知识为跨品牌的可解释读模型。 | Product Entity、比较规则、组合模板。 | AI Product Card、差异比较、候选搭配。 | 不保存实时库存，不生成交易指令。 |
| Scenario Engine | 将需求翻译为受控场景约束并执行匹配。 | 活动、环境、客户等级、显式限制。 | 硬过滤、适配分、补问项、风险提示。 | 高风险场景不作安全保证；未知不臆测。 |
| Recommendation Engine | 按 DATA-003.2 进行策略门控、库存校验和排序。 | 最小授权 Customer Context、产品候选、Retail snapshots。 | 可解释的推荐/替代建议。 | 库存过期、越权或未知时降级或抑制。 |
| Retail Brain | 结合销售、库存、门店数据生成经营洞察。 | 已批准知识及只读聚合业务信号。 | 门店/品类/生命周期经营建议。 | 只读分析；不触发采购、调拨或销售。 |
| Employee AI Assistant | 在权限内呈现产品答案、卡片与建议。 | 上述带 Citation 的响应。 | 员工可审阅的辅助内容。 | 不展示越权来源；人工确认后才可对客沟通。 |

PIP 作为 Knowledge Brain 与 DATA-003.2 之间的产品知识供给层：它不替代 Customer Brain 的同意决策，也不替代 Retail Brain 对实时可售状态的验证。

## 3. Universal Product Schema

`UniversalProduct` 是跨品牌的**可发布产品知识视图**，不是 SAP 商品主数据，也不承载实时库存、价格或原始客户信息。字段名保持统一；多值属性使用数组；所有可发布产品主张必须关联 `citation[]`。

| 字段 | 类型 / 最低规则 |
| --- | --- |
| `product_id` | 必填、稳定、不可复用的产品实体 ID；关联外部 SKU，不替代 SKU。 |
| `brand_id` | 必填的受控品牌 ID；无法确认时为 `unresolved`，不可发布。 |
| `series` | `{series_id?, name, evidence_refs[]}`；必须属于 `brand_id`。 |
| `category` | 受控品类 ID/层级，如 footwear、shell、pack；支持多级分类。 |
| `model` | `{name, model_code?, variant_scope?}`；颜色、尺码等变体不应混入稳定产品身份。 |
| `technology` | `[{name, mechanism?, benefit?, limitations[], evidence_refs[]}]`；机制与性能不可从营销语推断。 |
| `specification` | `[{name, value, unit?, condition?, evidence_refs[]}]`；保留单位、适用条件与原文证据。 |
| `scenario` | `ScenarioFit[]`：活动、环境、难度、适配状态、需求、限制及 Citation。 |
| `customer_profile` | 匿名适配画像（等级、目标、偏好、排除项）；绝不存放 PII 或客户记录。 |
| `selling_points` | `[{claim, audience?, condition?, claim_kind, evidence_refs[]}]`；区分产品事实、培训话术与建议。 |
| `competitors` | `[{competitor_product?, dimensions[], observed_at, evidence_refs[]}]`；仅同维度、可验证、带观察日期的比较。 |
| `gear_package` | `[{package_id, role, required?, alternatives[], condition?, evidence_refs[]}]`；是角色级搭配规则，不是套装交易。 |
| `citation` | 字段级 Citation 指针；至少含 asset/version/hash/locator/claim path。 |
| `version` | `{schema_version, entity_version, effective_from, effective_to?}`；发布后不可原地改写。 |
| `status` | `Draft | Review | Published | Archived`；只有 Published 且有效的版本参与默认智能服务。 |

## 4. Product Asset Workflow

```text
Upload
  ↓
AI Extraction
  ↓
Metadata
  ↓
Citation
  ↓
Product Entity
  ↓
Human Review
  ↓
Publish
```

| 阶段 | 处理与产物 | 质量及安全控制 |
| --- | --- | --- |
| Upload | 接收 PDF、DOCX、PPT/PPTX、Excel/XLSX、Markdown 与 Image；生成 `asset_id`、MIME、大小、SHA-256、上传人、来源声明及初始 ACL。 | 病毒/格式/加密/重复检测失败进入隔离区；客户端不可提升 ACL 或来源等级。 |
| AI Extraction | PDF/DOCX/PPT 提取结构化文本；Excel 提取 sheet/cell 范围；Markdown 提取标题/行；Image 通过 OCR 与区域检测生成候选。 | 产物仅为候选；保留解析器版本、置信度和原始坐标。 |
| Metadata | 标注品牌、资料类型、语言、产品范围、有效期、来源等级候选、版本和血缘。 | 受控词表与品牌目录校验；不确定项保留 `unresolved`。 |
| Citation | 为每条候选主张生成不可变定位。 | PDF 页码、DOCX 段落、PPT 页/图形、Excel sheet/cell range、Markdown heading/line、Image 区域坐标均须可回链到 hash。 |
| Product Entity | 进行产品、系列、型号、SKU 映射与冲突检测，形成 Draft 版本。 | 一对多/多对一 SKU 关系显式标记；`ambiguous`/`unresolved` 不进入推荐可售候选。 |
| Human Review | 商品经理/指定审核者校验事实、限制、竞争比较、ACL、来源等级和版本差异。 | AI 不可自行发布、合并冲突、提升来源等级或修改权限。 |
| Publish | 原子发布 Entity 与索引，记录审批人与生效时间。 | 仅 Published、有效、ACL 可见版本服务于搜索、卡片和推荐；保留审计与回滚指针。 |

## 5. Procurement Portal

采购员工作台是 PIP 的受控管理界面；它只管理产品知识与生命周期，不写 SAP/Core，不产生采购或订单行为。

| 功能 | 工作台能力 | 控制 |
| --- | --- | --- |
| 新品上传 | 上传支持格式、选择可信品牌/供应商、查看解析队列和失败原因。 | 上传仅创建 Draft Asset；品牌归属与 ACL 由服务端验证。 |
| 商品资料维护 | 编辑受控 Metadata、补充产品关系、SKU 映射候选、有效期和 Citation 说明。 | 事实字段编辑必须保留证据；变更生成新 Draft 版本。 |
| 产品审核 | 查看原件与定位、差异、冲突、置信度，批准/退回/要求补证。 | 审核人不得审核自身受限事项；所有决定审计化。 |
| 主推设置 | 在品牌/渠道/门店范围内维护“主推候选”及起止时间、业务理由和排序。 | 主推不是自动销售、默认成交或库存承诺；不得绕过场景、安全、预算和库存校验。 |
| 生命周期管理 | 管理 Draft、Review、Published、Archived，查看版本时间线并归档失效资料。 | Archived 内容不可成为默认检索或推荐依据；历史响应仍可引用其当时版本。 |

## 7. AI Product Card

AI Product Card 是 Employee AI Assistant 中的只读产品解释视图。卡片针对已发布产品版本自动生成，但每个区块必须携带事实/建议属性、Citation、Confidence、版本和人工复核事项。

| 区块 | 自动生成内容 | 呈现规则 |
| --- | --- | --- |
| 产品定位 | 品牌、系列、品类、目标需求、适用边界。 | 仅依据发布知识；不写成绝对性能承诺。 |
| 技术特点 | 技术/规格 → 作用 → 条件/限制。 | 技术事实与培训解释分区；每项可追溯。 |
| 适用场景 | 活动、环境、难度、`suitable/conditional/not_suitable` 与需确认项。 | 将高海拔、登山等风险提醒置于醒目位置。 |
| 客户画像 | 匿名适配等级与需求；当前客户只显示授权匹配理由。 | 不显示原始 PII，不因未知画像作推断。 |
| 搭配方案 | 角色、必选/可选、兼容/替代条件。 | 不自动加购、锁货或承诺套装价格。 |
| 销售话术 | 条件化、非贬损、可编辑的话术草稿。 | 不承诺安全、效果、价格、库存或竞品优劣。 |
| Citation | 资产版本、来源等级、定位符、检索时点。 | ACL 不足时不泄露来源正文、标题或 locator。 |
| Confidence | `high|medium|low`、分数、依据、局限和补问项。 | 衡量证据/输入完整性，不是成交概率。 |

卡片固定标注：`库存、价格、尺码及实时场景请由员工核实；本建议需人工确认后使用。`

## 8. Scenario Engine

```text
Product
  ↓
Activity
  ↓
Environment
  ↓
Customer Level
  ↓
Recommendation
```

| 输入层 | 受控值 / 规则 | 输出 |
| --- | --- | --- |
| Product | 已发布产品的品类、技术、规格、场景、限制和组合关系。 | 可匹配候选与不可满足的硬限制。 |
| Activity | `hiking` 徒步、`mountaineering` 登山、`camping` 露营、`skiing` 滑雪、`high_altitude` 高海拔、`urban_outdoor` 城市户外。 | 需求角色、场景过滤条件和风险提示。 |
| Environment | 如湿热、雨、风、雪、地形、海拔、城市近郊；仅使用显式输入或有证据的场景标签。 | 环境约束、需要确认的天气/路线信息。 |
| Customer Level | `newcomer|intermediate|professional|unknown`；仅限显式或授权的匿名画像。 | 复杂度适配与补问项；`unknown` 不增减分。 |
| Recommendation | 已通过硬过滤的产品候选及数据上下文。 | 适配等级、理由、排除原因、置信度和人工下一步。 |

**匹配顺序：** 先排除未发布/失效/越权知识、`not_suitable`、关键安全或兼容冲突、渠道不可售和超出显式预算的候选；再进行可解释排序。高海拔不自动等同专业登山：海拔、路线、天气、天数、体能或经验缺失时，降低置信度并要求人工确认。

```text
scenario_score = 0.40 × activity_fit
               + 0.25 × environment_fit
               + 0.15 × customer_level_fit
               + 0.10 × technology_fit
               + 0.10 × evidence_completeness
               - risk_penalties
```

缺少授权客户信息时，移除 `customer_level_fit` 并重分配至非个人因素；评分版本、各分项和排除理由必须随响应返回。

## 9. Recommendation Interface（DATA-003.2）

```text
Customer
  ↓
Scenario
  ↓
Product Intelligence
  ↓
Inventory
  ↓
Recommendation
```

1. **Customer：** DATA-003.2 先校验组织、门店范围、用途、同意、质量和新鲜度，并只传递最小化的授权上下文。
2. **Scenario：** PIP 以显式活动、环境、预算、客户等级等条件检索 Published 产品知识，并返回硬约束与 Citation。
3. **Product Intelligence：** 生成产品候选、产品卡要点、搭配角色、对比理由、置信度及未知项。
4. **Inventory：** Retail Brain/现有受治理只读事实源在授权 `store_scope` 中验证 SKU、可售状态、时间戳与库存健康；PIP 不查询后写入或预留库存。
5. **Recommendation：** DATA-003.2 排序并返回可回放响应；库存未知/陈旧/隔离时返回“请员工核实”或抑制，而不宣称有货。

响应必须包含：`recommendation_context_ref`、`product_id`、SKU 映射状态、`knowledge_version`、`citations[]`、`inventory_as_of?`、`rules_version`、`confidence`、`limitations[]`、`excluded_candidates[]`、`advisory_only_requires_human_approval=true`。

## 10. Retail Brain Integration

```text
Published Product Knowledge + Citation
       + Sales Data (read-only aggregate)
       + Inventory Data (read-only snapshot)
       + Store Data (authorized scope)
                    ↓
            Retail Brain
                    ↓
       经营建议 / human review queue
```

| 信号 | 允许用途 | 经营建议示例 | 禁止用途 |
| --- | --- | --- | --- |
| 销售数据 | 已授权时间窗内的聚合销量、转化、退换和样本质量。 | 识别某门店在特定徒步场景中知识覆盖不足或主推结构与需求不一致。 | 不自动改价、促销、采购或强制销售。 |
| 库存数据 | 可售状态、库存健康、鲜度、缺货/滞销风险。 | 提示员工核实替代品、提示商品经理审阅场景覆盖缺口。 | 不锁货、扣减、调拨或补货。 |
| 门店数据 | 授权门店范围、区域、陈列/培训状态、服务能力。 | 提示门店补充产品培训、更新陈列资料或核验过期知识。 | 不以门店数据推断客户敏感属性。 |

经营建议必须区分**观察事实**、**推断**与**人工建议**，并返回数据窗口、质量状态、Citation/血缘、规则版本和限制。建议进入人工 Review Queue，不自动执行。

## 11. Version Management

```text
Draft → Review → Published → Archived
```

| 状态 | 允许操作 | 可被搜索/推荐 | 转换控制 |
| --- | --- | --- | --- |
| Draft | 上传、抽取、编辑、补充 Citation、解决映射。 | 否。 | 由创建者/授权编辑者提交 Review。 |
| Review | 审核、评论、退回、要求补证、批准。 | 否。 | 需满足 Citation、ACL、品牌/产品映射与职责分离检查。 |
| Published | 只读服务、生成卡片、参加场景匹配与推荐。 | 是，需同时有效且 ACL 可见。 | 修订必须创建新的 Draft 版本；可被归档或撤销。 |
| Archived | 历史留存、审计与响应回放。 | 否（默认）。 | 不可原地恢复为 Published；需复制为新 Draft 并重新审核。 |

每个版本保存 `entity_version`、`content_hash`、`parent_version?`、`effective_from/to`、审批人/时间、变更原因、规则兼容版本和 Citation 集合。已有推荐保留当时版本引用，后续检索默认使用最新有效 Published 版本。

## 12. Permission System：RBAC + ACL

RBAC 决定角色可执行的能力；ACL 决定某一 Asset、Product Entity、Citation、品牌或门店范围是否可见。两者均必须通过才可读写；搜索筛选只能缩小而不能扩大访问范围。

| 角色 | RBAC 典型权限 | ACL 范围限制 |
| --- | --- | --- |
| 品牌商 | 上传己方资料、维护 Draft、查看己方审核反馈。 | 仅已验证 `brand_id`、授予组织和资产 ACL。 |
| 采购员 | 新品上传、资料维护、SKU 映射候选、提交审核、维护主推草稿。 | 所属组织、授权品牌/品类与采购范围。 |
| 商品经理 | 审核产品事实、场景、组合、主推及生命周期。 | 授权品牌/品类；敏感竞品资料按 ACL 限制。 |
| 运营 | 维护培训/话术资料、查看授权经营建议。 | 授权组织、门店及运营资料范围。 |
| 店长 | 查看本店授权产品卡、库存支持的建议及门店洞察。 | 仅 `store_scope` 与可见品牌。 |
| 员工 | 查询产品卡、场景和推荐，提交人工反馈。 | 本人门店/渠道、用途、客户同意与 Asset ACL 的交集。 |
| 管理层 | 查看跨范围的聚合经营洞察与审计摘要。 | 默认聚合/脱敏；明细仍须显式 ACL。 |

系统额外执行职责分离（例如上传者不得独自批准高风险发布）、组织隔离、品牌 ACL、部门/门店范围、字段掩码、用途/同意策略与审计日志。Citation 继承父资产 ACL；无权者既不可读取正文，也不可通过搜索、计数或 locator 推断其存在。

## 13. API Contract

所有接口由网关认证并从可信 `AuthContext` 获取组织、角色、品牌/门店范围和用途；客户端提交的范围只能缩小请求。响应统一返回 `correlation_id`、`policy_decision`、版本、Citation、置信度与 `advisory_only_requires_human_approval`（适用时）。以下为设计契约，非部署接口。

### `POST /api/product-assets/upload`

**用途：** 创建产品资产 Draft 并触发受控解析流程。支持 `multipart/form-data`（文件 + metadata）或经批准的预签名上传引用。

```json
{
  "brand_id": "brand:example",
  "asset_type": "product_manual",
  "source_version": "2026-Q3",
  "effective_from": "2026-07-01T00:00:00Z",
  "visibility": "brand_and_procurement",
  "file": "<binary: pdf|docx|pptx|xlsx|md|image>"
}
```

**响应：** `202 Accepted`，返回 `asset_id`、`status: "Draft"`、`content_hash`、`accepted_format`、`extraction_job_ref` 和继承后的 ACL 摘要。拒绝越权品牌、不可审查文件或不支持格式；不自动发布。

### `GET /api/product-intelligence/{id}`

**用途：** 读取调用者有权访问的 Published 产品智能视图或指定可见版本。

**响应：** `product_id`、Universal Product Schema、AI Product Card 区块、`citations[]`、`knowledge_version`、`status`、`confidence`、`limitations[]`。若资产已归档、版本无效或 ACL 不满足，返回不泄露资源存在性的授权/不可用结果。

### `GET /api/product-intelligence/search`

**用途：** 在 ACL 范围内搜索已发布产品知识。

**查询参数：** `q?`、`brand_id?`、`category?`、`activity?`、`environment?`、`customer_level?`、`store_scope?`、`status=Published`、`cursor?`、`limit?`。

**响应：** 分页的产品摘要、匹配字段、场景适配、Citation 摘要、版本和限制。搜索不返回 Draft/Review/Archived 默认内容，也不以 facet、命中数或补全泄露受限资产。

### `POST /api/product-intelligence/recommend`

**用途：** 向 DATA-003.2 提供产品候选、场景匹配理由和库存验证输入；最终排序遵从其策略门控。

```json
{
  "recommendation_context_ref": "ctx:rec:01J...",
  "scenario": {
    "activity": "hiking",
    "environment": ["wet", "urban"],
    "customer_level": "unknown",
    "budget": {"currency": "CNY", "max": 2000}
  },
  "store_scope": ["store:shenzhen:001"],
  "channel": "store"
}
```

**响应：** `candidates[]`（`product_id`、SKU 映射状态、硬过滤结果、场景分项、搭配角色、Citation、置信度、限制）、`inventory_validation_request`、`excluded_candidates[]`、`rules_version` 与 `advisory_only_requires_human_approval: true`。接口不执行库存查询写入、下单、报价或销售。
