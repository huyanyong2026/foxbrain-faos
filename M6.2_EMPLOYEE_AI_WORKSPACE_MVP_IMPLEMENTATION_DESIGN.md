# VAFOX M6.2 Employee AI Workspace MVP Implementation Design

**状态：** Implementation Design / 仅设计，不部署。  
**前置设计：** `M6.1_AI_RUNTIME_FOUNDATION_DESIGN.md`、`M5.2_EMPLOYEE_AI_WORKSPACE_DESIGN.md`、`M4.4.2_EMPLOYEE_AI_ASSISTANT_V2_DESIGN.md`、`M4.4.3_EMPLOYEE_AI_ASSISTANT_MVP_IMPLEMENTATION_DESIGN.md`。  
**目标：** 把 Employee AI Workspace 收敛为一个由 AI Runtime 治理的、只读且建议性的员工工作台 MVP。  
**负责人：** Codex（设计开发）；WorkBuddy（环境）；Marvis（管理）；CTO（验收）。

> 本文定义实施边界、模块、服务端契约、权限、审计及试点验证；不部署服务、不迁移数据、不接入生产写操作。所有输出均为 `advisory_only_requires_human_approval: true`：员工必须核实并在获批的权威系统中独立完成任何后续业务动作。

## 1. MVP 架构与实施范围

```text
Employee Portal
      ↓ authenticated request + optional narrowing filters
AI Runtime
      ↓ server-derived identity, policy, evidence and audit envelope
Employee Workspace
      ↓ read-only module orchestration
Business Brain
      ↓ authorized facts / knowledge / aggregate signals
AI Advice
      ↓ editable, cited, human-reviewable recommendation
Employee Action
      ↓ manual verification / follow-up preparation / external approved workflow
Feedback Loop
      ↓ triage and review only
Sales Memory Candidate
```

| 层 | MVP 职责 | 关键实现约束 |
| --- | --- | --- |
| Employee Portal | 呈现任务、机会、客户最小投影、助手、门店摘要、Citation、限制和反馈。 | 不直连 Business Brain；不保存或提交角色、权限、`employee_id`、客户范围或特权上下文。 |
| AI Runtime | 身份派生、权限决策、记忆/知识路由、Citation、审计和反馈入口。 | 复用 M6.1 的服务端门控；无执行工具和写入凭证。 |
| Employee Workspace | 编排模块 API，产生受控的查询计划和建议请求。 | 仅请求 Runtime 已允许的只读数据；UI 筛选器只能收窄结果。 |
| Business Brain | Customer、Product Intelligence、Sales Knowledge、Retail/Sales Memory 的授权读取。 | 每个 Brain 二次执行 ACL、字段策略、版本和时效检查。 |
| AI Advice | 将事实、分析、假设、未知项与建议明确分区。 | 建议必须可编辑、带 Citation/限制/置信度，不能构成承诺或指令。 |
| Employee Action | 员工人工核实、准备问题、记录反馈或前往获批系统操作。 | 不发送、下单、预留、采购、改 SAP/Core；采纳仅是意图记录。 |
| Feedback Loop | 接收反馈，AI 整理为审核候选。 | 反馈不能直接改变生产知识、模型、权限、排名或业务事实。 |

### 1.1 MVP 内外边界

**MVP 包含：** Today Tasks、Customer Opportunities、Customer 只读查询、Product Sales、Scenario、Cross Sell、Follow-up 建议、Store Summary、Runtime 集成、RBAC + ACL、Citation、审计及受审核反馈闭环。

**明确禁止：**

- 自动销售、自动报价、自动成交或自动营销发送；
- 自动下单、购物车创建、库存预留、调拨、补货或自动采购；
- 修改 SAP、SAP 主数据/交易、Core 生产同步、同步规则或生产数据；
- 以建议直接写入 CRM/客户画像、Sales Memory、权限、知识或员工绩效；
- 将实时库存、价格、天气、许可、医疗或现场安全判断伪装为已验证事实。

## 2. Runtime Integration

每个 Workspace 请求均通过 AI Runtime，按如下失败关闭的顺序执行：

```text
Token + purpose + correlation ID
  → Identity Manager 派生 Employee Context
  → Permission Engine（RBAC ∩ ACL ∩ scope ∩ purpose ∩ field policy）
  → Memory Router / Knowledge Router 形成最小检索计划
  → authorized Business Brain read calls
  → Citation Engine 覆盖检查
  → advisory-only response or controlled denial / needs_verification
  → Audit Engine append event
```

| Runtime 组件 | Workspace 集成方式 | 必须返回/执行 |
| --- | --- | --- |
| Identity Manager | 从认证主体、员工目录、组织和有效角色构建短期上下文。 | `identity_snapshot_id`、`employee_id`、组织/门店范围、角色和派生时间。 |
| Permission Engine | 对 API、模块、资源、字段、客户关系、同意和用途逐项判定。 | `authorization_decision_id`、`allow`/`deny`/`masked`/`aggregate_only` 和策略版本。 |
| Memory Router | 仅在授权后路由 Sales/Customer/Product Memory。 | 最小片段、来源引用、保留/同意状态和 `as_of`。 |
| Knowledge Router | 路由 published、effective、未撤销且 ACL 可见的产品、销售、路线知识。 | 知识版本、有效期、限制和 Citation 候选。 |
| Citation Engine | 将事实与建议关联到可见、有效、足够支撑的证据。 | `citation_id`、来源、版本、时间、鲜度、置信度、限制。 |
| Audit Engine | 追加记录请求、授权、路由、建议、Citation、版本和反馈。 | `audit_id` 与 `X-Correlation-Id` 关联；最小化敏感内容。 |

下游查询必须携带经服务端签发的范围令牌/上下文引用，且参数只能是授权范围的子集。任何身份解析、授权、同意、字段过滤、来源可见性或 Citation 覆盖失败均不得回退到未授权检索；应返回受控拒绝或 `needs_verification`。

## 3. Employee Context（仅服务端派生）

`EmployeeContext` 是单请求、短生命周期的策略快照，不是前端可编辑的员工档案。Portal 可传递声明用途和收窄筛选器，但同名身份/范围字段一律忽略。

| 字段 | 类型 | 服务端派生与使用 |
| --- | --- | --- |
| `employee_id` | string | 由认证主体映射的稳定员工引用；绑定任务、客户关系和审计主体。 |
| `store_id` | string[] | 来自员工目录及有效门店/区域授权；请求中的门店只能求交集。 |
| `role` | controlled string[] | `employee`、`store_manager`、`regional_manager`、`executive` 的有效集合。 |
| `permission` | controlled string[] | Runtime 能力上限，例如 `tasks:read`、`advice:generate`、`feedback:write`。 |
| `knowledge_scope` | object | 可见知识域、ACL、发布状态、版本、有效期与许可范围。 |
| `customer_scope` | object | 已分配/授权客户关系、同意、字段规则、用途与保留限制。 |

```json
{
  "employee_id": "emp_01J...",
  "store_id": ["store_chengdu_01"],
  "role": ["employee"],
  "permission": ["tasks:read", "opportunities:read", "customer:read", "advice:generate", "feedback:write"],
  "knowledge_scope": {"domains": ["product", "sales", "route"], "states": ["published"], "effective_at": "2026-07-22T00:00:00Z"},
  "customer_scope": {"relationship": "assigned", "consent_required": true, "fields": "minimal"},
  "policy_snapshot": "pol_01J..."
}
```

## 4. Employee Workspace Modules

### 4.1 Today Tasks

Today Tasks 是员工的人工服务优先级队列，不是自动触达名单、销售 KPI 或工作指令。任务由已授权的客户信号、人工待办、有效期和数据鲜度生成；证据不足时标为待核实。

| 字段 | 规则 |
| --- | --- |
| `task_id` | 不可变任务引用；不暴露内部源主键。 |
| `employee_id` | 仅服务端绑定的当前/授权员工；响应可省略或返回受控引用。 |
| `priority` | `high`/`medium`/`low`/`verification_required`；由可复核规则而非敏感属性推导。 |
| `reason` | 已验证事实、时效、限制和 Citation；不能将模型推测表述为事实。 |
| `suggested_action` | 人工补问、试穿准备、现场核实或待办草稿；没有发送或交易语义。 |
| `status` | `proposed`、`viewed`、`accepted`、`edited`、`dismissed`、`expired`；不等同业务执行完成。 |

### 4.2 Customer Opportunities

机会只面向员工已关联、用途允许、客户同意且 ACL 可见的客户。无权与不存在返回相同的 `404 RESOURCE_NOT_AVAILABLE`，避免枚举。

| 字段 | 规则 |
| --- | --- |
| `customer_id` | 受控客户引用，逐项通过关系、同意、用途、组织和字段 ACL。 |
| `profile` | 最小画像/别名、明确偏好和未知项；不得推断敏感属性。 |
| `history` | 授权购买、装备、互动摘要及 `as_of`；不复制完整 CRM 时间线。 |
| `interest` | 客户明确表达与待确认需求分开标识。 |
| `opportunity_reason` | 可解释的需求、装备缺口、预约或有效期信号，带 Citation。 |
| `next_action` | 人工确认问题、试穿或核实建议；不自动联系、营销或承诺。 |

### 4.3 Product Sales Assistant

助手连接 **Product Intelligence**（产品定位、适配、兼容性、限制）与 **Sales Knowledge**（需求发现、话术、异议、组合规则）。仅使用发布、有效、调用者可见且 Citation 完整的资料。

| 输出 | 内容与限制 |
| --- | --- |
| `product` | 产品/品类定位、适用与不适用边界；型号、尺码、价格和库存需现场核实。 |
| `scenario` | 已确认场景 → 可验证功能 → 条件限制，不夸大绝对性能。 |
| `selling_points` | 引用支持的功能价值点，区分事实与销售表达。 |
| `script` | 可编辑开场、补问、说明和异议处理草稿；不得施压、虚假承诺或自动发送。 |
| `citation` | 每项主张的来源、版本、`as_of`、鲜度、置信度与限制。 |

### 4.4 Scenario Assistant

支持受控标签：**川西、冈仁波齐、徒步、登山、露营**。目的地/活动用于补问和装备角色匹配，不构成路线、天气、许可、医疗、救援或现场安全结论。

| 输出 | 规则 |
| --- | --- |
| `destination` | 已选受控标签及是否由客户确认。 |
| `environment` | 温度、降水、海拔、天数、负重、住宿等已知/未知条件；未知时优先补问。 |
| `equipment` | 按角色列出的已有、缺失、待核实与不适用装备。 |
| `recommendation` | 候选与替代、限制、风险升级提示、Citation；不得替代专业安全判断。 |

### 4.5 Cross Sell Assistant

Cross Sell 是补齐**已确认**装备缺口的人工辅助，不按销售目标、推断偏好或未同意画像发起促销。

```text
existing_equipment → compatibility / coverage check → missing_equipment → bundle_recommendation
```

| 字段 | 规则 |
| --- | --- |
| `existing_equipment` | 客户确认、授权历史或当前会话声明的装备；标注来源状态。 |
| `missing_equipment` | 基于已确认场景和硬约束的角色级缺口；未知不能当作缺失。 |
| `bundle_recommendation` | 可选组合、替代、兼容性/预算/库存待核实项与 Citation；不自动加购、报价或预留。 |

### 4.6 Follow-up Assistant

Follow-up Assistant 只组织员工主动发起的客户服务准备与最小化互动记录，不拥有消息、日历、营销或 CRM 自动执行权限。

| 字段 | 规则 |
| --- | --- |
| `customer_followup` | 受权客户引用、目的、人工确认状态和到期提示；不含自动发送内容。 |
| `interaction` | 渠道类别、时间、结果码、最小化摘要和来源；自由文本先脱敏。 |
| `next_action` | 人工待办/补问草稿及到期时间；员工可编辑或拒绝，需在外部获批系统中自行执行。 |

### 4.7 Store Summary

Store Summary 是店长及以上在授权门店范围内的只读聚合卡片；员工仅见被授权的最小汇总。所有卡片显示窗口、`as_of`、鲜度、范围和不可用原因。

| 卡片 | MVP 内容 | 边界 |
| --- | --- | --- |
| `sales_summary` | 授权门店的销售趋势、品类/时段聚合和异常提示。 | 不替代财务权威报表；不用于自动考核或佣金归因。 |
| `inventory_alerts` | 只读库存异常、覆盖风险、数据时点和需人工核实 SKU。 | 不承诺可售，不触发预留、调拨、补货或采购。 |
| `customer_opportunities` | 授权、去标识化的机会数量、优先级分布和鲜度。 | 下钻客户明细仍逐项执行 ACL、关系和同意检查。 |

## 5. API Contract

**共同约束：** JSON、RFC 3339 UTC、`Authorization`、`X-Correlation-Id`、`X-Purpose` 必填。`employee_id`、角色、权限、门店、知识与客户范围均由服务端派生。每个成功响应包含 `audit_id`、`policy_snapshot`、`as_of`、`advisory_only_requires_human_approval: true`；所有列表分页并限制 `limit=1..50`。

### `GET /api/workspace/tasks`

读取当前主体可见的 Today Tasks。允许查询：`status?`、`priority?`、`store_id?`、`limit?`、`cursor?`；`store_id` 只能收窄派生范围。

```json
{
  "items": [{
    "task_id": "task_01J...",
    "priority": "medium",
    "reason": {"facts": ["已确认徒步兴趣"], "limitations": ["天数待确认"], "citation_ids": ["cit_01"]},
    "suggested_action": {"type": "confirm_context", "text": "人工确认路线、天数与负重"},
    "status": "proposed"
  }],
  "next_cursor": null,
  "audit_id": "aud_01J...",
  "policy_snapshot": "pol_01J...",
  "as_of": "2026-07-22T00:00:00Z",
  "advisory_only_requires_human_approval": true
}
```

### `GET /api/workspace/opportunities`

读取授权 Customer Opportunities。允许查询：`status?`、`store_id?`、`limit?`、`cursor?`。每个条目单独执行客户关系、同意、用途和字段 ACL。

```json
{"items":[{"customer_id":"cst_01J...","profile":{"display_name":"客户 A","unknowns":["预算"]},"history":{"equipment_summary":["徒步鞋"],"as_of":"2026-07-20T00:00:00Z"},"interest":{"confirmed":["川西徒步"],"to_confirm":["露营"]},"opportunity_reason":{"text":"已确认场景但雨具覆盖待核实","citation_ids":["cit_02"]},"next_action":{"type":"human_question","questions":["是否计划露营？"]}}],"next_cursor":null,"audit_id":"aud_01J...","policy_snapshot":"pol_01J...","as_of":"2026-07-22T00:00:00Z","advisory_only_requires_human_approval":true}
```

### `GET /api/workspace/customer/{id}`

返回一个授权客户的最小投影、互动/装备摘要和可见机会。路径 ID 不是搜索接口；无权或不存在均为 `404 RESOURCE_NOT_AVAILABLE`。不返回支付、原始 PII、完整时间线或未同意字段。

### `POST /api/workspace/advice`

生成 Product Sales、Scenario、Cross Sell 或 Follow-up 的只读建议。请求中的 `customer_id`、`product_ref`、`destination`、`activity`、`existing_equipment` 和门店仅作为候选，必须被范围和知识规则验证。

```json
{"advice_type":"scenario","customer_id":"cst_01J...","context":{"destination":"川西","activity":"徒步","existing_equipment":["徒步鞋"]}}
```

响应包含 `facts`、`analysis`、`assumptions`、`unknowns`、`recommendations`、`clarifying_questions`、`citations`、`confidence`、`limitations` 和 `advisory_only_requires_human_approval`。无足够证据时返回 `needs_verification` 而不是补造答案。

### `POST /api/workspace/feedback`

接收对任务、机会、建议或 Citation 的显式反馈。仅允许最小化结构化字段：`audit_id`、`advice_ref?`、`citation_id?`、`outcome`（`accepted|edited|rejected|incorrect|outdated`）、`comment?`。服务端脱敏、恶意内容扫描、关联审计并返回 `feedback_id` 和 `submitted`；不直接写入知识或 Sales Memory。

**错误契约：** `400 INVALID_REQUEST`、`401 UNAUTHENTICATED`、`403 POLICY_DENIED`、`404 RESOURCE_NOT_AVAILABLE`、`409 STALE_CONTEXT`、`422 NEEDS_VERIFICATION`、`429 RATE_LIMITED`、`503 DEPENDENCY_UNAVAILABLE`。错误不得泄露受限资源存在性、数量、名称、规则细节或客户数据。

## 6. 权限模型：RBAC + ACL

最终授权为：

```text
RBAC capability ∩ resource ACL ∩ organization/store scope ∩ customer relationship/consent ∩ purpose ∩ field policy ∩ effective time
```

| 角色 | 默认可用能力 | 默认范围 | 明确限制 |
| --- | --- | --- | --- |
| 员工 | 本人任务、授权客户机会/查询、产品/场景/连带/跟进建议、反馈。 | 本人及已授权门店、已分配且同意的客户最小投影。 | 无跨店、未关联客户、库存明细、采购或管理层数据。 |
| 店长 | 员工能力及门店摘要、授权团队聚合任务/机会。 | 授权门店及团队聚合；客户下钻仍按 ACL。 | 不自动排班、补货、采购或访问其他门店。 |
| 区域经理 | 区域内门店聚合、趋势、库存提示和去标识化机会。 | 有效区域/门店授权范围。 | 默认聚合；不因职位绕过同意或字段 ACL。 |
| 管理层 | 组织/区域经营汇总、趋势、风险和机会概览。 | 组织授权的去标识化聚合范围。 | 无交易权、无客户 PII 豁免、无 SAP/Core 修改权。 |

ACL 对门店、客户关系、知识文档、产品、Citation、审计事件和字段逐资源判定。客户端筛选只能求交；访问审计本身也受角色、范围和保留期控制。

## 7. Citation、审计与反馈闭环

### 7.1 Citation

每个事实、产品主张、库存/销售信号与建议均绑定 Citation：`citation_id`、`source`、`source_type`、`version`、`timestamp`/`as_of`、`freshness_status`、`confidence`、`scope`、`claim_refs`、`limitations`。不可见、过期、冲突或不足以支持主张的来源必须移除相应主张，或将响应降级为待核实。

### 7.2 Audit

Audit Engine 至少记录以下事件，使用追加式、完整性保护的最小化记录：

| 类别 | 最小审计字段 |
| --- | --- |
| 查询 | `audit_id`、关联 ID、主体/员工受控引用、模块/API、用途、时间、筛选摘要、结果码。 |
| 建议 | `advice_ref`、意图、模型/提示策略版本、事实/建议分区、置信度、限制和人工审批标志。 |
| 反馈 | `feedback_id`、关联 `audit_id`/Citation、脱敏状态、结果码、审核状态。 |
| Citation | `citation_id`、来源受控引用、版本、鲜度、可见性和覆盖结果。 |
| 版本 | `identity_snapshot_id`、`policy_snapshot`、知识/工具/模型版本和授权决定 ID。 |

不得把完整客户画像、无关 PII、受限文档正文或原始提示词作为普通审计默认内容；审计查询、导出、更正和删除请求同样必须审计。

### 7.3 Feedback Loop

```text
Employee feedback
      ↓
AI 整理（脱敏、去重、聚类、证据缺口；仅 Draft）
      ↓
审核（业务负责人 + 知识/合规，职责分离）
      ↓
Sales Memory Candidate（版本化候选）
      ↓ approved + published + effective 才可被未来检索
```

状态为 `submitted → triaged → draft → in_review → approved|rejected`。AI 整理不能覆盖原始审计，不能自动批准，不能将反馈写入生产 Sales Memory；批准后仍需发布、ACL、有效期与可回滚控制。

## 8. KAILAS Pilot Validation

试点仅验证员工获得的建议是否有证据、是否在权限范围内、是否明确要求人工核实；不执行销售、营销、订单、库存或任何生产修改。每题至少由业务负责人检查答案事实/建议分区、Citation、版本/鲜度、权限范围、限制和审计回放。

| 测试问题 | 预期 MVP 行为 | 验收证据 |
| --- | --- | --- |
| **MONT 适合什么客户？** | Product Sales Assistant 从 Product Intelligence + Sales Knowledge 给出适用需求、场景、限制和补问；不承诺库存/价格。 | 可见 Citation、知识版本、`as_of`、不适用边界和 `audit_id`。 |
| **第一次川西需要什么？** | Scenario Assistant 先标记路线/季节/天数/露营/负重未知，再给角色级准备候选与安全升级提示。 | 川西/徒步标签、补问、装备角色、限制、路线知识 Citation。 |
| **冲锋衣贵如何解释？** | Sales Assistant 给出可编辑、事实支持的价值说明、需求澄清和替代/预算讨论。 | 产品事实与话术分区、Citation、禁止虚假性能/价格承诺。 |
| **哪些 SKU 需要关注？** | Store Summary 根据授权门店库存/销售快照提示异常 SKU、窗口和数据鲜度，并要求人工核实。 | 门店授权决定、聚合范围、`as_of`、库存 Citation；无补货/采购动作。 |

**负向测试：** 伪造 `employee_id`/角色/跨店 `store_id`、访问未关联客户、请求草稿/过期知识、Citation 不可见、建议自动发送或写 SAP/Core，均必须被拒绝或降级，并形成审计记录。

## 9. 实施阶段与验收清单

| 阶段 | 交付设计 | 退出条件 |
| --- | --- | --- |
| P1 | Runtime 接口适配、Employee Context、公共响应/错误/Citation/Audit 包络。 | 所有请求服务端派生范围且无写工具。 |
| P2 | Tasks、Opportunities、Customer 查询和 Store Summary 只读编排。 | 客户 ACL/同意与聚合范围的正负测试通过。 |
| P3 | Product、Scenario、Cross Sell、Follow-up Advice。 | 输出字段完整、证据充分或正确降级。 |
| P4 | Feedback 整理与审核候选、KAILAS 测试剧本和回放。 | 未审核反馈不影响可检索知识，试点验收可复现。 |

- [x] **Workspace MVP：** 定义 Portal → Runtime → Workspace → Business Brain → Advice → Action → Feedback 架构与只读边界。
- [x] **Runtime Integration：** 定义 Identity、Permission、Memory、Knowledge、Citation、Audit 的门控顺序和契约。
- [x] **Customer Opportunity：** 定义客户最小投影、历史、兴趣、机会理由、下一动作及同意/ACL 控制。
- [x] **Product Assistant：** 定义 Product Intelligence + Sales Knowledge 输入及产品、场景、卖点、话术、Citation 输出。
- [x] **Scenario Assistant：** 覆盖川西、冈仁波齐、徒步、登山、露营及受控装备建议。
- [x] **Cross Sell：** 定义已有装备、缺失装备和组合推荐，禁止自动加购。
- [x] **Store Summary：** 定义销售摘要、库存提示、客户机会及聚合限制。
- [x] **Feedback Loop：** 定义员工反馈 → AI Draft → 审核 → Sales Memory Candidate，禁止自动发布。
- [x] **Permission：** 定义员工、店长、区域经理、管理层 RBAC + ACL + 范围/同意控制。
- [x] **Audit：** 定义查询、建议、反馈、Citation 和版本的最小可回放审计记录。

本设计完成后，任何实现均须保持“服务端派生范围、只读查询、可引用建议、人工核实”四项不变量；违反任一项不得进入部署或生产接入。
