# VAFOX M5.2 Employee AI Workspace Design

**状态：** Design Phase / 仅设计，不部署。  
**前置设计：** `M5.1_DIGITAL_EMPLOYEE_FOUNDATION_DESIGN.md`、`M4.4.2_EMPLOYEE_AI_ASSISTANT_V2_DESIGN.md`、`M4.4.3_EMPLOYEE_AI_ASSISTANT_MVP_DESIGN.md`。  
**负责人：** Codex（设计开发）；WorkBuddy（环境）；Marvis（管理）；CTO（验收）。

## 1. 目标、原则与边界

M5.2 将 M5.1 的受治理 Digital Employee Runtime 设计为面向员工的统一 AI Workspace：在用户实际权限内，按角色把今日待办、客户和销售机会、经营提醒、建议及学习内容组织为可解释、可编辑、可反馈的工作流。工作台只提出建议和人工待办；人类自行在获批准系统中执行业务动作。

1. **运行时先于界面。** 每次读取均由服务端从认证身份派生组织、角色、门店/品牌、用途、RBAC、ACL 和字段脱敏规则；UI 不能扩大范围。
2. **证据与时效可见。** 所有业务卡片标示 `as_of`、freshness、置信度、限制、数据依据及可回放 Citation；无充分依据时仅显示“请核实”。
3. **建议与执行严格分离。** 卡片可以被人工采纳、编辑、拒绝、延后或标记为“人工已完成”，但不会触发对外消息、交易或主数据写入。
4. **最小化、拒绝优先。** 客户数据须同时满足组织、关系、用途、同意和字段 ACL；无权或资源不存在返回同一受控结果，不暴露对象存在性。
5. **反馈须审核。** 反馈是候选信号，不自动改变知识、模型排序、业务事实、权限或员工评价。

### 明确禁止

本设计不部署服务、模型、连接器或生产同步，且禁止 AI：

- 自动执行销售、成交、下单、报价、锁货、预留或调拨；
- 自动营销发送、客户触达或任何外部消息；
- 自动采购、创建采购单、供应商订单或补货；
- 修改 SAP；
- 修改 Core 生产同步、同步规则或生产数据。

## 2. Workspace Architecture

```text
Digital Employee Runtime
  （可信身份、策略快照、只读工具、授权数据范围）
                              ↓
Employee AI Workspace
  （Today Tasks / Opportunities / Alerts / Recommendations / Learning）
                              ↓
Employee Action
  （人工查看、核实、编辑、采纳、拒绝；在批准系统独立执行）
                              ↓
Feedback Loop
  （最小化反馈 → AI 整理 → 人工审核 → 知识增强）
```

| 层 | 职责 | 输入/输出 | 强制控制 |
| --- | --- | --- | --- |
| Digital Employee Runtime | 计算不可篡改的上下文，编排只读 Brain 查询、规则和模型。 | 输入为认证主体、请求用途及策略；输出为授权观察和策略快照。 | allow-list 工具；无销售、营销、采购、SAP/Core 写能力。 |
| Employee AI Workspace | 将观察聚合为角色相关的卡片、队列和学习内容。 | 输出任务、机会、提醒、建议及 Citation。 | 每张卡均为 `advisory_only_requires_human_approval=true`。 |
| Employee Action | 人工审阅、补充事实、选择动作，并在权威系统独立完成事务。 | 输出受控动作状态、理由和可选权威结果引用。 | “完成”固定为 `completed_by_human`，不表示 AI 执行。 |
| Feedback Loop | 归集显式反馈、检测敏感内容、形成候选知识。 | 输出审核队列、质量指标和已审核知识版本。 | 未审核反馈永不进入生产知识或自动动作。 |

### 2.1 Workspace 信息架构

| 区块 | 首页目标 | 卡片共同字段 | 允许交互 |
| --- | --- | --- | --- |
| Today Tasks | 先处理需要人工确认或核实的事项。 | ID、优先级、来源、原因、建议动作、状态、`as_of`、Citation。 | 接受、编辑、拒绝、延后、人工完成、反馈。 |
| Customer Opportunities | 将授权客户的明确机会转成下一步问题或人工跟进候选。 | 客户最小投影、机会理由、下一动作、限制、Citation。 | 查看、创建人工待办、反馈；不发送消息。 |
| Sales Opportunities | 呈现产品/场景/组合的销售辅助候选。 | 产品、场景、匹配依据、组合、置信度、限制、Citation。 | 打开建议草稿、人工核实、反馈。 |
| Business Alerts | 聚焦库存、销售、商品、门店与风险信号。 | 严重度、影响范围、数据窗口、原因、建议动作、Citation。 | 确认、静默（理由+期限）、创建待办。 |
| AI Recommendations | 将跨模块判断解释为可选择的方案。 | 问题、数据依据、AI 判断、替代方案、建议动作、确认状态。 | 采纳、编辑、拒绝、要求核实。 |
| Learning Center | 提供每日短学习和与当前工作相关的知识复习。 | 学习主题、内容版本、Citation、完成状态。 | 开始、完成、反馈知识问题。 |

## 3. Today Tasks Module

### 3.1 AI 任务生成与生命周期

```text
授权观察 / 已发布规则 / 人工请求
             ↓ 去重、鲜度、Citation 与范围门控
AI 任务候选（原因 + 建议动作 + 优先级）
             ↓ 人工确认或编辑
人工行动与结果反馈
             ↓ 审核式学习
```

任务来源只能是授权数据观察、已发布知识/规则或用户显式请求。优先级由影响、紧迫性、数据新鲜度、置信度和风险策略组成，不得使用个人绩效、佣金、敏感属性或不可解释推断。

| 字段 | 设计 |
| --- | --- |
| `task_id` | 稳定任务 ID，保留去重键及关联建议 ID。 |
| `task_source` | `customer_signal`、`sales_signal`、`business_alert`、`recommendation`、`human_created`；附来源引用。 |
| `priority` | `P0`–`P3`；必须给出 `priority_reason` 与影响/截止因素。 |
| `reason` | 已观察事实、规则/知识依据、未知项和限制分区展示。 |
| `suggested_action` | 可由人执行的核实、沟通准备、复盘或升级动作；不可为自动执行命令。 |
| `status` | `proposed → accepted|edited|rejected|snoozed → completed_by_human|expired`。 |
| `feedback` | `accepted`、`edited`、`rejected`、`insufficient_context`、`citation_issue` 等受控值及可选最小化说明。 |

任务创建后按 `scope + task_source + subject_ref + action_type + time_window` 去重；资料过期、引用撤销、同意撤回或权限失效时，任务应抑制、失效或从视图移除。

## 4. Customer Opportunities

客户机会仅在客户同意、员工关系、目的限制和字段 ACL 全部满足时显示；否则返回聚合机会或不返回条目。购买历史与装备资料表达为“已授权历史/待确认”，不能推断客户仍拥有、喜欢或需要某件商品。

| 客户机会卡字段 | 设计 |
| --- | --- |
| `customer_id` | 授权范围内的稳定 ID；界面可使用受控显示名，日志优先记录引用/哈希。 |
| `customer_profile` | 最小画像：已确认偏好、经验、预算/场景声明及其来源状态；未知保持 `unknown`。 |
| `purchase_history` | 授权购买摘要、时间窗口、类别/产品引用与 `as_of`；非当前意愿证据。 |
| `activity_interest` | 客户明确表示或获授权记录的活动兴趣，含确认时间与置信度。 |
| `equipment_profile` | 已确认、历史购买、待确认的装备分别列示；不自动认定适配或完整。 |
| `opportunity_reason` | “观察事实 → 已发布规则/知识 → 机会假设”，附限制和 Citation。 |
| `next_action` | 人工可执行的开放式确认问题、需求核实或跟进待办；不含自动营销发送。 |

## 5. Sales Opportunities

销售机会把已发布产品知识、Sales Knowledge、授权客户需求和只读零售快照组合为候选；价格、尺码、实时库存、促销、政策和安全结论均须人工在权威系统核实。

| 字段 | 设计 |
| --- | --- |
| `product` | 产品/SKU 候选、适用角色、限制、产品知识版本及 Citation；不能承诺可售。 |
| `scenario` | 已确认或待确认的活动、环境、时长、负重等标签；未知项显式展示。 |
| `customer_match` | 匹配的是明确需求和已证实适配条件，而非敏感推断或成交预测。 |
| `bundle` | 互补角色、已有覆盖、缺口、替代与兼容性依据；不创建购物车或套装价格承诺。 |
| `sales_reason` | 场景/问题 → 产品事实 → 条件和限制的可解释链路。 |
| `confidence` | `low|medium|high` 加原因；低置信度仅推荐补问或核实。 |

## 6. Business Alerts

经营提醒是授权范围内的只读信号，不等同于执行命令。每项含 `alert_id`、`severity`、`scope`、`as_of`、数据窗口、原因、建议动作、限制、Citation 和确认/静默状态；静默必须记录原因、到期时间和操作者。

| 类型 | 发现内容 | 示例人工建议 | 硬边界 |
| --- | --- | --- | --- |
| 库存 | 低可售、异常波动、快照过期或库存与需求信号不一致。 | 核实库存、检查替代类别、提交人工复核。 | 不补货、调拨、预留或承诺库存。 |
| 销售 | 趋势变化、转化/品类异常、机会积压。 | 查看窗口和原因，安排复盘或确认需求。 | 不自动归因、奖惩或生成销售指令。 |
| 商品 | 生命周期、资料缺失、适配/版本冲突、主推规则异常。 | 核对商品资料，提交商品经理审核。 | 不改价格、主数据或发布状态。 |
| 门店 | 门店经营指标、任务积压、数据鲜度或运营异常。 | 核实数据源，创建人工处理事项。 | 不自动排班、调店或评价员工。 |
| 风险 | 权限、Citation、过期知识、同意撤回或异常访问风险。 | 停用卡片、升级审核或复核权限。 | 风险提醒不绕过既有审批流程。 |

## 7. AI Recommendation Module

Recommendation 是可解释的建议对象，而非指令。无 Citation、数据过期、来源撤销或策略不允许时，系统应隐藏建议或降级为“请人工核实”。

| 字段 | 设计 |
| --- | --- |
| `question` | 需要解决的经营、客户、商品或学习问题，明确范围和时间窗口。 |
| `data_basis` | 授权的观察、数据时点、知识/规则/产品版本和 Citation 列表。 |
| `ai_judgment` | 基于事实与规则的判断、置信度、假设、未知项和风险；不得伪装为事实。 |
| `suggested_action` | 有序、可编辑、可撤回的人工动作及替代方案。 |
| `human_confirmation` | `pending|accepted|edited|rejected|needs_verification`，记录操作者、时间和理由。 |

所有建议响应固定包含 `advisory_only_requires_human_approval: true`。采纳只记录人工意图/待办；实际销售、采购、营销或系统修改必须在相应获批系统中独立进行并以其审计为准。

## 8. Learning Center

Learning Center 每日提供可在工作中使用的微学习，按角色、已完成培训、当前任务与知识 ACL 筛选；仅检索 `Published`、有效且可见的知识版本。

| 每日学习主题 | 内容 | 完成标准 |
| --- | --- | --- |
| 产品知识 | 产品定位、功能、适配、限制、兼容性与版本变化。 | 员工查看/确认学习；不等同于具备专业认证。 |
| 销售知识 | 需求发现、条件化话术、异议处理、组合覆盖与禁止表述。 | 完成短学习并可提交知识问题。 |
| 路线知识 | 路线/场景的已审核准备信息、必问项及风险升级规则。 | 不替代天气、医疗、救援、许可或领队判断。 |
| 品牌知识 | 品牌定位、产品线、授权资产和已发布经营策略。 | 不自动授予品牌资料或发布权限。 |

学习卡记录 `learning_id`、主题、知识版本、Citation、有效期、完成状态及反馈引用。学习完成只能用于内容覆盖度提示，不得自动改变权限、绩效、排班、佣金或安全资质。

## 9. Role Based Workspace

角色决定可见工作区与默认聚合级别，而不构成数据豁免。用户仍须同时满足实际资源 ACL、组织范围、用途、同意与字段策略。

| 角色 | 重点 Workspace | 默认视图 | 明确限制 |
| --- | --- | --- | --- |
| 员工 | 客户和销售。 | Today Tasks、授权 Customer Opportunities、Sales Opportunities、个人学习。 | 仅已关联且同意的客户最小投影；不自动触达或成交。 |
| 店长 | 门店经营。 | 门店销售/库存/任务/风险提醒、聚合机会与团队学习覆盖。 | 不自动排班、考核、补货或下钻未经授权客户。 |
| 采购 | 商品趋势。 | 授权品牌/品类趋势、库存信号、生命周期风险与采购研究建议。 | 不创建采购单、供应商订单或 SAP 修改。 |
| 商品经理 | 品牌经营。 | 品牌/产品线经营、商品资料缺口、生命周期与知识审核待办。 | 不改主数据、价格、发布状态或 Core 同步。 |
| CEO | 经营驾驶舱。 | 组织/区域级汇总、风险、机会、趋势和待核实事项。 | 默认聚合、去标识化；不因职位绕过客户同意或字段 ACL。 |

## 10. Feedback Loop

```text
员工反馈（采纳/编辑/拒绝/结果/引用问题）
                    ↓
AI 整理（去重、脱敏、聚类、证据缺口标注；仅生成 Draft）
                    ↓
审核（业务负责人 + 知识/合规审核，按职责分离）
                    ↓
知识增强（版本化发布、Citation、回滚；后续请求才可使用）
```

| 阶段 | 输入与输出 | 控制 |
| --- | --- | --- |
| 员工反馈 | 对任务、机会、提醒、建议或学习内容提交受控反馈及最小化说明。 | 原始 PII/敏感自由文本检测、用途限制与保留期适用。 |
| AI 整理 | 生成反馈摘要、重复聚类、建议质量指标、知识候选 Draft。 | 不将摘要写成事实，不自动改规则/排序。 |
| 审核 | 审核证据、Citation、范围、偏差和合规性。 | `submitted → triaged → draft → in_review → approved|rejected`；保留审批记录。 |
| 知识增强 | 发布新的知识/规则版本或拒绝并给出理由。 | 仅已发布、有效、ACL 可见版本可用于未来建议；支持撤销与回滚。 |

## 11. API Contract

**共同约束：** JSON、RFC 3339 UTC、`Authorization`、`X-Correlation-Id`、`X-Purpose` 必填。组织、用户、角色、权限和范围由服务端派生；客户端不得提交特权角色、权限、原始客户 PII 或业务执行参数。所有响应都附 `policy_snapshot`、`as_of` 与 `advisory_only_requires_human_approval: true`。

### `GET /api/workspace/tasks`

读取当前用户可见的 Today Tasks。查询参数：`status?`、`priority?`、`scope?`、`limit=1..50`、`cursor?`；`scope` 必须是服务端授权范围的子集。

```json
{
  "items": [{
    "task_id": "aitask_01J...",
    "task_source": "customer_signal",
    "priority": "P2",
    "priority_reason": "已授权客户需求需在 48 小时内人工确认；数据新鲜且证据完整",
    "reason": {"facts": ["客户确认过雨天徒步兴趣"], "limitations": ["路线和预算待确认"]},
    "suggested_action": {"type": "human_follow_up_preparation", "text": "准备开放式需求确认问题"},
    "status": "proposed",
    "citations": ["cit_01"],
    "as_of": "2026-07-21T00:00:00Z"
  }],
  "next_cursor": null,
  "policy_snapshot": "pol_01J...",
  "advisory_only_requires_human_approval": true
}
```

### `GET /api/workspace/opportunities`

读取客户与销售机会。查询参数：`type=customer|sales`、`status?`、`store_id?`、`limit=1..50`、`cursor?`；客户条目逐项执行同意、关系、用途及字段 ACL。

```json
{
  "items": [{
    "opportunity_id": "opp_01J...",
    "type": "customer",
    "customer_id": "cst_01J...",
    "customer_profile": {"experience": "confirmed", "preference": "unknown"},
    "purchase_history": {"summary": "授权购买摘要", "as_of": "2026-07-01T00:00:00Z"},
    "activity_interest": {"value": "徒步", "confirmation": "customer_stated"},
    "equipment_profile": {"confirmed": [], "historical": ["gear_ref_01"], "unknown": ["雨天外层"]},
    "opportunity_reason": "明确兴趣与待确认装备缺口形成的候选，非购买意愿判断",
    "next_action": {"type": "human_question", "text": "确认路线、日期和已有防雨装备"},
    "citations": ["cit_02", "cit_03"],
    "confidence": "medium"
  }],
  "policy_snapshot": "pol_01J...",
  "advisory_only_requires_human_approval": true
}
```

### `GET /api/workspace/alerts`

读取库存、销售、商品、门店和风险提醒。查询参数：`category?`、`severity?`、`scope?`、`include_acknowledged=false`、`limit=1..100`、`cursor?`。

```json
{
  "items": [{
    "alert_id": "alert_01J...",
    "category": "inventory",
    "severity": "medium",
    "scope": {"store_id": "store_cd_01", "product_ref": "prd_01"},
    "reason": "库存快照与近期需求信号存在差异，需人工核实",
    "suggested_action": "在权威库存系统核实后决定后续处理",
    "freshness": {"as_of": "2026-07-21T00:00:00Z", "status": "fresh"},
    "citations": ["cit_04"],
    "status": "active"
  }],
  "policy_snapshot": "pol_01J...",
  "advisory_only_requires_human_approval": true
}
```

### `POST /api/workspace/feedback`

提交对卡片或学习内容的反馈；只创建反馈/审核候选，绝不触发业务写操作。请求中的 `actor`、组织、角色和范围被忽略并由服务端派生。

```json
{
  "subject_type": "task",
  "subject_id": "aitask_01J...",
  "action": "edited",
  "reason_code": "insufficient_context",
  "employee_summary": "已确认需要先核实路线。",
  "outcome_ref": "external_audit_ref_01",
  "citation_issue_ids": []
}
```

## 12. 权限、审计与 Citation

### 12.1 RBAC + ACL

访问决定为：`allow = org_match ∧ RBAC ∧ ACL ∧ scope ∧ purpose ∧ consent_if_required ∧ freshness/publication`。RBAC 定义可尝试的能力，ACL 定义资源及字段范围；任一条件不成立即拒绝或返回去标识化聚合。权限、调店、离职、品牌撤权、同意撤回在下一请求生效，高风险读取使用短会话与重新评估。

### 12.2 审计记录

审计日志为仅追加、访问受控的事件流。使用引用、哈希和最小摘要，避免在通用日志写入 PII 或敏感商业正文。

| 记录对象 | 最低字段 |
| --- | --- |
| 访问 | `request_id`、actor、组织、目的、资源/字段类别、allow/deny、脱敏决定、`policy_snapshot`、时间。 |
| 建议 | `advice_id`/卡片 ID、输入引用、规则/模型/提示版本、输出 hash、置信度、限制、Citation、时间。 |
| 反馈 | `feedback_id`、主体、受控动作、理由码、最小摘要 hash、审核状态、关联权威结果引用。 |
| 版本 | 知识、产品、规则、策略、模型、提示模板版本、有效期、审批、撤销/回滚关系。 |
| Citation | `citation_id`、资源、版本、不可变定位（字段/段/页/单元格）、引用时状态、访问决定。 |

Citation 必须可定位、版本化且在建议生成时有效；建议回放按当时输入与版本进行，但展示回放时重新评估当前查看者 ACL。异常访问、策略拒绝、Citation 失效和批量导出应产生治理告警并接受审计导出审批。
