# VAFOX Huyan CEO Knowledge Brain Architecture

版本：`1.0`  
定位域名：`huyan.vafox.com`  
状态：架构设计（不包含服务器、数据库或生产环境操作）

## 0. 架构总览

Huyan CEO Control Plane 的 Knowledge Brain 是为 CEO 建立的长期、可治理、可追溯的知识与决策记忆系统。它同时提供 CEO 私人知识空间和企业共享知识空间，但两者不是同一个数据池，也不能通过检索、Agent 或报表被隐式混合。

```text
gateway.vafox.com
        |
        v
huyan.vafox.com  ── CEO Control Plane
        |
        v
CEO Dashboard
        |
        v
Agent Hub
        |
        v
Knowledge Brain
  ├── CEO Private Brain
  ├── Enterprise Knowledge
  ├── Operational Knowledge
  ├── Decision Memory
  └── Knowledge Governance / Audit
```

Knowledge Brain 只提供经过身份、权限、范围和引用校验后的知识访问能力。它不是生产执行器，不是 Core Data 的副本，也不是绕开公司制度、审批和数据治理的通道。

## 一、Knowledge Brain 定位

### 1.1 定义

`huyan.vafox.com = CEO Private Brain + Enterprise Knowledge Brain`。

- **CEO Private Brain**：保存仅属于 CEO 的战略脉络、判断框架、私有思考与受限决策记忆，服务于 CEO 的连续思考和个人授权决策。
- **Enterprise Knowledge Brain**：保存经组织治理后可复用的制度、品牌、流程、培训和经营经验，服务于企业协作与标准化执行。
- **Knowledge Brain**：在上述边界内，对知识进行编目、审核、版本化、检索、引用、归档和审计的控制层。

### 1.2 为什么 CEO 需要独立知识空间

CEO 的知识并不等同于企业公开知识。战略假设、投资评估、组织判断、未公开谈判、反思记录和决策取舍往往具有高度敏感性、长期性与上下文依赖性。若这些内容与员工知识库共用同一空间，会造成三类风险：

1. **保密风险**：私有判断可能被非授权人员、Agent 或检索结果间接暴露。
2. **决策失真**：未经发布的假设、草案和反思被误当作公司正式政策或事实。
3. **记忆断裂**：CEO 无法在新的问题中可靠回溯“当时为何这样判断、依据是什么、结果如何”。

独立知识空间使 CEO 可以保留完整的思考上下文；而企业知识只有在审核、脱敏和发布后，才进入可共享的知识域。

### 1.3 设计原则

1. **私有优先、明确发布**：CEO 私有内容默认不共享；跨域共享必须有显式发布动作。
2. **事实与判断分离**：事实须带来源；分析、假设、建议与决定须明确标记。
3. **引用优先于结论**：每项回答应返回可验证的来源、时间、版本和可信度。
4. **最小权限、最小上下文**：每次检索只返回调用者在当前任务中所需且被授权的内容。
5. **人类权威不被替代**：Agent 可检索、归纳和起草，但不得把建议伪装为 CEO 决定或企业正式制度。
6. **Core 为业务数据权威**：Knowledge Brain 可引用经治理的 Core 数据摘要或证据，不取代 Core Data 的权威性。

## 二、知识分层模型

### 2.1 三层知识域

| 层级 | 目的 | 典型内容 | 默认可见性 | 进入共享域的条件 |
| --- | --- | --- | --- | --- |
| CEO Private Knowledge | 保留 CEO 连续思考与私有决策上下文 | CEO 战略、投资判断、管理思考、私有决策记录 | 仅 CEO | CEO 显式授权、完成必要脱敏与审核 |
| Enterprise Knowledge | 沉淀组织正式、可复用的共同知识 | 公司制度、品牌资料、流程文件、培训资料 | 按组织角色授权 | 内容负责人审核并正式发布 |
| Operational Knowledge | 沉淀一线运营经验与可验证解决方案 | 门店经验、供应链经验、历史问题解决方案 | 按业务范围授权 | 经业务负责人验证、标明适用条件 |

### 2.2 CEO Private Knowledge

该层是 CEO Private Brain 的核心，不因“有助于回答问题”而自动向员工、其他高管或 Agent 开放。建议的子类：

- **CEO 战略**：愿景、战略目标、优先级、阶段性假设、竞争判断与战略复盘。
- **投资判断**：项目机会、估值逻辑、风险收益分析、谈判要点、投资结论及其前提。
- **管理思考**：组织设计、人才评价、管理原则、授权边界、待验证的管理假设。
- **私有决策记录**：尚未公开的方案比较、会议思考、反对意见、决定草案与个人复盘。

每条私有知识均应拥有 `private` 默认标签、CEO 所有者、敏感级别、可见范围和共享状态。即使引用了 Enterprise 或 Operational 知识，私有记录本身的分析和结论仍保持私有。

### 2.3 Enterprise Knowledge

该层承载企业认可的“共同语言”和“正式做法”：

- **公司制度**：治理制度、授权矩阵、合规要求、人事与财务相关制度。
- **品牌资料**：品牌定位、视觉规范、产品叙事、对外沟通原则和素材使用规则。
- **流程文件**：标准作业流程、跨部门协作流程、审批路径、服务标准和异常处理规范。
- **培训资料**：岗位手册、课程、考核标准、案例与新员工学习路径。

发布到本层的内容必须有责任部门、审核人、生效时间、版本号和适用范围。草稿、已废止版本和仅供讨论材料不得被呈现为现行制度。

### 2.4 Operational Knowledge

该层关注“在什么场景下、谁用什么方法、获得了什么结果”的实践经验：

- **门店经验**：陈列、服务、转化、排班、活动执行和异常处置经验。
- **供应链经验**：采购、补货、库存、质量、交付、供应商协同与风险缓释经验。
- **历史问题解决方案**：问题现象、根因、处置步骤、验证结果、适用边界与复发预防。

运营经验必须标明场景、地区/门店或业务范围、时间窗口、前提条件和效果证据。未验证的经验应标记为“候选实践”，不得与已验证标准流程混淆。

## 三、数据隔离模型

### 3.1 逻辑隔离边界

```text
┌──────────────── CEO Private Brain ────────────────┐
│ CEO 私有内容、私有索引、私有会话记忆、私有审计轨迹 │
└───────────────────────────────────────────────────┘
                      × 默认不可横向读取
┌──────────────── 员工 AI 知识空间 ─────────────────┐
│ 企业知识、运营知识、角色授权的检索索引与会话上下文 │
└───────────────────────────────────────────────────┘
                      × 不作为业务事实源
┌────────────────── Core Data ──────────────────────┐
│ 受治理的企业事实、主数据、事件、指标和业务 API    │
└───────────────────────────────────────────────────┘
```

### 3.2 隔离规则

| 数据域 | 可读取主体 | 禁止读取主体 | 使用规则 |
| --- | --- | --- | --- |
| CEO Private Brain | CEO；经 CEO 对具体内容显式授权的 CEO Agent | 员工、高管（除非逐项授权）、员工 Agent、通用报表 Agent | 不建立员工可检索索引；不用于训练或自动摘要到共享域 |
| 员工 AI 知识 | 经角色授权的高管、员工及其 Agent | 未授权角色；CEO Private Brain 读取路径 | 只返回已发布且在业务范围内的内容 |
| Core Data | 通过获批 Core API 的 Data Agent / Report Agent | Knowledge Brain 直接写入；任意 Agent 直连底层数据存储 | 作为事实证据被引用，不复制为可变“事实真相” |

### 3.3 跨域规则

1. CEO 可读取已获授权的 Enterprise Knowledge、Operational Knowledge 和由 Core API 返回的受治理事实。
2. CEO Private Brain 向 Enterprise Knowledge 的迁移必须经过“选择内容 → 脱敏/拆分 → 审核 → CEO/责任人发布”的显式流程。
3. Agent 不得以摘要、向量检索、缓存、日志、提示词或失败信息的方式，将私有知识泄漏到员工 AI 知识空间。
4. Core Data 只经批准接口按需读取；Knowledge Brain 不修改 Core 数据、不回写业务记录、不创建平行事实库。
5. 跨域引用应保留原始域标识；若无读取权限，结果只可显示“存在受限证据”，不得显示标题、片段或推断内容。

## 四、知识生命周期

```text
创建 → 审核 → 发布 → 更新 → 归档
             ↘ 驳回 / 退回补充 ↗
```

| 阶段 | 目标 | 必填控制 | 产物状态 |
| --- | --- | --- | --- |
| 创建 | 记录新知识或导入材料 | 所有者、知识域、分类、来源、敏感级别、适用范围 | Draft |
| 审核 | 验证准确性、合规性与可分享性 | 审核人、事实核验、脱敏检查、冲突检查、有效期 | In Review / Rejected |
| 发布 | 使其在授权范围内可被检索 | 发布人、版本、生效时间、访问策略、引用标识 | Published |
| 更新 | 修正或扩展已发布知识 | 变更说明、继承版本、重新审核、旧版处置 | Superseding / Published |
| 归档 | 保留历史而停止常规使用 | 归档原因、替代版本、保留期限、只读策略 | Archived |

所有阶段都应产生不可抵赖的审计事件：谁在何时以何种权限创建、审核、发布、更新、撤回或归档了什么内容。已发布内容的实质修改应创建新版本，禁止静默覆盖历史版本。

## 五、知识权限模型

### 5.1 角色与默认权限

| 主体 | CEO Private Knowledge | Enterprise Knowledge | Operational Knowledge | Core Data | 发布/审批权限 |
| --- | --- | --- | --- | --- | --- |
| CEO | 读/写/授权/归档 | 读；可发起或批准战略级发布 | 读 | 经授权的只读视图/API | 可授权私有内容共享；可进行战略级最终批准 |
| 高管 | 无默认访问；仅逐项获授权后读 | 按职责读；可创建职责范围草稿 | 按业务范围读；可提交/审核经验 | 经职责授权的只读视图/API | 对所属知识域审核、发布建议 |
| 员工 | 无访问 | 仅已发布且岗位/部门授权内容 | 仅岗位或区域授权内容 | 无直接访问，使用受控业务应用 | 创建候选内容；无越权发布权 |
| Agent | 仅服务身份、任务范围和用户授权的最小读取 | 在调用者范围内检索 | 在调用者范围内检索 | 仅 Data Agent/Report Agent 经批准 API 读取 | 不拥有最终发布、审批或授权权 |

### 5.2 授权要求

- 授权决策至少由 `主体 + 角色 + 知识域 + 操作 + 业务范围 + 有效期` 组成。
- CEO 私有知识使用逐项或明确集合授权，不能仅凭“高管”或“管理员”角色获得访问权。
- 权限变更、临时授权和撤销应立即影响后续检索；缓存结果不得绕过撤销。
- Agent 必须以可审计的服务身份运行，并同时继承调用者权限和任务所需的最小权限，两者取交集。

## 六、Knowledge Agent 接口

### 6.1 统一调用契约

Agent Hub 向 Knowledge Brain 提交受控的知识请求，而不是直接访问存储层。概念接口如下：

```text
KnowledgeQuery {
  requester_identity, requester_role, agent_identity,
  task_id, purpose, knowledge_scope, query,
  filters: {time_range, version_policy, confidence_minimum},
  requested_action: search | cite | summarize | compare
}

KnowledgeResponse {
  answer_context, citations[], access_decision,
  freshness, confidence, omitted_restricted_count,
  audit_event_id
}
```

接口强制进行身份验证、权限计算、域过滤、版本选择、敏感信息过滤、引用生成和审计记录。`answer_context` 只能包含已授权的最小必要片段。

### 6.2 CEO Agent

CEO Agent 服务于 CEO 的提问、战略思考和决策复盘：

1. 确认当前请求来自 CEO 或具有明确委托的会话。
2. 优先检索 CEO Private Brain；可按 CEO 当前权限补充 Enterprise/Operational 知识与 Core 事实证据。
3. 将“事实、历史判断、当前推断、建议”分开呈现。
4. 对私有内容仅在 CEO 会话中返回；若 CEO 指定转为企业知识，创建待审核发布包而非直接公开。

### 6.3 Data Agent

Data Agent 只通过获批的 Core API 获取企业事实、指标和数据质量元数据：

1. 将业务问题转换为最小化、只读的数据请求。
2. 返回数据来源、查询时间、口径、覆盖范围、数据新鲜度和质量状态。
3. 可把已批准的数据摘要作为知识证据链接，但不把 Agent 推断写成 Core 事实。
4. 不读取 CEO Private Brain，除非 CEO 在该次任务中对明确条目授权；即使授权，也不得回写或扩散。

### 6.4 Report Agent

Report Agent 用于生成 CEO 简报、部门报告或复盘草稿：

1. 先确定报告受众和权限边界，再构造检索范围。
2. 从已发布知识、被授权运营经验和获批 Core 事实中取材。
3. 每个关键结论附带来源引用；无法验证时标记为推断或待确认。
4. 不将 CEO 私有分析、受限证据或未经发布的草稿写入面向员工的报告。

## 七、历史决策记忆

### 7.1 决策记录模型

每项需要长期回溯的 CEO 决策建立一份独立的 Decision Memory，而不是只保存最终结论：

| 字段 | 说明 |
| --- | --- |
| 问题 | 当时需要回答、批准、拒绝或取舍的核心问题 |
| 背景 | 业务环境、触发事件、约束、相关角色与时间窗口 |
| 分析 | 事实证据、方案比较、假设、风险、反对意见和不确定性 |
| 决定 | 最终选择、决策人、决策时间、授权范围和执行条件 |
| 结果 | 实际结果、指标变化、偏差、意外影响和复盘时间 |
| 经验 | 可迁移原则、适用条件、应避免做法和后续行动 |

补充元数据至少包括：决策 ID、私有/共享状态、关联知识引用、关联 Core 事实引用、版本、保密等级、复盘到期日和责任人。

### 7.2 记忆使用规则

- 决策记忆默认属于 CEO Private Brain，除非 CEO 明确将某一部分发布为组织经验。
- 后续提问可检索相似问题、当时假设和已知结果，但 Agent 必须说明当前环境是否已变化。
- “结果”与“经验”应在复盘后补充，避免把即时决定误写为已验证成功的方法。
- 被撤销或事实基础已失效的决定应保留历史状态与撤销原因，不删除或改写为从未发生。

## 八、检索与引用机制

### 8.1 检索流程

```text
问题 → 身份与任务校验 → 权限范围计算 → 域内检索
    → 版本/时效/可信度筛选 → 证据排序 → 引用组装 → 可审计回答
```

检索应优先选择：适用范围匹配、已发布或明确授权、版本有效、来源权威、时间新鲜且有证据支持的内容。不得为了生成流畅答案而跨越权限边界或省略不确定性。

### 8.2 引用卡片

每个被用于关键结论的知识片段都应携带以下最小引用信息：

| 字段 | 含义 |
| --- | --- |
| 来源 | 原始文档、制度、决策记录、运营案例或获批 Core 证据的标识 |
| 时间 | 创建、发布、最近更新与事实发生/查询时间 |
| 版本 | 内容版本、当前有效状态与是否已被替代 |
| 可信度 | 基于来源权威性、审核状态、证据充分性和时效性的等级 |

建议额外显示知识域、所有者、适用范围、审核状态和访问限制。对无权限的引用仅报告“受限来源存在”，不泄露可识别信息。

### 8.3 回答呈现规则

- **事实**：引用权威来源与时间，必要时显示数据口径与新鲜度。
- **分析/推断**：明确标识为 Agent 或人类分析，并说明支持它的证据。
- **建议**：说明建议依据和风险，不得表述为既定决定。
- **冲突知识**：并列显示关键冲突、各自版本和可信度，交由有权限的负责人确认。

## 九、安全边界

本设计的 Knowledge Brain 仅处理知识访问与治理，不具有基础设施或生产变更权限。以下动作明确禁止由 Knowledge Brain、Knowledge Agent 或其检索接口执行：

- SSH 登录或远程 Shell。
- 服务器操作、配置变更、服务重启、容器操作或文件系统改写。
- 数据库修改、迁移、删除、修复或直接写入 Core/SAP 数据。
- 生产自动执行、自动部署、自动审批或自动触发不可逆业务动作。

Knowledge Brain 可以生成知识摘要、证据引用、决策草稿、风险说明和待审批建议；任何执行性动作必须离开本架构边界，进入独立的、经人工审批与审计的受控工作流。即使 CEO Agent 提出了行动建议，也不得把建议转换为生产命令。

## 十、未来扩展

在不改变隔离和治理原则的前提下，Knowledge Brain 可逐步增加以下受治理的知识域：

| 扩展域 | 典型内容 | 关键治理要求 |
| --- | --- | --- |
| 合同知识 | 合同条款、义务、到期、风险与谈判历史 | 权限分级、法务审核、有效版本与保留策略 |
| 供应商知识 | 供应商能力、履约、质量、风险、合作复盘 | 与 Core 事实证据关联，区分评价与事实 |
| 品牌知识 | 品牌资产、渠道表达、市场反馈、品牌决策 | 品牌负责人审核，明确可对外使用范围 |
| 客户知识 | 客户洞察、服务案例、需求信号与关系历史 | 最小化、隐私合规、用途限制与访问审计 |
| 投资知识 | 行业研究、项目档案、估值、投后复盘 | 默认 CEO 私有或受限，严格信息隔离 |

扩展应遵循“先定义知识所有者、敏感级别、来源、生命周期和访问策略，再接入检索”的顺序。任何新域都不得自动获得 CEO Private Brain、员工 AI 知识或 Core Data 的横向访问权限。

## 结论

Huyan Knowledge Brain 的最终目标是形成 CEO 的长期记忆系统：既能保留 CEO 在战略、投资与管理中的完整决策脉络，又能把被验证、被授权的组织经验沉淀为企业可复用知识。其核心不是收集更多文本，而是在 CEO 私有空间、员工 AI 知识和 Core Data 之间建立长期有效、严格隔离、可引用、可审计且由人类最终负责的知识治理体系。
