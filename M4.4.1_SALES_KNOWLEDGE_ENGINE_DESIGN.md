# VAFOX M4.4.1 Sales Knowledge Engine Design

**状态：** Design Phase / 仅设计，不部署。
**前置设计：** `M4_KNOWLEDGE_DOMAIN_MODEL_REPORT.md`、`M4.3_PRODUCT_KNOWLEDGE_DESIGN.md`、`DATA003_ARCHITECTURE_DESIGN.md`、`DATA003.3_EMPLOYEE_AI_ASSISTANT_DESIGN.md`。
**负责人：** Codex（设计开发）；WorkBuddy（环境）；Marvis（管理）；CTO（验收）。

## 1. 目标、原则与边界

Sales Knowledge Engine（SKE）将经过审核的销售资料转化为可检索、可引用、可解释的销售知识，辅助员工发现需求、组合装备、说明产品并处理异议。它是**建议系统**，不是营销、交易、库存或主数据系统。

- **证据优先：** 每条建议、话术、比较和异议回应均绑定 `citation`、版本、审核状态和置信度。
- **产品事实优先：** 产品适配、规格、限制和替代品来自 M4.3 Product Intelligence；SKE 不复制、猜测或修改产品/SKU 事实。
- **先授权后检索：** API 先由服务端基于可信 `AuthContext` 执行组织隔离、RBAC + ACL、门店范围与用途校验，再返回知识或来源。
- **人工优先：** 员工须确认客户需求、尺码、价格、促销资格和库存；任何输出均不是报价、库存承诺或成交承诺。
- **最小化客户数据：** Customer Context 仅保留本次推荐需要的经授权、去标识化/最小化属性；原始 PII 不进入销售知识实体。

**明确禁止：** 自动发送营销；自动成交；自动下单；修改 SAP；修改 Core 生产同步；预留/调拨库存、改价或把模型判断写成客户、订单或库存事实。

## 2. Sales Knowledge Architecture

```text
Sales Asset（培训材料、场景手册、经审核员工经验、产品资料）
                         ↓ 导入、切分、来源哈希、ACL 继承
Sales Knowledge Entity（版本化、可引用、审批后的销售知识）
                         ↓ 受权限检索 + 场景/需求匹配
Reasoning Engine（规则优先的解释、组合与限制；生成仅整理表达）
                         ↓ 建议、问题、话术、引用、置信度、人工核实项
Employee AI Assistant（员工审阅、编辑、拒绝或采纳）
                         ↓ 显式反馈与受控结果码
Feedback Loop（脱敏反馈 → AI 整理草稿 → 人工审核 → 新知识版本）
```

| 层 | 输入与职责 | 输出 | 不得执行 |
| --- | --- | --- | --- |
| Sales Asset | 品牌/产品资料、销售 playbook、批准 FAQ、员工提交经验。 | 可追溯原始资产和切分片段。 | 直接成为可用知识或扩大 ACL。 |
| Sales Knowledge Entity | 将资产映射到场景、需求、方案、话术、异议和连单规则。 | 仅 `approved`、有效且授权的实体可检索。 | 作为价格、库存或订单事实源。 |
| Reasoning Engine | 以 Customer Context、场景和 Product Intelligence 做硬约束过滤、候选组合和解释。 | 方案、澄清问题、理由、引用、限制和置信度。 | 自动决定客户偏好、自动触达或交易。 |
| Employee AI Assistant | 将建议呈现为员工可编辑的服务辅助。 | 人工确认后的展示/跟进参考和反馈。 | 发送消息、提交订单或写 SAP/Core。 |
| Feedback Loop | 收集结果码、员工修订和知识缺口。 | 审核队列、质量指标、候选新版本。 | 未审核地写入知识库或改变事实。 |

Reasoning Engine 必须先排除：越权内容、未批准/失效版本、场景 `not_suitable`、安全约束冲突、缺少必要客户输入、无可用产品证据的组合。它应优先返回澄清问题或“请人工核实”，不能用生成内容填补未知规格、库存、价格或安全建议。

## 3. Sales Knowledge Schema

`SalesKnowledgeEntity` 由 M4 Knowledge Brain 的通用 envelope 承载组织、所有者、生命周期、ACL、审计和血缘；以下为领域载荷。`knowledge_id` 永不复用，逻辑知识的修订创建新 `version`。

| 字段 | 类型 / 必填 | 定义与约束 |
| --- | --- | --- |
| `knowledge_id` | string，必填 | 稳定、不可变的知识引用。 |
| `scenario` | controlled object[]，必填 | 活动、环境、季节、难度及适配边界；使用 M4.3 场景词表。 |
| `customer_type` | controlled string[]，必填 | 匿名客户类型，例如 `newcomer`、`family_camping`、`regular_hiker`；不得含 PII 或敏感画像。 |
| `customer_need` | object[]，必填 | 显式需求/待确认需求、优先级、条件、缺失信息与证据。 |
| `discovery_questions` | object[]，必填 | 4W3H 问题、询问目的、必答性和分支条件。 |
| `recommendation_logic` | object，必填 | 硬约束、匹配规则、排除规则、排序理由和人工核实点；每条业务主张有证据。 |
| `product_bundle` | object[]，必填 | 单品/类别角色、必选性、替代规则、兼容条件及 `product_ref`。 |
| `sales_script` | object[]，必填 | 可编辑的开场、解释、确认和收束话术；区分事实与建议。 |
| `objection_handling` | object[]，必填 | 异议类型、客观回应、澄清问题、可比较维度、禁止表述和引用。 |
| `cross_sell` | object[]，可空 | 基于已确认缺口的互补需求/组合；非自动促销规则。 |
| `citation` | CitationRef[]，必填 | 至少一个 primary 来源；含定位符、来源版本/哈希、访问控制和血缘。 |
| `version` | semver/string，必填 | 内容修订版本；与审核记录、有效期关联。 |
| `confidence` | object，必填 | `{level, score?, rationale, assessed_at}`；表示知识/上下文完整性，不是成交概率。 |

## 4. Customer Discovery Model: 4W3H

员工以非诱导、可跳过且与本次服务相关的方式询问；客户不愿回答或字段未知时，明确保留 `unknown`，不得推断。输出为本次请求的 `CustomerContext`，除非员工通过独立受治理流程记录为客户明确声明，否则不写入客户画像。

| 维度 | 核心问题 | 结构化输出 |
| --- | --- | --- |
| Where | 去哪里？路线/地形/海拔/室内外怎样？ | `location_type`、`terrain`、`altitude_band`、`environment` |
| When | 什么季节、日期、预期天气？ | `season`、`trip_date?`、`weather_status` |
| Who | 谁使用？经验、人数、同行对象？ | `user_level`、`party_size?`、`role_needs` |
| How | 怎么进行？交通、负重、强度、技术要求？ | `activity`、`difficulty`、`load_band`、`technical_requirements` |
| How long | 单次多久、是否过夜、总天数？ | `duration_hours?`、`days?`、`overnight` |
| How often | 使用频率、已有装备、复用场景？ | `frequency`、`existing_gear_refs`、`reuse_intent` |
| How much | 预算范围与必需/可选优先级？ | `budget`、`must_have`、`nice_to_have` |

> 注：需求列举的“4W3H”中仅显示六个维度；本设计补充 `How much` 作为第三个 H，以使模型可形成预算受限且不诱导的建议。

```json
{
  "context_id": "ctx_...",
  "activity": "hiking",
  "environment": ["humid", "urban_fringe"],
  "duration_hours": 6,
  "overnight": false,
  "user_level": "newcomer",
  "frequency": "monthly",
  "budget": {"currency": "CNY", "max": 1500},
  "confirmed": ["activity", "duration_hours", "budget"],
  "unknown": ["weather_status", "fit"],
  "source": "employee_entered_current_session",
  "retention": "request_scoped"
}
```

## 5. Scenario Sales Flow

```text
客户需求（4W3H Customer Context）
              ↓
场景判断（活动、环境、季节、难度；识别缺失/风险）
              ↓
装备需求（功能角色、必选/可选、兼容与安全限制）
              ↓
Product Intelligence（已审批产品知识 + 只读 SKU/库存快照）
              ↓
销售方案（可解释组合、话术、替代、引用、置信度、人工核实）
```

| 场景 | 场景判断重点 | 装备需求例 | 销售方案护栏 |
| --- | --- | --- | --- |
| 徒步 | 路线、时长、湿滑、负重、经验。 | 鞋、分层、防雨、背包、补水。 | 先试穿/确认尺码；库存只显示时点快照。 |
| 登山 | 地形、攀登技术、天气、负重和安全能力。 | 支撑/防护、保暖、背负及按批准资料匹配的专业装备。 | 缺少专业安全信息时抑制结论，建议专业人工确认。 |
| 露营 | 人数、车载/徒步、过夜、温度、营地条件。 | 庇护、睡眠、照明、收纳/烹饪类别。 | 不对气候安全、营地许可或燃料使用作无证据承诺。 |
| 高海拔 | 海拔、路线、天数、气温/风/紫外线、经验。 | 分层保暖、防风雨、行进支撑、补水/防晒角色。 | 高海拔不等同普通徒步；高反与路线风险必须人工确认。 |
| 滑雪 | 雪场、温度、雪况、滑行水平、既有装备。 | 保暖层、防水外层、手套、护目等角色。 | 仅按批准兼容资料匹配，不替代雪场安全规则。 |
| 城市户外 | 通勤/周末用途、天气、携带方式、复用频率。 | 轻量防雨、舒适鞋服、日用收纳。 | 不将城市轻量方案包装为技术山地方案。 |

## 6. Gear Bundle Knowledge

组合知识以“需求覆盖”而非强制捆绑销售为核心。实际单品由 Product Intelligence 在产品适配、兼容性、价格和门店库存快照都可用时填充；不可用时给出类别替代或“人工核实”。

```text
单品（有证据的产品/类别）
          ↓ compatible_with / complements
组合（角色、数量、替代与预算层）
          ↓ coverage check
完整装备系统（场景、风险、使用顺序、缺口与人工确认）
```

| 装备包 | 适用条件 | 最低系统角色 | 关键确认 |
| --- | --- | --- | --- |
| 川西 7 天装备包 | 多日徒步；山地、多变天气；天数/季节须确认。 | 行进鞋、分层衣物、防风雨层、背负、补水、照明/收纳等。 | 路线、海拔、温度、是否露营、负重、已有装备和预算。 |
| 冈仁波齐转山包 | 高海拔、多日、风/低温/强紫外线；需经验与计划确认。 | 保暖分层、防风雨、行进支撑、背负、补水/防晒等。 | 海拔适应、路线/天气、健康与专业安全要求必须人工确认。 |
| 深圳周末徒步包 | 低海拔、湿热、近郊、单日为默认假设但须确认。 | 透气衣物、轻量防雨、徒步鞋/替代、日用背包、补水。 | 实际路线、降雨、时长、鞋服尺码及已有装备。 |

每个 bundle 至少有：`bundle_id`、`scenario`、`roles[]`、`required`、`alternatives[]`、`compatibility_rules[]`、`coverage_gaps[]`、`budget_guidance`、`evidence_refs[]`、`version` 和 `review_status`。预算仅为建议区间；实时价格、促销和可售性仅由授权 Retail 只读事实在响应时附带，且不构成承诺。

## 7. Objection Handling Knowledge

异议处理是帮助员工客观澄清价值和约束的知识，不是操纵、贬低或承诺。不得攻击竞品、捏造比较、评价竞争品牌动机，或用未经证实的“唯一/最好/绝对”表述。

| 异议 | 建议流程 | 客观回应原则 |
| --- | --- | --- |
| 价格异议 | 确认使用场景、频率和必要功能 → 说明已证实功能/限制 → 给出同需求预算层替代。 | 解释适配、材料/保修等有引用维度；不承诺保值或最低价。 |
| 品牌比较 | 询问客户比较维度 → 并列展示可验证规格、适配与限制 → 邀请试穿/核实。 | 比较同类、同条件、同版本资料；不攻击竞品。 |
| 线上价格 | 确认型号、变体、渠道、时间、售后/库存条件 → 说明门店可验证的事实。 | 不猜测线上价格或政策；不未经授权匹配/改价。 |
| 客户犹豫 | 识别尚未确认的问题（尺码、用途、预算、天气）→ 给出可比较选项。 | 尊重不购买决定；不制造紧迫感或虚假稀缺。 |
| 预算不足 | 标记必需与可选角色 → 提供安全/适配不被破坏的类别替代或分期补齐建议。 | 不降低明确安全/兼容约束；不引导超预算。 |

每条 `objection_handling` 必须带 `evidence_refs`、`allowed_claims`、`prohibited_claims`、`clarifying_questions` 与 `requires_human_verification`。比较类回答还必须带比较维度、资料版本和适用条件；缺少同条件证据时只说明“当前无法做可验证比较”。

## 8. Sales Memory and Feedback Loop

```text
员工经验（显式提交；结构化结果码/最小化摘要）
                         ↓
AI 整理（去标识化、聚类、标出证据缺口，生成 draft）
                         ↓
人工审核（销售负责人 + 知识审核；必要时产品/合规复核）
                         ↓
销售知识库（新版本 approved 后才可检索）
```

允许回流：员工接受/编辑/拒绝建议、缺失问题、客户明确表示的非敏感需求摘要、已确认的组合缺口、引用纠错和结果码。不得把自由文本、原始 PII、未经确认的客户偏好、订单推断或竞争攻击语直接转为知识。

| 反馈字段 | 规则 |
| --- | --- |
| `feedback_id`、`knowledge_ref`、`version` | 关联到当时建议快照，支持回放。 |
| `action` / `reason_code` | `accepted|edited|rejected|insufficient_context|citation_issue|product_unavailable` 等受控值。 |
| `employee_summary` | 最小化、可选的人工摘要；敏感数据扫描并按策略删除/隔离。 |
| `outcome_ref?` | 仅引用已授权的权威销售事实；不能由员工或 AI 自行宣称成交。 |
| `review_state` | `submitted → triaged → draft → in_review → approved|rejected`。 |

质量指标使用聚合、去标识化数据，如引用完整率、审核周转、建议被编辑/拒绝原因、知识过期率和缺口覆盖率；不得用于自动员工考核或自动营销分群。负反馈不会自动降低知识置信度或下架，必须经审核判断。

## 9. Retail Integration

```text
销售数据（只读聚合/授权事实） + 产品智能（M4.3） + 库存（只读快照） + 客户画像（最小化授权投影）
                                               ↓
                                      销售建议（仅供员工确认）
```

| 数据域 | 可提供 | SKE 使用方式 | 限制 |
| --- | --- | --- | --- |
| 销售数据 | 授权的购买摘要、趋势、权威事实引用。 | 识别已有装备和复用需求，避免重复推荐。 | 不以推断替代客户确认，不自动归因成交。 |
| 产品智能 | 已批准产品适配、规格、限制、关系、Citation。 | 选择候选、解释功能和兼容性。 | 不写产品主数据，不生成无依据事实。 |
| 库存 | 调用时门店/SKU 可售状态、`as_of`、鲜度。 | 排除不可售候选或提供替代，展示“请核实”。 | 不预留、调拨、承诺或修改库存。 |
| 客户画像 | 经用途/同意授权的最小化画像或当前明确需求。 | 个性化排序或无个人化时的通用场景方案。 | 不接收原始 PII；无授权则降级通用建议。 |

输出 `SalesAdvice` 必须分开标识知识事实、客户明确需求、库存事实和推理建议，并包含 `as_of`、依赖鲜度、`citation`、`confidence`、`limitations` 与 `disposition: advisory_only_requires_human_approval`。

## 10. Employee AI Assistant Integration

员工可提出“客户怎么推荐？”“如何搭配？”“如何解释产品？”“如何连单？”等问题。助手将自然语言归一为 SKE 查询意图，要求必要的 4W3H 信息，调用授权检索，并返回可编辑建议而非行动按钮。

| 员工意图 | SKE 返回 | 员工必须确认 |
| --- | --- | --- |
| 客户怎么推荐？ | 场景判断、缺失问题、方案/替代、理由和限制。 | 客户需求、授权范围、尺码、天气、库存。 |
| 如何搭配？ | 角色组合、兼容/缺口、预算层与完整系统检查。 | 已有装备、实际使用条件、试用体验。 |
| 如何解释产品？ | 经引用的功能、适用条件、限制和中性比较维度。 | 当前型号/变体、客户关注点、价格与售后事实。 |
| 如何连单？ | 仅基于已确认缺口的互补类别与开放式确认问法。 | 客户是否需要、预算、兼容性和不强迫原则。 |

所有回答显示来源、知识版本、检索时间、置信度和限制。未授权客户数据、受限来源、过期知识或缺失关键信息时，助手应拒绝/脱敏/要求补问，不得暴露对象存在性或拼接未授权信息。

## 11. Citation, Version, Permission and Review

- **Citation：** 每个业务主张指向 `citation_id`；引文包含 `knowledge_id`、`source_id`、不可变 locator（页/段/行/单元格）、片段范围、来源版本、SHA-256、捕获时间和派生血缘。引用与正文同样经 ACL 过滤。
- **版本：** 生命周期为 `draft → in_review → approved → deprecated → archived`。只有有效期内的 `approved` 版本进入默认搜索/推理；新版本不覆盖旧版本，历史建议保留版本引用。
- **RBAC + ACL：** 服务端根据 JWT/可信身份评估 `organization_id`、角色权限、门店、部门、所有者、可见性、用途与资源 ACL。筛选只能缩小结果，不能授予访问；无权与不存在采用不泄露存在性的响应。
- **人工审核：** 批准前必须具备必填 schema、至少一条 primary Citation、可解析定位符/哈希、受控场景、ACL、冲突检查、审核人和审核时间。涉及安全、比较或价格/政策的知识须按职责路由复核。
- **审计：** 记录主体/委托主体、用途、策略结果、查询摘要、命中 ID/版本、引用 ID、鲜度、反馈和关联 ID；不记录完整客户资料、原始 PII、受限正文或不必要自由文本。

## 12. API Contract

**共同约束：** 所有 API 使用 JSON、RFC 3339 UTC、`Authorization`、`X-Correlation-Id`、`X-Purpose` 和服务器派生的组织/角色/门店范围。默认仅返回 ACL 可见、`approved`、有效内容；请求不接受客户端声明的员工角色或原始客户 PII。

### `GET /api/sales-knowledge/search`

**用途：** 搜索销售知识。查询参数：`q?`、`scenario?`、`customer_type?`、`need?`、`objection?`、`bundle?`、`store_id?`、`limit`（1–50）、`cursor?`、`as_of?`。

```json
{
  "items": [{"knowledge_id": "sk:...", "scenario": ["hiking"], "summary": "深圳周末徒步发现与组合", "version": "1.0.0", "confidence": {"level": "medium"}, "citations": ["cit_01"]}],
  "page": {"next_cursor": null},
  "freshness": {"as_of": "2026-07-21T10:00:00Z"}, "correlation_id": "corr_..."
}
```

### `GET /api/sales-knowledge/{id}`

**用途：** 读取单一授权知识版本。参数：稳定 `knowledge_id`，可选 `version?`、`include=discovery,bundle,script,objections,citations`。返回完整 `SalesKnowledgeEntity`、生命周期、有效期、授权后的 citation 和血缘；无权限/安全不可披露时返回 `403 POLICY_DENIED`。

### `POST /api/sales-advice/recommend`

**用途：** 为当前受授权员工生成只读、可解释销售建议。请求只接受最小化 `customer_context` 和授权范围内的 `customer_id?`；客户 ID 可选，缺少个性化授权时必须生成通用场景方案。

```json
{
  "store_id": "store_nanshan",
  "customer_id": "cst_...",
  "customer_context": {"activity": "hiking", "environment": ["humid"], "duration_hours": 6, "budget": {"currency": "CNY", "max": 1500}},
  "include_inventory": true
}
```

响应包含 `discovery_gaps`、`scenario_assessment`、`bundle_roles`、经 Retail 只读校验的候选/替代、`sales_script`、`objection_handling`、`cross_sell`、理由、引用、版本、鲜度、置信度和人工核实清单。若库存依赖不可用/陈旧，返回 `inventory.status: unavailable|stale|verification_required`，不得声称有货。

标准错误：`400 INVALID_REQUEST`、`401 UNAUTHENTICATED`、`403 POLICY_DENIED`、`404 NOT_FOUND`（仅安全可披露时）、`409 KNOWLEDGE_STALE`、`422 CONTEXT_INSUFFICIENT`、`429 RATE_LIMITED`、`503 DEPENDENCY_UNAVAILABLE`。三个端点均不提供创建订单、营销发送、SAP/Core 写入、库存预留或价格修改能力。
