# VAFOX M4.3 Product Knowledge Base Design

**状态：** Design Phase / 仅设计，不部署。
**前置设计：** `M4_KNOWLEDGE_DOMAIN_MODEL_REPORT.md`（M4.1）、`M4_BRAND_KNOWLEDGE_REPORT.md`（M4.2）、`DATA003.2_RECOMMENDATION_ENGINE_DESIGN.md`。
**目标：** 建立统一、可引用、受 ACL 约束的 Product Knowledge Model，将 Knowledge Brain 的已审批产品事实，与 Retail Brain 的只读商品/库存事实和 DATA-003.2 Recommendation Engine 的建议链路连接起来。

## 1. 设计原则与边界

1. **产品知识不是商品主数据。** Product Knowledge 保存产品事实、技术解释、适用边界和销售建议；`product_id`/`brand_id` 仅引用权威目录，不复制或成为 SAP/Core 的事实源。
2. **证据先于主张。** 每一项可检索产品结论均有版本化来源、定位信息和内容哈希；派生内容保留完整血缘。
3. **授权先于检索。** 所有搜索、详情、导入后查看和推荐上下文，先由网关 JWT/可信身份导出的 `AuthContext` 执行组织、部门、角色、所有者和可见性判断。
4. **知识与零售事实分离。** Knowledge Brain 提供已审批、有效的适配依据；Retail Brain 只读提供 SKU 映射、渠道可售和库存快照；Recommendation Engine 生成可解释建议，三者均不写回对方。
5. **建议而非执行。** 返回结果只供员工人工确认，绝非报价、库存预留、订单、销售承诺或自动触达。

### 明确禁止

本设计及其未来实现不得：修改 SAP；修改 Core 生产同步、来源事实或产品主数据；自动下单；自动销售、触达客户、锁定库存、调拨、补货或改价。库存“可售”只是一个带时点的只读事实，不能被解释为库存承诺。

## 2. 统一 Product Entity Schema

产品实体是 `metadata.knowledge.product` 的受控领域载荷，并由 M4.1 Memory envelope 承载其 `knowledge_id`、组织、所有者、可见性、生命周期和审计信息。一个逻辑产品可有多个知识版本；仅 `approved`、处于有效期且通过 ACL 的版本进入默认检索和推荐。

| 字段 | 类型 / 是否必填 | 定义与约束 |
| --- | --- | --- |
| `product_id` | string，必填 | 稳定产品引用；与 Retail Brain 商品/SKU 映射分离，不能由知识服务生成或修改。 |
| `brand_id` | string，条件必填 | 品牌产品必填；引用 M4.2 `brand_id`。无品牌或待确认时须显式标记 `unbranded`/`unresolved`。 |
| `category` | controlled string，必填 | 产品类目，如 `shell_jacket`、`backpack`、`tent`；支持分类树路径。 |
| `series` | string，可空 | 产品系列的稳定名称或 ID。 |
| `model` | string，可空 | 厂商型号；变体/SKU 不应塞入该字段。 |
| `product_name` | localized object，必填 | `{default, zh_CN?, en?}`；默认名用于展示，别名仅用于搜索。 |
| `positioning` | object，必填 | `{segment, price_band?, use_intent, lifecycle?}`；陈述必须有引用。 |
| `technology` | array，必填 | 技术特性 `{name, mechanism, benefit, limitations, evidence_refs[]}`；不得将营销语视为技术事实。 |
| `specification` | object，必填 | 结构化规格、单位、适用变体与 `evidence_refs`；未知值不得补全。 |
| `scenario` | array，必填 | 第 4 节的场景适配对象；至少一个经证据支持的适用场景。 |
| `customer_profile` | array，必填 | 适合/不适合的匿名画像与能力要求；不得含个人 PII。 |
| `selling_points` | array，必填 | `{claim, audience, condition, evidence_refs[]}`，须区分事实、解释和话术。 |
| `competitors` | array，可空 | `{product_ref?, brand_ref?, comparison_dimension, claim, evidence_refs[]}`；无证据不得贬损竞品。 |
| `recommendations` | object，可空 | 销售提示、互补装备、替代规则、风险/尺码/安全提示；仅建议。 |
| `source_traceability` | object，必填 | 第 7 节的来源、引文、血缘、哈希和审核状态；至少一个 primary 来源。 |

## 3. 产品知识结构与关系

```text
品牌 (BrandEntity / M4.2)
  ↓ has_series
产品系列 (ProductSeries)
  ↓ contains_product
单品 (ProductEntity)
  ↓ has_feature / has_specification
技术特点 (TechnologyFeature)
  ↓ suitable_for / unsuitable_for
使用场景 (ProductScenario)
  ↓ fits_profile
用户画像 (CustomerProfileRule)
  ↓ informs
销售建议 (SalesAdvice / GearPackage)
```

关系必须是显式、可引用的对象，至少带 `relation_type`、源/目标 ID、`evidence_refs[]`、`effective_from`、`effective_to?`。允许的关系为 `has_series`、`contains_product`、`has_feature`、`suitable_for`、`unsuitable_for`、`fits_profile`、`compatible_with`、`complements`、`alternative_to`、`informs`；禁止从共现文本自动推导为可销售关系。系列可包含多个单品；单品只能有一个主品牌，但可引用多个兼容品或替代品。

## 4. 产品场景模型

`ProductScenario` 将场景需求标准化，作为检索过滤、匹配硬约束和推荐理由的共同语言。受控 `activity` 值至少包含：`hiking`（徒步）、`mountaineering`（登山）、`camping`（露营）、`skiing`（滑雪）、`urban_outdoor`（城市户外）、`high_altitude_activity`（高海拔活动）。一个单品可以有多个场景，并必须分别声明适配强度与限制。

| 字段 | 规则 |
| --- | --- |
| `activity` | 上述受控活动，可附子类型如 `multi_day_hiking`；请求中的显式活动优先。 |
| `environment` | 数组，如 `urban`、`forest`、`alpine`、`snow`、`wet`、`wind`、`low_temperature`、`high_uv`；不从用户住址推断。 |
| `season` | `spring|summer|autumn|winter|all_season` 或日期范围；跨季不确定时为 `unknown`。 |
| `difficulty` | `easy|moderate|challenging|expedition`；表示活动条件，不等同客户能力。 |
| `user_level` | `newcomer|casual|regular|advanced`；能力未知时不作推断。 |
| `fit` | `suitable|conditional|not_suitable`；`conditional/not_suitable` 必须有 `limitations` 与证据。 |
| `requirements` | 结构化需求，如保暖、防水、容量、抓地、雪况兼容；用于产品匹配。 |

场景匹配先排除 `not_suitable`、冲突安全条件和未审批/失效知识，再以活动、环境、季节、难度和用户等级计算可解释适配度。高海拔不自动等同专业登山：海拔、路线、天气、天数、用户经验或装备要求缺失时，降低置信度并要求人工确认。

## 5. 产品组合（Gear Package）模型

`GearPackage` 是可引用的建议模板，不是 SKU 套装、库存预留或促销规则。它由已审批产品知识定义需求覆盖；实际可推荐品须再通过 Retail Brain 的实时只读可售校验。

| 字段 | 类型 / 规则 |
| --- | --- |
| `package_id`, `name` | 稳定 ID 与展示名。 |
| `activity` | 必填；使用第 4 节受控活动或子类型。 |
| `environment` | 必填数组；明确气候、地形或城市条件。 |
| `difficulty` | 必填；`easy` 至 `expedition`。 |
| `products` | 必填数组：`{role, product_id? / category, quantity, required, alternatives[], evidence_refs[]}`；至少一个 `required`。 |
| `budget` | 必填 `{currency, min, max, basis}`；仅预算指引，税费/价格/促销以 Retail Brain 当时事实为准。 |
| `reason` | 必填数组；覆盖需求、兼容性、限制和每项引用。 |

示例模板：

| Gear Package | 活动 / 环境 / 难度 | 组合意图 |
| --- | --- | --- |
| 冈仁波齐转山装备组合 | `high_altitude_activity`；高海拔、风、低温、强 UV；`challenging` | 分层保暖、防风雨、步行支撑、补水/防晒等角色；必须提示高反、路线和专业安全要求由人工确认。 |
| 川西徒步装备组合 | `hiking`；山地、多变天气、可能潮湿；`moderate` | 徒步鞋、外层、分层衣物、背包/补水等按季节和天数替换的类别组合。 |
| 深圳周末徒步组合 | `hiking`；低海拔、湿热、城市近郊；`easy` | 轻量防雨、透气衣物、日用背包、补水等；不把它表述为高海拔或技术登山配置。 |

## 6. Knowledge Brain、Retail Brain 与 Recommendation Engine 集成

```text
已认证请求 + 需求/授权上下文
              ↓
Knowledge Brain：批准、有效、ACL 可见的 ProductEntity / Scenario / GearPackage + Citation
              ↓
Product Matching：硬约束过滤与可解释候选
              ↓
Retail Brain：产品/SKU 映射、渠道可售、库存快照与鲜度（只读）
              ↓
DATA-003.2 Recommendation Engine：排序、理由、置信度、限制、人工确认
```

### 6.1 DATA-003.2 对接契约

M4.3 是 DATA-003.2 “Knowledge Retrieval”和“Product Matching”的产品知识供给方，不替代其 Customer Context、同意策略、库存校验、排序、审计或人工确认。推荐引擎传入已通过策略门的最小化上下文：`organization_id`、`store_scope`、`activity`、`environment?`、`season?`、`difficulty?`、`customer_level?`、`budget?`、`channel`、`as_of`、`correlation_id`；禁止在此接口传递原始 PII。

M4.3 返回每个候选的：

- **产品匹配：** `product_id`、场景/画像适配、硬约束结果、互补/替代关系及未入选原因；
- **库存检查输入：** 商品/SKU 映射需求和可替代规则，交给 Retail Brain 校验；M4.3 不查询、更不修改库存；
- **推荐理由：** 区分产品知识事实、客户显式需求和 Retail Brain 库存事实，不将三者混写；
- **引用来源：** `knowledge_id`、版本、citation ID、定位符、来源哈希、片段范围与检索时点；
- **置信度：** `high|medium|low` 及证据完整性、知识新鲜度、场景完整度依据。它不是成交概率。

库存不可用、陈旧、隔离或越权时，Recommendation Engine 必须降级为“员工核实”或抑制，不得输出“有货”。推荐响应保留 DATA-003.2 的 `advisory_only_requires_human_approval` 处置标识。

## 7. 产品资料导入、Citation 与 ACL

### 7.1 Import Pipeline

支持 DOCX、PDF、Markdown 和 Excel（`.xlsx`）资料。导入是受信任的后台/服务操作，不提供给 agent 的公开写接口。

```text
文件接收与病毒/格式校验
  → 内容提取与分段（DOCX paragraph；PDF page；Markdown line/heading；Excel sheet/cell range）
  → Metadata + SHA-256 + 文档版本/去重
  → 字段/受控词表/来源完整性校验
  → ProductEntity 草稿、Citation 与 ACL 继承
  → 人工审核：draft → in_review → approved
  → 仅批准且有效内容索引、可检索
```

每次导入自动生成：

| 产物 | 最低内容 |
| --- | --- |
| `Metadata` | `document_id`、原文件名、MIME、格式、大小、内容哈希、导入人/时间、语言、解析器版本、源版本、组织与生命周期。 |
| `Citation` | `citation_id`、`knowledge_id`、`source_id`、来源类型、不可变/版本化 locator、页/段/行/工作表/单元格范围、chunk offset、来源哈希、captured/observed 时间。 |
| `ACL` | 从可信 `AuthContext` 继承的 `organization_id`、`owner_id`、`visibility`、`department_id?`、`role_scope?`；导入者不能通过文件元数据扩大访问范围。 |

解析失败、恶意/加密而不可审查文件、缺少主来源、无法定位引文、跨组织引用、受控字段不合法时，导入保留为隔离/草稿并不进入索引。Excel 中单元格级证据使用例如 `.../catalog.xlsx#sheet=Shells&range=B12:F12`；PDF 使用页码，Markdown 使用行或标题锚点。

### 7.2 ACL 与来源输出规则

授权由服务器依据 JWT/网关身份生成：不同组织一律拒绝；同组织 `admin` 允许；所有者允许；`visibility=organization` 允许；`visibility=department` 仅匹配部门（或 owner/admin）允许；其他私有内容拒绝。产品、品牌、场景、来源或部门筛选只能缩小已授权结果，绝不授予权限。

引用与正文同样受 ACL 保护：调用者无权访问父知识项时，不能获知 locator、片段、产品存在性或受限来源文本。所有返回引用包含来源版本/哈希和 retrieval timestamp；派生摘要还返回上游 `knowledge_id` 与 `extract|summarize|translate|aggregate` 血缘。

## 8. Product Knowledge API（设计契约）

**基础路径：** `/api/products`。所有端点仅支持读取，要求认证、服务器派生的组织/角色/门店范围和 `X-Correlation-Id`；响应采用 JSON、RFC 3339 UTC 时间，默认仅返回 `approved` 且有效的 ACL 可见内容。

| Endpoint | 目的与参数 | 最低响应 / 控制 |
| --- | --- | --- |
| `GET /api/products/search` | `query?`、`brand_id?`、`category?`、`activity?`、`environment?`、`season?`、`difficulty?`、`customer_level?`、`limit?`、`cursor?`。 | 命中产品摘要、场景适配、来源引文、知识版本、置信度/鲜度；所有过滤先做 ACL。 |
| `GET /api/products/{id}` | 稳定 `product_id`；可选 `include=technology,specification,scenarios,recommendations,citations`。 | 完整已授权 ProductEntity、关系、生命周期/有效期、Citation；未知或无权时采用不泄露存在性的响应。 |
| `GET /api/products/recommend` | `activity` 必填；`environment?`、`season?`、`difficulty?`、`customer_level?`、`budget?`、`store_id?`、`channel?`、`include_packages?`。 | 只读候选/组合、匹配理由、引用、知识置信度和限制；若要库存结果，服务以调用者范围委托 DATA-003.2/Retail Brain，返回库存时点而不做预留。 |

`/recommend` 不接受客户原始身份信息，也不能替代 DATA-003.2 的 `POST /api/recommendation` 个性化策略入口。对于缺少库存权限、库存依赖不可用或数据过期的情况，响应 `inventory.status` 为 `unavailable|stale|verification_required`，仍清楚标注为建议。标准错误使用 `400 INVALID_REQUEST`、`401 UNAUTHENTICATED`、`403 POLICY_DENIED`、`404 PRODUCT_NOT_FOUND`（仅安全时）、`409 KNOWLEDGE_STALE` 和 `503 DEPENDENCY_UNAVAILABLE`。

## 9. 审计、生命周期与质量门

- **生命周期：** `draft → in_review → approved → deprecated → archived`；默认检索/推荐仅限 `approved` 且处于 `effective_from/effective_to` 内的项目。
- **审核门：** 批准前必须有有效产品/品牌引用（如适用）、至少一个 primary Citation、可解析定位符/哈希、受控场景、ACL、审核记录和内容哈希。
- **审计：** 记录有效主体、agent actor（如有）、请求用途、相关 ID、过滤器、结果/引用 ID、策略决定、版本、鲜度、相关 ID；不记录完整 PII、私有源正文或库存之外的来源事实。
- **质量降级：** 来源过期、场景不完整、冲突规格、无法映射商品或库存陈旧时，显示限制、降低置信度或抑制建议；禁止由模型静默补全产品安全、规格或库存事实。
