## 四、安全加固步骤

> 以下步骤仅供**批准后的授权执行人**在窗口内使用。每个步骤开始前记录变更前状态，完成后立即执行对应验证。禁止把示例参数直接替换成未审批的宽泛值。

### 4.1 推荐安全变更顺序

1. 建立并验证新的实名管理员密钥登录路径；保持现有会话不断开。
2. 创建受限账号和最小 sudo 策略，完成权限验证。
3. 通过语法检查后应用 SSH 基线，并在第二个会话验证密钥登录与拒绝策略。
4. 以来源受限 SSH 规则先行，应用 UFW 默认拒绝入站和最小端口放行；从批准的管理来源验证访问。
5. 配置时间同步、持久日志与审计；验证服务运行和事件留痕。
6. 创建变更后备份，完成第五节验收并填写安全验收报告。

### 4.2 SSH 加固

**目标状态**

| 控制项 | 目标 |
| --- | --- |
| 公钥认证 | 启用：`PubkeyAuthentication yes`；每位管理员使用独立、已登记的现代公钥。 |
| root 远程登录 | 禁止：`PermitRootLogin no`。 |
| 密码登录 | 禁止：`PasswordAuthentication no` 与 `KbdInteractiveAuthentication no`。 |
| SSH 管理范围 | 仅允许 `{{APPROVED_ADMIN_SOURCE_CIDR}}`；在云网络边界与 UFW 同步限制。 |
| 额外基线 | 使用经审批的管理组（如 `vafox-ssh-admin`）、`MaxAuthTries 3`、`LoginGraceTime 30`、`X11Forwarding no`；默认禁用未获批端口转发。 |

**受控实施检查表**

- [ ] 为 `{{VAFOX_ADMIN_ACCOUNT}}` 创建/登记独立公钥认证；私钥始终由管理员在受控密钥介质或密钥管理系统中保管，不上传服务器以外的任何 Artifact。
- [ ] 确保授权密钥目录与文件属主、权限正确，且只允许批准的账号和管理组使用。
- [ ] 将 SSH 基线写入可审查的配置片段；先运行 SSH 配置语法检查，再重载服务。
- [ ] 在**新的独立 SSH 会话**中验证实名管理员的密钥登录成功，才可关闭旧会话。
- [ ] 验证 root 登录、密码登录和未授权来源均被拒绝；不得以“临时允许密码”代替修复。

### 4.3 用户权限加固

| 身份 | 创建/配置要求 | sudo / 登录限制 |
| --- | --- | --- |
| `{{VAFOX_ADMIN_ACCOUNT}}`（实名管理员） | 创建非 root、可归属个人的管理账号；加入批准的 SSH 管理组；绑定已登记公钥。 | 只授予与系统维护职责匹配的 sudo 权限。完整 sudo 须双人审批；禁止无审批的 `NOPASSWD: ALL`。 |
| `vafox-exec` | 创建专用、不可共享的受控执行服务账号；独立家目录/工作目录/日志目录，严格属主和权限。 | 默认不可交互 SSH 登录、不可直接 root、不可拥有通用 sudo。若业务确需 sudo，仅允许 `{{APPROVED_EXEC_SUDO_COMMANDS_REFERENCE}}` 中逐条批准的绝对路径命令。 |

**最小权限要求**

- sudo 规则必须位于独立、版本可追溯的 `/etc/sudoers.d/` 片段；变更前后均运行 `visudo -c`。
- `vafox-exec` 不得读取管理员家目录、`/etc/shadow`、SSH 私钥、其他服务目录或未获授权的日志；不得拥有 Docker socket、shell 任意执行或通配符 sudo。
- 禁止共享账号；管理员账号、服务账号和未来 Agent 身份必须分离、可撤销、可审计。

### 4.4 UFW 防火墙加固

**目标策略：** 默认拒绝所有入站；默认允许出站；拒绝转发；仅允许以下入站服务：

| 服务 | 协议/端口 | 来源 | 备注 |
| --- | --- | --- | --- |
| SSH | TCP/22 | `{{APPROVED_ADMIN_SOURCE_CIDR}}` | 必须在应用默认拒绝前添加并从批准来源验证。不得向全网开放。 |
| HTTP | TCP/80 | 公网 | 为未来获批的 HTTP→HTTPS 跳转或证书流程保留；当前空白主机不应监听。 |
| HTTPS | TCP/443 | 公网 | 为未来获批 Web/API 入口保留；当前空白主机不应监听。 |

**受控实施检查表**

- [ ] 记录启用前 UFW 状态、编号规则、默认策略、IPv4/IPv6 状态与实际监听端口。
- [ ] 先添加来源受限的 SSH/22 规则，并从批准管理来源成功建立第二会话。
- [ ] 添加 TCP/80 与 TCP/443 规则；不得因此安装、启动或部署任何 Web 服务。
- [ ] 设置默认拒绝入站、默认允许出站；确认不启用路由/转发。
- [ ] 启用/重载 UFW 后，使用编号规则复核仅存在批准规则；云安全组/ACL 与此设计一致，但本 Artifact 不授权修改云侧规则。

### 4.5 系统安全：时间、日志与审计

| 控制域 | 目标状态 | 验收要求 |
| --- | --- | --- |
| 时间同步 | 使用 `systemd-timesyncd` 或已批准的 NTP 客户端；使用批准的可信时间源；系统和安全日志以 UTC 关联。 | 时间同步服务为活动状态；同步状态正常；时间源、偏差和检查时间已记录。 |

## 六、回滚方案

### 6.1 触发条件与通用规则

以下任一情形触发停止后续变更并评估回滚：无法通过批准来源建立新的管理员 SSH 会话；SSH 配置语法失败；UFW 阻断受控管理路径；sudo 权限异常；日志/审计不可用；备份不可读取；出现未批准端口或权限扩大。

回滚由审批指定的负责人执行，并使用第三节的加密变更前备份。优先恢复受控管理访问和已知安全基线；**不得**通过永久开放公网 SSH、启用密码认证、允许 root 远程登录或禁用审计来“快速恢复”。每项回滚后均须在独立会话验证并记录。

| 回滚域 | 恢复步骤 | 验证 |
| --- | --- | --- |
| SSH 恢复 | 从变更前备份恢复 SSH 主配置与片段；执行语法检查；通过现有会话或已确认的带外控制台谨慎重载 SSH。 | 从批准来源以原先已验证的非 root 账号建立会话；确认 root/密码策略仍符合原批准基线。 |
| sudo 恢复 | 恢复 `/etc/sudoers` 与 `/etc/sudoers.d/` 的变更前版本、属主和权限；运行 `visudo -c`。 | 以授权管理员执行最小必要 sudo 测试；确认 `vafox-exec` 未获得通用 sudo。 |
| UFW 恢复 | 依据变更前编号规则和配置恢复默认策略与规则顺序；若必须暂时停用 UFW 进行带外恢复，须有紧急审批并立即恢复到已知规则集。 | 从批准 CIDR 确认 SSH；复核仅存在变更前批准规则及实际监听端口。 |
| 配置恢复 | 恢复时间同步、journald、rsyslog、audit、logrotate、APT 安全更新及相关 systemd 配置；按需重载服务。 | 服务 active、时间同步正常、日志/审计可写可检索；恢复后的配置校验和与备份一致。 |
