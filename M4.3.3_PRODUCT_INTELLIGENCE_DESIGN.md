# VAFOX M4.3.3 Product Intelligence MVP Design

**状态：** Design Phase / 仅设计，不部署。  
**前置设计：** `M4.3_PRODUCT_KNOWLEDGE_DESIGN.md`、`M4.3.2_KNOWLEDGE_ASSET_CENTER_DESIGN.md`、`M4.3_PRODUCT_IMPORT_PIPELINE_DESIGN.md`、`DATA003.2_RECOMMENDATION_ENGINE_DESIGN.md`。  
**目标：** 在已审批、可引用的产品知识之上，建立“产品 × 场景 × 客户 × 库存 × 销售”的只读产品智能层，为员工提供可解释、可审核的产品卡片、场景匹配和经营建议。

## 1. 原则、范围与边界

1. **知识有证据。** 产品属性、技术解释、场景适配、竞品比较和销售话术均须回链已发布资产的版本化 Citation；未知、冲突或证据不足必须明确展示。
2. **事实不搬家。** Product Intelligence 是派生读模型：SAP/ERP、POS、WMS 和 CRM 保留事实权威，Core 保留受治理事实中心；本层只读取经现有治理链路提供的快照和引用。
3. **场景优先且安全保守。** 先以活动、环境、难度和限制筛选，再以授权客户画像、库存与销售信号排序。高海拔、登山等高风险场景不把产品建议表述为安全保证。
4. **客户最小化。** 仅使用已授权、用途匹配且未过期的 `customer_id` 与聚合画像；不接收或返回原始 PII，缺失信息不由 AI 猜测。
5. **建议不执行。** 所有输出均为 `advisory_only_requires_human_approval`；员工必须复核尺寸、天气、路线、实时库存及客户需求后才能沟通。

### 明确禁止

- 修改 SAP，或修改 Core 生产同步、同步规则或来源事实；
- 自动下单、锁定/扣减库存、调拨、补货、改价或报价；
- 自动销售、自动触达客户或替员工完成交易；
- 把 AI 生成内容、未审批资产或缺少 Citation 的主张发布为产品事实。

## 2. Product Intelligence Architecture

```text
Product Asset（已版本化资料）
        ↓ extraction + citation + human review
Product Entity（稳定产品实体 / SKU 只读映射）
        ↓ approved scenario tags + technical claims
Scenario Understanding（活动、环境、难度、限制）
        ↓ policy + consent + customer facets
Customer Matching（授权画像、显式需求、预算、已有装备）
        ↓ inventory + sales snapshots + store scope
Recommendation Context（可回放的时点化上下文）
        ↓ explainable rules / ranked proposals
AI Product Intelligence（产品智能卡、装备组合、经营建议）
```

| 层级 | 输入 | 责任与输出 | 强制控制 |
| --- | --- | --- | --- |
| Product Asset | KAC 中 `Published`、有效、ACL 可见的资料 | 提供原件版本、来源等级、内容 hash、locator 和 Citation。 | Draft、Review、Archived、越权资产不可作为默认依据。 |
| Product Entity | 稳定 `product_id`、品牌/系列/技术及只读 SKU 映射 | 汇聚经审核的产品主张、场景、搭配和限制。 | 不反向覆盖 SAP 商品主数据；`unresolved/ambiguous` SKU 不可用于可售推荐。 |
| Scenario Understanding | 活动、环境、难度、季节/时长（若声明） | 输出需求角色、产品硬约束、适配程度和风险复核项。 | 受控枚举；未声明天气、路线或经验时不作推断。 |
| Customer Matching | 显式需求 + 授权客户聚合特征 | 生成客户适配信号、预算/已有装备缺口和排除条件。 | 用途、同意、质量和门店范围先于画像读取。 |
| Recommendation Context | 产品/场景/客户匹配 + Retail 快照 | 固化输入版本、来源、鲜度、约束、候选及排除原因。 | 缺库存、陈旧或越权时降级/抑制，不宣称有货。 |
| AI Product Intelligence | 上述受控上下文 | 输出产品卡、组合建议、销售辅助与经营建议。 | 每一结论标注事实/建议/未知、Citation、置信度、版本及人工下一步。 |

## 3. Product Intelligence Schema

`ProductIntelligenceEntity` 是产品实体的可发布智能视图，不是新的商品主数据，也不保存实时库存或客户明细。多值字段以数组表示；所有可发布主张都必须有 `citation[]`。

| 字段 | 类型与规则 |
| --- | --- |
| `product_id` | 必填，稳定产品实体 ID；关联而非替代 SAP SKU。 |
| `brand_id` | 必填或显式 `unresolved`；使用受控品牌 ID。 |
| `series` | 系列 ID/名称；须与品牌关系一致，可为 `unresolved`。 |
| `technology` | `[{name, mechanism, benefit, limitation, evidence_refs}]`；规格/机制不得由营销语臆测。 |
| `scenario` | `ScenarioFit[]`：活动、环境、难度、适配状态、需求和限制。 |
| `customer_profile` | 匿名适配画像，如等级、目标、偏好条件、排除项；不是 PII 或客户记录。 |
| `usage_environment` | 温度、降雨、海拔、地形等受控/有单位条件及适用边界；未知保持未知。 |
| `difficulty_level` | `easy|moderate|challenging|expedition`；描述行程条件，不等同客户能力。 |
| `gear_package` | 关联审核过的 `package_id`、产品角色、兼容/替代关系与条件。 |
| `selling_points` | `[{claim, audience, condition, claim_kind, evidence_refs}]`；将产品事实与培训话术区分。 |
| `competitor` | `CompetitorComparison[]`：仅已证实、同维度、带观察日期的客观比较。 |
| `recommendation` | 场景匹配规则、推荐理由模板、限制、排序版本；是建议，不是交易指令。 |
| `citation` | 字段级 Citation 指针，至少含 asset/version/hash/locator/claim path。 |
| `confidence` | `score`、`level`、`basis`、`limitations`；衡量证据与输入完整性，不是成交概率。 |

{

  "brand_id":"brand:example",
  "series":{"series_id":"series:example:alpine","name":"Alpine"},
  "technology":[{"name":"waterproof membrane","mechanism":"approved claim","benefit":"rain protection","limitation":"verify care and conditions","evidence_refs":["cit:ast:manual:v3:page-12"]}],
  "scenario":[{"activity":"hiking","environment":["shenzhen"],"difficulty_level":"easy","fit":"suitable","requirements":["breathability","rain_protection"],"limitations":["confirm weather and size"],"evidence_refs":["cit:ast:manual:v3:page-8"]}],

## 4. AI Product Card

产品智能卡是面向员工的只读呈现。每张卡必须显示知识版本、生成/审核状态和 `requires_human_approval=true`，并把“产品事实”“场景建议”“零售事实”分区呈现。

| 卡片区块 | 内容与展示规则 |
| --- | --- |
| 产品定位 | 品牌、系列、品类、目标需求与适用边界；不可将定位写成绝对性能承诺。 |
| 适用场景 | 活动、环境、难度、`suitable/conditional/not_suitable`、限制和需确认事项。 |
| 适合用户 | 已审批的匿名能力画像；对当前客户仅显示授权、聚合的匹配理由。 |
| 技术解释 | “技术/机制 → 作用 → 条件/限制”的通俗解释，区分官方事实与培训解释。 |
| 搭配产品 | 来自 `gear_package` 或经审核兼容关系的角色、必选/可选、替代项；不自动加购。 |
| 销售话术 | 条件化、非贬损的话术建议；不可承诺安全、效果、库存或价格。 |
| 竞品比较 | 相同产品维度的客观表格；显示来源日期、不可比较项和“请按客户需求选择”。 |
| 来源 Citation | 每个区块的 Citation、资产版本、来源等级与 locator；ACL 不足时不泄露来源正文。 |

卡片结尾固定输出：`库存请以门店实时核实为准`、`建议需员工确认后再沟通`，以及低置信度时的补问项（如路线、天气、尺码、已有装备）。

## 5. Scenario Matching Engine

```text
Activity
   ↓
Environment
   ↓
Customer Level
   ↓
Product Match
```

### 5.1 受控输入与匹配规则

| 阶段 | 受控值/输入 | 输出与规则 |
| --- | --- | --- |
| Activity | `hiking` 徒步、`mountaineering` 登山、`camping` 露营、`skiing` 滑雪、`high_altitude` 高海拔、`urban_outdoor` 城市户外 | 确定核心需求角色与禁止/条件性类目。高海拔可与徒步叠加，但不自动等同专业登山。 |
| Environment | `shenzhen`、`western_sichuan`、`tibet`、`snow_mountain`；可补充经证据支持的湿热、风雨、雪、昼夜温差 | 产出环境约束；天气与路线为时变信息，未接入权威数据时必须要求人工确认。 |
| Customer Level | `newcomer|intermediate|professional|unknown`，仅使用显式或授权画像 | 与难度/技术复杂度匹配；`unknown` 不扣分也不提升，改为补问。 |
| Product Match | 已审批产品场景、技术、限制、预算、渠道和库存状态 | 先执行硬过滤，再进行可解释排序，返回适配、排除与置信度。 |

**硬过滤：** 知识未批准/失效、`not_suitable`、关键安全或兼容限制、不可售渠道、超出显式预算、库存隔离或陈旧均不得进入“可成交首选”。库存未知可作为“人工核实的替代候选”，不能标注为有货。

**MVP 评分：**

```text
match_score = 0.35 × scenario_fit
            + 0.20 × technology_fit
            + 0.15 × customer_level_fit
            + 0.15 × inventory_fit
            + 0.10 × budget_fit
            + 0.05 × sales_signal_fit
            - evidence_or_risk_penalties
```

缺少客户授权时，移除 `customer_level_fit` 并将权重重分配到场景/技术/库存；销售只作为供给稳定性信号，不能压过场景、安全、证据和库存硬约束。规则版本、各分项、排除原因和数据时点必须随响应返回。

## 6. Gear Package Integration

`GearPackage` 是审核通过的角色级装备模板，不是 SKU 套装、价格承诺、库存预留或自动加购。流程如下：

```text
活动
  ↓
装备组合（需求角色、必选/可选、限制）
  ↓
产品推荐（知识匹配 → 库存验证 → 人工确认）
```

| 活动模板 | 场景与难度 | 装备组合（角色级） | 推荐控制 |
| --- | --- | --- | --- |
| 深圳周末徒步 | `hiking` / `shenzhen` / `easy` | 透气层、轻量雨具、日用背包、补水容器、防滑徒步鞋。 | 依据湿热、短途、降雨风险匹配；员工确认天气、尺码和路线。 |
| 川西7天 | `hiking` / `western_sichuan` / `moderate` | 分层保暖、硬壳、徒步鞋、40L 级背包、睡眠与补水角色。 | 多日、温差和天气变化需逐项补齐；预算和路线变动时重算。 |
| 冈仁波齐转山 | `high_altitude` / `tibet` / `challenging` | 保暖层、防风雨层、支撑鞋、遮阳/补水、应急角色。 | 仅装备辅助；强制人工确认高反风险、天气、路线、体能与专业安全要求。 |

每个角色至少关联一个已批准产品候选或受控品类；替代品需要兼容性 Citation。组合内任一必选角色无合格候选、库存状态不可核验或关键限制不明时，输出“组合不完整，需要人工核实”，而不是以无关商品填充。

## 7. Competitor Knowledge

竞品知识用于帮助员工理解取舍，不用于攻击、贬损或作无法验证的优劣断言。它是版本化的比较主张，来源可为品牌官方公开资料、经审核采购分析或可验证技术资料，并遵循 KAC 来源等级与 ACL。

| 维度 | 最小内容 | 表达规则 |
| --- | --- | --- |
| 品牌定位 | 声明的目标活动、产品线或设计取向 | 使用可引用定位描述，避免“更高端/更差”等无定义词。 |
| 价格区间 | 币种、适用地区/渠道、观察日期、价格类型 | 标注为历史/观察值，不替代实时价签或报价。 |
| 用户群 | 资料支持的目标能力/需求画像 | 不依据敏感属性标签客户，也不作排他判断。 |
| 产品差异 | 同类目、同用途、同条件下的材料、重量、功能、保修等可验证维度 | 对比维度、版本和不可比较项必须明确；证据冲突时展示冲突。 |
| 销售建议 | 基于客户需求的选择问题、适合何种取舍及何时建议核实 | “按需求比较”，不做竞品攻击、虚假稀缺或绝对胜负结论。 |

推荐话术示例：**“这两款针对的环境和取舍不同；我们可以先确认您的路线、天气和背负需求，再结合当前可售状态选择。”** 该话术不构成对竞品性能、库存或价格的承诺。

## 8. Retail Brain Integration

```text
已审批产品知识 + Citation
              + 销售数据（只读聚合）
              + 库存（只读快照）
              + 门店（授权范围/渠道）
                         ↓
                    经营建议
```

| 输入 | 使用方式 | 输出示例 | 约束 |
| --- | --- | --- | --- |
| 产品知识 | 识别场景覆盖、技术需求、组合缺口和可解释卖点。 | “该门店的深圳周末徒步组合缺少经审核的轻量雨具候选。” | 不以生成文本补足产品事实。 |
| 销售数据 | 使用明确窗口、样本充分性和异常标记的聚合销量。 | “建议复核过去 30 天该场景角色的销售覆盖。” | 不是需求预测、绩效结论或自动补货命令。 |
| 库存 | 使用 `sellable`、健康状态、时点、质量和门店范围。 | “优先核实库存健康为 watch 的替代品。” | 不预留、扣减或移动库存；陈旧即降级。 |
| 门店 | 结合授权门店/渠道及可售范围生成本地化建议。 | “为该店准备经审核的徒步场景产品卡与培训复核。” | 不跨越组织/门店 ACL，也不自动创建任务。 |

经营建议必须带 `knowledge_as_of`、`inventory_as_of`、`sales_window`、`model/rule_version`、证据引用、置信度和 `human_action_required`。允许的建议包括复核陈列/培训覆盖、人工检查场景组合缺口、请计划人员评估补货信号；不允许生成采购订单、调拨或价格修改。

## 9. DATA-003.2 Recommendation Integration

```text
Customer
   ↓ consent / purpose / authorized profile
Scenario
   ↓ controlled activity + environment + difficulty
Product Intelligence
   ↓ approved product facts + citations + matching rules
Inventory
   ↓ authorized sellable snapshot + freshness + store scope
Recommendation
   ↓ ranked advisory proposal + human approval
```

1. **Customer：** 接受 `customer_id` 或匿名上下文；按照 DATA-003.2 的同意、用途、组织与门店策略生成最小特征。无个性化授权时提供非个性化场景推荐。
2. **Scenario：** 客户显式活动、环境、难度、预算优先；未知字段保留 `unknown` 并转为员工补问项。
3. **Product Intelligence：** 仅检索 `Published`、有效、可访问且 Citation 完整的实体与装备模板；输出候选、限制、评分分项、知识版本和证据。
4. **Inventory：** 对候选执行门店/渠道 `sellable` 与鲜度校验，连接销售速度聚合；数据不可用时抑制“可成交首选”。
5. **Recommendation：** 返回单品和组合的理由、事实时间、数据质量、置信度、替代项和人工确认要求；不创建订单、报价、任务或客户消息。

## 10. Human Review, Versioning & Citation

### 10.1 内容生命周期

```text
AI draft → in_review → approved/published → deprecated/archived
             │
             └──→ rejected / returned_for_revision
```

AI 可提取候选实体、生成产品卡草稿、建议场景标签和比较摘要；它**不能**自动发布、替换已批准版本、提升来源等级、修改 ACL 或裁决冲突。每次编辑、审核或重新生成创建不可变版本并记录输入资产/实体版本、规则或提示模板版本、审核人、时间、决定和原因。

### 10.2 发布门槛

- 字段级 Citation 具备 `citation_id`、`asset_id`、`asset_version`、`content_sha256`、`locator`、`chunk_id`、`claim_path`、`captured_at` 和提取血缘；
- 产品实体、场景枚举、搭配关系、竞品比较和销售话术均已通过对应业务/领域审核；
- 资产为 `Published`、在有效期内、可访问，且冲突已解决或显式标注；
- 高风险场景的安全限制、人工确认文案和低置信度降级路径已审核；
- 每个输出保留来源等级、知识版本、Citation、置信度、限制与审计关联 ID。

失效、撤销或归档上游资产时，受影响智能实体立即停止默认检索并进入复审；历史回答仅在授权审计查询中保留，并标明过期状态。

## 11. API Contract

所有 API 为 JSON 只读决策支持接口，使用认证身份、服务端派生 ACL、`X-Correlation-Id`、`X-Purpose`、UTC RFC 3339 时间和审计记录。客户端不得指定或扩大组织、门店、角色、客户可见性或零售事实范围。

### 11.1 `GET /api/product-intelligence/{id}`

读取一个已批准产品智能实体与卡片。需要 `product_intelligence:read`；可选参数：`store_id`、`channel`、`include=card,scenarios,gear_packages,competitors,citations`、`as_of`。响应包含 `entity`、`ai_product_card`、字段级 `citations`、`knowledge_version`、`confidence`、`limitations`、`freshness` 和 `disposition: advisory_only_requires_human_approval`。无权限时不得泄露实体存在性。

### 11.2 `GET /api/product-intelligence/search`

按已审批智能实体检索。参数：`q?`、`brand_id?`、`series?`、`activity?`、`environment?`、`difficulty_level?`、`customer_level?`、`store_id?`、`channel?`、`cursor?`、`limit=1..50`。响应返回 ACL 过滤的摘要、场景适配、Citation 指针和知识鲜度；不以搜索结果承诺库存或价格。

### 11.3 `POST /api/product-intelligence/recommend`

生成一次即时、可回放的只读建议。需要 `product_intelligence:recommend`，且用途为获批准的客户服务/导购/分析用途。

```json
{
  "customer_id":"cst_01J...",
  "scenario":{"activity":"hiking","environment":["shenzhen"],"difficulty_level":"easy"},
  "store_id":"store_nanshan",
  "channel":"store",
  "budget":{"currency":"CNY","min":500,"max":2000},
  "include_gear_package":true,
  "limit":5
}
```

成功响应至少包含 `recommendation_id`、`context_snapshot`、`recommendations`、`gear_packages`、`excluded_candidates`、`recommendation_reason`、`citations`、`confidence`、`freshness`、`inventory_status`、`sales_signal`、`policy`、`rule_version`、`limitations`、`correlation_id` 和 `disposition`。不接受电话、微信号或其他原始 PII；不执行任何业务写入。

标准错误：`400 INVALID_REQUEST`、`401 UNAUTHENTICATED`、`403 POLICY_DENIED`、`404 NOT_FOUND`（仅可安全披露时）、`409 CONTEXT_OR_KNOWLEDGE_STALE`、`422 REVIEW_REQUIRED|CONSENT_REQUIRED`、`503 RETAIL_DEPENDENCY_UNAVAILABLE`。`inventory_status=stale|unavailable|verification_required` 时，响应必须抑制“有货”结论。
