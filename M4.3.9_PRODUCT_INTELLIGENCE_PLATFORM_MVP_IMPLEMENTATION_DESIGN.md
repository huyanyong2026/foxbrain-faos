# VAFOX M4.3.9 Product Intelligence Platform MVP Implementation Design

**状态：** Implementation Design / 仅设计，不部署。  
**前置设计：** `M4.3_PRODUCT_KNOWLEDGE_DESIGN.md`、`M4.3.2_KNOWLEDGE_ASSET_CENTER_DESIGN.md`、`M4.3.3_PRODUCT_INTELLIGENCE_DESIGN.md`、`M4.3.8_PRODUCT_INTELLIGENCE_PLATFORM_DESIGN.md`。  
**目标：** 把 Product Intelligence Platform（PIP）定义为可运行的 MVP 能力边界、数据契约、处理流水线、接口、审核与验证方案，为后续实施提供单一、可审计的设计基线。

> 本文不部署、不创建运行环境、不改动现有生产服务。MVP 是只读决策支持能力：AI 可以生成候选和草稿，不能发布知识、修改 SAP/Core 生产同步，也不能执行采购、订单、库存或销售动作。

## 1. MVP Architecture

```text
Product Asset Upload
        ↓
Document Processing
        ↓
Product Entity Extraction
        ↓
AI Product Card Generator
        ↓
Human Review
        ↓
Knowledge Publish
        ↓
Search / Recommendation API
        ↓
Retail Brain (read-only business advice)
```

| 组件 | MVP 职责 | 输入 | 输出 | 强制约束 |
| --- | --- | --- | --- | --- |
| Asset Upload | 接收、校验、隔离并版本化保存原始产品资料。 | 文件、受服务端验证的身份、品牌/来源声明。 | `ProductAsset`、SHA-256、初始 ACL、处理任务。 | 上传不等于发布；客户端不能扩大 ACL 或确认官方来源。 |
| Document Processing | 解析结构、OCR、分块并生成可定位文本。 | 已通过安全检查的资产版本。 | `metadata`、`chunks`、locator、解析质量。 | 加密、损坏、恶意或不可解析文件进入隔离/失败状态。 |
| Product Entity Extraction | 从 chunk 提取品牌、系列、型号、技术、规格、场景和关系候选。 | 解析产物、受控词表、已有实体。 | Draft `ProductEntity`、冲突/去重结果、Citation 候选。 | 抽取是候选；不得从营销文案推断未声明的技术或安全性能。 |
| AI Product Card Generator | 依据已审核或待审实体生成结构化卡片草稿。 | 实体版本、字段级 Citation、模板版本。 | Draft `ProductCard`、置信度、限制、补问项。 | 不得虚构 Citation；没有足够证据的字段必须是 `unknown`。 |
| Human Review | 审核事实、来源、冲突、权限、适用限制和发布决定。 | Draft、原始资产、差异、审计上下文。 | `approved`、`returned`、`rejected` 决议。 | 审核人与受限/自身提交事项隔离；AI 不可审批。 |
| Knowledge Publish | 原子发布经批准的实体、卡片与索引版本。 | 已通过门槛的审核决议。 | `Published` 版本、检索索引、回滚指针。 | 仅 Published、有效、ACL 可见、Citation 完整内容可服务。 |
| Search / Recommendation API | 在调用者权限内返回可引用、可解释的只读结果。 | 查询/场景、服务端身份、授权门店范围、只读快照。 | 搜索命中、产品卡、推荐或降级说明。 | 不返回 Draft/Review/Archived/越权内容，不写回业务系统。 |

### 1.1 运行边界与依赖

- PIP 保存产品知识的派生读模型和对外部 SKU 的**只读映射**；SAP/ERP、Core、POS、WMS、CRM 仍是商品、价格、销量、库存和客户事实的权威来源。
- 文档原件与其 hash 不可变；更正通过新版本完成，并用 `supersedes_version_id` 关联，绝不覆盖既有 Citation。
- 搜索与推荐的默认知识集合只包含 `Published`、未过期、ACL 可见且字段 Citation 完整的版本。
- 运行时库存、价格、可售状态和天气不写入产品卡事实；需要时由已授权的只读事实快照提供，并在结果中声明 `as_of` 与鲜度。

## 2. MVP Implementation Scope

### 2.1 Phase 1 必须实现

| 能力 | MVP 最小交付 | 完成定义 |
| --- | --- | --- |
| 1. Product Asset Upload | 支持上传、文件安全/格式/大小检查、SHA-256、不可变版本、初始 ACL、异步处理状态。 | 受权用户可上传支持格式；失败可查询原因；上传资产不会进入默认搜索。 |
| 2. AI Extraction Pipeline | 格式解析、OCR、元数据、chunk、候选实体/主张、Citation、schema/词表/冲突校验与 Review 队列。 | 每个候选主张可回链至 hash 与精确 locator；不可引用内容不得通过发布门槛。 |
| 3. Product Card Generation | 为一个 Published 产品实体生成规范化的可解释卡片；支持草稿重生成。 | 输出所有第 5 节字段、字段 Citation、Confidence、限制和版本。 |
| 4. Citation Generation | 支持页、段、幻灯片、单元格、行及图像区域定位。 | API 中的每个产品事实字段均能返回可访问 Citation 或明确未知。 |
| 5. Product Search API | ACL 过滤的关键词/结构化筛选、分页、排序、Citation 摘要。 | 只能检索 Published 有效版本；越权/草稿/归档内容不泄露。 |
| 6. Recommendation API | 场景约束、硬过滤、可解释排序、库存/销售快照降级、替代品和人工确认。 | 返回只读建议、证据、规则/知识版本与 `advisory_only_requires_human_approval=true`。 |

### 2.2 非 MVP / 后续阶段

- 自动实体合并、自动发布、自动来源等级提升或自动 ACL 调整；
- 写入 SAP/Core、修改生产同步、库存预留/调拨、自动采购/下单、自动报价/改价、自动销售或客户触达；
- 自动采购建议执行、实时价格引擎、完整竞品爬取、客户 PII 画像仓库；
- 未经人工确认的高风险户外安全结论或安全保证。

## 3. Data Model

所有时间使用 UTC RFC 3339；所有 ID 为稳定、不可复用标识；状态变迁和审核决定必须写入不可变审计记录。`organization_id`、`brand_id`、`store_scope` 和 ACL 均由服务端认证上下文派生或校验，不能信任客户端自报值。

### 3.1 Core entities

| 模型 | 关键字段 | 规则 |
| --- | --- | --- |
| **Product Asset** | `asset_id`、`asset_version`、`organization_id`、`brand_id?`、`asset_type`、`mime_type`、`content_sha256`、`size_bytes`、`storage_ref`、`source`、`acl_id`、`processing_status`、`review_status`、`effective_from/to`、`created_at` | 原件存证单元；`brand_id=unresolved`、无 hash、无来源或未通过安全检查时不可发布。 |
| **Product Entity** | `product_id`、`entity_version`、`brand_id`、`series`、`model`、`category`、`sku_mappings[]`、`attributes`、`scenario_fits[]`、`evidence_refs[]`、`review_status` | 产品知识实体，不替代 SAP SKU；`ambiguous/unresolved` SKU 映射不可作为可售推荐候选。 |
| **Product Intelligence** | `intelligence_id`、`product_id`、`entity_version_ref`、`claims[]`、`technology[]`、`customer_profiles[]`、`gear_packages[]`、`limitations[]`、`confidence`、`citation_ids[]` | 聚合且可解释的读模型；每条 `claim` 记录 `claim_kind`（fact / interpretation / sales_guidance）和 Citation。 |
| **Product Card** | `card_id`、`product_id`、`intelligence_version_ref`、`sections`、`citation_ids[]`、`confidence`、`generator_version`、`review_status` | 只读展示投影；重新生成创建新 Draft，不静默改写已发布卡。 |
| **Citation** | `citation_id`、`asset_id`、`asset_version`、`content_sha256`、`chunk_id`、`locator`、`claim_path`、`source_rank`、`captured_at`、`extractor_version` | `locator` 必须精确回链到版本化原件；ACL 继承 Asset，未授权者不得看到标题、正文或 locator。 |
| **Version** | `version_id`、`resource_type`、`resource_id`、`revision`、`parent_version_id?`、`supersedes_version_id?`、`schema_version`、`effective_from/to`、`created_by`、`created_at` | 发布版本不可原地修改；历史响应引用当时版本。 |
| **Review Status** | `Draft`、`Review`、`Published`、`Archived`、`Rejected` | 默认服务仅使用 `Published` 且在有效期内的实体/智能/卡片；`Rejected` 仅供审计和授权协作。 |

### 3.2 关系与完整性约束

```text
ProductAsset (1) ──< Citation (N)
ProductAsset (N) ──< Chunk (N)
ProductEntity (1) ──< ProductIntelligence (N versions)
ProductIntelligence (1) ──< ProductCard (N versions)
ProductEntity / ProductIntelligence / ProductCard ──< Citation (N)
Version (1) ── belongs_to ── exactly one governed resource version
ReviewDecision ── approves/returns/rejects ── Version
```

发布门槛：产品身份、品牌、品类、版本、ACL、有效期、至少一条有效 Citation、字段级 evidence、审核记录完整；有冲突的主张必须被裁决或保留为非发布字段。高风险场景还必须有明确限制和人工确认提示。

### 3.3 规范化状态机

```text
Draft → Review → Published → Archived
  │       │
  └──────→ Rejected / returned_to_draft
```

- `Draft`：上传、解析、提取和卡片草稿可见于授权协作者，但不可进入默认索引。
- `Review`：冻结待审版本，审核人可通过、退回或拒绝；不得作为回答事实。
- `Published`：原子写入实体、卡片和搜索索引；只可由新版本取代。
- `Archived`：从默认搜索/推荐移除，保留审计、血缘和授权历史查询。

## 4. Asset Processing Pipeline

支持 **PDF、DOCX、PPT/PPTX、Excel/XLSX、Markdown、图片（PNG/JPEG/WebP）**。处理是可重试的异步工作流；每一步写入任务 ID、输入/输出版本、工具版本、耗时、错误码和相关 ID。

```text
上传
  ↓
文件解析
  ↓
Metadata
  ↓
Chunk
  ↓
Extraction
  ↓
Validation
  ↓
Review
  ↓
Publish
```

| 阶段 | 处理 | 格式定位 / 产物 | Gate 与失败处理 |
| --- | --- | --- | --- |
| 上传 | 鉴权、MIME 魔数/扩展名一致性、大小/恶意/加密检查、重复 hash 检测、原件隔离保存。 | `asset_id`、`asset_version`、hash、ACL、`processing_status=queued`。 | 无权、恶意、加密、损坏或超限：`quarantined/failed`，不解析、不索引。 |
| 文件解析 | PDF 文本/页；DOCX 标题/段落；PPT 幻灯片/shape；Excel sheet/表头/单元格；MD 标题/行；图片 OCR/区域。 | 统一 `document_node` 与原始 locator。 | 低 OCR、空文本、表格结构不明保留质量标记并进入人工处理。 |
| Metadata | 识别语言、资料类型、候选品牌/系列、版本、来源等级、产品范围、有效期和血缘。 | 受控 metadata 候选。 | `unresolved` 保留，不能被模型补齐为事实或通过发布。 |
| Chunk | 按语义边界切块，保留父文档节点、页/段/单元格范围、字符偏移及表头上下文。 | `chunk_id`、内容 hash、embedding 输入、locator。 | chunk 不跨越 ACL；表格缺表头或图片缺区域标为 `needs_review`。 |
| Extraction | 提取产品、品牌、系列、技术、规格、场景、客户画像、搭配和销售建议候选。 | Candidate entity/claim + evidence refs。 | AI 输出标记为 `generated_candidate`，不得成为唯一高影响证据。 |
| Validation | JSON Schema、受控词表、单位/范围、品牌目录、实体去重、版本冲突、Citation 与 ACL 完整性校验。 | validation report、冲突集合、置信度。 | 无 Citation、无 hash、字段不合规、ACL 冲突或模糊 SKU：阻断发布。 |
| Review | 商品经理/领域审核人对原件定位、差异、风险、权限和来源进行决议。 | `ReviewDecision`、理由、审计记录。 | 退回或拒绝时保留草稿；审核者不得提升自己无权资料。 |
| Publish | 原子发布 Product Entity、Intelligence、Card 和检索索引；记录生效时间。 | Published version、索引事件、回滚指针。 | 任何一个发布门槛失败均不部分发布；撤销上游资产触发依赖复审与索引撤除。 |

### 4.1 Citation locator contract

| 格式 | locator 示例 |
| --- | --- |
| PDF | `#page=12&chars=120-420` |
| DOCX | `#heading=护理&paragraph=3` |
| PPT/PPTX | `#slide=7&shape=2` |
| Excel/XLSX | `#sheet=面料&range=B12:F12` |
| Markdown | `#heading=装备&lines=42-58` |
| 图片 | `#image=1&region=x1,y1,x2,y2` |

每个 locator 必须与 `asset_version + content_sha256 + chunk_id` 一同使用。Citation 访问必须先执行 Asset ACL；搜索摘要只可展示调用者可读的文本片段。

## 5. Product Card Generator

### 5.1 Input and generation contract

生成器只接收已授权的 Product Entity / Product Intelligence 版本、其 Citation、适用场景和受控模板。它按字段生成“事实”“解释”“建议”三类内容，保存 `generator_version`、`prompt/template_version`、输入版本和生成时间。若证据不完整，输出 `unknown`、低 Confidence 和需人工补充项，而不是推断。

| 输出区块 | 结构化字段 | 生成规则 |
| --- | --- | --- |
| 产品名称 | `product_name`、`model_code?` | 取已发布稳定身份；颜色/尺码不混入产品型号。 |
| 品牌 | `brand_id`、`brand_name` | 必须通过品牌目录与 Citation/来源审核。 |
| 系列 | `series` | 无明确证据则显示未知。 |
| 定位 | `positioning`、`target_need`、`boundaries[]` | 条件化描述，不声明绝对性能。 |
| 技术 | `technologies[]`、`specifications[]`、`limitations[]` | 技术事实、作用及限制分开；每项带 Citation。 |
| 场景 | `scenario_fits[]`、`requirements[]`、`risk_notes[]` | 返回 `suitable/conditional/not_suitable` 与需确认项。 |
| 客户画像 | `customer_profiles[]` | 仅匿名等级/需求；不包含或推断 PII。 |
| 搭配 | `gear_packages[]`、`alternatives[]` | 角色级搭配，不自动加购、锁货或承诺套装价格。 |
| 销售建议 | `sales_guidance[]` | 作为可编辑话术草稿，明确不能承诺价格、库存、安全或竞品优劣。 |
| Citation | `citations[]` | 字段级引用，并受 ACL 过滤。 |
| Confidence | `confidence`、`score`、`basis[]`、`limitations[]`、`questions[]` | 表示证据完整度和输入质量，不表示成交概率。 |

所有卡片固定显示：**“库存、价格、尺码及实时场景请由员工核实；本建议需人工确认后使用。”**

### 5.2 Card response example

```json
{
  "card_id": "pcd:kailas:mont:v1",
  "product_id": "prd:kailas:mont",
  "knowledge_version": "pi:kailas:mont:v1",
  "product_name": "MONT",
  "brand": "KAILAS",
  "positioning": {"text": "基于已批准资料的条件化产品定位。", "citation_ids": ["cit:..."]},
  "scenario_fits": [{"activity": "hiking", "fit": "conditional", "limitations": ["确认路线、天气和尺码"], "citation_ids": ["cit:..."]}],
  "sales_guidance": [{"text": "请先确认客户的活动、环境与个人需求。", "claim_kind": "sales_guidance", "citation_ids": ["cit:..."]}],
  "confidence": {"level": "medium", "score": 0.78, "basis": ["关键字段有证据"], "limitations": ["实时库存未含"], "questions": ["使用场景是什么？"]},
  "advisory_only_requires_human_approval": true
}
```

## 6. API Contract

### 6.1 Shared rules

所有接口使用 JSON、认证身份、服务端 ACL、`X-Correlation-Id`、`X-Purpose`、UTC 时间与审计。客户端不可指定或扩大组织、品牌、门店、角色、客户或来源的可见范围。所有成功的推荐响应均包含 `advisory_only_requires_human_approval: true`。

标准错误：`400 invalid_request`、`401 unauthenticated`、`403 forbidden`、`404 not_found_or_not_authorized`、`409 version_conflict`、`413 file_too_large`、`415 unsupported_media_type`、`422 validation_failed`、`429 rate_limited`、`503 dependency_unavailable`。为避免资源枚举，无法访问与不存在均可返回 `404`。

### 6.2 `POST /api/product-assets/upload`

**用途：** 创建 Draft Product Asset 与处理任务。`multipart/form-data` 包含 `file` 和可选的最小 metadata；身份、最大 ACL、组织与品牌授权由服务端校验。

```json
{
  "asset_type": "product_manual",
  "brand_id": "brand:kailas",
  "source": {"source_type": "brand_partner", "declared_version": "2026.1"},
  "effective_from": "2026-07-21T00:00:00Z"
}
```

**`202 Accepted`：**

```json
{
  "asset_id": "ast:kailas:01J...",
  "asset_version": "1",
  "review_status": "Draft",
  "processing_status": "queued",
  "content_sha256": "<sha256>",
  "job_id": "job:asset-process:01J...",
  "links": {"status": "/api/product-assets/ast:kailas:01J..."}
}
```

### 6.3 `GET /api/product-intelligence/{id}`

**用途：** 返回调用者有权访问的 Published 产品智能及当前/指定发布版本的产品卡。`id` 为 `product_id`；可选 `version` 仅允许访问者可读取的历史 Published 版本。

**`200 OK` 必含：** `product_id`、`knowledge_version`、`review_status=Published`、产品卡字段、字段级 `citations[]`、`confidence`、`limitations[]`、`knowledge_as_of`、`advisory_only_requires_human_approval`。如果没有可访问的已发布版本，返回 `404`。

### 6.4 `GET /api/product-intelligence/search`

**用途：** 在可访问的已发布知识中执行文本和结构化搜索。

| 参数 | 说明 |
| --- | --- |
| `q` | 可选关键词；与结构化过滤至少提供其一。 |
| `brand_id`、`category`、`series` | 可选受控筛选；服务端再执行 ACL。 |
| `activity`、`environment`、`customer_level` | 可选场景筛选，未知不被自动猜测。 |
| `store_id?` | 仅用于服务端授权范围内的可售上下文，不改变知识 ACL。 |
| `limit`、`cursor` | 分页，`limit` 最大值由服务端限制。 |

**`200 OK`：** 返回 `results[]`（`product_id`、名称、简短定位、匹配原因、Citation 摘要、confidence、knowledge_version）、`next_cursor?`、`query_interpretation`、`excluded_count` 和 `knowledge_as_of`。结果必须过滤无效、归档、草稿、审核中和越权内容。

### 6.5 `POST /api/product-intelligence/recommend`

**用途：** 对显式场景做只读、可解释的产品/搭配建议；可接受匿名或已通过用途/同意策略的最小客户上下文。

```json
{
  "scenario": {
    "activity": "hiking",
    "environment": ["rain", "wind"],
    "difficulty": "unknown",
    "destination": "川西",
    "budget": {"currency": "CNY", "max": 3000}
  },
  "store_id": "store:cd:001",
  "customer_context": {"mode": "anonymous", "level": "unknown"},
  "include_inventory": true,
  "include_sales_signals": true
}
```

**处理顺序：** ACL/用途策略 → Published/有效/Citation 完整知识 → 场景硬过滤（`not_suitable`、安全/兼容冲突、预算）→ 已授权只读库存/销售快照 → 可解释排序 → 降级/替代建议。

**`200 OK`：** 返回 `recommendation_context_ref`、`candidates[]`、`excluded_candidates[]`、`questions[]`、`limitations[]`、`rules_version`、`knowledge_as_of`、`inventory_as_of?`、`sales_window?`、`data_quality`、`citations[]`、`confidence` 和人工确认标记。库存未知、陈旧、隔离或无权时，响应必须降级为“请员工核实”，不得宣称可售。

## 7. KAILAS MVP Validation

验证只使用经授权、可追溯的 KAILAS 样本资产与只读业务快照；不使用真实客户 PII，不写入 SAP/Core/POS/WMS。每条预期结论应由商品经理确认，不预设“正确”销售答案。

| 验证问题 | 测试输入 | 预期 MVP 行为 / 通过标准 |
| --- | --- | --- |
| 产品问题：**MONT适合什么客户？** | KAILAS MONT 已发布资料 + 匿名客户等级/需求（如有）。 | 返回带 Citation 的产品定位、适配画像、场景条件、限制与补问项；客户信息未知时不猜测，不做安全保证。 |
| 场景问题：**第一次川西需要什么装备？** | `activity=hiking`、`destination=川西`、`customer_level=unknown`。 | 返回角色级装备候选与风险/缺失信息，不把地区经验泛化为路线事实；要求确认季节、天气、路线、天数、体能和安全条件。 |
| 经营问题：**哪些SKU需要重点推广？** | 授权门店/周期内的聚合销售、产品知识、主推候选和数据质量。 | 输出观察、推断、人工建议三段，显示 SKU 映射、数据窗口、Citation、规则版本；不自动创建促销、价格或销售指令。 |
| 库存问题：**哪些商品需要补货？** | 只读库存健康、销量趋势、门店范围、产品覆盖关系。 | 输出待人工评估的补货信号、鲜度和限制；不创建采购单、调拨、库存预留或自动补货。 |

### 7.1 Validation procedure

1. 上传覆盖手册、培训材料、场景资料和表格的 KAILAS 样本，确认每种格式产生正确 hash、locator 和处理状态。
2. 抽样核验每个回答的产品事实和场景关系都能打开至授权原件版本；撤销一份上游资产后验证相关结果从默认索引移除。
3. 使用采购员、品牌商、商品经理、运营、员工五种身份重复同一查询，验证 ACL 过滤和最小可见原则。
4. 用库存可用、陈旧、不可用三种只读快照调用推荐接口，验证 `inventory_as_of`、降级文字和禁止可售承诺。
5. 对四个问题由商品经理/运营审核“观察—推断—建议”边界、Citation、Confidence、人工确认文案和禁止执行行为。

## 8. Retail Brain Integration

```text
Published Product Intelligence + Citation
       + Sales Data (read-only aggregate)
       + Inventory (read-only snapshot)
       + Store (authorized scope)
                    ↓
              Retail Brain
                    ↓
      经营建议 / Human Review Queue
```

| 输入 | 最小化使用 | 输出示例 | 绝不执行 |
| --- | --- | --- | --- |
| Product Intelligence | 已发布产品、场景适配、搭配角色、Citation、知识版本。 | 识别某场景的产品知识/培训覆盖缺口。 | 把 AI 草稿当产品事实。 |
| Sales Data | 授权时间窗内的聚合销量、转化、退换和数据质量。 | 标示某 SKU/品类在指定门店的观察信号。 | 自动促销、改价或销售。 |
| Inventory | 授权门店的可售状态、库存健康、鲜度、缺货/滞销风险。 | 提醒人工复核替代品、补货信号或陈列覆盖。 | 锁货、扣减、调拨、补货或下单。 |
| Store | 授权门店范围、区域、陈列/培训状态和服务能力。 | 建议复核门店培训/陈列资料。 | 以门店信息推断客户敏感属性。 |

Retail Brain 结果必须将内容分为 **Observation（可验证事实）**、**Inference（规则化推断）** 与 **Human recommendation（待人工决定）**，并返回：`knowledge_as_of`、`inventory_as_of`、`sales_window`、`data_quality`、`rules_version`、`citation_ids`、`store_scope`、`limitations[]` 和 `human_action_required=true`。结果只进入人工 Review Queue。

## 9. Permissions (RBAC + ACL)

### 9.1 RBAC role baseline

| 角色 | Asset | Draft/Review | Publish | Search / Card | Recommendation / Retail |
| --- | --- | --- | --- | --- | --- |
| 采购员 | 在授权品牌/组织上传采购资料。 | 编辑自己的 Draft、提交审核。 | 否，除非另授商品经理发布权限。 | 仅 ACL 可见内容。 | 仅授权门店的只读结果。 |
| 品牌商 | 在授权品牌范围上传官方资料。 | 编辑自己的 Draft、响应退回。 | 否；不得自证官方等级或提升 ACL。 | 仅自身被授权内容。 | 不可访问其他品牌或门店经营数据。 |
| 商品经理 | 上传/维护授权产品资料。 | 审核实体、Citation、冲突、SKU 映射与卡片。 | 可发布授权品牌/品类，需审计且避免自审受限事项。 | 授权范围内。 | 可看授权经营建议，不可执行。 |
| 运营 | 上传培训/运营资料。 | 可审核运营/销售话术类内容；产品事实按领域授权。 | 仅授权内容类型。 | 授权范围内。 | 可查看授权门店的只读经营建议。 |
| 员工 | 通常不上传；如获授权仅创建个人 Draft。 | 不可审批/发布。 | 否。 | 仅 Published、有效、ACL 可见卡片/搜索。 | 使用非执行型推荐，须人工确认。 |

### 9.2 ACL enforcement

ACL 是每个 Asset 的组织、品牌、所有者、角色、门店/部门范围以及显式 allow/deny 的服务端策略。Chunk、OCR、缩略图、Citation、实体、卡片、搜索命中和缓存均继承或收紧 Asset ACL；标签、筛选条件、角色或模型置信度绝不扩大可见性。推荐必须同时通过知识 ACL 与 Retail 数据的用途/组织/门店范围策略；取二者交集。

审计事件至少记录：主体、角色、组织/门店范围、动作、资源/版本、决定、理由、Correlation ID、用途、时间、策略版本和拒绝原因。对 Citation 的访问也必须审计。

## 10. Explicit Boundaries

以下事项在 MVP 及本文设计范围内均**禁止**：

- 修改 SAP，或修改 Core 生产同步、同步规则、来源事实与生产数据；
- 自动采购、自动补货、自动下单、自动调拨、库存锁定或扣减；
- 自动销售、自动报价、自动改价、自动触达客户或替员工完成交易；
- 自动发布 AI 内容、自动裁决冲突、自动提升来源等级或自动扩大权限；
- 将 Draft、Review、Archived、无 Citation、失效或越权内容作为默认搜索/推荐依据；
- 把产品建议表达为安全、可售、价格、库存或效果保证。

## 11. MVP Acceptance Checklist

| 验收项 | 可验证证据 |
| --- | --- |
| □ Upload | 支持文件上传、隔离/错误路径、hash、版本、ACL 与异步处理状态。 |

## 12. Implementation Handoff Sequence

1. 先建立 schema、生命周期、ACL 继承和审计契约，再接入上传与解析工作流。
2. 以 Citation 完整性作为实体、智能和卡片发布的硬门槛，并先实现 Review/Publish 原子性。
3. 在不连接写入路径的前提下实现搜索，再实现场景推荐与只读 Retail snapshot 适配器。
4. 以 KAILAS 受权样本执行第 7 节验证；只有验收清单全部通过，才进入后续部署规划。本文不包含部署动作。
