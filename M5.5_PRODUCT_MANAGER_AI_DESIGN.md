# VAFOX M5.5 Product Manager AI Design

**状态：** Design Phase / 仅设计，不部署。  
**前置设计：** M5.1 Digital Employee Foundation、M5.2 Employee AI Workspace、M5.3 Store Manager AI、M5.4 Procurement AI、M4.3 Product Intelligence Platform。  
**负责人：** Codex（设计开发）；WorkBuddy（环境）；Marvis（管理）；CTO（验收）。

## 1. 目标、原则与边界

M5.5 定义 **Product Manager AI**：在经过授权的品牌、品类、门店、商品和知识范围内，把品牌知识、产品情报、销售、库存和客户洞察组织为可解释的商品策略建议，帮助商品经理进行品牌、品类、SKU 生命周期和门店商品策略的人工决策。它是只读的分析与建议能力，不是交易、定价或主数据系统。

1. **范围先于推理。** 每次请求由认证身份在服务端派生上下文；客户端提供的筛选条件只能缩小 `brand_scope`、`category_scope`、`store_scope` 与 `product_scope`，不能扩大可见范围。
2. **事实、推断和建议分离。** 输出固定区分 `facts`、`analysis`、`assumptions`、`unknowns`、`recommendations`；趋势相关性不得表述为因果或业绩承诺。
3. **时点、证据和质量可见。** 每个结论须携带 `as_of`、比较窗口、指标定义、数据质量、Citation、知识/规则/模型版本及置信度。数据过期、冲突或不足时标记 `needs_verification`。
4. **人类拥有决策权。** AI 建议可由有权人员接受、编辑、拒绝或要求核实；接受仅记录决策，不执行采购、下单、调价或系统写入。
5. **反馈受治理。** 反馈经脱敏、去重、审核和版本化后才可成为知识或规则候选；反馈不能直接改变生产事实、排序、模型或策略。

### 明确禁止

本设计不部署服务、模型、连接器或生产同步，Product Manager AI 严禁：

- 自动采购、自动补货或自动创建采购建议单；
- 自动下单、自动向供应商承诺、自动执行商品交易；
- 自动调价、自动促销、自动清货或自动变更陈列；
- 修改 SAP；
- 修改 Core 生产同步、同步规则或生产数据；
- 修改商品主数据、库存、价格、生命周期权威状态或品牌资产发布状态。

## 2. Product Manager AI Architecture

```text
Brand Knowledge（已发布、有效、ACL 控制的品牌定位/产品资料/策略）
                                  ↓
Product Intelligence（商品主档投影、属性、SKU 映射、生命周期与类目关系）
                                  ↓
Sales Data（授权销售、销量、折扣、退货、售罄及趋势快照）
                                  ↓
Inventory Data（授权库存、可售状态、覆盖、在途投影及数据时点）
                                  ↓
Customer Insights（经授权的聚合客群、反馈、场景与需求信号）
                                  ↓
Product Manager AI（Context + 策略门控 + 分析编排 + Citation + 审计）
                                  ↓
Product Strategy Recommendation（品牌/品类/SKU/门店策略候选与风险）
                                  ↓
Human Decision（人工核实、接受/编辑/拒绝；在权威系统独立执行）
                                  ↓
Feedback Loop（最小反馈 → 脱敏/审核 → 版本化知识/规则候选）
```

| 层 | 职责 | 输出 | 强制控制 |
| --- | --- | --- | --- |
| Brand Knowledge | 读取已发布品牌定位、产品线、季节策略、培训与活动资料。 | 版本化资产及 Citation。 | 仅有效、授权版本；不发布或修改资产。 |
| Product Intelligence | 提供商品、SKU、属性、类目、替代关系和权威生命周期投影。 | 商品事实、映射与规则。 | 只读；AI 候选不能覆盖权威状态。 |
| Sales Data | 提供门店/渠道授权范围的聚合销售事实。 | 增长、速度、售罄、退货趋势。 | 标识促销、缺货等混杂因素；不暴露未授权明细。 |
| Inventory Data | 提供库存、覆盖、状态及可选在途投影。 | 库存健康和供给风险。 | 不承诺实时可售；不写库存。 |
| Customer Insights | 提供脱敏、最小必要、聚合的客户画像、反馈和场景信号。 | 客群与场景匹配依据。 | 禁止敏感属性推断、个人级画像或越权再识别。 |
| Product Manager AI | 融合信号，执行 ACL、鲜度、冲突、Citation 和策略检查。 | 仅咨询性的策略建议、风险、待核实项。 | 只读工具 allow-list；所有输出可审计。 |
| Human Decision | 有权人员审阅和独立决策。 | 决策状态、理由、外部流程引用。 | 不由接受动作触发业务执行。 |
| Feedback Loop | 管理显式反馈和结果观察。 | 审核队列、评估指标、版本化 Draft。 | 未审核反馈不得影响生产行为。 |

### 2.1 请求门控与输出契约

```text
Authenticated user + X-Purpose + requested filters
  ↓ server-derived ProductManagerContext / organization isolation
RBAC + resource ACL + field filtering + knowledge effective-date check
  ↓
authorized source snapshots → freshness / reconciliation / conflict / citation checks
  ↓
advisory recommendation or needs_verification
  ↓
immutable audit event + optional human feedback
```

无身份、组织不匹配、用途不当、资源 ACL 不允许、必需证据缺失或数据时点不满足时，服务必须拒绝、返回允许的聚合结果，或降级为 `needs_verification`；错误信息不得泄露受限品牌、门店或商品是否存在。每项输出含 `policy_snapshot`、`as_of`、`citations` 与 `advisory_only_requires_human_approval: true`。

## 3. Product Manager Context

`ProductManagerContext` 是一次请求的服务端派生安全边界，不接受客户端赋予角色或权限。

| 字段 | 类型 | 定义与控制 |
| --- | --- | --- |
| `product_manager_ai_id` | string | 会话/编排稳定 ID，关联模型、规则、策略与审计版本；不是人员身份。 |
| `owner` | object | 认证主体引用及其有效角色；日志使用受控引用或哈希。 |
| `brand_scope` | object | 可读品牌、产品线、区域、时间窗及字段策略；请求只能取子集。 |
| `category_scope` | object | 允许的品类树版本、品类及聚合粒度。 |
| `store_scope` | object | 允许的门店/区域/门店类型、渠道及数据粒度。 |
| `product_scope` | object | 允许的款式、SKU、商品线及生命周期视图范围。 |
| `permission` | object | RBAC、ACL、用途、导出、脱敏及策略快照。 |
| `knowledge_scope` | object | 可检索的已发布、有效、未撤销知识版本与 Citation 范围。 |


## 4. Brand Strategy Assistant

品牌策略助手在品牌范围内形成策略**候选**，而不是替代品牌负责人或市场决策。它按品牌、区域、渠道和观察期展示以下六类内容：

| 模块 | 分析设计 | 输出与限制 |
| --- | --- | --- |
| 品牌定位 | 汇总已发布品牌价值、目标场景、价格带、产品线及区域策略。 | 定位一致性/缺口；草案或过期资料不能作为当前事实。 |
| 品牌表现 | 对比销售、增长、毛利、售罄、退货、库存和活动窗口。 | 指明基线、促销与缺货干扰，不承诺增长结果。 |
| 用户画像 | 使用授权的聚合客群、需求、价格带和场景偏好。 | 不展示个人画像，不推断敏感属性。 |
| 市场机会 | 结合授权市场资料、未满足需求、区域/渠道空白及品类趋势。 | 记录地域、采集日期、许可和不确定性。 |
| 竞争分析 | 按可比价格带、场景、功能、覆盖和已授权资料比较。 | 显示比较维度与资料时点；不构成竞争事实断言。 |
| 品牌发展建议 | 输出维持、扩张、收缩、测试或研究候选及成功/停止指标。 | 必须由品牌负责人/管理层人工审批。 |

## 5. Category Planning Assistant

品类规划助手支持 **服装、鞋、背包、装备、睡眠系统**，并允许按已批准品类树扩展。每一品类均按销售、需求、趋势、库存与场景分析，输出具有证据链的品类机会建议。

| 品类 | 重点分析 | 机会建议示例 |
| --- | --- | --- |
| 服装 | 季节、尺码/颜色结构、穿着场景、售罄与退货。 | 补齐场景/价格带、调整深度候选。 |
| 鞋 | 尺码覆盖、功能、地形/活动场景、试穿反馈。 | 尺码结构、场景缺口、试点门店候选。 |
| 背包 | 容量、背负系统、通勤/徒步场景、搭配销售。 | 容量带和场景组合机会。 |
| 装备 | 功能系统、耐用性、季节/活动、关联商品。 | 关键功能覆盖与组合陈列候选。 |
| 睡眠系统 | 温标、体积/重量、露营季节、系统兼容性。 | 温标段、套装/配件关联机会。 |

**统一分析框架：** `销售`（量、额、速度、毛利、退货）→ `需求`（授权聚合需求、搜索/咨询/反馈信号）→ `趋势`（多窗口、季节、活动影响）→ `库存`（覆盖、积压、缺货与时点）→ `场景`（使用场景、区域和门店类型）。输出须包含机会范围、证据、影响假设、风险、优先级、建议验证动作、Citation 和置信度；不能直接生成采购、调价或商品主数据动作。

## 6. SKU Lifecycle Management

SKU 生命周期是受治理的分析视图；权威状态由 Product Intelligence 管理，AI 只能提出复核候选。

```text
New → Growth → Hero → Stable → Risk → Retire
```

| 阶段 | 观察信号 | 策略输出 |
| --- | --- | --- |
| `New` | 上市期、资料完整度、早期销售、退货和门店反馈。 | **培育**：小范围验证、观察窗口、资料缺口。 |
| `Growth` | 连续增长、需求扩展、库存覆盖与场景匹配。 | **培育**：扩大验证候选与供给/门店核实项。 |
| `Hero` | 多窗口高贡献、稳定售罄、充分需求证据。 | **主推**：保护可得性、重点陈列和替代方案候选。 |
| `Stable` | 销售、毛利、周转与库存接近已批准基线。 | **控制**：维持深度、监控偏离与季节变化。 |
| `Risk` | 低周转、积压、退货异常、趋势反转或资料不确定。 | **控制**：风险原因拆解、缩减/验证候选。 |
| `Retire` | 已发布退市计划或经审批的持续退出信号。 | **退出建议**：剩余库存、替代覆盖和退出计划待审。 |

所有阶段结论记录 `lifecycle_source`（权威事实或 AI 候选）、时间窗口、阈值/规则版本、异常因素与 Citation。冲突时权威状态优先；AI 仅显示差异和人工待办。

## 7. Product Health Model

Product Health Model 将商品健康度作为解释性评分，不将分数当作自动动作触发器。评分按 SKU/款式/品类的授权粒度计算，并输出总分、分项、权重版本、时间窗、缺失数据和置信度。

| 评分维度 | 解释性指标 | 评分保护 |
| --- | --- | --- |
| 销售速度 | 相对类目/季节基线的销量、售罄、趋势。 | 标识缺货和促销造成的不可比性。 |
| 库存健康 | 覆盖天数、可售/不可售、积压与缺货风险。 | 显示库存 `as_of`，在途与未知库存分开。 |
| 毛利 | 已授权聚合毛利率/贡献及变化。 | 受字段 ACL 限制，不能据此自动调价。 |
| 周转 | 周转率、库龄、销售与库存匹配。 | 区分新品观察期和历史商品。 |
| 场景价值 | 对核心场景、季节和商品组合的覆盖价值。 | 是规则化假设，须披露规则版本。 |
| 客户匹配 | 授权聚合客群需求、反馈与价格带匹配。 | 不使用个人数据或敏感推断。 |

建议公式：`health_score = Σ(normalized_dimension_score × approved_weight) − data_quality_penalty`。权重由已批准规则版本提供；缺失维度不得以零替代，而应降低置信度并显示 `insufficient_data`。分数等级仅用于“关注/复核/机会”排序，不能自动改变 SKU 状态、订单、价格或陈列。

## 8. Store Product Strategy

门店商品策略针对不同门店类型、区域、客群和空间约束，生成以下四类可人工验证的建议：

| 维度 | 设计 | 输出 |
| --- | --- | --- |
| 商品深度 | 结合门店需求、销售速度、尺码/颜色结构、库存覆盖及空间。 | 各品类/SKU 深度候选和证据缺口。 |
| 陈列策略 | 结合场景、关联商品、活动、季节和已发布品牌规范。 | 场景化陈列、组合与重点位置建议；不自动修改陈列。 |
| 客户匹配 | 比较授权聚合客群、价格带、场景需求与现有商品覆盖。 | 客群/场景缺口及验证问题，不输出个人级判断。 |
| 推广重点 | 基于生命周期、库存健康、品牌活动和门店适配。 | 主推/培育/控制候选、时间窗与停止条件；不自动促销或调价。 |

门店卡片必须显示门店范围、比较基线、数据时点、空间/活动等假设、风险及 Citation。跨门店比较应按门店类型或规模标准化，避免将不可比门店简单排名。

## 9. Brand Performance Analysis

品牌表现分析按授权品牌、区域、渠道和时间窗提供五个方面：

| 指标 | 设计 |
| --- | --- |
| 销售 | 销售额、销量、售罄、价格带和商品贡献，并显示比较期。 |
| 增长 | 环比/同比/季节调整候选及基线，解释促销、开店、缺货等影响。 |
| 库存 | 覆盖、库龄、积压/缺货风险、在途投影和库存时点。 |
| 客户反馈 | 脱敏聚合的满意/退货原因/场景反馈及样本限制。 |
| 活动表现 | 活动前后窗口、参与范围、商品/门店差异及归因限制。 |

输出 `brand_scorecard`、异常、机会、风险和建议验证清单。活动表现仅提供关联分析，不能把短期提升直接认定为活动因果效果。

## 10. Product Manager AI Workspace

工作台以“今日最重要、证据完整、需要人工处理”为序展示授权范围内容。

| 区块 | 内容 | 卡片必填字段 | 允许操作 |
| --- | --- | --- | --- |
| 今日商品洞察 | 销售/库存/健康度变化、生命周期与策略待办。 | 范围、影响、原因、`as_of`、Citation、置信度、限制。 | 查看、筛选、创建人工复核、反馈。 |
| 新品 | 新品资料发布、早期表现、上市窗口和验证候选。 | 资产版本、适用范围、有效期、Citation。 | 打开分析、标记需核实、反馈。 |
| 商品风险 | 缺货、积压、退货、低毛利、资料过期或映射冲突。 | 严重度、影响范围、规则版本、数据时点。 | 确认、静默（理由和到期）、升级复核。 |
| 机会 | 品牌/品类/场景/门店机会和组合候选。 | 证据、假设、优先级、风险、Citation。 | 审阅、建立验证待办、反馈。 |
| 品牌动态 | 品牌资产、策略、季节/活动资料的发布、撤销和到期。 | 品牌、版本、变更摘要、有效期、ACL。 | 查看受权资料、确认阅读。 |
| 门店反馈 | 门店提交的脱敏反馈、场景问题和资料缺口。 | 来源类型、范围、时间、审核状态、证据。 | 复核、分类、提交知识审核；AI 不直接发布。 |

排序因素仅限影响、紧迫性、数据鲜度、证据完整度和已批准风险策略；不得隐藏使用个人绩效、敏感客户属性或不可解释偏好。

## 11. API Contract

**共同约束：** JSON、RFC 3339 UTC；`Authorization`、`X-Correlation-Id`、`X-Purpose: product_strategy` 必填。服务端派生 `ProductManagerContext`。所有 GET 端点只读，`brand_id`、`category_id`、`store_id`、`product_id`、`sku_id` 只能是授权范围的子集。所有响应都携带 `policy_snapshot`、`as_of`、`citations` 和 `advisory_only_requires_human_approval: true`。

### `GET /api/product-manager/dashboard`

读取工作台；可选参数：`brand_id?`、`category_id?`、`store_id?`、`date?`、`include=insights,new_products,risks,opportunities,brand_updates,store_feedback`。

### `GET /api/product-manager/product-health`

读取商品健康度；可选参数：`product_id?`、`sku_id?`、`brand_id?`、`category_id?`、`store_id?`、`window=7d|30d|90d`。返回分项分数、总分、权重版本、缺失数据、生命周期候选与建议，不返回可执行指令。

### `GET /api/product-manager/category-insight`

读取品类洞察；可选参数：`category_id`、`brand_id?`、`store_id?`、`market?`、`window?`。返回销售、需求、趋势、库存、场景分析与机会建议。

### `GET /api/product-manager/brand-analysis`

读取品牌表现；可选参数：`brand_id`、`region?`、`channel?`、`store_id?`、`window?`。返回销售、增长、库存、客户反馈和活动表现，并披露归因限制。

### `POST /api/product-manager/feedback`

仅记录对已返回建议/洞察的人工反馈；不修改商品、价格、库存、SAP 或 Core。请求体包括：

```json
{
  "recommendation_id": "pmrec_01J...",
  "decision": "accepted",
  "reason_code": "validated_with_store",
  "comment": "已由商品经理复核，进入人工策略会议。",
  "external_decision_ref": "optional_governed_reference"
}
```

`decision` 为 `accepted|edited|rejected|needs_verification`。服务端记录 `actor_ref`、上下文、输入/策略版本和时间；反馈先进入审核队列。示例通用响应：

```json
{
  "items": [{"recommendation_id": "pmrec_01J...", "status": "needs_verification", "citations": ["cit_sales_12"]}],
  "as_of": "2026-07-21T09:00:00Z",
  "policy_snapshot": "pol_01J...",
  "advisory_only_requires_human_approval": true
}
```

## 12. Permission（RBAC + ACL）

RBAC 决定动作类别，ACL 决定特定品牌、品类、门店、商品、字段和知识资产是否可见；最终权限为两者与目的、组织和保密策略的交集。

| 角色 | 可执行的受权动作 | 明确不能做 |
| --- | --- | --- |
| 商品经理 | 阅读策略洞察、提交/审阅本范围反馈、创建人工复核。 | 越权查看、自动交易、改 SAP/Core/商品主数据。 |
| 采购 | 阅读与采购计划相关的商品/品类洞察，提交采购侧反馈。 | 以本 AI 自动采购、下单或承诺供应商。 |
| 品牌负责人 | 阅读本品牌策略与表现，审阅品牌建议和品牌资料关联反馈。 | 绕过资产发布、自动调价或变更主数据。 |
| 管理层 | 阅读获授权的跨范围聚合看板和决策摘要。 | 读取受限明细或以看板触发自动执行。 |

导出、跨品牌汇总、商业字段、客户反馈明细和反馈审核均需独立 ACL 与用途检查。权限变更、知识撤销或范围收缩后，缓存和会话必须重新授权。

## 13. Audit

每次洞察、建议、人工决策和反馈都写入不可篡改、可关联的审计事件。审计记录最小必要信息，不复制不必要的客户、合同或商业原文。

| 记录项 | 内容 |
| --- | --- |
| 建议 | `recommendation_id`、类型、范围、事实/分析/假设/风险、状态、置信度。 |
| 数据来源 | 数据集/资产引用、`as_of`、窗口、质量、访问决策和 Citation。 |
| 版本 | 模型、提示词/规则、品类树、健康度权重、知识、ACL 与策略快照版本。 |
| Citation | 精确 Citation ID、来源版本、有效期和检索时间。 |
| 反馈 | 操作者引用、决策、理由、编辑差异、审核状态、时间和外部决策引用。 |

审计事件还应包含 `correlation_id`、`product_manager_ai_id`、授权结果、输出哈希与保留策略。审计用于回放、争议处理、质量评估和合规复核，不用于未经批准的人员绩效推断。
