# VAFOX Control Plane Phase 3 Agent Hub 建设方案

> **文档性质：仅设计方案（Design Only / Not Deployed）。** 本方案定义 `control.vafox.com` 的 Phase 3 Agent Hub 建设标准，不执行服务器操作、不创建运行时资源、不启动 Docker 容器、不接入生产数据、不变更网关或权限配置。

| 项目 | 设计值 |
| --- | --- |
| 目标环境 | `control.vafox.com` |
| 前置状态 | Phase 1 Security **Completed**；Phase 2A Foundation **Completed**；Phase 2B Service Framework **Designed** |
| 目标 | 将现有 VAFOX Agent 体系以可注册、可审计、可审批、可回滚的方式接入 Control Plane |
| 初始接入对象 | Health、Connectivity、Report、Data、CEO 五个 Agent |
| 默认运行状态 | `Draft`；未通过逐项审批不得运行 |
| 数据边界 | 不接入生产数据；仅允许已批准的本地、离线、非生产、合成、脱敏或聚合样本（按 Agent 契约进一步收窄） |

---

## 一、Phase 3 目标

### 1.1 为什么需要 Agent Hub

现有 Agent 已具有独立的规范、模板和最小权限边界，但若直接以本地脚本、独立容器或点对点调用方式运行，会产生身份不可统一验证、版本不可统一追溯、权限难以持续校验、报告缺乏来源链路以及停用/回滚不一致的风险。

Agent Hub 是 `control.vafox.com` 内唯一受控的 Agent 接入与运行治理层。它将 Agent 的身份、版本、状态、能力、权限、审批、运行记录和报告证据连接为一条可验证控制链，以实现：

- **统一治理：** Agent 必须先注册、后审批、再激活；Registry 是 Agent 身份与状态的唯一权威来源。
- **默认拒绝：** 无法验证 Agent ID、版本、状态、权限、能力、审批或输入范围时，Hub 必须拒绝任务。
- **最小权限：** Agent 仅获得完成已批准任务所需的最小数据、工具、网络和文件范围；注册不等于访问或执行授权。
- **可审计运行：** 每次调用、授权判定、执行结果、错误与报告均保留关联 ID、版本和证据引用。
- **可控演进：** 通过固定镜像、声明式配置、审批门禁和版本回滚，在不降低安全控制的前提下扩展 Agent 能力。

### 1.2 Phase 3 完成定义

Phase 3 的完成是“Agent Hub 的设计标准、接入契约、治理门禁和验收方法完整且可审查”，而不是默认将任何 Agent 或生产集成上线。各 Agent 只有在其自身版本完成审批、隔离验证和上线放行后，才可从 `Approved` 进入 `Active`。

## 二、Agent Hub 架构

### 2.1 逻辑架构

```text
经认证的内部调用方 / 已批准调度器
                |
                v
          Agent Hub（统一入口与策略执行点）
                |
                v
 Registry（身份、版本、状态、权限、能力、审批与审计依据）
                |
                v
 Agent Runtime（隔离容器中的已批准 Agent 版本）
                |
                v
 Reports（结构化结果、证据引用、人工可读报告）
```

### 2.2 组件职责与强制边界

| 组件 | 主要职责 | 必须执行的控制 | 明确不负责 |
| --- | --- | --- | --- |
| Agent Hub | 接收经认证的内部任务；校验请求；编排已批准运行；返回受控结果引用。 | 请求 ID、身份认证、Registry 实时校验、策略判定、限流、超时、审计。 | 不绕过 Registry；不代替审批；不直接授予数据或执行权限。 |
| Registry | 保存 Agent 身份、版本、生命周期、权限、能力、责任人与审批结论。 | 全局唯一且不可复用的 Agent ID；状态/版本变更审计；默认拒绝。 | 不执行任务；不保存凭据、原始 payload 或生产数据。 |
| Agent Runtime | 在隔离环境执行固定版本的 Agent；产生结构化执行结果。 | 固定镜像、非 root、只读根文件系统、最小 capability、资源限制、健康检查与短时身份。 | 不访问未获批系统；不持有 Docker socket、宿主机 shell 或共享高权限凭据。 |
| Reports | 存放经策略过滤的结果、摘要和证据引用，供获授权人员读取。 | 来源、Agent/version、时间、关联 ID、输入范围、授权结论和数据新鲜度。 | 不伪造源结论；不作为审批或审计唯一存储；不保存 Secret。 |

### 2.3 请求处理控制链

1. 调用方通过受控 Gateway/内部入口提交版本化请求，并提供 `X-Request-ID`、调用方身份、目标 Agent 与任务类型。
2. Agent Hub 验证调用方身份、受众、时间有效性、速率限制与请求 schema；验证失败即拒绝并审计。
3. Hub 向 Registry 查询目标 Agent 的 Agent ID、精确版本、状态、许可级别、能力、数据范围、审批引用与紧急停止状态。
4. Hub 使用 RBAC、ABAC 与 Capability 三重校验决定是否允许该具体任务；任何一项不匹配均为拒绝。
5. 仅将经过最小化和策略验证的输入交给对应 Runtime；Runtime 不可自行扩大数据源、网络范围或动作集合。
6. Runtime 产出结构化结果、状态和脱敏证据引用；Hub 写入执行审计并将可发布结果交由 Reports 管理。
7. 调用方仅获得其角色和属性允许读取的报告/结果；错误信息不得泄露内部拓扑、凭据或原始敏感内容。

### 2.4 网络与接口原则

- 仅允许通过经批准的 Gateway/内部 API 进入 Hub；禁止绕过 Gateway 直接访问 Runtime、Registry 或 Reports。
- 内部 API 使用版本化路径（例如 `/internal/v1/...`）、TLS/mTLS 或同等批准机制，以及短时效、受众受限的工作负载身份。
- Agent Runtime 位于内部隔离网络；默认无公网出站能力。任何新增外部连接、消息队列或数据源必须作为独立变更重新评审。
- 所有服务间请求必须携带关联 ID；审批或变更相关请求还必须关联审批/变更引用。

## 三、Agent 迁移策略

### 3.1 迁移目标

```text
本地 VAFOX-Control
        |
        v
control.vafox.com Agent Hub
```

迁移是**治理与运行入口迁移**，不是把本地文件、凭据、真实数据或历史运行环境整体复制到服务器。迁移前后 Agent ID 保持稳定；每个可运行版本、配置与权限必须重新受到 Registry、审批和审计控制。

### 3.2 分阶段迁移步骤

| 步骤 | 迁移活动 | 输出/门禁 |
| --- | --- | --- |
| 1. 清单与冻结 | 盘点五个 Agent 的规范、模板、Agent ID、版本、输入/输出、能力、禁止事项和负责人；冻结待迁移版本。 | 清单完整；不存在未声明的网络、数据或执行路径。 |
| 2. 契约标准化 | 将本地定义转换为 Hub 可校验的 Registry manifest、能力清单、权限策略、数据范围、报告 schema 与健康契约。 | schema 校验通过；无凭据或真实生产数据进入 manifest。 |
| 3. Draft 登记 | 在目标 Registry 以 `Draft` 记录元数据与工件摘要；不创建任务、调度、数据连接或公网入口。 | ID 唯一、版本可追溯、责任人已指定。 |
| 4. 隔离验证 | 使用批准的离线/非生产样本验证默认拒绝、输入边界、报告脱敏、审计、超时、停止与回滚契约。 | 验证证据完整；无外部/生产系统访问。 |
| 5. 审批与发布准备 | 对精确版本、镜像 digest、配置引用、权限、能力、数据范围、报告与回滚点进行联合审批。 | 状态可变更为 `Approved`；批准不自动启动运行。 |
| 6. 受控激活 | 仅由 Hub 启动被批准的精确版本；先进行低风险、无生产数据的受控任务，持续监测健康和审计。 | `Active` 仅在所有运行门禁通过后生效。 |
| 7. 稳定化与旧路径收敛 | 验证报告来源、权限拒绝、暂停和回滚；撤销旧的直接运行/直接调用入口。 | 不存在绕过 Hub、Gateway 或 Registry 的并行生产路径。 |

### 3.3 迁移原则

- 先迁移**元数据与治理**，再迁移运行入口；不得以“已有本地 Agent”为由跳过注册或审批。
- 每个扩大权限、能力、数据源、网络范围、输出范围或执行方式的变更均为新版本，必须重新注册审查。
- 不迁移明文 Secret、真实 `.env`、生产数据、未脱敏日志、宿主机访问权或 Docker socket 权限。
- 迁移失败或证据不足时保持/恢复 `Draft` 或 `Suspended`，而不是降级安全控制继续运行。

## 四、Agent Registry 迁移

### 4.1 Registry 最小注册记录

Registry 是 Phase 3 的唯一 Agent 身份与治理目录。每条记录至少应包含以下字段，并保留每次变更的前后值、操作者、时间、原因与审批引用：

| 字段 | 要求 |
| --- | --- |
| Agent ID | 全局唯一、稳定、不可复用；禁用/弃用后不得分配给其他 Agent。 |
| Name / Type / Owner | 名称、类型、业务责任人、技术责任人和安全责任人；责任人缺失不得进入 `Registered`。 |
| Version | 语义版本与不可变工件标识（如镜像 digest）；仅允许精确已批准版本运行。 |
| Status | `Draft`、`Registered`、`Approved`、`Active`、`Suspended` 或 `Deprecated`。 |
| Permission | 许可级别、角色、属性条件、资源/数据/文件/网络范围、时效和拒绝策略。 |
| Capability | 允许动作、禁止动作、任务类型、输入/输出 schema、资源与速率限制。 |
| Audit | 创建、修改、审批、状态切换、授权决策、调用、执行、错误、暂停、回滚与读取审计引用。 |
| Data & Runtime Scope | 数据分类/来源/字段边界、目标环境、网络策略、容器基线、健康检查和报告路径。 |
| Approval & Emergency Control | 所需审批类型/有效期、批准证据、紧急停止状态、最后验证时间和回滚版本。 |

### 4.2 Registry 迁移流程

1. 从当前本地 Agent 规范和 YAML 模板提取最小元数据，不把本地运行状态视为目标环境的激活授权。
2. 为每个 Agent 建立 `Draft` Registry 记录，校验 Agent ID 唯一性、版本格式、责任人、能力和权限字段完整性。
3. 为对应版本绑定不可变工件摘要、配置 schema/引用、报告 schema 与批准的输入输出范围。
4. 校验禁止能力显式存在（包括远程 shell、未授权网络、生产数据、数据修改和自动化执行等适用项）。
5. 将历史文档、测试与审批材料以证据引用方式关联；不将原始敏感内容复制入 Registry。
6. 经完整性审核后进入 `Registered`；经安全、数据、测试、技术、业务和权限审核后才可进入 `Approved`。
7. 每次状态、版本、权限、能力或配置变化均写入追加式审计记录；运行时由 Hub 实时读取/验证 Registry 结论。

## 五、Agent 运行模型

### 5.1 Docker 运行设计

每个 Agent 版本以独立 Docker 容器运行，并适用以下最小安全基线：

- 使用已批准、固定 digest 的镜像；禁止 `latest`、未审查来源与运行时下载不受控代码。
- 以非 root 用户运行，优先只读根文件系统，挂载最小化的只读配置和服务专属可写临时/报告卷。
- 删除所有非必要 Linux capabilities，禁止 privileged 容器、`network_mode: host`、宿主机路径广泛挂载和 `/var/run/docker.sock`。
- 默认置于内部网络且无公网出站；容器间访问由显式 allowlist 控制。
- 限制 CPU、内存、进程数、文件大小、任务并发和执行超时；资源超限即失败并审计，不自动扩权重试。
- 运行时身份、配置和 Secret 仅经批准的引用机制注入；不得写入镜像、Git、报告、日志或 Registry。

### 5.2 Service 管理设计

- Agent Runtime 由 Hub 的受控编排/服务管理层管理；单个 Agent 不得自行注册、启动其他服务、修改 Compose 或控制 Docker。
- 服务定义、镜像、配置、策略和依赖均需版本控制；配置变更与代码/镜像变更分离审批。
- 仅 `Active` 且审批仍有效的精确版本可被调度；`Suspended`、`Deprecated`、审批失效或版本不匹配时禁止启动新任务。
- 失败处理采用“停止任务、保留证据、告警、人工判定”的模式；不得由 Agent 自行重启服务、修复配置或触发部署/回滚。

### 5.3 Health Check 设计

每个 Agent Runtime 必须提供并由 Hub 记录以下检查，结果使用 `healthy`、`degraded`、`unhealthy` 或 `unknown`；`unknown` 不得视为健康：

| 检查类型 | 检查内容 | 失败处置 |
| --- | --- | --- |
| Liveness | 进程、容器和最小运行循环是否存活。 | 标记不健康、停止接收新任务、记录错误。 |
| Readiness | Agent 是否已加载已批准版本、配置与策略，且能接受受控请求。 | 不路由新任务；不以跳过校验方式恢复。 |
| Dependency | 对 Registry、Hub、报告写入契约等允许依赖的认证后可达性。 | 标记降级/不健康；不得降级为匿名或缓存过期授权。 |
| Policy | Agent ID/版本/状态/审批/权限/能力/配置摘要是否一致。 | 立即拒绝任务；必要时切换 `Suspended` 并告警。 |
| Reporting | 结构化日志、审计事件和报告引用是否可写、可关联且已脱敏。 | 阻断需审计的运行，升级处理。 |

## 六、五个 Agent 接入方案

所有 Agent 初始均以其已定义的 `0.1.0`、`Draft` 工件进入本方案的登记流程；以下内容是接入设计，不构成运行授权。

| Agent | Registry 身份 | Phase 3 接入范围 | 许可基线 | 输出与运行限制 |
| --- | --- | --- | --- | --- |
| **Health Agent** | `agt_ops_health_phase1_001` | 接收已批准的控制面健康快照、脱敏健康指标和告警摘要，生成可审计健康结论与人工建议。 | 规范当前为 `read_only`；迁移时映射为本方案 L1 本地/内部只读分析基线，任何提升需重新审批。 | 仅 Observe/Analyze/Recommend；禁止 SSH、远程连接、Docker 操作、服务重启、数据写入、部署和回滚。 |
| **Connectivity Agent** | `agt_ops_connectivity_phase1_001` | 分析已批准的离线、非生产、无真实端点连接样本与设计元数据，生成连接性设计/报告。 | L0。 | 非联网、非变更；禁止网络探测、远程命令、云访问、网络配置修改与自动修复。 |
| **Report Agent** | `agt_ops_report_phase1_001` | 读取获批本地 Health/Connectivity 报告和 Registry 元数据，生成受控每日健康汇总。 | L1。 | 仅允许批准路径和固定报告 schema；禁止修改源报告、远程访问、数据库、Docker 和生产变更。 |
| **Data Agent** | `agt_ops_data_phase1_001` | 在隔离环境分析已批准的离线、非生产、合成、脱敏或聚合样本，生成不可执行洞察。 | L0。 | 禁止 Core、SAP、数据库、生产数据、远程端点、数据增删改导出与自动执行。 |
| **CEO Agent** | `agt_ceo_decision_phase1_001` | 分析批准的既有本地报告与 Registry 元数据，生成高管摘要和人工决策模板。 | L1。 | 仅 Analyze/Recommend；禁止 Huyan/AI/Core/SAP/云访问、工作流触发、数据修改和自动业务执行。 |

### 6.1 接入顺序与依赖

1. **Connectivity Agent 与 Data Agent：** 先完成 L0 离线、非联网验证，作为默认拒绝与样本边界的基线。
2. **Health Agent：** 在已批准健康快照契约下完成只读健康报告验证，不提供修复能力。
3. **Report Agent：** 在源 Agent 的报告 schema、来源和审计引用稳定后接入，验证只读取允许路径且不改写来源。
4. **CEO Agent：** 最后接入，仅消费已批准报告/Registry 元数据，不直接连接任何业务或运行系统。

此顺序是建议的验证依赖顺序，不是自动激活队列。每个 Agent 仍须独立完成其状态迁移和审批。

## 七、权限模型

### 7.1 权限等级

| 等级 | 定义 | Phase 3 默认边界 |
| --- | --- | --- |
| **L0** | 离线、非联网、非变更的设计/样本分析。 | 仅本地批准的非生产、合成、脱敏或聚合样本；无远程连接、无生产数据、无写入/执行。 |
| **L1** | 受控本地/内部只读分析与受限报告生成。 | 仅批准的本地/内部报告、元数据或健康快照；允许固定输出路径；无远程系统与生产修改。 |
| **L2** | 受控的只读服务集成。 | 仅在独立方案、数据分类、网关、字段级策略、隔离测试和审批完成后启用；Phase 3 初始不授予。 |
| **L3** | 人工批准下的受限建议执行/任务编排。 | 必须具备每次操作审批、职责分离、可逆性、速率/范围限制和强审计；本方案不默认启用。 |
| **L4** | 高风险、生产影响的特权动作。 | 默认禁止。任何可能的例外必须单独立项、独立风险评审、双人/多方审批、紧急停止和恢复演练；不属于本次五 Agent 初始接入范围。 |

### 7.2 RBAC、ABAC 与 Capability 联合校验

| 模型 | 作用 | Phase 3 校验示例 |
| --- | --- | --- |
| RBAC | 定义主体可承担的职责。 | `agent-runtime` 只能执行其自身注册的任务；`registry-admin` 可管理元数据但不拥有执行授权；`report-reader` 仅可读授权报告。 |
| ABAC | 根据主体、资源、环境和请求上下文作细粒度决策。 | 校验 Agent 状态=`Active`、精确版本匹配、审批未过期、环境=`control.vafox.com`、数据分类允许、请求时间/来源/关联 ID 合法。 |
| Capability | 将 Agent 可调用动作限制为明确、不可伪造、可过期的最小能力集合。 | 例如 `generate_report` 不隐含 `network_probe`；`analyze_health_snapshot` 不隐含 `restart_service`。 |

允许决定必须同时满足 RBAC、ABAC 和 Capability；任一缺失、过期、冲突或无法验证即拒绝。权限授予以“动作 + 资源 + 条件 + 时效”为单位，不以 Agent 名称或管理员身份做宽泛放行。

## 八、审批流程

### 8.1 Agent 上线生命周期

```text
Draft
  |
  v
Registered
  |
  v
Approved
  |
  v
Active
```

异常、审批失效、未授权变更、健康/审计失败或治理指令发生时，应转为 `Suspended`；永久下线后转为 `Deprecated`。`Suspended`/`Deprecated` 不得接受任务，且 Agent ID 不得复用。

### 8.2 状态门禁

| 状态 | 允许活动 | 进入条件 | 禁止活动 |
| --- | --- | --- | --- |
| Draft | 文档、模板、schema 和离线设计审查。 | Agent 定义存在，但未完成注册完整性检查。 | Gateway 接入、任务调度、数据访问、正式报告生成。 |
| Registered | 受控注册验证、隔离的非生产测试。 | ID 唯一；责任人、版本、能力、权限、审计和禁止项完整。 | 生产数据、外部系统、自动调度、生产任务。 |
| Approved | 配置已批准的最小权限和上线准备。 | 安全、数据、测试、技术、业务和权限审批均有可追溯证据。 | 因审批自动运行；任何超出已批范围的任务。 |
| Active | 在精确已批版本/范围内处理经 Hub 授权任务。 | 运行前健康、策略、审计、紧急停止和回滚验证通过。 | 自扩权、绕过 Gateway/Registry、未经审批的版本/数据/动作。 |

### 8.3 审批职责分离

- **技术责任人：** 确认工件、接口、运行模型、健康检查和回滚可行性。
- **安全责任人：** 审核身份、网络、容器隔离、Secret 引用、审计与禁止能力。
- **数据责任人：** 审核数据分类、来源、字段、保留、脱敏和输出范围。
- **业务责任人：** 确认任务目的、报告使用方式和人工决策责任。
- **权限/治理责任人：** 批准最小权限、能力、有效期、紧急停止与例外记录。

提交者不得单独批准自身的生产影响权限或高风险变更。审批应绑定精确 Agent ID、版本、镜像摘要、配置摘要、数据范围、能力集合、有效期和回滚目标。

## 九、日志审计

日志使用 UTC、结构化格式、关联 ID 与统一脱敏策略。不得记录 Secret、令牌、授权头、Cookie、完整敏感输入、原始生产数据或可被用于绕过安全控制的内容。

| 日志类别 | 最小字段 | 目的与访问规则 |
| --- | --- | --- |
| **调用日志** | UTC 时间、请求 ID、调用方/工作负载身份、Agent ID、请求版本、任务类型、策略/授权结论、拒绝原因分类、延迟。 | 证明谁请求了什么以及是否被允许；仅授权运维/审计人员读取。 |
| **执行日志** | 执行 ID、请求 ID、Agent ID、精确版本/工件摘要、Runtime 实例、开始/结束时间、输入/输出摘要、能力使用、资源/超时状态、报告/证据引用。 | 证明获批任务如何运行；不得包含原始敏感 payload。 |
| **错误日志** | 请求/执行 ID、错误类别、组件、版本、脱敏上下文、重试/停止结果、升级引用。 | 支持诊断与趋势分析；不得输出堆栈中的 Secret 或内部敏感拓扑。 |
| **审计日志** | 注册、版本、权限、能力、审批、状态、配置引用、暂停、回滚与日志读取事件；操作者、理由与前后值。 | 追加式、访问严格受限、保留策略明确，并具备防篡改/外部归档设计。 |

审计记录应在执行前记录授权判定，并在执行后补充结果；日志丢失、审计管道不可写或关联 ID 缺失时，需审计的任务不得继续执行。

## 十、备份回滚

### 10.1 备份范围

| 对象 | 备份内容 | 明确排除 |
| --- | --- | --- |
| Agent 版本 | 已批准 manifest、镜像 digest/签名引用、依赖清单、报告 schema、测试与审批证据引用。 | 未审查镜像、运行时下载内容、明文 Secret。 |
| Registry | 元数据、状态/版本/权限/能力历史、审批与审计引用、校验和与导出版本。 | 凭据、完整原始任务 payload、生产数据。 |
| 配置 | 非敏感配置、策略、Compose/服务声明、Secret **引用**、校验和与批准版本。 | 实际 Secret 值、真实 `.env`、私钥、令牌。 |

备份须加密、受访问控制、带校验和、保留一个不依赖目标主机的受控副本，并在隔离非生产环境定期验证可恢复性。

### 10.2 回滚设计

| 回滚类别 | 触发条件 | 受控路径 | 验证 |
| --- | --- | --- | --- |
| Agent 版本回滚 | 健康失败、契约不兼容、安全缺陷、资源异常或未授权行为。 | 停止新任务；保留证据；将 Hub 路由恢复到最近批准的精确版本/digest；必要时将问题版本 `Suspended`。 | 版本/签名一致、权限未扩大、健康/审计/报告验证通过。 |
| Registry 恢复 | 元数据损坏、错误状态转换、策略误配置或审计存储故障。 | 使用最近经过校验的 Registry 快照恢复；重放/保留后续审计证据；恢复后禁止基于过期缓存授权。 | ID、版本、状态、审批、权限、能力和审计链完整且一致。 |
| 配置恢复 | 配置导致认证、授权、网络、日志或运行异常。 | 恢复最近批准的配置和 Secret 引用版本；不以明文 Secret 或宽泛临时权限替代。 | schema、校验和、最小权限、Runtime 健康和审计均通过。 |

每次回滚必须有触发原因、影响范围、批准人、操作者、起止 UTC 时间、恢复版本、证据和后续问题记录。若无法在安全范围内恢复，保持停止/`Suspended` 并升级处理。

## 十一、验收标准

| 验收域 | 通过标准 | 必需证据 |
| --- | --- | --- |
| Agent 注册 | 五个 Agent 均可建立唯一、完整、可追溯的 `Draft`/`Registered` 记录；Agent ID、版本、负责人、状态、权限、能力和审计字段完整。 | Registry schema 校验、唯一性校验、manifest 摘要和审计样例。 |
| Agent 运行 | 仅精确已批准、`Active` 的版本可被 Hub 路由；容器隔离、非 root、只读/最小挂载、资源限制、健康检查和紧急停止均可验证。 | 运行策略检查、健康检查结果、拒绝测试、暂停/恢复演练记录。 |
| 报告生成 | Health/Connectivity/Report/CEO/Data 的输出均具备 Agent ID、版本、UTC、关联 ID、来源/证据、数据新鲜度、权限结论和脱敏校验；Report 不修改源报告。 | 报告 schema 校验、来源追溯、脱敏扫描与样例报告（无敏感数据）。 |
| 权限验证 | RBAC、ABAC 和 Capability 三者均需通过；未注册、非 Active、审批失效、版本不匹配、越权数据/动作和绕过 Gateway 的请求均被拒绝并审计。 | 正反向授权测试、拒绝审计、策略评审与有效期测试。 |
| 审批与审计 | `Draft → Registered → Approved → Active` 每一步有职责分离的批准与不可缺失审计；审批不自动触发运行。 | 审批记录、状态转换记录、关联 ID 链路和访问审计。 |
| 备份与回滚 | Agent 版本、Registry 与配置均有校验备份和隔离恢复验证；回滚不降低权限或删除证据。 | 校验和、恢复演练、回滚记录和恢复后健康/授权检查。 |

验收未完成、存在未批准例外、审计不可用、回滚不可验证或任何禁止项被违反时，保持 `Design Only / Not Deployed`，不得进入正式上线。

## 十二、禁止事项

Phase 3 设计、验证、迁移和后续实施均禁止以下行为：

1. **未经审批 Agent 上线：** 禁止未完成 `Draft → Registered → Approved → Active` 门禁、审批失效、版本不一致或责任人缺失的 Agent 接收任务或运行。
2. **生产数据接入：** 禁止连接、复制、处理、导出或保留 SAP、Core、AI、Huyan、数据库、消息队列、对象存储或其他生产业务数据；不得以测试、调试或备份为由规避。
3. **绕过 Gateway：** 禁止直接调用 Agent Runtime、Registry、Reports 或内部服务来规避受控入口、身份校验、限流、策略和审计。
4. **绕过权限：** 禁止共享高权限凭据、匿名写入、管理员权限替代审批、缓存过期授权、隐式工具调用或 Agent 自我提权。
5. **越权执行：** 禁止 SSH、远程 shell、网络探测、Docker/Compose/Kubernetes 修改、服务控制、部署、自动修复、回滚、数据增删改迁移或自动业务执行，除非未来以独立版本和独立授权明确批准。
6. **安全控制降级：** 禁止 privileged 容器、Docker socket 挂载、host 网络、未固定镜像、明文 Secret、未审查公网暴露和未记录的例外。
7. **审计规避：** 禁止删除、篡改、关闭或绕过调用、执行、错误、审批、状态和权限审计；审计不可用时不得继续需要审计的 Agent 运行。

本方案形成 VAFOX Control Plane Agent Hub 的建设与治理标准。任何超出本文定义的网络、数据、权限、执行或生产影响能力，必须另行设计、风险评估、测试、审批和验收，不得由本方案隐含授权。
