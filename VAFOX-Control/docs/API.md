# Control API V1

All application routes are versioned under `/api/v1`; the OpenAPI document is available at `/api/v1/openapi.json` when the local API is running. V1 is a registry/control metadata API, **not** an orchestration or remote-execution API.

| Resource | Endpoints | Responsibility |
| --- | --- | --- |
| Server Registry | `POST`, `GET /servers`; `POST /servers/{id}/health` | Declare a control-plane-visible server and record its reported health. |
| Service Registry | `POST`, `GET /services`; `POST /services/{id}/health` | Associate a service with a registered server and declare capabilities. |
| Deployment Registry | `POST`, `GET /deployments` | Record a non-production deployment intent tied to a service, artifact digest, and change reference. |
| Platform health | `GET /health/live`, `GET /health/ready` | Container liveness/readiness only; no dependency or remote host probes. |

## Health check contract

A server or service reports `{ "status": "healthy|degraded|unhealthy", "latency_ms": 12, "detail": "optional" }`. `checked_at` is generated by the Control API, and the current registry health state is updated atomically by the repository adapter. Future agents must authenticate and sign their reports; V1 accepts no production agent connections.

## Guardrails

- No endpoint accepts credentials, SSH commands, SAP payloads, or Dify configuration.
- Deployment environments are constrained to `local`, `development`, or `staging`; `production` is rejected.
- The present repository adapter is in-memory. The PostgreSQL model in `infra/postgres/001-control-plane.sql` is an unexecuted migration specification.
