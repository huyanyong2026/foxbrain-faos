# FireFox Enterprise AI Operation Center V1 建设报告

## 1. 系统架构

本阶段在现有 `ai.vafox.com` 应用上增量建设 AI 经营助手，没有重写已有 Agent、知识、任务、反馈或身份模块。

数据链路固定为：

`SAP-PROD -> core.vafox.com SAP Mirror -> Core API -> ai.vafox.com -> 规则分析 -> Excel -> 人工确认`

`ai.vafox.com` 不包含 SAP 地址、SAP 账号或数据库连接代码。经营分析只调用 `core.vafox.com` 的 HTTPS 只读接口。

## 2. 数据来源

新增 Core API 固定只读端点：

- `GET /api/v1/operation/snapshot?store=...&as_of=...`

端点只执行预定义的 `SELECT` 查询，并按门店返回：

- 商品编码、名称、品牌、品类
- 当前库存、已承诺库存、在途库存、平均成本
- 最近 30、60、90、180 天净销量
- 最后销售日期

销售汇总包含销售单并扣除退货单。接口不接受任意 SQL，不提供新增、修改、删除或建表能力。

## 3. 功能完成情况

### AI 智能补货

- 页面：`/operation/replenishment`
- 目标库存等于近 30 天净销量
- 建议补货等于 `目标库存 - 可售库存 - 在途库存`，结果最小为 0
- 展示 30、60、90 天销量、趋势、建议数量及可读原因
- 生成 `门店_智能补货建议_日期_记录号.xlsx`
- 每次结果默认处于“待人工确认”状态

### AI 库存健康

- 页面：`/operation/inventory-health`
- 以最后销售日期作为第一阶段库存活跃度依据
- 90 天黄色、180 天橙色、270 天红色、360 天黑色
- 展示门店库存金额、风险商品数、分级明细与处理建议
- 生成 `门店_积压库存分析_日期_记录号.xlsx`
- 调拨、促销等建议必须人工确认，系统不自动执行

### 历史记录

新增数据表：

- `operation_analysis_runs`：分析类型、门店、Core 来源、数据更新时间、结果、摘要、Excel 路径、人工审核状态
- `operation_analysis_audit`：创建、确认、拒绝等操作审计

Excel 文件使用独立持久目录 `/data/operation_exports`，Docker 示例已配置服务器挂载路径。

## 4. 身份与权限

- CEO：查看全部门店及两类经营助手
- 管理层：查看全部门店，可人工审核
- 采购员：查看全部门店补货建议与库存信息
- 店长：只查看身份中心绑定的本人门店
- 普通员工：无经营分析权限

下载 Excel、查看历史和提交分析都会重新检查岗位权限与门店数据范围。AI 能力继承当前用户权限。

## 5. 测试结果

- 全仓库自动测试：85 项通过
- 经营中心专项测试：7 项通过
- Core API 专项测试：5 项通过
- Python 编译检查：通过
- Flask 路由注册：42 条路由加载成功
- 三个新增模板编译：通过
- 补货公式样例：近 30 天销售 10、可售库存 3、在途 2，建议补货 5，结果正确
- 库存风险样例：180 天以上命中橙色、无销售记录命中黑色，结果正确
- 两种 Excel：均可由 `openpyxl` 重新打开并读取表头
- Core API：无令牌读取被拒绝，写方法返回只读错误

## 6. 安全检查

- 未连接 SAP-PROD
- 未增加 SAP 权限
- 未加入 SAP 主机、账号或密码
- 未实现 SAP 写入、自动采购、自动调拨或自动促销
- Core API 经营快照仅允许已授权的 `facts:read` 令牌访问
- SQL 参数使用参数化绑定，仓库名称不能拼接为 SQL
- 所有建议保留 Core 来源表、镜像更新时间和人工审核轨迹

## 7. 发布状态与下一步

本阶段已完成代码、测试和 PR 准备，尚未部署生产环境。应先合并 Identity Center PR，再合并本阶段 PR，随后配置服务器端 Core API 令牌和导出目录，使用真实只读镜像做门店数量、销售数量、退货数量与库存金额对账。

下一阶段建议在人工验收 V1 结果后增加：安全库存天数、季节系数、跨店调拨候选、采购包装量及建议采纳效果反馈。所有扩展继续保持人工确认边界。
