# VAFOX 2.0 企业 AI 操作系统总体架构设计审查清单

> **用途：** 作为 VAFOX 2.0 企业 AI 操作系统总体架构的设计审查依据。  
> **审查原则：** 品牌一致、职责清晰、数据可信、权限最小化、AI 可治理、移动优先、可持续扩展。  
> **审查方式：** 对每项填写 `PASS`、`NEEDS REVIEW` 或 `N/A`，并在备注中记录证据、问题、责任人和整改期限。  
> **边界：** 本文仅用于架构设计审查，不包含系统操作、部署、配置变更或生产执行步骤。

## 四、Control Plane 检查

> **审查目标：** 确认 Control Plane 是统一的 AI 编排、治理与观测层，Agent 不绕过 Control Plane 直接扩张权限或执行高风险动作。

| 能力域 | 应有职责边界 | 审查检查项 | 状态 | 证据 / 备注 |
| --- | --- | --- | --- | --- |
| Agent Hub | Agent 的发现、岗位定义、配置、生命周期与可见性管理。 | 是否记录 Agent 身份、职责、版本、所有者、工具、数据范围及状态。 |  |  |
| Registry | Agent、模型、工具、工作流、连接器和策略的受控登记。 | 是否有唯一标识、版本、依赖、批准状态、责任人和退役规则。 |  |  |
| Runtime | 任务编排、模型路由、工具调用、会话上下文、限流与隔离运行。 | 是否仅执行已登记、已授权、可审计的运行单元。 |  |  |
| Approval | 人类在环的审批、升级、驳回、回退和授权记录。 | 是否覆盖资金、主数据、合同、对外沟通、人事和高风险生产变更。 |  |  |
| Reports | 运行质量、成本、风险、性能、审计与业务结果的报告。 | 是否区分运行事实、AI 推断、建议和已批准行动。 |  |  |

- [ ] Agent Hub、Registry、Runtime、Approval、Reports 的职责无重叠，并具备明确交接关系。`状态：____`
- [ ] 未登记或已退役的 Agent、工具、模型、工作流和连接器不能进入 Runtime。`状态：____`
- [ ] Approval 不能被 Agent、服务账户或普通用户绕过；审批结果可追溯、可撤销或可升级。`状态：____`
- [ ] Reports 不成为业务数据或审批事实的替代权威源，而是消费受控事实并输出可审计观测结果。`状态：____`
- [ ] Control Plane 的管理权限与业务使用权限隔离，且管理操作进入审计范围。`状态：____`

## 五、Agent 体系检查

> **统一要求：** 每个 Agent 必须具备明确的岗位职责、允许输入、受控输出、独立身份、工具范围、数据范围、审批条件、审计策略和停用 / 升级路径。

| Agent | 职责 | 输入是否明确 | 输出是否明确 | 权限是否明确 | 审批 / 人工在环 | 状态 | 证据 / 备注 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| Health Agent | 监测平台、数据任务、集成和 Agent 运行健康。 | 服务指标、任务状态、日志、告警规则。 | 健康评分、异常告警、初步诊断、升级工单。 | 只读观测；可创建告警 / 工单；不可修改生产数据。 | 生产修复与高风险变更须人工批准。 |  |  |
| Connectivity Agent | 监测系统连接、接口可用性、同步状态与失败重试。 | 集成元数据、接口状态、消息事件。 | 连接状态、失败报告、重试建议、依赖影响。 | 凭据不可见；仅在受控范围执行重试。 | 高风险连接或配置变更须人工批准。 |  |  |
| Report Agent | 将可信数据生成标准经营报告和角色摘要。 | 已认证指标、报告模板、时间范围、授权上下文。 | 日报、周报、月报、异常说明、数据引用。 | 仅只读经授权数据产品；不可修改指标口径。 | 发布范围或敏感报告按策略审批。 |  |  |
| Data Agent | 执行数据查询、质量检查、血缘解释和受控数据准备。 | 数据请求、语义模型、质量规则、权限令牌。 | 查询结果、质量问题、指标解释、可审计数据集。 | 按行、列、用途读取；写入仅限隔离派生层。 | 写入、导出或源系统操作须审批。 |  |  |
| CEO Agent | 汇总经营信号，准备决策材料，跟踪授权行动。 | 报告、风险、战略知识、CEO 指令、授权数据。 | CEO 摘要、行动选项、任务建议、决策复盘上下文。 | 限 CEO / 明确委托范围；不可代表 CEO 最终审批。 | 高风险行动、对外承诺和最终决策由人批准。 |  |  |
| People Agent（未来） | 支持成长、岗位辅导与组织能力发展。 | 角色、技能、学习资源、授权反馈。 | 成长计划、学习建议、能力洞察。 | 仅处理授权人才信息；敏感信息受严格分类。 | 人事决策和敏感操作须人工批准。 |  |  |

- [ ] 每个 Agent 以独立服务身份运行，不共享高权限凭据或跨 Agent 的无限制访问令牌。`状态：____`
- [ ] 每个 Agent 的输入来源均已列明可信等级、数据 / 知识时效和授权条件。`状态：____`
- [ ] 每个 Agent 的输出均标明事实、推断、建议、未知项和来源 / 血缘。`状态：____`
- [ ] 每个 Agent 都有失败、超时、越权、低置信度和异常升级时的安全降级行为。`状态：____`
- [ ] 未来新增 Agent 遵循相同岗位定义、注册、审批、审计、评估和退役流程。`状态：____`

## 六、数据流检查

### 6.1 目标数据链路

```text
SAP B1
  ↓
Core Data Brain
  ↓
Data Agent
  ↓
Report Agent
  ↓
CEO Agent
  ↓
Huyan
```

| 链路节点 | 审查检查项 | 状态 | 证据 / 备注 |
| --- | --- | --- | --- |
| SAP B1 → Core Data Brain | SAP B1 是否被定义为核心业务记录的权威来源；接入是否经过受控连接、校验、标准化、权限控制和血缘记录。 |  |  |
| Core Data Brain | 是否统一主数据、指标口径、数据分类、质量规则、刷新频率和可消费的数据产品。 |  |  |
| Core Data Brain → Data Agent | Data Agent 是否只接收已授权的数据产品与语义上下文，而非绕过 Core 直连源系统。 |  |  |
| Data Agent | 查询、质量检查、数据准备和解释是否保留请求者、用途、范围、时间和血缘。 |  |  |
| Data Agent → Report Agent | 结构化结果是否包含可信度、时间范围、异常说明、指标定义和引用。 |  |  |
| Report Agent | 是否仅以已认证指标与批准模板生成报告，且不自行改变指标口径。 |  |  |
| Report Agent → CEO Agent | 报告、趋势、异常和证据是否完整传递，并能区分事实与建议。 |  |  |
| CEO Agent → Huyan | Huyan 是否呈现决策摘要、行动选项、任务与证据，并保存决策上下文。 |  |  |

- [ ] 数据链路中每一跳均定义输入格式、输出格式、责任方、时效、失败处理与审计事件。`状态：____`
- [ ] 敏感数据在每一跳执行分类、最小化、脱敏、加密与用途限制。`状态：____`
- [ ] 数据质量异常、同步延迟、口径变更或来源不可用时，Huyan 和报告中不会将信息误呈现为实时或确定事实。`状态：____`
- [ ] 此数据链路不自动写回 SAP B1；任何写回均经独立审批、验证、回滚与审计流程。`状态：____`
