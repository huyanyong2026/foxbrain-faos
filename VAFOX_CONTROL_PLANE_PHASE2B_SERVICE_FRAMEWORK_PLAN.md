# VAFOX Control Plane Phase 2B 服务框架设计计划

> **文档性质：仅设计计划。** 本文为 `control.vafox.com` 的 Phase 2B 服务基础框架定义目标架构、控制边界与验收条件；不执行服务器操作、不创建目录、不启动容器、不写入配置，也不授权任何部署。

| 项目 | 设计值 |
| --- | --- |
| 目标环境 | `control.vafox.com` |
| 前置状态 | Phase 1 Security **Completed**；Phase 2A Foundation **Completed** |
| 规划根目录 | `/opt/vafox-control/` |
| 本阶段产物 | 受版本控制的服务模板、配置模板、Compose 模板、健康检查与运维流程定义 |
| 后续阶段 | Phase 3 Agent Hub（仅在本计划验收、审批和单独放行后） |
| 默认状态 | `Design Only / Not Deployed` |

## 一、Phase 2B 目标

### 1.1 为什么需要 Service Framework

Phase 2A 已提供 Docker、Docker Compose 与 `/opt/vafox-control/` 的基础承载能力，但尚未定义可重复、可隔离、可审计地承载控制面能力的服务边界。Service Framework 用于在引入任何 Agent 或生产业务数据之前，建立以下共同基础：

- **一致性：** 为注册、审批、报告、仪表盘和健康检查提供统一的目录、镜像、配置、日志、健康检查与发布约定。
- **隔离性：** 将服务间通信限制在专用 Docker 网络；不以服务框架为由扩大宿主机、业务系统或数据面的访问权限。
- **可审计性：** 将审批决策、服务事件和执行记录按职责分流，并要求关联请求、审批和变更引用。
- **可恢复性：** 使服务、Compose 定义和非敏感配置可独立版本化、备份及回滚。
- **可演进性：** 为 Phase 3 Agent Hub 预留受控接入点，但本阶段不部署 Agent Hub、不注册 Agent、不执行任务。

### 1.2 本阶段完成定义

Phase 2B 的完成是“**框架可审查、可验证、可回滚**”，而不是“业务能力已上线”。所有服务均为最小模板或接口契约实现；对外业务路由、真实工作负载、生产数据源和自动化执行均不在范围内。

## 二、服务架构

### 2.1 规划目录

```text
/opt/vafox-control/
├── services/
│   ├── registry/
│   ├── approval/
│   ├── report/
│   ├── dashboard/
│   └── health/
├── config/
├── compose/
├── logs/
├── backups/
└── releases/
```

每个 `services/<service>/` 仅规划保存服务源码或最小模板、`Dockerfile`、依赖清单、只读运行配置模板、API 契约、健康检查说明及测试说明。运行时可写数据不得写入镜像层或源码目录；其位置必须通过受控 volume 明确声明。

### 2.2 服务职责与边界

| 服务 | 职责 | 允许的输入/输出 | Phase 2B 边界 |
| --- | --- | --- | --- |
| `registry` | 维护受控服务与未来 Agent 的**元数据注册**、版本、状态和所有者引用。 | 已审批的元数据请求；返回注册查询结果。 | 不注册正式 Agent；不保存凭据、任务 payload 或生产数据。 |
| `approval` | 提供审批请求、审批状态、审批证据引用和不可变决策记录的接口。 | 经认证的审批请求；返回状态及审计引用。 | 不执行被审批动作；不绕过人工/既定审批链。 |
| `report` | 汇总健康、审批和框架事件的报告元数据及受控报告引用。 | 已脱敏事件与报告引用；返回报告状态。 | 不抓取生产数据；不充当日志唯一存储。 |
| `dashboard` | 提供只读框架状态视图的后端/BFF 契约。 | 已授权的只读查询；返回聚合状态。 | 不在本阶段正式发布 UI；不提供控制、执行或审批写入入口。 |
| `health` | 汇总服务、Docker、内部 API 与日志管道的健康结论。 | 只读探针结果；返回统一健康摘要。 | 不自动重启、修复或改变其他服务状态。 |

### 2.3 服务交互原则

```text
dashboard (read-only) ──► report ──► health
                             │
registry ──► approval ───────┘
```

- 服务仅通过版本化内部 API 通信，禁止共享源码目录、容器 socket 或宿主机 shell。
- `approval` 是任何未来执行链路的强制决策来源；`registry`、`report` 与 `dashboard` 不得替代或伪造审批结论。
- `health` 只读取经允许的状态端点和日志管道状态，不读取业务内容、Secret 或生产数据。
- 后续引入异步队列、数据库或外部网关时，必须另行设计、风险评审和审批，不能隐含在本框架内。

## 三、Docker Compose 框架

### 3.1 Compose 定义

规划一个独立、版本控制的 Compose 文件（例如 `compose/compose.phase2b.yml`），其中定义 `registry`、`approval`、`report`、`dashboard` 和 `health` 五项服务。每项定义必须包含：

- 固定的镜像 digest 或经批准的固定版本标签；禁止 `latest` 和未审查镜像来源。
- `restart` 策略、非 root 运行用户、只读根文件系统（无法满足时须记录经批准例外）、最小 Linux capabilities 与资源限制。
- 明确的内部 `expose` 端口；默认不得将任一框架服务端口直接发布到公网。
- 明确 `healthcheck`、超时、重试次数与启动宽限期；服务依赖以健康状态而非仅启动顺序表示。
- 只读配置挂载、服务专属可写 volume、结构化日志输出与受控环境变量来源。

### 3.2 Network 设计

| 网络 | 成员 | 用途 | 限制 |
| --- | --- | --- | --- |
| `vafox-control-internal` | 全部 Phase 2B 服务 | 默认内部 API 通信。 | `internal: true`；不承载公网入口。 |
| `vafox-control-edge`（预留） | 仅未来获批的反向代理与 `dashboard` 只读入口。 | 未来经批准的受控入口。 | Phase 2B 不暴露公网端口；接入需独立变更。 |

服务不得使用 `network_mode: host`、Docker 默认 bridge 作为隐式跨服务通道，或挂载 `/var/run/docker.sock`。如未来健康检查确需读取 Docker 状态，应采用最小权限的受控只读代理，并单独评审。

### 3.3 Volume 设计

| Volume/挂载类别 | 用途 | 权限与数据规则 |
| --- | --- | --- |
| `*-config`（bind mount） | 已批准的非敏感服务配置。 | 容器内只读；源文件受版本控制或受控配置管理。 |
| `*-state`（named volume） | 单服务的非生产框架状态。 | 每服务独立；不得跨服务共享；生命周期与备份策略明确。 |
| `logs`（集中挂载或日志驱动） | 结构化运行日志的输出路径。 | 追加写入；访问最小化；禁止记录 Secret。 |
| `backups`（仅备份流程） | 加密/受控备份产物的临时或目标位置。 | 非服务常规读写路径；备份权限独立控制。 |

### 3.4 Environment 设计

- Compose 只引用变量名及受控来源，例如 `${VAFOX_CONTROL_ENV}`、`${APP_CONFIG_PATH}` 与 `${SECRET_REFERENCE_*}`；不得提供秘密默认值。
- 环境变量按服务最小化，并将非敏感默认值、必填变量、格式与是否可变写入配置契约。
- Secret 不进入镜像、Dockerfile、Compose 文件、`.env.example`、日志、报告或 Git 历史。
- 需要 Secret 的未来服务须通过经批准的 Secret 管理系统或 Docker secret 的**引用/挂载**取得；引用名称与轮换责任人可记录，Secret 值不可记录。

## 四、配置管理

### 4.1 `config/` 结构

```text
config/
├── common/
│   ├── logging.example.yaml
│   └── api-policy.example.yaml
├── registry/
├── approval/
├── report/
├── dashboard/
└── health/
```

- 仅将可公开审查的 schema、非敏感默认值、服务端点别名、日志级别、保留期限和策略开关纳入版本控制。
- 生产生效配置应具有版本号、变更单引用、审批人、创建时间、校验和及回滚版本引用。
- 配置变更与镜像变更分离审查；任何涉及认证、授权、网络、日志保留或 Secret 引用的变更均为受控变更。

### 4.2 `.env` 模板

提供 `.env.example`（或同等模板），只包含变量名、无敏感示例值的占位符、用途、是否必填及格式说明。例如：

```dotenv
# Non-secret deployment identity
VAFOX_CONTROL_ENV=<environment-name>
VAFOX_CONTROL_RELEASE=<approved-release-id>

# References only; never place secret values in this file.
APPROVAL_SIGNING_KEY_REF=<secret-manager-reference>
INTERNAL_API_CLIENT_SECRET_REF=<secret-manager-reference>
```

实际 `.env` 文件必须被忽略、权限受限、不得提交，并且不得成为备份中可读的明文 Secret 容器。

### 4.3 Secret 引用与轮换

| 要求 | 设计规则 |
| --- | --- |
| 引用 | 使用逻辑引用名、版本/轮换标识和受控存储位置，不记录 Secret 值。 |
| 注入 | 仅在运行时由批准的 Secret 机制注入；优先文件挂载而非可被轻易导出的环境变量。 |
| 访问 | 按服务身份最小授权；服务之间不得复用不必要的 Secret。 |
| 轮换 | 定义所有者、轮换周期、失效处理、兼容窗口和轮换后健康验证。 |
| 审计 | 记录引用变更、访问失败和轮换事件，不记录秘密内容。 |

## 五、Health Check 框架

健康检查由 `health` 服务汇总，并输出 `healthy`、`degraded`、`unhealthy` 或 `unknown`；`unknown` 不得被当作健康。探针结果必须包含 UTC 时间、服务版本、关联 ID 与错误分类，但不包含敏感请求或响应内容。

| 检查域 | 检查内容 | 判定与处置边界 |
| --- | --- | --- |
| 服务状态 | 进程/容器存活、应用 `/healthz` 或 `/readyz`、依赖可达性和版本一致性。 | 单项失败标记降级/不健康；仅告警和报告，不自动修复。 |
| Docker 状态 | 容器运行、Compose 声明一致性、健康检查结果、资源阈值与重启次数。 | 读取范围必须受限；不授予容器控制 Docker socket 的权限。 |
| API 状态 | 内部 API 的认证前后可达性、响应码、响应时间、schema/版本兼容性。 | 使用专用低权限探针身份；不执行写入或业务操作。 |
| 日志状态 | 日志管道可写、最近事件时间、结构化格式、轮转/保留状态与错误率。 | 不扫描或暴露日志中的业务内容；发现 Secret 疑似泄露时升级安全事件。 |

每项服务必须区分：`liveness`（进程是否可运行）、`readiness`（是否可以接受内部请求）和 `dependency`（依赖是否可用）。健康摘要 API 仅供内部读取；未来公开展示需独立审查脱敏规则。

## 六、API 基础设计

### 6.1 内部 API 规范

- 基路径使用版本化形式：`/internal/v1/...`；禁止未版本化的隐式契约。
- 使用 HTTPS/mTLS 或同等经批准的加密传输；仅限 `vafox-control-internal` 网络内的服务身份通信。
- 请求必须携带 `X-Request-ID`；审批、变更或执行相关请求还必须携带关联引用（如 `X-Approval-Ref`、`X-Change-Ref`）。
- 统一使用 JSON、UTC ISO 8601 时间、明确的 schema、分页和兼容性策略；错误响应不泄露内部拓扑、Secret 或堆栈详情。
- 为每个端点定义方法、调用方、权限、数据分类、幂等性、超时、速率限制与审计事件。

### 6.2 认证方式

服务到服务认证采用短时效、受众限制的工作负载身份令牌，并结合 mTLS（如经批准的身份平台支持）。令牌签发、验证、撤销、轮换、时钟偏差和失败处理必须有明确策略。禁止共享静态 API key、在 URL 中传递凭据、匿名内部写入和将认证材料写入日志。

### 6.3 权限校验

采用默认拒绝的 RBAC/ABAC 组合：

| 角色/身份 | 最小权限 |
| --- | --- |
| `health-reader` | 读取限定的健康摘要与探针结果。 |
| `report-writer` | 写入已验证的框架报告元数据，不可修改审批记录。 |
| `approval-reviewer` | 提交或确认审批决策；所有决策需要审计并遵循职责分离。 |
| `registry-reader` / `registry-admin` | 分别读取或管理受控注册元数据；管理权限不等同于执行权限。 |
| `dashboard-reader` | 仅读取经脱敏、经授权的聚合状态。 |

任何未来“执行者”身份必须在 Phase 3 单独定义；Phase 2B 不定义可执行动作的 API 权限。服务端必须在每次请求校验身份、受众、scope/角色、资源归属、审批状态（如适用）和过期时间。

## 七、日志体系

| 日志类别 | 来源与内容 | 保存/访问规则 |
| --- | --- | --- |
| Service logs | 每个服务的结构化运行事件、请求 ID、版本、级别、延迟和错误分类。 | 统一收集、按服务分流、轮转与保留；仅授权运维读取。 |
| Audit logs | 认证/授权结果、审批决策、配置引用变更、管理员操作和访问拒绝。 | 追加式、时间同步、访问严格受限；应具备防篡改或外部归档策略。 |
| Execution logs | **仅预留契约**：未来经审批任务的请求、审批引用、开始/结束、结果与证据引用。 | Phase 2B 不产生 Agent 执行记录；不得将其替代审批审计。 |

所有日志字段应定义脱敏清单（凭据、token、Cookie、授权头、私密 URL 参数、个人/生产业务数据等）。日志级别变更、导出、删除、保留例外和审计日志访问必须可追溯。日志时间统一使用 UTC，并设置容量阈值与告警，而不是依赖无限增长。

## 八、备份体系

### 8.1 备份范围与要求

| 备份对象 | 范围 | 保护与验证 |
| --- | --- | --- |
| Service config | 已批准的非敏感配置、schema、策略与 Secret **引用**。 | 加密、校验和、版本/审批引用；不包含明文 Secret。 |
| Compose files | Compose 模板、固定镜像清单、网络/volume 声明与部署说明。 | 存入版本库及受控离机/异地副本；定期验证可读性。 |
| Environment templates | `.env.example`、变量契约与 Secret 引用映射（不含值）。 | 纳入版本控制并校验；真实 `.env` 不作为可读明文备份。 |

备份设计应定义频率、保留期限、加密与密钥责任、存储位置、访问控制、校验和、恢复责任人和恢复演练周期。备份成功不等于可恢复：必须通过批准的非生产/隔离验证来确认恢复步骤、配置完整性和所引用 Secret 的可用性。

### 8.2 备份边界

- 不备份生产数据、Agent memory、真实任务 payload 或未经批准的日志内容。
- 不以备份便利为由导出、复制或明文化 Secret。
- 备份与运行目录分离，且至少保留一个不依赖目标主机的受控副本。

## 九、回滚设计

回滚以“恢复到最近批准、可验证的框架基线”为目标，先停止扩大影响，再恢复声明与配置，不以降级安全控制、删除审计证据或绕过审批作为恢复手段。

| 回滚类别 | 触发示例 | 设计路径 | 完成验证 |
| --- | --- | --- | --- |
| Service rollback | 健康检查失败、接口不兼容、安全缺陷或资源异常。 | 选择上一批准镜像 digest/服务模板；保留故障日志与版本证据。 | 服务就绪、内部 API 契约兼容、无未预期写入。 |
| Compose rollback | 网络、依赖顺序、volume 或安全选项错误。 | 恢复上一批准 Compose 文件与锁定镜像清单；重新进行声明校验。 | 所有服务仅在预期网络/volume 中运行，且无额外暴露。 |
| Config rollback | 配置导致认证、授权、日志或健康异常。 | 恢复上一批准配置快照及对应 Secret 引用版本；不回退为明文 Secret。 | 配置校验通过、权限符合最小化、健康与审计完整。 |

每次回滚必须记录触发条件、批准人、责任人、影响范围、起止 UTC 时间、恢复版本、验证证据和后续问题单。若回滚无法在既定恢复时间目标内安全完成，应停止服务并升级，不得用未批准临时配置继续运行。

## 十、验收标准

| 验收域 | 通过标准 | 证据要求 |
| --- | --- | --- |
| Docker 正常 | Compose 模板语法与策略校验通过；镜像固定、服务无公网直曝、网络/volume/healthcheck 定义完整。 | 校验报告、批准的镜像清单与 Compose 版本引用。 |
| 目录正常 | 所有规划目录及服务模板职责明确；运行时写入边界、属主和最小权限方案完整。 | 目录清单、权限设计与范围复核记录。 |
| 服务模板正常 | 五个服务具有职责、接口、健康端点、配置契约、依赖和禁止事项；无 Agent 或生产集成。 | 服务设计评审记录与模板测试结果。 |
| 日志正常 | 三类日志的 schema、脱敏、访问、保留、轮转和审计规则可验证。 | 日志字段规范、样例（无敏感内容）与检查记录。 |
| 备份正常 | 三类规定对象可生成受控备份、校验完整性，并完成隔离恢复验证设计/演练。 | 备份清单、校验和、恢复演练记录与责任人确认。 |

放行进入 Phase 3 前，还必须确认：所有适用验收项通过；无未批准例外；审批与回滚责任人明确；以及本计划中的禁止事项未被违反。未通过时保持 `Design Only / Not Deployed`，不得以“框架已存在”为由开始 Agent Hub 部署。

## 十一、禁止事项

Phase 2B 无论在设计、模板验证或后续实施准备中，均禁止：

- **Agent 正式部署：** 不部署、注册、启用或调度 Agent Hub、CEO Agent 或其他生产 Agent。
- **生产数据接入：** 不连接 SAP、Core、AI、Huyan 或任何生产数据库、消息队列、对象存储和业务 API；不复制真实业务数据进行测试。
- **绕过审批：** 不提供匿名写入、自动批准、审批结果伪造、审批后静默变更，或以管理员权限替代审批的路径。
- 不执行未获单独批准的服务器操作、容器启动、镜像拉取、端口暴露、网络规则修改、真实 Secret 注入或生产发布。
- 不将 Docker socket、宿主机特权、共享 root 凭据或宽泛跨服务权限作为实现捷径。

## Phase 3 进入门槛

仅当本计划已转换为经审批的实施 Artifact、完成独立安全/运维复核，并满足第十节所有验收标准后，才可另行提交 Phase 3 Agent Hub 部署计划。Phase 3 计划必须重新明确 Agent 身份、审批链、执行权限、数据边界、生产接入、风险评估、回滚和验收；本文件不构成该等授权。
