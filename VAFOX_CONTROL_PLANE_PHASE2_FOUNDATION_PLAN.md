# VAFOX Control Plane Phase 2 基础环境建设计划

- **目标主机**：`control.vafox.com`
- **公网 IP**：`114.132.55.178`
- **内网 IP**：`172.16.16.6`
- **前置状态**：Phase 1 Security Hardening 已完成。
- **本文性质**：仅为 Phase 2 的设计、实施标准与验收计划；**不构成服务器操作指令，不执行任何服务器变更**。

## 1. Phase 2 目标

### 1.1 定义

Phase 2 建设 **VAFOX Control Plane 基础运行环境**：在既有安全基线上建立可重复部署、可审计、可备份、可回滚的容器化运行底座，为后续业务服务和 Agent 能力上线提供统一标准。

### 1.2 交付范围

1. 统一 Docker 容器运行时及 Compose 编排规范。
2. 建立 `/opt/vafox-control/` 的标准化持久化目录与权限边界。
3. 规划 Agent Registry、Agent Hub、Approval Center、Report Center、Dashboard Backend 的基础服务边界。
4. 建立配置、密钥、日志、审计、备份与回滚的治理标准。
5. 定义从准备到上线的受控部署流程和可量化验收标准。

### 1.3 非目标

- 不在本阶段直接修改生产业务数据或生产应用逻辑。
- 不以本计划替代应用层架构、具体数据库选型或业务 API 设计。
- 不绕过审批流程、变更记录和上线验证。

## 2. 安全基线与前提

Phase 1 已完成，Phase 2 必须持续遵循以下主机安全状态：

- SSH 仅允许密钥认证。
- 禁止 `root` 远程登录。
- 禁止 SSH 密码登录。
- UFW 已开启，仅按最小暴露原则放行必要端口。
- Fail2Ban 持续运行并纳入运行检查。

Docker 与各服务不得降低上述控制措施。任何新增监听端口、出站访问规则、服务账号或权限调整，均须通过变更审批并形成审计记录。

## 3. Docker 基础环境

### 3.1 运行时与编排

| 项目 | 设计标准 |
| --- | --- |
| Docker Engine | 使用受支持的稳定版本；固定主版本和批准的补丁版本，避免无计划自动升级。 |
| Docker Compose | 使用 Docker Compose v2 插件；Compose 文件按服务版本纳入版本控制。 |
| 镜像来源 | 仅使用经过批准的官方镜像、内部镜像仓库或已审查镜像；镜像必须使用不可变版本标签或 digest。 |
| 编排方式 | 每个服务使用独立 Compose 项目/命名空间；共享依赖通过受控网络与显式接口连接。 |
| 持久化 | 所有业务持久数据映射到 `/opt/vafox-control/` 下的指定目录；禁止依赖匿名 volume 保存关键数据。 |

### 3.2 版本管理

- 建立受控的版本清单，记录 Docker Engine、Compose、操作系统依赖、镜像名称、镜像 tag/digest、部署日期和变更单号。
- 升级采用“测试环境验证 → 审批 → 备份 → 生产变更 → 验证”的流程；不得直接使用浮动标签（如 `latest`）。
- Compose 定义、环境变量模板、服务配置和部署说明必须存放在受保护的 Git 仓库中，并通过受审查的提交发布。
- 为每个上线版本保留可复现的 release 标记，确保能够定位并恢复至上一稳定版本。

### 3.3 Docker 安全配置

- 容器默认以非 root 用户运行；确有例外时必须记录原因、范围和补偿控制。
- 采用最小权限：移除不必要 Linux capabilities，使用只读根文件系统（可行时），只挂载必需目录。
- 禁止 `privileged: true`、挂载 Docker socket、宿主机网络模式及任意主机目录挂载；例外必须经安全审批。
- 网络按服务域隔离；仅对外暴露经批准的入口，数据库、内部 API 和管理端点仅允许内部网络访问。
- 配置容器资源限制、健康检查、重启策略和日志轮转，避免单一服务耗尽宿主机资源或日志空间。
- 部署前执行镜像漏洞扫描和许可证审查；高风险漏洞须处置、豁免或阻止上线。

## 4. 标准目录结构

根目录统一为 `/opt/vafox-control/`，仅由指定的部署服务账号和受控运维流程写入。目录设计如下：

```text
/opt/vafox-control/
├── agents/                 # Agent 运行时工作目录、受控定义与隔离数据
├── registry/               # Agent Registry 的持久化数据与受控元数据
├── approvals/              # 审批记录、待处理请求及不可变归档
├── reports/                # 报告生成结果与受控发布内容
├── artifacts/              # 构建产物、证据附件和版本化交付物
├── logs/                   # 服务、Agent、执行和审计日志的落盘位置
├── backups/                # 本地临时备份、备份清单和恢复验证记录
└── config/                 # 非密钥配置、Compose 定义、环境变量模板及版本清单
```

### 4.1 目录治理

- 各服务只获得其必需子目录的读写权限；服务之间不得共享可写工作目录。
- `config/` 不得保存明文密钥；仅保存模板、非敏感配置和密钥引用。
- `artifacts/`、`reports/`、`approvals/` 的保留期、访问权限与删除机制必须可配置并可审计。
- `backups/` 仅作为加密备份的受控暂存或恢复介质；生产备份必须复制到独立、受保护的备份目标。
- 建议使用统一所有者组、最小文件权限和定期权限审计；禁止以宽松的全员可写权限解决访问问题。

## 5. 基础服务规划

| 服务 | 主要职责 | 数据与接口边界 | 运行要求 |
| --- | --- | --- | --- |
| Agent Registry | 保存 Agent 身份、版本、能力声明、状态与注册记录。 | 仅向已认证的内部服务提供查询/注册接口；持久化数据置于 `registry/`。 | 具备数据备份、变更审计和健康检查。 |
| Agent Hub | 负责 Agent 发现、路由、调度入口和受控通信。 | 仅调用已登记的 Agent；不得直接授予生产系统写权限。 | 采用服务身份认证、网络隔离、请求追踪与限流。 |
| Approval Center | 管理高风险动作、发布、权限和例外申请的审批流。 | 审批记录置于 `approvals/`；审批结果须可被下游验证且不可被静默篡改。 | 强制身份认证、角色授权、时间戳、审批人和决策理由审计。 |
| Report Center | 汇总运行、执行、风险和审计报告。 | 输出存于 `reports/`，引用经授权的日志和元数据。 | 报告访问按角色控制，敏感字段脱敏。 |
| Dashboard Backend | 向受控前端或运营接口提供汇总状态与受权操作入口。 | 只通过内部服务 API 访问数据；不得直接暴露数据库。 | 统一认证授权、操作审计、速率限制和健康检查。 |

服务之间应使用明确的 API 契约、服务身份和最小网络连通性。服务上线顺序建议为 Registry → Hub → Approval Center → Report Center → Dashboard Backend；后置服务不得绕过前置的身份、注册或审批控制。

## 6. 配置管理

### 6.1 环境变量

- 使用版本化的 `.env.example` 或等效模板描述非敏感变量、默认值策略、必填项和所属服务。
- 实际环境变量文件不得提交到 Git，不得出现在镜像层、容器 inspect 输出、日志或报告中。
- 各环境（开发、测试、生产）使用独立配置集合；生产配置变更须走审批和版本记录。
- 启动时对必填变量、格式、允许值及依赖连通性进行校验，失败时安全退出而非使用隐式默认值。

### 6.2 Secret 管理

- 密钥、令牌、数据库凭据、证书私钥等只保存在受控 Secret 管理系统或等效加密存储中。
- 应用通过短期凭据、受限挂载或运行时注入获取 Secret；禁止写入 Compose 文件、Git、镜像、报告和普通日志。
- 每个服务使用独立服务身份和独立 Secret；禁止共享密钥。
- 建立 Secret 轮换、吊销、泄露响应和访问审计制度；轮换必须经过验证且支持回退。

### 6.3 配置版本管理

- 配置即代码：Compose、模板、策略和非敏感服务配置均纳入 Git 管理。
- 合并须经代码审查；生产发布必须关联批准的提交、版本标签与变更单。
- 在部署记录中保存配置版本、镜像 digest、数据库迁移版本和审批编号，形成可追溯发布快照。

## 7. 日志与审计

### 7.1 日志分类

| 类型 | 覆盖范围 | 最低记录内容 |
| --- | --- | --- |
| Docker 日志 | Docker daemon、容器 stdout/stderr、生命周期和健康状态。 | 时间、容器/镜像版本、服务名、级别、事件标识。 |
| Agent 日志 | Agent 注册、调用、状态变更、错误与策略决策。 | Agent ID、请求关联 ID、操作者/服务身份、结果和错误码。 |
| 执行日志 | 工作流、任务执行、部署与恢复动作。 | 发起人、审批引用、输入摘要、目标、开始/结束时间、结果。 |
| 审计日志 | 登录、授权、审批、配置变更、密钥访问和管理操作。 | 不可抵赖的主体、动作、对象、来源、时间、结果和理由。 |

### 7.2 日志控制

- 统一采用结构化、带时区的时间戳和全链路关联 ID，日志落盘至 `logs/` 并按服务分区。
- 严禁记录密码、令牌、私钥、完整敏感载荷或可还原的 Secret；日志采集前实施脱敏。
- 对容器日志配置大小上限、滚动策略、保留周期和容量告警，防止磁盘耗尽。
- 审计日志应具备访问控制、完整性保护和独立留存；任何删除、导出或访问行为均应记录。
- 建立日志检索、异常告警和定期审查机制，重点关注认证失败、权限提升、审批绕过、镜像漂移和备份失败。

## 8. 备份策略

### 8.1 备份范围

| 对象 | 内容 | 策略 |
| --- | --- | --- |
| 配置备份 | Compose 定义、非敏感配置、版本清单、策略与环境模板。 | 每次批准的配置变更前后创建版本化快照，并保留 Git 历史。 |
| 数据库备份 | Registry、审批、报告等服务使用的数据库及迁移元数据。 | 按恢复目标定义全量与增量/日志备份；备份需一致性校验和定期恢复演练。 |
| Artifact 备份 | 构建产物、报告附件、审批证据及其校验和与元数据。 | 按保留策略复制到独立备份目标，保证可追溯和可验证。 |

### 8.2 备份控制

- 备份应加密传输、加密存储，并保存校验和、创建时间、来源版本和保留期限。
- 至少保留一份脱离主机的备份副本；备份目标的访问权限独立于生产运行权限。
- 备份任务成功、失败、覆盖、删除和恢复操作必须进入审计日志并触发相应告警。
- 定期执行恢复演练：验证配置、数据库和 Artifact 可在隔离环境中恢复，并记录恢复时间与数据完整性结果。

## 9. 部署流程

### 9.1 准备

1. 确认 Phase 1 安全控制持续有效，检查主机容量、时间同步、网络策略和 DNS。
2. 审查变更单、架构设计、镜像来源、漏洞扫描、端口清单、服务账号和 Secret 访问策略。
3. 固化发布清单：镜像 digest、Compose 版本、配置提交、迁移步骤、备份点、验证步骤和回滚责任人。
4. 在非生产环境完成功能、安全、备份恢复和回滚演练，并取得上线审批。

### 9.2 安装

1. 按批准版本安装或验证 Docker Engine 与 Docker Compose，记录版本清单。
2. 创建标准目录、最小权限服务账号、隔离网络和受控配置挂载。
3. 从批准来源获取并验证镜像，应用经审查的 Compose 定义和运行时配置。
4. 按依赖顺序启动服务；数据库迁移或初始化必须可追踪、可重试并有回滚预案。

### 9.3 验证

1. 确认容器处于预期版本、健康状态、资源限制和网络隔离状态。
2. 验证 Agent 注册、Hub 路由、审批链路、报告生成和 Dashboard 后端授权的最小闭环。
3. 验证日志写入、脱敏、轮转、审计事件和监控告警。
4. 验证已创建可用备份并在隔离环境完成抽样恢复验证。

### 9.4 上线

1. 由批准的变更负责人确认验证结果、版本快照和监控基线。
2. 在受控窗口内切换流量或启用服务，持续观察健康、错误率、延迟、资源和安全告警。
3. 将上线结果、实际版本、异常及处置记录关联至变更单；达到稳定观察期后关闭变更。

## 10. 回滚方案

### 10.1 触发条件

发生安全告警、服务健康检查持续失败、数据完整性异常、核心链路不可用、性能显著劣化或验收项不通过时，应停止扩大影响并启动已审批的回滚流程。

### 10.2 Docker 回滚

- 停止或隔离异常版本，保留必要诊断证据和日志。
- 使用上一稳定 release 的 Compose 定义、镜像 digest 和环境配置重建服务；禁止以浮动标签替代版本回滚。
- 逐项验证容器健康、网络策略、资源限制和服务依赖后再恢复流量。

### 10.3 配置恢复

- 从部署前的批准配置快照恢复非敏感配置，并通过 Secret 管理系统恢复对应的受控 Secret 版本或引用。
- 配置恢复须记录操作者、来源版本、变更理由和验证结果；不得将 Secret 以明文方式从备份写入配置库。

### 10.4 服务与数据恢复

- 按服务依赖顺序恢复数据和服务，优先恢复 Registry、审批与身份相关控制，再恢复 Hub、报告与 Dashboard。
- 数据库恢复必须采用已验证的一致性备份；如涉及迁移，使用经过演练的反向迁移或恢复点策略。
- 回滚完成后执行功能、安全、日志和备份验证，形成事故/变更复盘，并在批准后恢复正常变更节奏。

## 11. 安全边界

以下行为在 Phase 2 设计与实施中明确禁止：

- **直接修改生产**：不得跳过受控发布流程、版本管理、变更审批和验证直接变更生产服务、配置或数据。
- **绕过审批**：不得通过脚本、内部接口、管理员账户或人工流程绕过 Approval Center 的适用审批规则。
- **明文密码**：不得在代码、Git、Compose 文件、环境模板、镜像、日志、报告、聊天记录或工单中存储或传输明文密码与 Secret。
- **共享密钥**：不得在服务、人员、环境或自动化任务之间复用身份密钥、令牌或数据库凭据。

补充边界：不得暴露未认证管理端点；不得将内部数据库直接开放至公网；不得使用未审查镜像或未固定版本；不得以便利性为由授予容器宿主机级权限。

## 12. 验收标准

| 验收领域 | 通过标准 |
| --- | --- |
| Docker 正常 | Docker Engine 与 Compose 版本符合批准清单；所有基础服务运行在固定镜像版本上，健康检查、资源限制、重启策略和网络隔离有效。 |
| 目录正常 | `/opt/vafox-control/` 及规定子目录完整存在；数据映射符合设计；服务账号权限最小化，配置与数据边界清晰。 |
| 日志正常 | Docker、Agent、执行与审计日志均可生成、检索和关联；已验证脱敏、轮转、保留、容量告警和审计完整性。 |
| 备份正常 | 配置、数据库和 Artifact 备份按策略成功生成、加密、校验、异地/独立留存；已完成记录在案的恢复演练。 |
| 安全正常 | Phase 1 SSH/UFW/Fail2Ban 控制仍有效；无明文 Secret、共享密钥、未批准端口、特权容器或未授权管理入口；审批与变更审计可追溯。 |
| 服务闭环正常 | Registry、Hub、Approval Center、Report Center 和 Dashboard Backend 的最小授权链路可用，且不能绕过身份和审批控制。 |
| 回滚可用 | 已验证可从上一稳定版本恢复 Docker、配置和必要数据；回滚步骤、责任人、触发条件和验证记录完整。 |

## 13. Phase 2 完成定义

当本计划中的基础环境、服务边界、配置治理、日志审计、备份恢复、受控部署与回滚能力均通过验收，并且所有上线证据已归档至相应系统后，VAFOX Control Plane Phase 2 才可标记为完成，并作为后续功能阶段的强制基础建设标准。
