# VAFOX Agent 生命周期标准

**文档状态：** 规范（Normative）  
**适用范围：** VAFOX 内所有 Agent、Agent Registry、控制平面、网关，以及向 Agent 提供数据、工具或执行能力的系统。  
**规范性用语：** “必须”表示强制要求；“应”表示推荐要求；“可”表示可选实现。

---

## 1. 目标与原则

本标准定义 Agent 从设计、登记、审核、启用到停用的受控生命周期。其目标是确保每个 Agent 均具备明确负责人、可验证身份、最小权限、适当审批、完整审计和可立即停用的路径。

生命周期管理必须遵循以下原则：

1. **默认拒绝：** 状态、身份、版本、权限或审批无法验证时，必须拒绝请求。
2. **最小权限：** 权限必须按 Agent、版本、环境、资源和动作授予；批准不等同于无限期执行授权。
3. **职责分离：** 提交人不得独自批准其高风险 Agent 或高风险权限申请。
4. **先验证后启用：** 未完成安全审核、测试和正式审批的 Agent 不得上线。
5. **持续治理：** Active 不是永久信任；运行时必须继续校验状态、权限、审批和策略。
6. **可追溯、可停用：** 状态、权限、版本和负责人变更必须留存审计证据，并可快速暂停。

## 2. 状态模型

标准主路径如下：

```text
Draft
  ↓
Registered
  ↓
Approved
  ↓
Active
  ↓
Suspended
  ↓
Deprecated
```

`Suspended` 是风险处置和整改状态，不代表所有 Agent 必须在正常退役前停留于该状态。正常停用时，Active Agent 可在完成停用控制后直接进入 Deprecated；发生安全事件、重大异常、未授权变更或治理要求时，必须先进入 Suspended。Suspended Agent 完成整改、复测和重新审批后，可回到 Approved，再按上线门禁进入 Active。Deprecated 不得直接恢复；如确有业务需要，必须以新版本或新注册记录重新走完整流程。

## 3. 各阶段控制要求

| 阶段 | 进入条件 | 审批要求 | 权限变化 | 退出条件 |
|---|---|---|---|---|
| **Draft** | 已提出业务需求，并指定业务负责人、技术负责人和初始用途。 | 无正式上线审批；负责人必须确认需求和设计可进入注册准备。 | 无运行时身份、数据、工具或执行权限；不得接入网关或接收任务。 | 注册包完整：元数据、版本、能力范围、风险分级、数据与工具清单、权限申请、审计与停用设计均已提交。 |
| **Registered** | Registry 已生成唯一且不可复用的 Agent ID；注册包通过完整性校验。 | 必须完成安全审核；安全审核发现的问题必须关闭或取得书面风险接受。 | 仅允许受控的注册验证和隔离测试所需的临时最小权限；禁止生产数据访问、生产执行和自动任务分派。 | 安全审核通过，测试计划获确认，且测试环境、测试数据和回滚方案就绪。 |
| **Approved** | 安全审核和测试均已通过；测试证据、审计证据及遗留风险处置已归档。 | 业务、技术、安全和权限责任方必须按风险等级完成正式审批；高风险执行能力须取得额外动作审批。 | 可配置已批准的生产最小权限，但默认保持未启用；不得因 Approved 自动获得生产任务或执行权。 | 上线变更获批准，生产配置、监控、告警、限额、审计和紧急停止机制均已验证。 |
| **Active** | 已满足所有上线门禁，并已完成受控上线。 | 上线批准必须在有效期内；需要人工审批的具体动作仍须逐次或按已批准策略审批。 | 启用已批准范围内的身份、数据、工具和执行权限；每次请求仍须做运行时状态、权限与审批校验。 | 发生安全/合规/稳定性风险时立即转 Suspended；业务终止、替代完成或退役计划获批时转 Deprecated。 |
| **Suspended** | 因安全事件、异常行为、审批失效、未授权变更、负责人失效、治理要求或计划维护而暂停。 | 暂停可由安全、治理或授权运维责任方紧急发起；恢复必须完成原因分析、整改、复测及重新审批。 | 必须阻断新任务、数据访问、工具调用和执行；撤销或禁用凭据、会话、令牌及自动调度，仅保留受控审计查询。 | 整改证据和复测通过后回到 Approved；确认不再使用、已完成迁移和保留要求后进入 Deprecated。 |
| **Deprecated** | 已批准停用，或 Suspended Agent 确认不再恢复；依赖方和替代方案已识别。 | 停用与数据/审计保留计划必须由业务和治理责任方确认；涉及高风险数据时须安全确认。 | 永久撤销运行、数据、工具、执行和调度权限；不得接收新任务或新授权，仅保留合规所需的受控审计与迁移访问。 | 完成留存期内的审计、迁移和删除/归档义务后关闭记录；不得恢复为其他状态。 |

## 4. 状态转换规则

1. 状态变更必须由 Registry 记录 Agent ID、版本、原状态、新状态、原因、申请人、审批人、时间（UTC）和关联变更单或事件号。
2. 任何能力、数据范围、工具范围、执行范围、负责人或版本的重大变更，都必须重新评估风险；扩大权限或影响范围时，必须至少回到 Registered 重新审核。
3. 仅 Active Agent 可在已批准范围内服务生产请求。即使处于 Active，未获相应权限或动作审批的请求仍必须被拒绝。
4. 任何人员或自动化控制发现高风险异常时，都必须能够触发 Suspended；紧急暂停不得等待常规审批完成。
5. Deprecated 的 Agent ID、凭据和执行授权不得复用。

## 5. Agent 上线流程

上线流程必须依次执行，且每一步都应保留可审计的输入、结论、责任人与时间记录：

| 步骤 | 必需活动 | 主要责任方 | 产出与门禁 |
|---|---|---|---|
| 1. **创建** | 定义业务目标、适用边界、负责人、版本、能力草案、风险等级、禁止动作和停用方案。 | 业务负责人、Agent 负责人 | 建立 Draft；不得申请超出业务目标的泛化权限。 |
| 2. **注册** | 向 Registry 提交最小元数据、身份与认证方案、数据/工具清单、权限申请、依赖关系、审计设计和变更记录。 | Agent 负责人、平台治理方 | 生成唯一 Agent ID，进入 Registered；缺少必填信息不得注册。 |
| 3. **安全审核** | 审查身份认证、密钥管理、提示词/工具边界、数据分类与脱敏、最小权限、执行风险、日志、限额、紧急停止和供应链风险。 | 安全负责人、权限责任方 | 审核结论、问题清单和风险接受记录；高风险问题未关闭不得进入测试。 |
| 4. **测试** | 在隔离或非生产环境验证身份、授权拒绝、审批门禁、审计链路、异常处理、限额、超时、重试、回滚/补偿及暂停能力。 | 技术负责人、测试负责人、安全负责人 | 测试报告和缺陷处置证据；不得使用未经授权的生产数据或生产执行权限。 |
| 5. **审批** | 汇总安全审核、测试、风险评估、权限范围、上线与回滚计划，执行业务、技术、安全和权限审批。 | 审批责任方 | 进入 Approved；审批必须绑定 Agent、版本、环境、权限范围及有效期。 |
| 6. **上线** | 实施受控变更，启用最小生产权限、监控、告警、审计、限额、值守和紧急停止措施。 | 发布负责人、平台运维责任方 | 状态变为 Active；任一上线门禁缺失时必须回退并保持 Approved。 |
| 7. **监控** | 持续监控可用性、错误率、拒绝率、权限异常、审批异常、工具调用、成本/配额、业务影响和安全事件。 | Agent 负责人、安全与运维责任方 | 定期复核记录；异常达到阈值时自动或人工暂停并进入 Suspended。 |
| 8. **停用** | 停止任务分派，撤销凭据和授权，终止调度，迁移依赖，通知使用方，归档审计并执行保留策略。 | Agent 负责人、平台治理方、安全负责人 | 进入 Deprecated；确认无新调用、无残留授权且留存要求已落实。 |

## 6. 上线与运行门禁

进入 Active 前，以下条件必须全部满足：

- Registry 中的 Agent ID、版本、负责人、能力和风险等级完整且一致；
- 安全审核和测试均已通过，或存在已批准且有期限的风险接受记录；
- 生产权限按最小权限配置，并与批准的资源、动作、环境和有效期一致；
- 所需人工审批策略已生效，高风险或不可逆动作默认不得绕过人工审批；
- 审计日志可关联 Agent ID、版本、调用、权限判定、审批引用、结果和异常；
- 监控、告警、速率/影响限额、回滚或补偿方案以及紧急停止机制均已验证；
- 负责人和值守/升级路径有效，且依赖方已知晓上线影响。

运行期间，必须至少在敏感访问和执行前校验 Agent 的 Active 状态、身份、版本、权限和审批有效性。任一校验失败时，系统必须拒绝请求、记录审计事件，并在适用时触发暂停或告警。

## 7. 停用与证据保留

停用不得仅删除配置。进入 Suspended 或 Deprecated 时，必须阻断运行入口并撤销有效凭据、会话、令牌、调度和权限绑定。应保留满足合规、调查和迁移所需的注册记录、审批记录、状态变更、审计日志、测试证据和风险处置记录；保留期限及删除方式应遵循适用法规和 VAFOX 数据治理中更严格的要求。

本标准与其他 Agent 注册、权限、审批及审计标准并行适用；如要求不一致，应采用更严格的控制。
