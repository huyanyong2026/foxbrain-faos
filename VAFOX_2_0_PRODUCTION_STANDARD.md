# VAFOX 2.0 企业级 AI 操作系统生产标准

## 0. 文档目标与适用范围

本文档用于建立 VAFOX 2.0 企业级 AI 操作系统的生产标准，指导 VAFOX 从 1.0 实验建设阶段升级到 2.0 企业生产阶段。

本标准只定义架构、角色、目录、发布、配置、权限、Agent 接入、备份、安全、扩展与验收规范；不包含任何服务器登录、部署、变更或执行命令操作。

## 1. VAFOX 2.0 总体架构

### 1.1 总体原则

VAFOX 2.0 采用“统一入口、分层控制、智能执行、数据中枢”的企业级 AI 操作系统架构：

- `gateway.vafox.com` 是统一生产入口与流量网关。
- `huyan.vafox.com` 是 CEO Control Plane，负责经营控制、审批、指令与管理闭环。
- `ai.vafox.com` 是 AI Workforce Platform，负责 AI 员工、机器人、任务执行与自动化协作。
- `core.vafox.com` 是 Enterprise Data Brain，负责企业数据、知识库、业务事实与数据服务。

### 1.2 节点关系

| 域名 | 当前节点 | 定位 | 标准角色 |
| --- | --- | --- | --- |
| `gateway.vafox.com` | 统一入口 | Production Gateway | 统一入口、TLS 终止、路由、访问控制、审计入口 |
| `huyan.vafox.com` | `140.143.207.194` | CEO Control Plane | CEO 控制台、审批中心、指令中心、经营驾驶舱 |
| `ai.vafox.com` | `1.13.254.217` | AI Workforce Platform | AI 员工平台、Agent 编排、任务执行、机器人工作台 |
| `core.vafox.com` | `139.199.174.36` | Enterprise Data Brain | 数据中枢、知识库、业务事实库、统一数据服务 |

### 1.3 标准访问链路

```text
用户 / CEO / 员工 / Robot
        ↓
gateway.vafox.com
        ↓
 ┌───────────────┬───────────────┬───────────────┐
 ↓               ↓               ↓
huyan.vafox.com  ai.vafox.com    core.vafox.com
CEO控制面        AI劳动力平台     企业数据大脑
        ↓               ↓               ↑
        └──── 审批 / 任务 / 数据 / 审计 ────┘
```

### 1.4 架构边界

- Gateway 不承载业务逻辑，只负责入口、路由、安全与审计接入。
- Huyan 不直接修改底层生产数据，所有高风险动作必须进入审批与审计流程。
- AI 不绕过 Huyan 的权限与审批机制执行生产动作。
- Core 不承载前端控制面与 Agent 编排，只提供可信数据、知识与事实服务。

## 2. 服务器角色标准

### 2.1 Huyan：CEO Control Plane

Huyan 是 VAFOX 2.0 的 CEO 控制面，标准职责包括：

- CEO 经营驾驶舱。
- 企业指令发布与跟踪。
- 审批中心与授权控制。
- 关键业务状态查看。
- AI 员工工作结果验收。
- 高风险生产动作的人类确认入口。
- 企业级审计查询入口。

Huyan 不应承担大规模模型推理、长期数据存储、底层数据库职责。

### 2.2 AI：AI Workforce

AI 是 VAFOX 2.0 的 AI 劳动力平台，标准职责包括：

- AI 员工注册、运行与管理。
- Agent 能力声明与任务编排。
- 自动化工作流执行。
- Robot 用户运行环境。
- 任务日志、执行结果与失败重试。
- 与 Huyan 审批链路、Core 数据服务对接。

AI 不得绕过权限系统直接执行生产变更，不得保存未受控的长期企业核心数据副本。

### 2.3 Core：Data Brain

Core 是 VAFOX 2.0 的企业数据大脑，标准职责包括：

- 结构化业务数据库。
- 企业知识库与向量知识资产。
- 业务事实库与指标口径。
- 统一数据服务 API。
- 数据备份、恢复、校验与审计。
- 数据访问权限控制。

Core 不应承担 CEO 控制台展示逻辑，也不应直接运行未经注册的业务 Agent。

## 3. 目录标准

### 3.1 `/opt/vafox-control`

适用节点：Huyan / CEO Control Plane。

标准职责：

- 存放 CEO 控制面应用程序。
- 存放控制台后端服务、前端构建产物与启动配置。
- 存放审批流、控制面配置、经营视图配置。
- 存放与 AI、Core 的受控集成配置。

目录要求：

- 生产运行文件与发布包来源必须可追溯。
- 不允许在该目录直接手工修改生产代码。
- 配置文件必须与 Secret 分离。
- 日志不得与应用代码混放。

### 3.2 `/opt/ai-vafox`

适用节点：AI / AI Workforce Platform。

标准职责：

- 存放 AI Workforce 平台应用程序。
- 存放 Agent 注册表、能力声明、任务编排配置。
- 存放 Robot 运行时、工作流执行器、调度器。
- 存放任务执行缓存与临时工作区。

目录要求：

- Agent 必须通过注册机制接入，不允许散落脚本直接成为生产机器人。
- 任务执行临时目录必须具备生命周期清理策略。
- Robot 凭证不得硬编码在脚本、配置或仓库中。
- 执行结果必须进入标准日志与审计链路。

### 3.3 `/opt/core`

适用节点：Core / Enterprise Data Brain。

标准职责：

- 存放数据服务应用程序。
- 存放数据库连接配置模板。
- 存放知识库服务、索引服务、数据同步服务。
- 存放数据校验、备份、恢复与迁移工具。

目录要求：

- 数据文件、备份文件、应用代码必须分层管理。
- 生产数据不得由未授权用户直接读取或删除。
- 数据迁移脚本必须具备版本、回滚说明与验收记录。
- 知识库索引必须可重建、可校验、可审计。

## 4. Release 标准

### 4.1 禁止事项

VAFOX 2.0 生产环境禁止以下发布方式：

- 禁止在生产服务器直接修改代码。
- 禁止在生产服务器直接拉取未验收分支覆盖运行版本。
- 禁止跳过构建、校验、备份、审批直接重启生产服务。
- 禁止使用无法追溯来源的二进制、脚本或压缩包。
- 禁止将开发环境配置、测试 Secret 或个人凭证带入生产。

### 4.2 标准发布链路

所有生产发布必须遵循以下链路：

```text
Artifact
   ↓
Checksum
   ↓
Installer
   ↓
Verify
```

### 4.3 Artifact 标准

Artifact 是唯一允许进入生产的发布载体，必须满足：

- 由受控 CI/CD 或受控构建流程生成。
- 包含版本号、构建时间、Git Commit、构建环境说明。
- 不包含明文 Secret。
- 可重复获取、可审计、可回滚。

### 4.4 Checksum 标准

每个 Artifact 必须生成 Checksum，必须满足：

- 使用标准哈希算法生成校验值。
- 发布前、传输后、安装前必须校验一致。
- Checksum 与 Artifact 版本绑定保存。
- 校验失败必须立即停止发布。

### 4.5 Installer 标准

Installer 是生产安装唯一入口，必须满足：

- 支持幂等执行。
- 支持预检查、安装、回滚或失败退出。
- 明确写入版本记录。
- 不在执行过程中静默吞掉错误。
- 不绕过权限、审批与审计。

### 4.6 Verify 标准

每次发布完成后必须执行 Verify，至少包括：

- 服务进程状态检查。
- HTTP 健康检查。
- 版本号检查。
- 关键 API 冒烟测试。
- 依赖连接检查。
- 日志错误扫描。
- 回滚条件确认。

## 5. 配置标准

### 5.1 环境变量

生产环境配置必须通过标准环境变量或受控配置系统注入：

- 不允许将生产环境变量硬编码到代码中。
- 环境变量必须按服务、环境、用途分组。
- 必须区分 `production`、`staging`、`development`。
- 关键变量必须有默认禁止策略，即缺失时拒绝启动，而不是使用危险默认值。

### 5.2 Secret 管理

Secret 包括数据库密码、API Key、Token、私钥、机器人凭证等，必须满足：

- 不得提交到 Git 仓库。
- 不得写入镜像、Artifact 或普通配置文件。
- 必须使用 Secret 管理系统、受控环境变量或加密文件注入。
- 必须支持轮换、吊销、最小权限与访问审计。
- Secret 泄露必须触发立即吊销、替换与事故复盘。

### 5.3 权限管理

配置权限必须遵循最小权限原则：

- 普通服务只获得运行所需权限。
- 发布权限与运行权限分离。
- 审批权限与执行权限分离。
- 数据读取权限与数据修改权限分离。
- Robot 用户不得默认拥有管理员权限。

### 5.4 日志管理

生产日志必须具备可观测、可审计、可追踪能力：

- 应用日志、访问日志、错误日志、审计日志分层记录。
- 日志必须包含时间、服务、请求 ID、用户或 Robot 身份、动作、结果。
- 日志不得输出明文 Secret、Token、密码、身份证明材料。
- 关键日志必须集中收集、保留、检索与告警。
- 日志保留周期应满足企业审计与合规要求。

## 6. 权限标准

### 6.1 CEO 用户

CEO 用户是最高业务决策身份，但不等同于无限系统 Root 权限。

标准权限：

- 查看企业经营全局状态。
- 发起、审批、驳回关键业务动作。
- 查看 AI 员工工作结果与审计记录。
- 授权高风险生产变更。

限制：

- 不直接操作底层数据库。
- 不直接绕过发布流程修改生产。
- 高风险动作仍必须进入审计记录。

### 6.2 员工用户

员工用户是企业业务协作身份。

标准权限：

- 访问与岗位相关的业务功能。
- 提交任务、查看授权范围内的数据。
- 对 AI 结果进行反馈或验收。

限制：

- 不得访问岗位无关数据。
- 不得执行未授权生产变更。
- 不得使用共享账号。

### 6.3 Robot 用户

Robot 用户是 Agent、自动化任务、系统集成的运行身份。

标准权限：

- 只访问自身任务所需 API、数据与工具。
- 通过注册、授权、审计机制执行任务。
- 具备明确的能力边界与调用范围。

限制：

- 不得默认拥有管理员权限。
- 不得共享 CEO 或员工个人账号。
- 不得绕过审批执行高风险动作。
- 不得保存超出任务需要的敏感数据。

### 6.4 `vafox-exec` 用户

`vafox-exec` 是生产执行用户，用于受控执行 Installer、服务启动、健康检查与标准运维动作。

标准权限：

- 只能执行经过批准的生产安装、启动、停止、验证脚本。
- 只能访问职责范围内的应用目录、日志目录与运行目录。
- 所有操作必须进入审计记录。

限制：

- 不作为日常登录用户使用。
- 不直接持有数据库超级管理员权限。
- 不绕过 Artifact、Checksum、Installer、Verify 发布链路。

## 7. Agent 接入标准

未来所有机器人、AI 员工、自动化 Agent 必须通过标准接入流程进入 VAFOX 2.0。

### 7.1 注册

每个 Agent 必须注册以下信息：

- Agent ID。
- Agent 名称。
- 所属业务域。
- 负责人。
- 运行身份。
- 版本。
- 上线时间。
- 回滚或停用方式。

### 7.2 声明能力

每个 Agent 必须声明能力边界：

- 可执行任务类型。
- 可访问系统。
- 可读取数据范围。
- 可写入或修改的数据范围。
- 可调用外部服务。
- 失败处理方式。

### 7.3 权限控制

Agent 权限必须满足：

- 默认无权限，按任务授权。
- 高风险动作必须审批。
- 数据读取、数据修改、外部调用分离授权。
- 权限变更必须记录原因、审批人、时间与影响范围。

### 7.4 审计记录

Agent 每次生产动作必须记录：

- Agent ID 与运行身份。
- 触发来源。
- 输入摘要。
- 执行动作。
- 影响对象。
- 输出结果。
- 成功或失败状态。
- 错误信息与重试记录。
- 审批单号或授权依据。

## 8. 备份标准

### 8.1 数据库备份

数据库备份必须覆盖：

- 业务主库。
- 用户与权限数据。
- 任务与审批记录。
- 审计日志索引。
- 关键配置表。

标准要求：

- 定期全量备份。
- 高频增量或日志备份。
- 备份加密存储。
- 跨节点或异地保存。
- 定期恢复演练。
- 恢复结果必须验收记录。

### 8.2 知识库备份

知识库备份必须覆盖：

- 原始文档。
- 清洗后文本。
- 向量索引。
- 元数据。
- 版本信息。
- 权限标签。

标准要求：

- 原始资料必须可追溯。
- 向量索引必须可重建。
- 知识库变更必须记录来源与操作者。
- 删除知识必须具备审批与恢复机制。

### 8.3 配置备份

配置备份必须覆盖：

- 服务配置模板。
- 环境变量清单。
- 非 Secret 配置。
- 权限配置。
- 路由配置。
- 告警配置。

标准要求：

- Secret 不得以明文形式进入普通配置备份。
- 配置变更必须有版本记录。
- 配置恢复必须经过验证。

### 8.4 工作流备份

工作流备份必须覆盖：

- 审批流。
- Agent 编排流。
- 自动化任务定义。
- 调度规则。
- 失败重试策略。
- 业务触发条件。

标准要求：

- 工作流必须可导出、可恢复、可比对。
- 生产工作流变更必须经过审批。
- 废弃工作流必须保留停用记录。

## 9. 安全标准

### 9.1 禁止未审批修改生产

任何生产变更必须经过授权与记录，禁止：

- 未审批直接修改生产代码。
- 未审批直接修改生产配置。
- 未审批变更数据库结构或关键数据。
- 未审批上线、下线或修改 Agent。

### 9.2 禁止直接删除数据

生产环境禁止直接删除数据。确需删除时必须满足：

- 有明确业务理由。
- 有审批记录。
- 有影响评估。
- 有备份或可恢复方案。
- 有执行记录与复核结果。

### 9.3 禁止绕过审批

任何人员、系统、Robot、Agent 均不得绕过审批流程执行高风险动作，包括：

- 权限提升。
- 数据批量修改。
- 数据批量导出。
- 生产发布。
- 停止核心服务。
- 修改审计日志。
- 修改备份策略。

### 9.4 安全基线

VAFOX 2.0 生产安全基线包括：

- 强制身份认证。
- 最小权限授权。
- 全链路审计。
- 关键操作审批。
- Secret 加密管理。
- 生产发布可追溯。
- 数据备份可恢复。
- 高风险动作可阻断。

## 10. 扩展标准

VAFOX 2.0 架构必须支持未来横向扩展与专业机器人接入。

### 10.1 GPU 服务器

未来 GPU 服务器用于模型推理、训练、嵌入生成或高性能 AI 任务，必须：

- 通过 AI Workforce 注册为受控计算资源。
- 不直接暴露给普通用户或未授权 Agent。
- 具备任务队列、资源隔离、配额与成本记录。
- 推理结果与任务日志进入审计链路。

### 10.2 BI 服务器

未来 BI 服务器用于经营分析、报表、指标看板，必须：

- 从 Core 获取统一口径数据。
- 不绕过 Core 直接读取未授权数据源。
- 指标定义、数据刷新、报表权限必须可审计。
- CEO 控制面只展示已授权的 BI 结果。

### 10.3 供应链机器人

供应链机器人必须：

- 注册为 Robot 用户与 Agent。
- 声明采购、库存、补货、供应商协同等能力边界。
- 关键采购、调价、供应商变更必须审批。
- 所有动作进入审计记录。

### 10.4 财务机器人

财务机器人必须：

- 采用更高安全等级的权限控制。
- 涉及付款、对账、发票、成本、利润数据时必须强审计。
- 高风险财务动作必须人工审批。
- 不得保存未加密的财务 Secret 或敏感凭证。

### 10.5 门店机器人

门店机器人必须：

- 按门店、岗位、任务范围授权。
- 支持门店运营、巡检、补货、员工协作等场景。
- 不得跨门店访问未授权数据。
- 门店异常动作必须上报 Huyan 控制面。

## 11. 验收标准

VAFOX 2.0 生产完成必须同时满足以下条件。

### 11.1 架构验收

- `gateway.vafox.com`、`huyan.vafox.com`、`ai.vafox.com`、`core.vafox.com` 的角色边界清晰。
- Gateway、Huyan、AI、Core 的访问关系符合本标准。
- CEO Control Plane、AI Workforce、Data Brain 分层运行。

### 11.2 发布验收

- 所有生产服务均通过 Artifact 发布。
- 每个 Artifact 均有 Checksum。
- 生产安装通过 Installer 执行。
- 每次发布后均完成 Verify。
- 禁止生产直接修改的制度已落实。

### 11.3 配置与权限验收

- 环境变量、Secret、权限、日志管理符合标准。
- CEO 用户、员工用户、Robot 用户、`vafox-exec` 用户边界明确。
- 高风险动作具备审批机制。
- 权限变更具备审计记录。

### 11.4 Agent 验收

- 所有生产 Agent 均完成注册。
- 所有 Agent 均声明能力边界。
- Agent 权限按最小权限原则授予。
- Agent 执行动作可审计、可追溯、可停用。

### 11.5 备份与恢复验收

- 数据库、知识库、配置、工作流均有备份策略。
- 备份文件具备加密、保留与访问控制。
- 已完成至少一次恢复演练。
- 恢复演练具备记录与验收结论。

### 11.6 安全验收

- 未审批修改生产、直接删除数据、绕过审批均被制度禁止。
- Secret 不在代码、仓库、Artifact 中明文保存。
- 关键生产动作具备日志、审计与告警。
- 高风险动作具备阻断与人工确认机制。

### 11.7 扩展验收

- 架构支持 GPU 服务器、BI 服务器扩展。
- 架构支持供应链机器人、财务机器人、门店机器人接入。
- 新节点与新机器人必须遵循注册、授权、审计、备份、安全标准。

### 11.8 完成定义

当且仅当以下条件全部满足时，VAFOX 2.0 可判定为企业生产阶段完成：

1. 四层架构角色明确并按标准运行。
2. 生产发布链路完成 Artifact、Checksum、Installer、Verify 闭环。
3. 目录、配置、权限、日志标准落地。
4. CEO、员工、Robot、`vafox-exec` 用户权限边界落地。
5. 所有生产 Agent 完成注册、能力声明、权限控制与审计记录。
6. 数据库、知识库、配置、工作流具备可恢复备份。
7. 安全禁止项已制度化并具备技术或流程控制。
8. 未来扩展节点与机器人接入有统一标准。
9. 每次生产变更都可追溯、可回滚、可审计。
10. CEO 可通过 Huyan 控制面查看、审批、验收 VAFOX 2.0 的关键生产动作。
