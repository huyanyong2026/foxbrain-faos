# VAFOX M6.1 AI Runtime Foundation Design

**状态：** Design Phase / 仅设计，不部署。<br>
**前置设计：** M5.1 Digital Employee Foundation、M5.2 Employee AI Workspace、M5.3 Store Manager AI、M5.4 Procurement AI、M5.5 Product Manager AI、M5.6 CEO AI Cockpit Design。<br>
**负责人：** Codex（设计开发）；WorkBuddy（环境）；Marvis（管理）；CTO（验收）。

## 1. 目标、原则与非目标

M6.1 定义 FoxBrain **AI Runtime**：一个位于 AI Workspace 与各 Business Brain 之间的、服务端治理的只读编排层。它为每次问答建立可信 AI 身份与上下文，执行权限、数据范围、知识/记忆/工具路由、Citation、审计与反馈闭环，并输出可解释的经营辅助建议。Runtime 不替代业务系统、审批流程或人类决策。

### 1.1 设计原则

1. **服务端身份与最小权限。** 用户认证、组织、角色、范围和策略快照仅能由服务端从可信身份、目录、策略与资源授权派生；客户端的过滤器只能缩小范围。
2. **读、析、建议与执行分离。** Runtime 只允许查询、分析与建议。建议是待人工核实的内容，不构成交易或经营指令。
3. **证据优先、时点明确。** 每项可验证事实与建议均附来源、版本、时间戳和置信度；来源过期、冲突、不可见或不足时必须降级为 `needs_verification` 或拒绝回答。
4. **拒绝优先且不泄露。** RBAC、ACL、数据范围、字段策略、用途、同意和有效期必须同时通过。拒绝信息不得暴露受限资源存在性、数量或名称。
5. **审计可回放、反馈须审核。** 请求、上下文快照、路由、证据、回答和反馈可关联回放；反馈只能形成候选，不能直接改变生产知识、权限、模型或业务事实。

### 1.2 非目标与硬边界

本文件只描述设计契约，不部署 Runtime、模型、连接器、数据管道或生产同步。AI Runtime 严禁直接或间接：

- 修改 SAP、SAP 主数据或 SAP 交易；
- 修改 Core 生产同步、同步规则或生产数据；
- 自动销售、报价、营销发送、成交、锁货、预留、调拨；
- 自动采购、补货、创建/提交采购单、自动下单；
- 自动付款、审批或执行任何经营动作。

即使用户接受建议，Runtime 也只记录人工意图或待办；真正的业务动作必须由有权人员在获批准的权威系统内独立完成并由该系统审计。

## 2. AI Runtime Architecture

```text
Human User
    ↓ authenticated request + declared purpose
AI Workspace
    ↓ question / allowed filters / correlation ID
AI Runtime
├── Identity Manager
├── Permission Engine
├── Memory Router
├── Knowledge Router
├── Tool Router
├── Citation Engine
├── Audit Engine
└── Feedback Engine
    ↓ authorized, read-only context and evidence requests
Business Brains
├── Memory Brain
├── Knowledge Brain
├── Product Intelligence
├── Customer Brain
└── Retail Brain
```

### 2.1 组件职责与强制控制

| 组件 | 职责 | 输入 / 输出 | 强制控制 |
| --- | --- | --- | --- |
| Human User | 提出业务问题、核实和反馈。 | 认证后的自然人主体、请求用途。 | 不能声明或提升 AI 身份、角色、组织或范围。 |
| AI Workspace | 呈现问答、证据、限制和人工建议。 | 问题与可选收窄筛选；返回结构化回答。 | 不持有特权；不直连 Business Brain。 |
| Identity Manager | 验证认证主体并派生 AI Identity。 | 令牌、目录、组织归属、有效角色。 | 拒绝客户端提交的角色/范围；短期上下文、策略版本绑定。 |
| Permission Engine | 合并 RBAC、ACL、Data Scope、用途和字段策略。 | AI Identity、请求与资源元数据；返回 allow/deny/mask。 | 默认拒绝、逐资源/字段判定、不得泄露存在性。 |
| Memory Router | 按问题意图选择授权记忆域。 | 已授权问题、身份、时间窗；返回最小记忆证据。 | 记忆检索先授权后召回；客户记忆另验同意与关系。 |
| Knowledge Router | 选择可见、已发布、有效的知识域和版本。 | 问题、知识范围；返回知识片段及版本。 | 草案、撤销、过期或无 ACL 知识不可使用。 |
| Tool Router | 以 allow-list 调度只读 Brain 查询/分析工具。 | 经过策略核准的工具计划；返回工具结果。 | 无事务型工具；参数受范围约束、超时/配额/输出校验。 |
| Citation Engine | 将回答主张绑定至来源与质量信息。 | 证据、回答草案；返回 Citation 集合与覆盖检查。 | 无支持主张不得伪装成事实；Citation 不可见则移除/降级。 |
| Audit Engine | 记录不可篡改的请求、决策与输出证据链。 | 运行事件；返回 `audit_id`。 | 最小化记录敏感数据，访问审计记录也受权限和保留期限制。 |
| Feedback Engine | 接收反馈、脱敏、分流、审核和知识候选。 | 反馈与关联审计；返回状态。 | 不自动写入知识、模型、权限、业务事实或执行系统。 |
| Business Brains | 提供授权的领域只读事实、知识或分析。 | Scope-bound 查询；返回快照/证据。 | 每个 Brain 再次执行自身 ACL 与字段策略，Runtime 不是绕过路径。 |

### 2.2 请求生命周期

```text
认证请求 → Identity Manager 派生 AI Identity
         → Permission Engine 建立 policy snapshot / 可访问资源集合
         → Memory + Knowledge Router 生成授权检索计划
         → Tool Router 仅调用 allow-list 只读工具
         → Citation Engine 验证主张、来源、版本、时点与置信度
         → 生成 advisory-only 回答 / needs_verification / controlled denial
         → Audit Engine 记录可回放事件 → 可选 Feedback Engine
```

若任何必需检查失败，后续组件不得接收未授权数据。模型看到的上下文必须是已通过字段脱敏、范围限制与用途验证后的最小投影；模型输出在返回前再次经策略、敏感信息、Citation 覆盖和执行意图检查。

## 3. AI Identity Model

`AIIdentity` 是单次 Runtime 请求的服务端派生安全上下文，不是可由前端保存、编辑或复用的用户配置，也不是独立的人事身份。它将认证的人类用户与其当前允许使用的 AI 角色、资源和策略绑定。

| 字段 | 定义 | 服务端派生规则 |
| --- | --- | --- |
| `ai_employee_id` | 稳定的 AI 员工/编排身份引用。 | 由认证主体、组织、允许角色模板和 Runtime 策略映射产生；不得由客户端传入。 |
| `role` | 当前有效业务角色，可为受控多角色集合中的本次活动角色。 | 来自身份目录与角色分配；冲突或失效角色拒绝，不能由 UI 选择特权角色。 |
| `organization` | 租户、法人/组织单元及适用区域。 | 来自认证租户与组织目录，作为所有查询的不可跨越边界。 |
| `permission_scope` | 经 RBAC/ACL/用途计算的能力与字段权限。 | 与策略版本、资源授权和会话时点绑定；请求参数只能求交集。 |
| `knowledge_scope` | 可读取的知识域、文档状态、版本、有效期和许可范围。 | 由知识 ACL、组织、角色和用途派生，仅含 published/effective/not-revoked 内容。 |
| `business_scope` | 可查询的品牌、门店、区域、渠道、品类、产品和时间粒度。 | 由业务资源 ACL 和 Data Scope 派生；默认最小聚合粒度。 |
| `memory_scope` | 可检索的企业、客户、销售和商品记忆域及字段/保留限制。 | 由记忆 ACL、客户同意/关系、用途与保留策略派生。 |
| `tool_scope` | 可调用的只读工具、参数边界、速率和输出等级。 | 由角色、用途、策略 allow-list 与风险级别派生，永不包括执行工具。 |

示例（仅表示契约，不表示可由客户端提交）：

```json
{
  "ai_employee_id": "aie_01J...",
  "role": ["store_manager"],
  "organization": {"tenant_ref": "org_cn_01", "region_refs": ["CN-SZ"]},
  "permission_scope": {"purpose": "store_operations_analysis", "export": false, "policy_snapshot": "pol_01J..."},
  "knowledge_scope": {"domains": ["brand", "product", "sales", "route"], "states": ["published"], "effective_at": "2026-07-22T00:00:00Z"},
  "business_scope": {"store_refs": ["store_sz_01"], "brand_refs": ["brand_a"], "granularity": "store"},
  "memory_scope": {"domains": ["enterprise", "sales", "product"], "customer_detail": "denied"},
  "tool_scope": {"allowed": ["retail.read_aggregate", "product_intelligence.read"], "execution": "denied"}
}
```

Identity Manager 对每次请求重新求值，并记录 `identity_snapshot_id`、`policy_snapshot`、目录版本和派生时间。令牌失效、角色变更、组织转移、同意撤回、策略撤销或会话超时后，旧上下文不得继续使用。

## 4. Permission Engine

Permission Engine 的决定为：`allow`、`deny`、`allow_masked` 或 `allow_aggregate_only`。一个资源的最终允许条件为：

```text
RBAC capability ∩ resource ACL ∩ organization/data scope ∩ purpose ∩ field policy ∩ effective time
```

### 4.1 三层授权模型

| 层 | 解决的问题 | 设计 |
| --- | --- | --- |
| RBAC | 该角色原则上能做什么？ | 决定 Runtime 查询、聚合、导出、反馈审核等能力上限。 |
| ACL | 该主体能访问哪一个具体资源？ | 针对门店、品牌、文档、客户关系、产品、记忆记录和审计事件逐项判定。 |
| Data Scope | 可见到何种组织范围、时间/字段粒度？ | 约束组织、区域、门店、品牌、品类、渠道、客户关系、时间窗以及明细/汇总等级。 |

角色不等于资源授权。所有角色仍受组织隔离、资源 ACL、用途、字段掩码、客户同意与数据时效约束；CEO 也不能因职位绕过客户同意或字段 ACL。

| 角色 | 默认可用分析范围 | 默认输出粒度 | 明确限制 |
| --- | --- | --- | --- |
| 员工 | 本人任务、已关联且获同意的客户最小画像、授权门店的销售/商品辅助。 | 客户最小投影、个人任务。 | 不见未关联客户、采购/组织级敏感数据或员工数据。 |
| 店长 | 授权门店/团队的销售、库存、任务、商品和风险信号。 | 门店及授权团队聚合。 | 不越店、不下钻未授权客户、不自动排班/补货。 |
| 采购 | 授权品牌/品类/区域的趋势、库存与供应研究。 | 品类/品牌/区域聚合。 | 不创建订单、采购单或供应商承诺。 |
| 商品经理 | 授权品牌、产品线、SKU/品类与门店商品表现。 | 商品/品类/授权门店。 | 不修改价格、主数据、生命周期或发布状态。 |
| CEO | 组织、区域和业务单元经营汇总、风险与趋势。 | 默认去标识化聚合。 | 不绕过数据范围、客户同意或字段策略；无交易权。 |

### 4.2 判定与拒绝规则

1. 先验证认证、组织和用途，再解析意图；未认证或用途不允许时不检索。
2. 客户资源必须同时通过组织、关系、同意、用途、记录 ACL 与字段 ACL；失败返回同一受控拒绝或允许的聚合结果。
3. 客户端的 `store_id`、`brand_id`、`date_range`、`domain` 等只能与派生范围求交，空交集为拒绝，不做自动扩展。
4. 对导出、高敏字段、跨范围比较和审计查看执行更严格的显式权限；默认不导出原始 PII。
5. 每个决定产生不可变 `authorization_decision_id`，其中仅记录资源引用/哈希、规则版本、结果、掩码和原因代码。

## 5. Memory Router

Memory Router 在权限门控后按问题意图、实体、时效和身份的 `memory_scope` 选择一个或多个记忆域；它不把所有记忆拼入上下文，也不以语义相似度绕过 ACL。

| 记忆域 | 适用问题信号 | 可用内容 | 关键控制 |
| --- | --- | --- | --- |
| 企业记忆 | 政策、流程、组织约定、已发布经营决议。 | 已批准企业规则、流程摘要、受控历史决议。 | 按组织、文档 ACL、版本和有效期过滤。 |
| 客户记忆 | 明确授权客户及其关系/需求问题。 | 同意范围内的明确偏好、沟通/购买事实、待确认项。 | 需客户同意 + 员工关系 + 用途 + 字段 ACL；未通过时不检索。 |
| 销售记忆 | 销售趋势、活动复盘、机会或门店表现问题。 | 授权销售快照、已审核复盘、趋势和时间窗。 | 不以个人客户明细替代聚合分析；标明 `as_of`。 |
| 商品记忆 | 产品特征、兼容性、历史商品问题。 | 产品事实、已审核商品经验、版本/生命周期引用。 | 以 Product Intelligence 权威事实为准；过期资料只能作为历史线索。 |

路由过程：`意图/实体识别 → 候选域 → Scope 与 ACL 过滤 → 鲜度/保留/冲突检查 → 最小片段检索 → Citation 候选`。多域结果必须保留域别与来源；无授权域或证据不足时返回“无足够授权记忆，请人工核实”，不能用未授权记忆补全答案。

## 6. Knowledge Router

Knowledge Router 自动在可见知识集合中选择下列知识域，并为每个片段附带版本、发布状态、有效期、许可和 Citation 引用。

| 知识域 | 自动调用条件 | 典型用途 | 路由限制 |
| --- | --- | --- | --- |
| 品牌知识 | 品牌定位、活动、规范、产品线或品牌策略问题。 | 解释已发布品牌定位、授权策略和资产规则。 | 仅已发布且当前有效的授权品牌资料。 |
| 产品知识 | 产品功能、适配、兼容、SKU、材料或使用限制问题。 | 产品比较、场景匹配、资料缺口说明。 | 不能将知识当作实时库存、价格或可售承诺。 |
| 销售知识 | 销售辅助、需求发现、组合、异议处理或销售流程问题。 | 生成可编辑的人工核实问题和建议。 | 不生成自动营销、成交或价格承诺。 |
| 路线知识 | 户外路线、活动场景、准备与风险提示问题。 | 提供已审核准备信息、必问项和升级规则。 | 不替代天气、医疗、救援、许可或现场安全判断。 |

检索只面向 `published`、`effective`、未撤销、许可证允许且 ACL 可见的版本。多个版本冲突时，按权威性、适用组织和生效时间选择，并在回答中提示冲突或未知；草案、用户反馈和模型生成摘要不得作为权威知识直接引用。

## 7. Tool Router

Tool Router 是唯一允许 Runtime 调用 Business Brain 能力的入口。它根据 `tool_scope`、请求用途、工具风险级别、资源参数和配额创建声明式工具计划；工具返回的内容必须再次经过字段过滤和 Citation 检查。

| 工具类别 | 状态 | 允许示例 | 约束 |
| --- | --- | --- | --- |
| 查询 | 允许 | 检索已授权知识、读取商品/零售/客户最小投影、读取审计。 | 只读、参数为授权范围子集、分页/速率限制。 |
| 分析 | 允许 | 对授权销售/库存快照做趋势、异常、覆盖和对比分析。 | 不修改源数据；显示时间窗、基线、假设和不确定性。 |
| 建议 | 允许 | 生成核实问题、方案候选、风险提示和人工待办草案。 | 固定 `advisory_only_requires_human_approval=true`，不得触发外部动作。 |
| 交易执行 | 禁止 | 创建/修改订单、报价、调拨、库存预留、审批。 | Runtime 不注册此类工具，策略层与网络/凭证层双重拒绝。 |
| 自动采购 | 禁止 | 自动补货、采购建议单/采购单创建、供应商下单。 | 无写入凭证、无工作流触发器。 |
| 自动下单 | 禁止 | 电商/门店订单、支付、配送、客户通知。 | 不得构造可执行指令或调用外部执行 API。 |

每次工具调用记录 `tool_call_id`、工具/版本、许可决定、经过掩码的参数摘要、开始/结束时间、结果摘要、数据来源引用、错误代码和关联 `audit_id`。工具不可用、超时、返回越范围字段或来源无法引用时，Runtime 必须失败关闭或降级为 `needs_verification`。

## 8. Citation Engine

所有 AI 输出必须具有 Citation 包络。对于每条事实或建议，Citation Engine 将可追溯的证据与生成主张连接，并检查来源在请求时仍可见、有效且足以支撑主张。

| 必填字段 | 含义 |
| --- | --- |
| `source` | 来源系统/知识域及受控资源引用；不暴露用户无权看到的内部路径或 PII。 |
| `version` | 知识、规则、数据集、工具或模型的可追溯版本。 |
| `timestamp` | 数据观测、知识生效或检索的 RFC 3339 UTC 时间；语义必须由 `timestamp_type` 指明。 |
| `confidence` | `low`、`medium`、`high` 及原因；反映证据充分性、鲜度、冲突和适用性，不代表业务保证。 |

建议 Citation 对象另含 `citation_id`、`source_type`、`as_of`、`freshness_status`、`scope`、`claim_refs` 与可选 `limitations`。回答固定区分 `facts`、`analysis`、`assumptions`、`unknowns`、`recommendations`：事实和可验证分析必须有 Citation；无证据的内容仅能作为明确标注的假设或待核实项。Citation 缺失、过期、被撤销、范围不匹配或相互冲突时，系统移除不受支持主张并返回受控限制说明。

## 9. Audit Engine

Audit Engine 记录运行时可回放事件，以支持安全调查、质量评估、合规审查和反馈处理；它不是面向普通用户的原始提示词/PII 数据仓库。

| 记录类别 | 最小记录内容 |
| --- | --- |
| 用户与身份 | `audit_id`、主体受控引用/哈希、`ai_employee_id`、组织引用、角色快照、用途、策略/身份快照。 |
| 问题与请求 | 规范化问题或受控摘要、请求时间、关联 ID、客户端声明的收窄筛选、拒绝/掩码结果。 |
| 路由与数据来源 | Memory/Knowledge 域选择、工具/版本、授权决定、来源引用、版本、`as_of`、鲜度和 Citation ID。 |
| AI 回答与建议 | 回答受控存档/摘要、模型/提示策略版本、事实/建议分区、置信度、限制、`advisory_only` 标志。 |
| 反馈与后续审核 | 反馈 ID、类型、最小化内容、脱敏状态、审核状态、审批人引用、发布/拒绝结果。 |

审计事件以追加式、时间排序和完整性保护方式保存；访问、导出、更正或删除请求同样审计，并遵守数据分类、保留期限、法务保全和客户同意撤回规则。审计查询只返回调用者有权查看的最小字段，默认隐藏原始客户标识、敏感文本和安全策略细节。

## 10. Feedback Engine

```text
用户反馈（评分 / 引用问题 / 采纳、编辑、拒绝 / 最小化说明）
                              ↓
脱敏、恶意内容检测、去重与证据关联
                              ↓
审核（业务负责人 + 知识/合规审核）
                              ↓
知识优化候选（版本化 Draft） → 批准发布 / 拒绝 / 回滚
```

| 阶段 | 输入 / 输出 | 强制控制 |
| --- | --- | --- |
| 接收 | 关联 `audit_id`/`citation_id` 的结构化反馈与可选最小说明。 | 不接受权限变更、执行指令或未经验证的事实作为直接更新。 |
| 整理 | 脱敏、敏感内容扫描、去重、聚类、问题分类和证据缺口标注。 | AI 摘要只是 Draft，不得写成事实或覆盖原始审计。 |
| 审核 | 业务、知识与合规责任人评估证据、范围、偏差、版权与安全影响。 | 职责分离；状态为 `submitted → triaged → draft → in_review → approved|rejected`。 |
| 知识优化 | 形成版本化知识/规则候选、评估记录和发布计划。 | 仅 `approved + published + effective` 的版本可影响未来检索；支持撤销和回滚。 |

反馈可用于衡量 Citation 覆盖、回答有用性、拒绝质量、过期知识和待补证据，但不得自动改变模型权重、权限、员工绩效、客户画像、业务事实或任何交易/经营系统。

## 11. Runtime API Contract

**共同约束：** JSON；RFC 3339 UTC；`Authorization`、`X-Correlation-Id`、`X-Purpose` 必填。组织、AI Identity、角色、权限、范围和工具能力均由服务端派生。客户端不得提交 `ai_employee_id`、角色、`permission_scope`、任意资源白名单、原始客户 PII 或执行参数。所有成功回答包含 `policy_snapshot`、`audit_id`、`as_of`、`citations` 与 `advisory_only_requires_human_approval: true`。

### `POST /api/runtime/query`

在授权范围内进行只读问答、分析和建议。`filters` 与 `domains` 只能缩小服务器已派生范围；不允许请求执行型 action。

```json
// request
{
  "question": "本店本周徒步鞋销售变化有哪些需要人工核实的原因？",
  "domains": ["sales", "product", "brand"],
  "filters": {"store_id": "store_sz_01", "time_range": {"from": "2026-07-15", "to": "2026-07-21"}},
  "response_format": "structured"
}
```

```json
// response
{
  "audit_id": "aud_01J...",
  "answer": {
    "facts": [{"text": "授权门店快照显示徒步鞋销量较上一可比窗口变化。", "citation_ids": ["cit_01"]}],
    "analysis": [{"text": "促销与缺货状态需要一并核实，不能据此认定单一原因。", "citation_ids": ["cit_01", "cit_02"]}],
    "assumptions": [],
    "unknowns": ["竞争活动与现场陈列状态未纳入当前授权快照"],
    "recommendations": [{"text": "由店长人工核实库存、促销和陈列记录。", "type": "human_verification", "confidence": "medium"}]
  },
  "citations": [{"citation_id": "cit_01", "source": "Retail Brain:sales_snapshot", "version": "retail_v42", "timestamp": "2026-07-21T23:00:00Z", "timestamp_type": "as_of", "confidence": "high"}],
  "policy_snapshot": "pol_01J...",
  "as_of": "2026-07-21T23:00:00Z",
  "advisory_only_requires_human_approval": true
}
```

### `GET /api/runtime/context`

返回当前用户可见的**摘要化**运行时上下文，供 Workspace 解释可用能力与限制；不返回原始 ACL、隐含资源清单、内部策略规则或其他主体数据。

查询参数：`include=identity|scopes|capabilities|freshness`（默认最小集合）。响应包括 `ai_employee_id`、有效角色、组织引用、允许域、聚合粒度、允许的只读能力、禁止执行能力、`policy_snapshot` 和上下文过期时间。

### `GET /api/runtime/audit`

读取调用者有权查看的审计摘要。查询参数：`from?`、`to?`、`audit_id?`、`status?`、`limit=1..100`、`cursor?`。普通用户仅能读取自己的事件；管理/合规角色须同时通过专门 ACL 和数据范围。响应使用受控问题/回答摘要、路由、Citation、反馈状态和时间，不默认返回原始敏感内容。

### `POST /api/runtime/feedback`

提交与一次 Runtime 回答、建议或 Citation 关联的反馈；反馈创建审核候选而非直接知识更新。

```json
// request
{
  "audit_id": "aud_01J...",
  "citation_id": "cit_01",
  "feedback_type": "citation_issue",
  "rating": "not_helpful",
  "comment": "数据窗口需要人工确认"
}
```

```json
// response
{
  "feedback_id": "fb_01J...",
  "status": "submitted",
  "audit_id": "aud_01J...",
  "next_step": "triage",
  "knowledge_updated": false
}
```

### 11.1 错误与安全响应

| 情况 | HTTP / 代码 | 安全行为 |
| --- | --- | --- |
| 无认证或会话失效 | `401` / `unauthenticated` | 不暴露租户、角色或资源信息。 |
| 无权限、范围为空或用途不允许 | `403` / `access_denied` | 使用统一消息，不确认资源是否存在。 |
| 请求执行、交易或写入 | `403` / `execution_not_supported` | 拒绝且审计；不生成可执行调用。 |
| 无足够授权证据 | `200` / `needs_verification` 或 `422` / `insufficient_evidence` | 仅返回允许的限制说明和人工核实建议。 |
| 过期/冲突来源 | `200` / `needs_verification` | 明示鲜度或冲突，不给出确定性结论。 |

## 12. 安全边界与验收

### 12.1 不可绕过的安全边界

1. Runtime 和 Business Brain 使用只读身份、网络路径和凭证；写入型系统端点不在工具注册表中。
2. 模型输出不是授权依据。所有数据检索、工具调用和最终响应均由确定性策略层门控。
3. 不信任用户输入、检索文本和工具输出中的指令；它们不能改变 Scope、调用未注册工具或解除禁止动作。
4. 客户数据不因聚合、角色名称或模型上下文而自动可见；同意撤回与 ACL 变化必须立即影响后续路由。
5. 安全/权限、Citation、工具执行拒绝和反馈发布均须可审计；审计本身受最小权限与保留策略约束。

### 12.2 验收清单

- [x] Runtime Architecture：定义 Human User → AI Workspace → AI Runtime → Business Brains 及八个运行时组件。
- [x] Identity Model：定义服务端派生的 `ai_employee_id`、角色、组织及四类 Scope。
- [x] Permission Engine：定义 RBAC + ACL + Data Scope、五类角色及拒绝优先策略。
- [x] Memory Router：定义企业、客户、销售、商品记忆的意图路由与授权过滤。
- [x] Knowledge Router：定义品牌、产品、销售、路线知识的自动调用与版本门控。
- [x] Tool Router：仅允许查询、分析、建议；明确禁止交易执行、自动采购和自动下单。
- [x] Citation Engine：所有输出包含 `source`、`version`、`timestamp`、`confidence`。
- [x] Audit Engine：记录用户、问题、AI 回答、数据来源、建议和反馈。
- [x] Feedback Engine：定义用户反馈 → 审核 → 知识优化的非自动闭环。
- [x] API Contract：定义四个 Runtime API、共同安全约束、样例与错误契约。

本设计的验收标准是：任何实现必须保持只读、建议性、人类审批和服务端范围派生这四项不变量；不满足任一不变量时不得进入部署或生产接入阶段。
