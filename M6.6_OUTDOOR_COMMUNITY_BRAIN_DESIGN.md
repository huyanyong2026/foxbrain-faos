# VAFOX M6.6 Outdoor Community Brain Design

**状态：** Design Phase / 仅设计，不部署。  
**前置设计：** `M6.5_CUSTOMER_BRAIN_ONLINE_DESIGN.md`、`DATA003.4_GROWTH_LOOP_DESIGN.md`、`DATA-004 Outdoor Community`、`DATA-005 Brand Partner`、`M5 Digital Employee System`。  
**负责人：** Codex（设计开发）；WorkBuddy（环境）；Marvis（管理）；CTO（验收）。

> Outdoor Community Brain 是 Customer Brain 的受治理社区能力层：在客户明确授权、活动与社群可见、且员工关系合法的前提下，提供户外身份、社群/活动候选、内容贡献和成长闭环的**只读查询与待审核提交**。它不替代 Customer Brain、活动系统、Growth Loop、品牌伙伴系统或 SAP/Core 的权威事实；不部署服务、不导入数据，也不改变生产同步。

## 1. 目标、原则与硬边界

### 1.1 目标

1. 以 `organization_id + customer_id` 为边界，将 Customer Brain 中获授权的户外画像转化为可解释的 Outdoor Identity。
2. 支持徒步、登山、露营、滑雪和高海拔场景下的城市、兴趣、经验、路线、活动与社群候选匹配。
3. 连接火狐狸、品牌、俱乐部和目的地活动，并向客户与授权员工提供人工确认前的建议。
4. 连接 DATA-003.4 的积分、等级和邀请事实，形成可回放、反作弊、可撤回的成长闭环。
5. 支持客户自愿提交照片、视频、路线和装备体验，并按平台、可见性、版权、肖像和安全规则治理。

### 1.2 设计原则

- **身份、用途、同意优先：** 服务端先解析认证主体、组织、角色、客户关系、目的、Consent 与字段 ACL，再检索或推荐；URL 中的 ID 不能授权访问。
- **事实不搬家：** Customer Brain 仍拥有客户受治理投影；活动/俱乐部拥有活动与成员事实；Growth Loop 拥有不可变积分账本和等级规则；DATA-005 拥有品牌伙伴关系；SAP/Core 保持只读事实源。
- **安全不推断：** 经验、装备覆盖和路线匹配不构成健康、天气、领队资格、雪况、高反适应或现场安全结论。未知一律返回 `needs_verification`。
- **建议与执行分离：** 匹配、排序和员工下一动作均为 `advisory_only_requires_human_approval`；系统不自动报名、入会、触达、奖励、交易或写回主数据。
- **最小化、可撤回、可审计：** 仅输出完成当前用途所需字段；记录来源、版本、`as_of`、策略决定与审计引用；撤回同意后立即停止后续个性化处理与公开展示。

### 1.3 明确禁止

本设计禁止：

- 未授权营销、批量触达、自动私信、自动社交传播或规避退订/频控；
- 自动报名、自动入群、自动确认活动资格、安全能力或领队资格；
- 自动交易、报价、加购、锁货、库存预留、下单、支付、发券或兑现权益；
- 修改 SAP、SAP 主数据/交易、Core 生产同步或其同步规则；
- 以购买金额、社交互动或模型猜测代替户外安全能力、客户同意或人工审核。

## 2. Community Brain Architecture

```text
Customer Brain
      ↓  identity + consent + minimum outdoor profile
Outdoor Identity
      ↓  city + interests + verified experience + constraints
Community Matching
      ↓  route / club / activity candidates
Activity Engine
      ↓  participation facts and verification state
Club Network
      ↓  governed community relationship
Content Sharing
      ↓  moderated voluntary contribution
Growth Points
      ↓  DATA-003.4 ledger and Lv1–Lv5 projection
Referral Loop
      └──────────────────────────────→ New Customer / Customer Brain
```

| 层 | 核心职责 | 权威来源 | 输出与控制 |
| --- | --- | --- | --- |
| Customer Brain | 身份、Consent、购买/装备与客户关系的最小投影。 | M6.5 / DATA003.1。 | 只读、按目的和字段 ACL 过滤。 |
| Outdoor Identity | 归一化户外身份与证据质量。 | 客户声明、核验活动、获权装备事实。 | 画像候选不能静默回写 Customer Brain。 |
| Community Matching | 在城市、兴趣、经验和路线约束下生成社群候选。 | Outdoor Identity + Club Network。 | 不公开其他成员，不自动入群。 |
| Activity Engine | 已发布活动、报名/签到/完成状态与资格核验工作流。 | 活动事实源 / DATA-004。 | 不以推荐结果替代容量、天气或领队决定。 |
| Club Network | 俱乐部伙伴、路线专长、认证和状态目录。 | 俱乐部伙伴主数据。 | 认证过期/暂停的俱乐部不可被推荐。 |
| Content Sharing | 客户主动内容的审核、可见性和渠道投影。 | 客户提交、平台授权、审核记录。 | 不代发、不抓取私密内容。 |
| Growth Points | 积分、等级、贡献与奖励建议。 | DATA003.4 Growth Loop。 | 只读账本投影；奖励必须审核。 |
| Referral Loop | 自愿分享、归因、反作弊、奖励建议。 | Growth Loop + Identity。 | 不自动邀请或自动入账。 |

**统一处理链：** `认证 → 服务端组织/关系范围 → RBAC + ACL + Consent → 最小投影 → 匹配/推荐 → 输出复检 → 追加式审计`。任一条件拒绝、过期或依赖不可用时不得降级为未授权读取；返回受控拒绝或 `needs_verification`。

## 3. Outdoor Identity Model

### 3.1 实体定义

`OutdoorIdentity` 是客户可解释户外画像投影，不是医疗、安全资质或活动资格的权威记录。每一个属性必须带有 `source_type`、`source_ref`、`observed_at`、`quality_state`、`allowed_purposes`、`expires_at` 和 `profile_version`；缺少证据或有效授权时省略字段而非猜测填充。

| 字段 | 类型/受控值 | 定义与控制 |
| --- | --- | --- |
| `customer_id` | opaque string，必填 | 复用 Customer Brain 不可变客户 ID；不得使用手机号、会员号或社交账号。 |
| `city` | city reference，选填 | 客户声明或获授权事实的常用活动城市；订单地址不得推定为常住地。 |
| `experience_level` | `newcomer/casual/regular/advanced/unknown` | 仅描述已验证的户外经验；不等于安全资格。 |
| `activity_preference` | 受控标签数组 | `hiking`、`mountaineering`、`camping`、`skiing`、`high_altitude`；来自客户选择或允许的证据。 |
| `favorite_destination` | destination reference[] | 明确收藏、完成或客户声明的目的地；精确轨迹默认不返回。 |
| `equipment_level` | `entry/developing/complete_for_declared_scenario/unknown` | 已知装备覆盖程度；不推断磨损、适配性或实时可用性。 |
| `interest_tags` | 受控标签数组 | 如 `weekend_hike`、`family_camping`、`snow_travel`；标记来源及置信度。 |

```json
{
  "customer_id": "cst_01J...",
  "city": {"value": "成都", "quality_state": "customer_confirmed"},
  "experience_level": {"value": "regular", "source_ref": "journey_...", "quality_state": "verified"},
  "activity_preference": ["hiking", "camping"],
  "favorite_destination": [{"destination_id": "dst_siguniang", "relationship": "saved"}],
  "equipment_level": {"value": "developing", "verification_state": "partial"},
  "interest_tags": ["weekend_hike", "photography"],
  "policy": {"purpose": "member_service", "decision": "allow_minimum"},
  "as_of": "2026-07-22T00:00:00Z"
}
```

### 3.2 维护和更正

- 客户可提交城市、兴趣、目的地或装备声明的**待审核候选**；员工只能在获授权服务关系内协助提交，不能直接覆盖事实。
- 已核验活动完成可更新经验候选；取消、撤销、争议或来源过期时，以新版本/失效状态处理，不静默改写历史。
- 高海拔、滑雪、登山等标签只能表达偏好或已验证经历，不能推断身体状况、技能证书或允许参与特定活动。

## 4. Community Matching Engine

```text
用户
  ↓  identity / purpose / consent gate
城市
  ↓  visible city scope
兴趣
  ↓  hiking | mountaineering | camping | skiing | high_altitude
经验
  ↓  verified level + activity-specific requirements
路线
  ↓  published, visible, seasonally applicable route candidates
社群
```

### 4.1 匹配输入与规则

| 阶段 | 输入 | 处理 | 输出/禁止事项 |
| --- | --- | --- | --- |
| 用户 | 当前认证客户或获批员工委托。 | 强制组织、客户关系和用途检查。 | 不向无权主体暴露客户或社群存在性。 |
| 城市 | 客户确认城市、本次手选城市或匿名公开目录范围。 | 先应用社群/活动地域 ACL。 | 城市未知时询问或给公开目录，不能猜测。 |
| 兴趣 | `activity_preference`、`interest_tags`、本次选择。 | 映射五类受控活动标签。 | 不以消费或社交关系推断兴趣。 |
| 经验 | 经核验 `experience_level`、活动特定门槛。 | 仅筛选/标记，不能自动判定资格。 | 健康、天气、技能、名额均待确认。 |
| 路线 | 已发布、可见、适季的路线及难度版本。 | 路线与目的地、活动、俱乐部交叉关联。 | 不公开限制路线、精确坐标或安全运营细节。 |
| 社群 | 有效状态的俱乐部/社群。 | 返回可解释候选、差距和核验问题。 | 不返回其他成员名单/联系方式，不自动入群。 |

### 4.2 输出对象和排序

`CommunityMatch` 至少包括：`match_id`、`customer_id`、`community_ref`、`city`、`supported_activities`、`route_refs`、`fit_score_band`、`fit_reasons[]`、`gaps[]`、`eligibility_status`、`verification_questions[]`、`rule_version`、`as_of` 与 `policy`。

排序只可使用当前用途获准的城市、兴趣、客户主动选择、已验证经验、公开活动可见性与俱乐部有效状态。`fit_score_band` 仅为 `high/medium/low/insufficient_evidence`，不得作为自动准入或营销分群。状态为：

- `potential_fit`：基础匹配，仍需活动/俱乐部确认；
- `needs_verification`：经验、装备、季节、名额、领队或安全条件未确认；
- `not_matched`：在当前已授权信息中没有候选；
- `not_eligible`：已发布的客观规则不满足；应提供最小解释，避免泄露内部策略。

## 5. Activity Engine

### 5.1 活动实体

| 字段 | 类型/约束 | 说明 |
| --- | --- | --- |
| `activity_id` | opaque string，必填 | 活动系统的唯一引用。 |
| `title` | string，必填 | 已发布标题；草稿或受限活动不对无权主体返回。 |
| `destination` | destination reference，必填 | 目的地公开投影，精确集合点可按权限延后披露。 |
| `difficulty` | `easy/moderate/hard/expert/unknown` | 版本化活动难度，不等于安全许可。 |
| `season` | season/date-window | 适用季节和报名/发生时间窗；实时天气另行核验。 |
| `organizer` | organization reference | 火狐狸、品牌、俱乐部或目的地组织方。 |
| `brand_partner` | partner reference，可选 | DATA-005 已批准品牌伙伴引用。 |
| `equipment_requirement` | requirement[] | 已发布的装备/安全要求；仅用于提示和人工核验。 |
| `participants` | aggregate + membership state | 默认只显示名额/候补摘要和当前客户状态，绝不公开名单。 |

### 5.2 活动类型与生命周期

支持以下来源：**火狐狸活动、品牌活动、俱乐部活动、目的地活动**。所有活动遵循：`draft → submitted_for_review → published → registration_open → closed → in_progress → completed`，可转为 `cancelled/postponed/archived`。仅 `published` 且在可见范围、季节和组织状态有效的活动可以进入匹配或推荐。

`POST /api/activity/join` 只创建报名申请或候补申请：`requested → eligibility_review → approved|waitlisted|rejected|cancelled`。它不自动扣积分、预留库存、确认资格或向外部系统写入交易。活动完成、签到和领队确认是事实源事件；只有可信完成事件可被 Growth Loop 消费。

## 6. Club Network

### 6.1 俱乐部伙伴模型

| 字段 | 类型/约束 | 说明 |
| --- | --- | --- |
| `club_id` | opaque string，必填 | 俱乐部唯一引用。 |
| `location` | city/region reference | 服务城市或区域；精确集合地受 ACL 控制。 |
| `specialty` | controlled tags[] | 徒步、登山、露营、滑雪、高海拔等已审核专长。 |
| `leader` | leader reference | 仅返回获授权的公开资料；个人联系信息默认不返回。 |
| `routes` | route reference[] | 已发布、审核和可见的路线引用与版本。 |
| `certification` | certification summary | 认证类型、签发方、状态、到期时间；证据文件受限。 |
| `status` | `pending/active/suspended/expired/retired` | 非 `active` 俱乐部不可被自动推荐。 |

### 6.2 治理

俱乐部认证、领队资质、路线安全和活动责任由相应审核工作流/权威伙伴记录决定。Community Brain 只读取其最小状态投影；认证到期、暂停、投诉或路线下线应使后续匹配立即失效。客户社群关系应保存 `membership_status`（`none/requested/active/paused/left`）和来源，但不构造社交图谱供任意员工浏览。

## 7. Activity Recommendation

```text
客户
  ↓  consented Outdoor Identity
兴趣
  ↓
装备
  ↓  verified coverage + declared gaps
历史活动
  ↓  verified participation, not private location history
活动推荐
```

推荐候选先经活动发布状态、组织/城市可见性、活动类型、季节和活动规则过滤，再以兴趣、已核验装备覆盖、已完成活动与客户本次选择排序。每一候选必须返回：`activity_id`、`fit_reasons[]`、`equipment_gap[]`、`experience_gap[]`、`verification_questions[]`、`citation_refs[]`、`confidence`、`as_of` 和 `advisory_only_requires_human_approval: true`。

- 装备“足够”只可表示针对已声明场景的已知覆盖，活动要求、磨损、尺码、兼容性、天气和安全装备必须再次核验。
- 历史活动只有 `verified` 完成状态可影响排序；客户自报、已取消报名或未审核媒体不可作为经验事实。
- 推荐不自动报名、不自动推荐商品、不发送通知；客户主动请求之外的个性化通知需要独立营销/通知 Consent。

## 8. Content Sharing

### 8.1 用户内容模型

客户可主动提交以下内容：**照片、视频、路线、装备体验、分享渠道**。`CommunityContent` 包括 `content_id`、`customer_id`、`activity_id`（可选）、`content_type`、`media_refs[]`、`route_ref`、`equipment_experience`、`channels[]`、`visibility`、`consent_snapshot`、`moderation_state`、`reward_ref`、`created_at`、`audit_ref`。

支持渠道：**视频号、小红书、公众号、社群**，枚举为 `video_account`、`xiaohongshu`、`official_account`、`community_group`。渠道连接或发布证据只能在客户明确授权、平台许可与可见性允许时处理；不读取私密内容、不保存访问令牌到日志、不代客户发布、不自动转发或私信。

### 8.2 审核、可见性和撤回

内容生命周期：`draft → submitted → moderation_pending → approved|restricted|rejected → published_or_private → withdrawn`。审核至少包括重复/刷量、版权、肖像/同行人同意、隐私、危险路线坐标、户外安全误导和平台规则。精确位置、原始媒体 URL、同行人信息和未审核装备评价默认不可公开。

客户撤回内容、撤回平台授权或内容下架后，系统停止新的公开分发、匹配用途和奖励计算；已入账奖励根据 DATA-003.4 规则进入待审冲正，审计按保留政策最小保存。

## 9. Growth & Points

Community Brain 连接 `DATA003.4_GROWTH_LOOP_DESIGN.md`，不建立第二个可写积分账本。它只读取积分/等级摘要，并把合格活动、内容和邀请事件提交为带幂等键的待审核建议。

| 积分来源 | 合格事实 | 控制 |
| --- | --- | --- |
| 购买 | SAP/Core 已认证、结算且符合规则的购买事实引用。 | 退款、取消、赠品、重复或归属不明不得自动计分。 |
| 活动 | 报名、签到和组织方/领队确认的完成事实。 | 报名本身、取消或争议状态不计分。 |
| 分享 | 客户主动提交且通过授权、内容和平台审核的内容。 | 禁止用自动营销发送或私密抓取内容计分。 |
| 打卡 | 可验证、频控、反作弊通过的打卡事实。 | 位置最小化；异常进入审核。 |
| 邀请 | 有效归因窗口内、身份解析与反作弊通过的邀请案件。 | 禁止自邀、重复奖励和自动入账。 |
| 贡献 | 经审核的路线、装备体验、领队/社群贡献。 | 版权、安全、利益冲突与质量审核。 |

成长等级为 **Lv1–Lv5**。等级阈值、权益、证据窗口和冲正规则完全由 DATA-003.4 的版本化规则决定；本系统仅显示 `level`、`points_summary`、`progress`、`rule_version`、`review_state` 与 `as_of`。等级不是安全资质、交易折扣或自动带队权。

## 10. Referral Loop

```text
会员
  ↓  voluntary share link/code
分享
  ↓
好友
  ↓  explicit attribution + identity resolution
新会员
  ↓  qualified event and fraud review
积分奖励
```

`ReferralCase` 应包含 `referral_id`、`referrer_customer_id`、`invitee_customer_id`、`referral_code_id`、`shared_at`、`attribution_evidence`、`qualified_event_ref`、`fraud_flags`、`reward_rule_version`、`state`、`review_ref` 与 `audit_ref`。状态为 `shared → attributed → new_member → qualified → reward_pending_review → rewarded`，或 `rejected/reversed/expired`。

分享必须由会员主动发起且可撤销；好友必须在允许的归因窗口内明确接受或使用有效链接。系统应检测自邀、重复归因、异常批量、员工利益冲突、退款/撤销和高风险身份信号；高风险仅进入审核队列，不自动定罪，也不向普通运营人员泄露敏感信号。不得因客户拒绝营销而剥夺其已合法获得的服务性权益。

## 11. Brand Partner Integration

DATA-005 Brand Partner 是品牌伙伴身份、合作状态、授权范围和合作条款的权威来源。Outdoor Community Brain 只接收处于允许合作状态的 `brand_partner_ref` 和活动/体验资格投影。

| 品牌能力 | 输入 | Community Brain 输出 | 必须边界 |
| --- | --- | --- | --- |
| 发布活动 | 已审核品牌、活动草稿/发布事实、可见范围。 | 品牌活动目录、客户候选和审核申请。 | 品牌不能读取无关客户资料或自动报名。 |
| 装备体验 | 产品/体验计划、资格条件、客户明确兴趣。 | 装备体验候选、已确认体验内容引用。 | 不自动发样、下单、调库存或将购买历史用于未授权营销。 |
| 会员活动 | 合作权益、名额、有效期与规则版本。 | 资格建议、候补/报名申请草稿。 | 权益确认、资格和奖励均须权威系统/人工审核。 |

品牌伙伴仅可在合同、组织隔离、目的和 Consent 的交集内看到去标识化聚合数据或经批准的最小客户服务上下文；不得导出成员名单、个人联系方式、精确路线或原始媒体。

## 12. Employee Integration

M5 Digital Employee System 经 M6.5 Customer Brain 的服务端策略门控访问 Community Brain；员工不得凭客户 ID 直接绕过 Customer Brain 获取完整社群资料。

| 员工查看/动作 | 可显示 | 必须限制 |
| --- | --- | --- |
| 客户兴趣 | 脱敏客户身份、获授权兴趣/城市/经验摘要和数据时点。 | 无客户关系、目的或 Consent 时不显示；无原始身份标识。 |
| 活动机会 | 可见活动候选、匹配理由、装备/经验缺口与核验项。 | 仅建议；员工不得自动报名或承诺资格。 |
| 社群关系 | 客户本人成员状态、可见俱乐部候选和关系摘要。 | 不显示其他成员、群聊内容或联系方式。 |
| 推荐动作 | 可编辑的服务问题、确认清单、人工待办草案。 | 不自动发送、入群、写画像、发积分或交易。 |

员工所有个人级访问应显示 `policy.decision`、`freshness`、`lineage` 和 `advisory_only_requires_human_approval`。高风险路线、认证争议、内容投诉、奖励反作弊或权限提升必须升级给相应审核人。

## 13. API Contract

**基础约定：** JSON、UTC RFC 3339。服务端从认证上下文派生 `actor_id`、`organization_id`、角色、客户关系与用途；客户端的 ID/范围声明不能扩大权限。所有响应包含 `correlation_id`、`audit_id`、`policy`、`freshness.as_of`、`lineage` 和 `disposition`。写入接口必须携带 `Idempotency-Key`，且仅创建草稿或待审核案件。

| API | 权限/用途 | 请求与响应 | 关键行为 |
| --- | --- | --- | --- |
| `GET /api/community/profile/{id}` | `community:profile:read`；本人自助或获批服务用途。 | 返回最小 Outdoor Identity、本人社群关系、成长摘要和策略元数据。 | 无权/不存在统一为 `403 POLICY_DENIED`；不返回他人数据。 |
| `GET /api/community/match/{id}` | `community:match:read`；本人自助或获批服务/个性化。 | 返回 CommunityMatch 候选、理由、缺口、核验问题和规则版本。 | 无 Consent 时仅可提供客户主动查询的可见公开目录。 |
| `GET /api/activity/recommend/{id}` | `activity:recommend:read`；本人自助或获批服务/个性化。 | 返回 ActivityRecommendation 候选和 `advisory_only` 标记。 | 不承诺天气、资格、名额、装备状态或安全。 |
| `POST /api/activity/join` | `activity:join:request`；客户本人或获权代办。 | 请求含 `activity_id`、`customer_id`、最小确认项；`202` 返回申请/审核 ID。 | 仅创建申请；不自动报名、扣分、付款或写 SAP/Core。 |
| `POST /api/content/share` | `community:content:submit`；内容处理 Consent。 | 请求含内容引用、渠道、可见性、活动/路线引用；`202` 返回审核 ID。 | 需版权/肖像/安全/平台审核；不自动发布。 |
| `POST /api/community/feedback` | `community:feedback:submit`；客户本人或获权员工。 | 提交画像更正、活动/俱乐部/内容反馈；`202` 返回案件 ID。 | 只创建待审核反馈，不直接改写事实、认证或积分。 |

统一错误码：`400 INVALID_REQUEST`、`401 UNAUTHENTICATED`、`403 POLICY_DENIED`、`409 CONTEXT_STALE`、`422 CONSENT_OR_PURPOSE_REQUIRED`、`422 REVIEW_REQUIRED`、`429 RATE_LIMITED`、`503 DEPENDENCY_UNAVAILABLE`。审计记录主体、委托主体、组织、目的、资源类型、策略决定、规则版本、关联 ID 与时间；不得记录原始身份标识、访问令牌、完整媒体或完整响应体。

## 14. Privacy & Permission

### 14.1 RBAC + ACL + Consent

| 控制层 | 规则 |
| --- | --- |
| RBAC | 客户仅查看本人；员工按角色和门店/组织范围；活动审核、内容审核、增长审核和品牌伙伴角色分离。 |
| ACL | 活动、路线、俱乐部、内容和媒体均具有独立可见范围；先 ACL 后检索，避免侧信道泄露。 |
| Consent | 逐目的记录 `member_service`、`community_matching`、`activity_registration`、`content_processing`、`platform_connection`、`personalization` 和 `marketing`；营销 Consent 不得被其他用途隐含替代。 |
| Audit | 所有读取、匹配、推荐、提交、审核、撤回与策略拒绝均为追加式最小审计事件，可按 `correlation_id` 回放。 |

撤回 `community_matching` 或 `personalization` Consent 后，停止新的个性化匹配/推荐；撤回 `content_processing` 或平台授权后，停止处理和公开投影；撤回不会自动删除依法需保留的最小审计与已验证事实，但必须按保留政策限制后续用途。Break-glass 访问需限时审批、明确原因、最小字段和事后复核。

## 15. 非目标与验收

### 15.1 边界确认

本交付为架构和接口设计，**不部署**任何服务、连接器、同步任务、活动报名、内容分发或积分发放。它不修改 SAP、不修改 Core 生产同步，也不具备自动营销、自动社交传播或自动交易能力。

### 15.2 验收清单

- [x] Outdoor Identity：定义了客户、城市、经验、偏好、目的地、装备和兴趣标签及其证据治理。
- [x] Community Matching：定义了“用户→城市→兴趣→经验→路线→社群”流程，并覆盖徒步、登山、露营、滑雪、高海拔。
- [x] Activity Engine：定义了活动实体、四类活动、生命周期和待审核加入流程。
- [x] Club Network：定义了俱乐部伙伴字段、认证状态与关系隐私边界。
- [x] Content Sharing：定义了照片、视频、路线、装备体验与视频号/小红书/公众号/社群渠道治理。
- [x] Growth Points：连接 DATA-003.4，覆盖购买、活动、分享、打卡、邀请、贡献和 Lv1–Lv5。
- [x] Referral Loop：定义会员分享至奖励审核的归因、反作弊和撤回链路。
- [x] Brand Integration：连接 DATA-005 的品牌活动、装备体验和会员活动，并限定数据边界。
- [x] Employee Integration：定义员工兴趣、机会、关系和建议动作视图及限制。
- [x] Privacy Governance：定义 RBAC + ACL、Consent、Audit 和禁止自动化边界。
