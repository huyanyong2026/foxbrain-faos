# VAFOX M4.3.2 Knowledge Asset Center Design

**状态：** Design Phase / 仅设计，不部署。  
**前置设计：** `M4_KNOWLEDGE_DOMAIN_MODEL_REPORT.md`（M4.1）、`M4_BRAND_KNOWLEDGE_REPORT.md`（M4.2）、`M4.3_PRODUCT_KNOWLEDGE_DESIGN.md`（M4.3）。  
**目标：** 建立统一的 Knowledge Asset Center（KAC），使品牌资料、采购资料、员工经验、外部知识和户外场景资料以可追溯、分级、可审核、受权限约束的方式进入 VAFOX Knowledge Brain。

## 1. 原则、范围与边界

1. **资产先治理、知识后发布。** 文件或网页采集物是 Asset；经解析、引用、分级与审核后才可形成可检索 Knowledge Entity。Asset 不等于已批准事实。
2. **证据和版本不可丢失。** 每个事实、摘要和关系都必须能回到版本化的原始资产、内容哈希和精确 locator；AI 生成内容保留其输入血缘。
3. **来源等级随回答传递。** AI 只能基于调用者可访问的资产回答，并逐项返回 `source_rank`、Citation、版本、生命周期和警告；不得把 C 级整理内容表述为品牌官方事实。
4. **授权先于解析结果可见性。** 解析文本、缩略图、chunk、Citation、搜索命中和知识实体都继承资产 ACL；标签和搜索条件只能缩小结果，不能扩大权限。
5. **人工决定发布。** AI 可以提取、标记、去重建议和生成草稿，不能自动发布、提升来源等级、修改 ACL 或将经验写成官方结论。
6. **与主数据和执行系统解耦。** KAC 仅保存知识引用和只读业务 ID；不成为 SAP/Core 主数据源或同步改写方。

### 明确禁止

本设计不修改 SAP，不修改 Core 生产同步，不创建 SAP/Core 写入路径；不进行自动采购、自动销售、下单、报价、改价、库存预留、客户触达或其他业务执行。产品推荐仅为带来源的人工决策支持，实际商品、价格、库存和交易状态仍须经既有只读事实源与人工确认。

## 2. Knowledge Asset Center Architecture

```text
多角色资料入口 / 受控采集
        │ Asset Upload
        ▼
文件安全、格式和 ACL 校验 → 隔离区（失败/不可审查/越权）
        ▼
AI Extraction（文本、表格、图片 OCR、结构、候选实体与主张）
        ▼
Metadata（身份、范围、版本、哈希、标签、权限、血缘）
        ▼
Citation（页/段/行/单元格/区域/chunk 精确定位）
        ▼
Source Ranking（S / A / B / C，及其判定理由）
        ▼
Human Review（事实、风险、权限、版本、等级、冲突审核）
        ▼
Knowledge Base（仅 Published、有效、ACL 可见的实体与索引）
        ▼
VAFOX Knowledge Brain（检索、问答、产品智能；只读、带 Citation）
```

| 层 | 责任 | 输出 | 不允许的行为 |
| --- | --- | --- | --- |
| Asset Upload | 接收资料、验证身份和文件、记录原件 | `asset_id`、原件、初始 ACL、上传审计 | 用客户端字段授予更高权限。 |
| AI Extraction | 抽取内容和候选结构 | chunk、OCR、候选 Metadata/Tags/Entity/Citation | 直接发布或改写源文件。 |
| Metadata & Citation | 固化检索、版本和证据契约 | 受控元数据、hash、locator、lineage | 缺少定位时制造引用。 |
| Source Ranking | 依据来源身份和验证状态分级 | `source_rank`、理由、审批人 | 仅因文本质量提高等级。 |
| Human Review | 校验事实、冲突、权限和适用边界 | review 决议、状态转换 | 批准不可访问或不可引用资料。 |
| Knowledge Base | 提供已发布资产的受控检索 | Knowledge Entity、索引、响应 Citation | 返回 draft、archived 或越权内容。 |

## 3. 多角色资料入口模型

所有入口使用服务端认证的 `AuthContext` 生成 `organization_id`、角色和可授予的最大 ACL。上传者提交的品牌、供应商、可见性或等级均为待校验声明，不是授权依据。

### 3.1 Brand Partner（品牌商）

必填字段：`brand_id`、`supplier_id`、`asset_type`、`version`、`upload_time`。系统另生成 `asset_id`、`uploader_id`、`content_sha256`、`source_rank=S`（待品牌身份/版本核验）、`status=Draft`。

| 支持资料 | `asset_type` 建议值 | 最低审核要求 |
| --- | --- | --- |
| 产品手册 | `product_manual` | 品牌与版本、产品范围、页级 Citation。 |
| 培训资料 | `brand_training` | 适用角色、有效期、事实/话术区分。 |
| 技术资料 | `technical_document` | 技术主张、限制、测试/标准定位。 |
| 陈列资料 | `merchandising_guide` | 渠道/门店适用范围与版本。 |
| 活动资料 | `brand_activity_material` | 活动日期、地区、状态；计划与结果分离。 |

`brand_id` 和 `supplier_id` 必须映射到已授权的合作方目录或保持 `unresolved`，后者不能 Published。品牌官方身份无法验证时，等级降为待审而非自动 S 级。

### 3.2 Procurement（采购）

| 支持资料 | `asset_type` | 默认候选等级 | 审核要点 |
| --- | --- | --- | --- |
| 订货资料 | `procurement_order_material` | A | 只作为采购证据；不可推导实时可售或自动采购。 |
| 商品规划 | `merchandise_plan` | A | 规划与已经执行的事实分开。 |
| 竞品分析 | `competitive_analysis` | A | 每项比较必须保留外部来源与观察日期。 |
| 供应商资料 | `supplier_profile` | A | 验证供应商、版本和授权范围。 |
| 市场判断 | `market_assessment` | A | 标明作者、方法、时效和不确定性。 |

### 3.3 Operation / Training（运营 / 培训）

支持 `sales_talk_track`（销售话术）、`service_process`（服务流程）、`store_standard`（门店标准）、`training_material`（培训资料）。这些资料默认进入 A 级候选池；其中经验或话术不是产品技术事实，必须明确 `claim_kind=practice|opinion|procedure`，并由相应业务所有者审核后发布。

### 3.4 External Knowledge（外部知识）

支持 `official_publication`（官方公开资料）、`public_technical_reference`（公开技术资料）、`route_guide`（路线攻略）、`outdoor_media`（户外媒体）、`community_experience`（社区经验）。外部来源须记录 URL/采集时间/版权和使用许可；网页保存版本化快照或可验证归档 locator。官方公开资料可在身份与版本核验后为 S；公开技术资料、路线攻略和媒体通常为 B，社区经验为 B。无可验证来源的 AI 摘要仅为 C。

## 4. Source Ranking 与回答契约

| 等级 | 名称 | 合格来源 | 使用规则 |
| --- | --- | --- | --- |
| S | 品牌官方资料 | 已验证品牌方提交的原厂资料，或经验证的官方公开资料。 | 可支撑品牌/产品官方主张，仍受版本、有效期和 ACL 限制。 |
| A | 采购 / 内部验证资料 | 经授权采购、运营、培训或内部专家审核的资料。 | 可支撑内部流程、判断和已验证观察；不能覆盖更高优先级官方规格。 |
| B | 用户经验和活动内容 | 用户经验、活动复盘、媒体、路线攻略及未验证外部材料。 | 作为情境/经验参考，必须标注主观性、时效和风险，不能作为安全保证。 |
| C | AI 整理资料 | AI 提取、总结、翻译、聚合或待复核草稿。 | 永远保留派生标记和全部上游 Citation；不得单独成为高影响结论的唯一依据。 |

冲突处理顺序为 S > A > B > C；同级冲突按版本有效性、审核状态、直接性、时效和证据完整性比较，而不是静默合并。系统保留 `conflicts_with` 关系，向审核者展示，不自动删除较低级来源。

**AI Answer Contract：** 每个结论必须返回 `claim`、`source_rank`、`citation_ids`、`asset_id`、`asset_version`、`knowledge_status` 和 `confidence`。一个答案混合多等级资料时，逐条显示最低支撑等级，不得只展示最高等级。没有可访问且 Published 的证据时，回答“未找到已批准证据”，并可建议提交审核，不得以模型常识补足。

```json
{
  "claim": "该结论仅作为川西徒步经验参考，不构成安全保证。",
  "source_rank": "B",
  "citation_ids": ["cit:asset:route-guide-v2:section-3"],
  "asset_id": "ast:external:route-guide-001",
  "asset_version": "2.0",
  "knowledge_status": "Published",
  "confidence": "medium",
  "limitations": ["路线和天气会变化", "需人工确认个人能力与安全条件"]
}
```

## 5. Knowledge Asset Schema

Asset 是存证和治理单位；Knowledge Entity 是由 Asset 支撑的可检索实体/主张。每个 Entity 都通过 `evidence_refs[]` 指向 Citation，不能只指向文件名。

| 字段 | 规则 |
| --- | --- |
| `asset_id` | 必填、不可变；例如 `ast:<organization>:<uuid>`。 |
| `asset_type` | 必填受控值；见第 3 节。 |
| `brand` | `brand_id` 或 `unresolved`；品牌资料须可验证。 |
| `source` | 必填对象：来源机构/作者、`source_type`、原始 URI/交付渠道、`source_rank` 和核验状态。 |
| `version` | 必填；供应方版本、发布日期或受控修订号，不得用上传时间替代。 |
| `creator` | 必填对象：原作者（若已知）、上传者、创建组织；记录而非赋权。 |
| `permission` | 必填 ACL 引用：组织、所有者、可见性、部门/角色条件、显式 allow/deny。 |
| `citation` | 必填数组；每个可用主张至少一条精确 Citation。 |
| `status` | `Draft`、`Review`、`Published`、`Archived`。 |

建议扩展字段：`title`、`language`、`mime_type`、`content_sha256`、`size_bytes`、`uploaded_at`、`effective_from/to`、`tags`、`parser_version`、`extraction_status`、`review`、`lineage`、`retention_policy`、`legal_basis/license`、`supersedes_asset_id`。

Citation 最小结构为：`citation_id`、`asset_id`、`asset_version`、`content_sha256`、`locator`、`chunk_id`、`claim_path?`、`captured_at`、`extractor_version`。locator 格式示例：PDF `#page=12&chars=120-420`，DOCX `#heading=护理&paragraph=3`，PPT `#slide=7&shape=2`，Excel `#sheet=面料&range=B12:F12`，Markdown `#heading=装备&lines=42-58`，图片 `#image=1&region=x1,y1,x2,y2`。

## 6. AI 解析流程

支持 PDF、DOCX、PPT/PPTX、Excel/XLSX、Markdown 和图片（PNG/JPEG/WebP 等）。原件在隔离扫描通过前不可解析或索引；加密、损坏、恶意、无授权或无法审查的文件维持隔离状态。

```text
接收原件 → MIME/恶意文件/大小/ACL 校验 → 保存不可变原件 + SHA-256
→ 格式解析（PDF 页、DOCX 段、PPT 幻灯片、Excel 单元格、MD 行、图片 OCR 区域）
→ 分块与语言识别 → AI 候选提取 → 词表/Schema/重复/冲突校验
→ 自动生成产物 → Review 队列 → 通过后建立 Published 索引
```

| 自动产物 | 内容 | 审核规则 |
| --- | --- | --- |
| Metadata | 格式、语言、版本、时间、范围、hash、解析器及 ACL 继承。 | 缺少版本、来源或 hash 不可 Published。 |
| Tags | 品牌、产品、活动、地点、季节、资料类型等受控标签。 | 标签是辅助检索，不代替实体关系或权限。 |
| Citation | 每个 chunk/表格/图像区域的精确 locator。 | AI 找不到定位时，主张标为不可引用。 |
| Knowledge Entity | Brand、Product、Supplier、Procedure、Destination、Activity、Experience、Technology 等候选实体与关系。 | 仅引用充分、无冲突或经人工裁决的候选可发布。 |

抽取器对表格保留表头和单元格坐标，对图片保留 OCR 置信度和区域；低 OCR 置信度、模糊表格或不确定实体匹配必须标记 `needs_human_review`。AI 翻译/摘要/聚合生成新 Asset 时，`source_rank=C`，并以 `lineage[]` 引用所有输入资产；它不得覆盖原件或删除原 Citation。

## 7. Version Management

```text
Draft → Review → Published → Archived
          │        │
          └──→ Draft（退回修改）
                   Published ──→ Archived（被新版本替代或失效）
```

| 状态 | 可做操作 | 检索规则 |
| --- | --- | --- |
| Draft | 上传者/授权编辑者补充元数据、替换草稿、提交审核。 | 不进入默认索引；仅授权协作者可见。 |
| Review | 审核人核实来源、等级、Citation、ACL、实体和冲突。 | 不作为 AI 默认证据。 |
| Published | 冻结原件和版本；仅通过新版本更正。 | 仅在有效期、ACL 和状态均合格时可检索。 |
| Archived | 保留审计、血缘与历史引用；新知识不再引用。 | 默认隐藏；仅显式历史查询且已授权时可见，并带归档警告。 |

上传相同逻辑资料的新版时新建 `asset_id` 或新不可变 revision，使用 `supersedes_asset_id` 建立关系，绝不静默覆盖已有 Citation。发布门槛：内容 hash、可解析 locator、来源等级理由、至少一个有效主来源、ACL、版本、审核记录完整；高风险安全/技术主张还需指定领域审核人。撤销/失效以 Archived 处理并触发索引撤除和受影响 Knowledge Entity 复审。

## 8. Geo + Activity Knowledge

Destination Entity 为地点与活动知识的受控对象，供场景检索、产品适配解释和风险提示使用；它不是导航、天气预报或救援服务。

| 字段 | 类型与规则 |
| --- | --- |
| `destination_id` | 稳定 ID，支持地点别名与层级区域。 |
| `location` | 地理名称、行政/区域范围和可选几何范围；避免保存用户位置。 |
| `altitude` | 数值/范围、单位、来源与测量/观察时点；未知不得推断。 |
| `season` | 适用季节、日期范围及来源版本；天气为动态信息需另行核验。 |
| `difficulty` | `easy|moderate|challenging|expedition`，说明针对的路线/活动。 |
| `risk` | 风险类型、描述、严重性、时效、Citation 和人工确认提示；不替代专业安全意见。 |
| `recommended_equipment` | 类别/角色级装备需求、适用条件、限制和证据；不是 SKU、库存或销售承诺。 |
| `activity` | 受控活动，如 hiking、trekking、mountaineering、camping。 |

初始目的地词表至少覆盖：川西、冈仁波齐、四姑娘山、雨崩、武功山。每个名称可关联多条路线/季节/活动记录；不得把地区级经验自动泛化为任何路线或日期。对高海拔、极端天气、多日徒步和技术登山，输出必须显示 B/C 级经验的限制，并要求人工确认路线、季节、天气、身体状况、向导/许可及安全要求。

## 9. Product Enrichment 与 Product Intelligence

```text
已审核产品资料（优先 S/A）
        + 已审核外部场景（S/A/B，保留等级）
        + 已审核用户经验（B，保留主观性）
        ↓ 证据关联、冲突检测、人工复核
Product Intelligence（适用情境、限制、装备角色、解释性 Citation）
```

Product Intelligence 仅增强 M4.3 Product Knowledge 的解释和检索：例如将产品资料的明确性能/限制与 Destination 的环境需求、活动资料和用户经验建立带 `evidence_refs[]` 的 `suitable_for`、`conditional_for`、`not_suitable_for` 或 `experienced_in` 关系。它不得把外部经验升级为产品规格，也不得将“推荐装备”转化为自动商品推荐。若产品和场景来源冲突、缺版本、无 Citation 或等级不足，关系保持 Draft/Review 并在回答中降级为“待人工核实”。

## 10. 权限设计：RBAC + ACL

RBAC 决定用户可执行的动作，ACL 决定其对特定 Asset/Entity/Citation 的可见与可操作范围；两者必须同时满足。授权由可信 JWT/网关 `AuthContext` 在服务端计算，客户端不可传入或伪造 `organization_id`、角色、owner 或 ACL 扩权字段。

| 角色 | RBAC 能力 | ACL 约束 |
| --- | --- | --- |
| 品牌商 | 上传其授权品牌资料；查看自身提交和被授予资料；回复退审。 | 仅其组织/品牌范围，不能查看采购私有或其他品牌资料。 |
| 采购员 | 上传采购/供应商/市场资料；提交审核；查看授权采购域。 | 受组织、供应商、品牌和显式 ACL 限制。 |
| 运营 | 上传/维护运营、门店标准和培训资料；参与审核。 | 仅获授门店、部门或组织范围。 |
| 员工 | 搜索和使用已 Published 的被授权知识；提交经验草稿。 | 默认无审核、发布、ACL 修改或受限来源浏览权。 |
| 管理层 | 查看组织范围已授权资产、审计和审批队列；可委派审核。 | 不跨组织；敏感资产仍受显式 deny 和最小权限约束。 |

ACL 对象至少含 `organization_id`、`owner_id`、`visibility`（`private|department|organization|partner`）、`department_id?`、`brand_scope?`、`supplier_scope?`、`allow_principals[]`、`deny_principals[]`。默认拒绝；跨组织拒绝；显式 deny 优先；所有搜索结果在向量召回后重新读取 Asset 状态与 ACL。Citation 和派生内容沿用最严格的父资产访问约束，不能因摘要而泄露受限原文、URL、名称或存在性。

## 11. API Contract（设计）

所有接口要求可信认证、请求 ID 和审计日志；这些接口是治理平台契约，不部署、不修改现有生产接口。响应不返回调用者无权访问的 Asset、Citation 或存在性信息。

### `POST /api/assets/upload`

接收 multipart 原件或受控对象存储引用，以及 `asset_type`、`brand_id?`、`supplier_id?`、`version`、`source`、`permission_request?`。服务端记录上传时间并计算 hash，返回 `202 Accepted`：`asset_id`、`status=Draft`、`extraction_job_id`、拒绝/隔离原因（如有）。上传成功不表示解析成功、等级确认或发布。

### `GET /api/assets/{id}`

返回授权范围内的 Asset metadata、版本、状态、审核摘要、解析状态、可访问 Citation 和血缘；默认不返回全文或原始二进制。`Archived` 仅在显式 `include_archived=true`、具有历史访问权限时返回并标注状态。

### `GET /api/assets/search`

查询参数：`q?`、`asset_type?`、`brand_id?`、`supplier_id?`、`source_rank?`、`status=Published`（默认）、`destination_id?`、`activity?`、`tags?`、`version?`、`cursor?`、`limit?`。服务端先 ACL 与状态过滤、再检索和排序；响应每项包含 `asset_id`、标题、摘要、`source_rank`、版本、状态、Citation 摘要、置信度、更新时间和 cursor。筛选参数只能缩小可见结果。

### `POST /api/assets/review`

仅具备审核 RBAC 且具有 Asset ACL 的用户调用。请求为 `asset_id`、`decision`（`approve|request_changes|archive`）、`review_note`、`verified_source_rank`、`citation_checks[]`、`entity_decisions[]`、`effective_period?`。服务端验证状态迁移、版本/hash 不变性、审核人职责和发布门槛；返回新的状态、审核记录和索引动作。禁止用此接口改写原件、扩大 ACL 或越权提升等级。

## 12. 审计、质量与验收

每次上传、解析、查看、搜索、导出 Citation、等级变更、审核、发布、归档和权限变更记录 `request_id`、有效主体、代理主体（如有）、时间、资产/版本/hash、决策、理由和结果；日志不存储不必要的原文或 PII。质量监控包括解析失败率、Citation 完整率、无版本资产、待审时长、冲突率、过期 Published 资产、按等级的回答覆盖率和 ACL 拒绝审计。
